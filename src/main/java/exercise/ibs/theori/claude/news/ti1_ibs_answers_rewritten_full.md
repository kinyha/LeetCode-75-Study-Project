## Секция 2. Java Core

<a id="q2-1"></a>
### 2.1 Разница между && и &, || и |

`&&` и `||` — логические операторы с **short-circuit**: Java не вычисляет правую часть, если результат уже понятен по левой. Поэтому `user != null && user.isActive()` безопасно: если `user == null`, второй вызов не выполнится. `&` и `|` на `boolean` тоже дают логический результат, но **всегда вычисляют обе части**. На числах `&` и `|` — это уже **побитовые операции**.

<a id="q2-2"></a>
### 2.2 Назовите все методы Object

У `Object` есть базовые методы, которые наследует почти любой объект: **`equals`**, **`hashCode`**, **`toString`**, **`getClass`**, **`clone`**, **`finalize`**, **`wait`**, **`notify`**, **`notifyAll`**. У `wait` есть перегрузки. Важно помнить не только список, но и смысл: `equals/hashCode` нужны для сравнения и коллекций, `toString` — для текстового представления, `wait/notify` — для работы с монитором объекта. `finalize` сейчас считается устаревшим механизмом.

<a id="q2-3"></a>
### 2.3 Ошибки от неправильных equals/hashCode на уровне метода / сервиса / приложения

Если нарушить контракт **`equals` + `hashCode`**, объект начинает странно вести себя в `HashMap`, `HashSet` и других коллекциях. `contains()` может не найти объект, хотя он логически “такой же”. В сервисах ломается дедупликация, кэширование и `distinct()` в стримах. С JPA-сущностями особенно больно: можно получить дубли в `Set`, проблемы с detached/proxy-объектами и нестабильное поведение после сохранения. Главное правило: переопределять **оба метода вместе** и использовать **стабильные поля**, а у entity аккуратно работать с `id`, особенно пока он `null`.

<a id="q2-4"></a>
### 2.4 this и super: когда применять, правила

`this` — ссылка на **текущий объект**. Чаще всего нужен, когда параметр перекрывает поле: `this.name = name`. Ещё через `this(...)` можно вызвать другой конструктор этого же класса. `super` — обращение к **родительскому классу**: вызов конструктора `super(...)`, метода `super.method()` или поля родителя. В конструкторе `this(...)` или `super(...)` должен быть **первой строкой**, и использовать можно только один из них. В `static`-контексте `this` и `super` недоступны, потому что там нет конкретного объекта.

<a id="q2-5"></a>
### 2.5 🟣 Почему HashMap неэффективна при плохой хэш-функции: метод / сервис / приложение + оптимизации

`HashMap` быстрая, пока ключи нормально распределяются по бакетам. Если `hashCode()` плохой и много ключей попадает в один бакет, поиск превращается из условного **O(1)** в перебор цепочки. В Java 8+ длинный бакет может стать **красно-чёрным деревом**, но это всё равно хуже нормального распределения. В сервисе это даёт тормоза в кэшах, дедупликации и in-memory индексах. Лечится нормальным `hashCode`, правильным `equals`, заданием `initialCapacity`, если размер известен, и использованием `ConcurrentHashMap` там, где есть конкурентный доступ.

<a id="q2-6"></a>
### 2.6 Fail-fast vs fail-safe итераторы

**Fail-fast** итераторы, например у `ArrayList` или `HashMap`, пытаются быстро заметить, что коллекцию поменяли во время обхода, и бросают `ConcurrentModificationException`. Это не железная гарантия, а защита от типичных ошибок. Правильно удалять элементы во время обхода через `Iterator.remove()` или `removeIf()`. **Fail-safe / weakly consistent** итераторы работают иначе: `CopyOnWriteArrayList` итерируется по снимку массива, а `ConcurrentHashMap` может видеть часть изменений и не падает.

<a id="q2-7"></a>
### 2.7 Абстрактный класс vs интерфейс, когда что

**Интерфейс** описывает контракт: что объект умеет делать. Его удобно использовать, когда нужны разные реализации и слабая связность. **Абстрактный класс** нужен, когда у наследников есть общая реализация, состояние, поля, конструктор или protected-методы. После появления `default`-методов интерфейсы стали гибче, но главное отличие осталось: интерфейс — про **контракт**, абстрактный класс — про **общую базу и состояние**. В бизнес-коде чаще стараются зависеть от интерфейсов.

<a id="q2-8"></a>
### 2.8 Как работает «перегрузка» статического метода

Статический метод можно **перегружать**: компилятор выбирает нужную версию по параметрам, как и с обычными методами. Но статический метод нельзя нормально **переопределить**. Если в наследнике объявить static-метод с тем же именем, это будет **method hiding**, а не override. Вызов выбирается по типу ссылки на этапе компиляции, а не по реальному объекту. Поэтому static-методы лучше вызывать через имя класса.

<a id="q2-9"></a>
### 2.9 Проблемы try-with-resources: метод / сервис / приложение

`try-with-resources` сам по себе обычно не проблема, а защита от проблем. Он гарантирует вызов **`close()`** у ресурсов, которые реализуют `AutoCloseable`. Важно понимать, что ресурс закрывается в обратном порядке объявления, а исключение из `close()` может попасть в **suppressed exceptions**. Если try-with-resources не использовать, можно словить утечки файлов, сокетов или соединений БД. На уровне приложения это превращается в исчерпанный connection pool или ошибку вроде `too many open files`.

<a id="q2-10"></a>
### 2.10 Механизм AutoCloseable, последствия для файлов и БД

`AutoCloseable` — интерфейс с методом **`close()`**. Когда объект объявлен в try-with-resources, Java автоматически вызовет `close()` даже при исключении. Для файлов это освобождает дескриптор. Для БД важный нюанс: `Connection.close()` при использовании пула обычно **не закрывает физическое соединение**, а возвращает его в пул. Если забыть закрыть `Connection`, `Statement` или `ResultSet`, пул постепенно исчерпается, и сервис начнёт зависать на ожидании свободного соединения.

<a id="q2-11"></a>
### 2.11 ClassLoader, ClassNotFoundException

`ClassLoader` загружает классы в JVM. Обычно работает иерархия: **Bootstrap → Platform → Application ClassLoader**. По модели parent delegation загрузчик сначала спрашивает родителя, и только потом пробует загрузить сам. `ClassNotFoundException` появляется, когда код явно пытается загрузить класс, например через `Class.forName`, но класса нет в runtime classpath. Похожая, но другая ошибка — `NoClassDefFoundError`: класс был виден при компиляции, но исчез или не смог инициализироваться в рантайме. На практике проверяют зависимости, scope, конфликт версий и итоговый classpath.

<a id="q2-12"></a>
### 2.12 Проблемы рефлексии: производительность, безопасность, защита

Рефлексия позволяет читать классы, поля, методы и вызывать их во время выполнения. Цена — **медленнее обычного вызова**, сложнее для JIT-оптимизаций и хуже читаемость. По безопасности проблема в том, что через `setAccessible(true)` можно обойти инкапсуляцию и трогать private-поля. Защита — не строить критичные инварианты только на приватности, использовать JPMS-модули, аккуратно открывать пакеты через `opens`, кэшировать reflective-объекты. Spring и Hibernate активно используют рефлексию, но прячут эту сложность внутри фреймворка.

<a id="q2-13"></a>
### 2.13 var — что это и когда уместен

`var` — это **локальный вывод типа** на этапе компиляции, а не динамическая типизация. Тип всё равно строгий и известен компилятору. `var` можно использовать только для локальных переменных с инициализатором, нельзя для полей, параметров метода и `null`. Уместно, когда тип очевиден справа: `var users = new ArrayList<User>()`. Плохо, когда справа непонятный метод вроде `var result = service.process()`, потому что читающий код теряет контекст.

<a id="q2-14"></a>
### 2.14 Variable shadowing

**Shadowing** — ситуация, когда переменная из внутренней области видимости скрывает переменную с таким же именем снаружи. Самый частый пример: параметр конструктора `name` скрывает поле `name`, поэтому пишут `this.name = name`. Shadowing бывает и с локальными переменными, и с полями родителя. Это не ошибка компиляции, но хороший способ родить баг, который потом вся команда будет искать с лицом человека, пожалевшего о выборе профессии.

<a id="q2-15"></a>
### 2.15 final vs effectively final

`final` означает, что переменную **нельзя переприсвоить** после инициализации. **Effectively final** — переменная не помечена `final`, но фактически тоже не меняется. Такие переменные можно захватывать в лямбдах и анонимных классах. Ограничение нужно потому, что локальная переменная живёт на стеке метода, а лямбда может жить дольше; Java захватывает значение, и разрешение менять его создало бы странную рассинхронизацию.

<a id="q2-16"></a>
### 2.16 🔵 Generics, примеры, PECS

Generics дают **типобезопасность на этапе компиляции**: `List<String>` лучше, чем сырой `List`, потому что не нужно постоянно кастить и ловить `ClassCastException`. В рантайме работает **type erasure**: информация о `T` в основном стирается, поэтому нельзя сделать `new T()` или проверить `instanceof List<String>`. Правило **PECS**: если коллекция только отдаёт значения — `? extends T` (**producer extends**), если в неё кладём — `? super T` (**consumer super**). Классический пример: копируем из `List<? extends T>` в `List<? super T>`.

<a id="q2-17"></a>
### 2.17 Циклические зависимости: примеры и инструменты

Циклическая зависимость — когда A зависит от B, а B зависит от A. На уровне классов это часто признак, что ответственность размазана, и нужен третий объект или интерфейс. В Spring constructor injection такой цикл обычно падает на старте, что хорошо: ошибка видна сразу. Костыли вроде `@Lazy` или setter injection иногда помогают, но правильнее разорвать цикл: вынести общую логику, использовать события, интерфейс, отдельный сервис-координатор. В микросервисах похожая проблема проявляется как взаимные синхронные вызовы.

<a id="q2-18"></a>
### 2.18 Suppressed exceptions

**Suppressed exceptions** появляются, например, в try-with-resources. Если основной код бросил исключение, а потом `close()` тоже бросил исключение, Java сохраняет главное исключение, а ошибку закрытия добавляет как suppressed. Посмотреть их можно через `getSuppressed()`. Это важно, потому что до Java 7 исключение из `finally` могло просто затереть исходную причину.

<a id="q2-19"></a>
### 2.19 Кастомная аннотация, retention

Кастомная аннотация объявляется через `@interface`. Важные мета-аннотации — **`@Target`**, где аннотацию можно ставить, и **`@Retention`**, как долго она живёт. `SOURCE` нужен только компилятору, `CLASS` попадает в байткод, но обычно не доступен через рефлексию, `RUNTIME` доступен во время выполнения. Если аннотацию должен читать Spring, AOP или твой код через reflection, почти всегда нужен **`RetentionPolicy.RUNTIME`**.

<a id="q2-20"></a>
### 2.20 @SneakyThrows и @SuppressWarnings

`@SneakyThrows` из Lombok позволяет бросить checked exception без объявления в `throws`. JVM это допускает, а Lombok просто обходит проверку компилятора. Удобно в редких местах, но опасно, потому что метод скрывает свой настоящий контракт по ошибкам. `@SuppressWarnings` — обычная Java-аннотация для подавления предупреждений компилятора, например `unchecked` или `deprecation`. Её лучше ставить как можно уже: на конкретный метод или маленький участок, а не на весь класс от отчаяния.

<a id="q2-21"></a>
### 2.21 🟣 Optional: когда и какие конструкции

`Optional` — способ явно сказать: **результат может отсутствовать**. Лучше всего использовать его как возвращаемое значение, например `findById()`. Основные методы: `map`, `flatMap`, `filter`, `orElse`, `orElseGet`, `orElseThrow`, `ifPresent`. Важный нюанс: `orElse()` вычисляет аргумент сразу, а `orElseGet()` лениво. Не стоит использовать `Optional` как поле entity, параметр метода или замену пустой коллекции. И почти всегда плохой стиль — писать `optional.get()` без проверки.

<a id="q2-22"></a>
### 2.22 Утечки памяти в Java и как избежать

В Java память утекает не потому, что GC “забыл” объект, а потому что на объект всё ещё есть **живая ссылка**. Типичные источники: static-коллекции, кэши без лимита, listener'ы без отписки, `ThreadLocal` в пуле потоков без `remove()`, незакрытые ресурсы, внутренние классы, которые держат ссылку на outer. Диагностика: heap dump, dominator tree, профайлеры. Профилактика простая, но почему-то человечество всё равно страдает: лимитированные кэши, try-with-resources, отписка listener'ов и очистка `ThreadLocal` в `finally`.

<a id="q2-23"></a>
### 2.23 Records

`record` — компактный способ описать **immutable data carrier**. Java сама создаёт final-поля, конструктор, аксессоры, `equals`, `hashCode` и `toString`. Хорошо подходит для DTO, value object, ключей в map, результатов запросов. Можно добавить валидацию в compact constructor. Ограничения: record не наследуется от другого класса, поля фактически final, и это не замена полноценной mutable entity с жизненным циклом.

<a id="q2-24"></a>
### 2.24 Sealed классы

`sealed` ограничивает, кто может наследоваться от класса или интерфейса. Это удобно для **закрытых доменных иерархий**, например `PaymentResult`, где возможны только `Success`, `Declined`, `Error`. Наследник должен быть `final`, `sealed` или `non-sealed`. Плюс sealed-иерархий — компилятор может проверять полноту `switch`, особенно вместе с pattern matching. То есть меньше `default`, который молча прячет забытый кейс.

<a id="q2-25"></a>
### 2.25 Последняя версия Java, что добавлено

На июль 2026 актуальная линия такая: **JDK 26** — последняя feature-release версия, а **JDK 25** — последняя LTS. На интервью безопасно сказать: в проде чаще встречаются 17/21/25 LTS, а за новыми версиями стоит следить по JEP и release notes. Из последних крупных тем: virtual threads и pattern matching из Java 21, развитие structured concurrency/scoped values, улучшения производительности и памяти. Главное — не пытаться перечислить всё подряд, а назвать 2–3 вещи и объяснить, зачем они нужны в реальном коде.

<a id="section-3"></a>
## Секция 3. Алгоритмы, структуры данных, паттерны, чистый код и ООП

<a id="q3-1"></a>
### 3.1 Основные интерфейсы коллекций + EnumSet/EnumMap

В Java основные коллекции идут от `Collection`: **`List`** хранит порядок и дубли, **`Set`** хранит уникальные элементы, **`Queue/Deque`** нужны для очередей и стеков. `Map` стоит отдельно: это ключ-значение. Для enum есть специальные коллекции: **`EnumSet`** и **`EnumMap`**. Они работают быстрее и компактнее обычных `HashSet/HashMap`, потому что внутри используют ordinal enum'а, фактически массив или битовую маску. Если ключ или значение — enum, часто это лучший выбор.

<a id="q3-2"></a>
### 3.2 Внутренняя структура HashMap, разрешение коллизий

`HashMap` внутри — массив бакетов. По `hashCode()` ключа вычисляется индекс бакета. Если несколько ключей попали в один бакет, возникает **коллизия**. Сначала элементы хранятся цепочкой, а при длинной цепочке в Java 8+ бакет может превратиться в **красно-чёрное дерево**, чтобы поиск был ближе к O(log n), а не O(n). Когда элементов становится слишком много относительно capacity и load factor, происходит **resize**: массив увеличивается, элементы перераспределяются.

<a id="q3-3"></a>
### 3.3 🟣 Красно-чёрное дерево: как работает, где применяется

Красно-чёрное дерево — самобалансирующееся бинарное дерево поиска. У каждого узла есть цвет, и за счёт правил по цветам дерево не превращается в длинную палку. Поэтому поиск, вставка и удаление остаются **O(log n)**. В Java это используется в `TreeMap`, `TreeSet` и в длинных бакетах `HashMap`. На интервью не обязательно доказывать все свойства, но важно сказать: смысл в **сохранении баланса** через повороты и перекраску.

<a id="q3-4"></a>
### 3.4 Основные структуры данных, что применяю в работе

В реальной backend-разработке чаще всего используются не экзотические структуры, а базовые: **`ArrayList`** для списков, **`HashMap`/`HashSet`** для быстрого доступа и дедупликации, **`Deque`** для очередей/стека, **`PriorityQueue`** для задач с приоритетом, **`TreeMap`** когда нужен отсортированный доступ. В бизнес-коде типичный пример — `Map<EventType, Handler>` для выбора обработчика без огромного `if-else`. Главное — уметь объяснить, почему структура подходит по скорости, памяти и читаемости.

<a id="q3-5"></a>
### 3.5 Fail-fast и fail-safe коллекции

Fail-fast коллекции пытаются заметить изменение во время итерации и бросают `ConcurrentModificationException`. Это помогает быстро поймать ошибку, когда коллекцию меняют не через итератор. Fail-safe или weakly consistent коллекции не падают при конкурентных изменениях: `CopyOnWriteArrayList` даёт снимок данных, `ConcurrentHashMap` позволяет читать и менять структуру параллельно. Выбор зависит от сценария: для частого чтения и редкой записи подходит copy-on-write, для конкурентного доступа — `ConcurrentHashMap`.

<a id="q3-6"></a>
### 3.6 Hashtable vs ConcurrentHashMap

`Hashtable` — старый потокобезопасный map, где почти всё синхронизировано одним монитором. Из-за этого при конкуренции потоки быстро начинают ждать друг друга. `ConcurrentHashMap` устроен тоньше: чтение обычно не блокируется, запись блокирует только нужный участок структуры, плюс есть атомарные операции вроде `putIfAbsent`, `compute`, `merge`. `ConcurrentHashMap` не разрешает `null`, потому что в конкурентной среде `null` создаёт неоднозначность: ключа нет или значение реально null. В новом коде `Hashtable` почти не используют.

<a id="q3-7"></a>
### 3.7 🟣 Big O: сложность по времени и памяти

Big O показывает, как растёт стоимость алгоритма при росте входных данных. **O(1)** — почти постоянное время, **O(log n)** — например бинарный поиск, **O(n)** — один проход, **O(n log n)** — хорошие сортировки, **O(n²)** — вложенные циклы по тем же данным. Есть ещё сложность по памяти: сколько дополнительной памяти нужно алгоритму. Важно помнить, что Big O не учитывает константы, поэтому на маленьких данных простой O(n²) иногда быстрее сложного O(n log n), но на больших данных асимптотика побеждает.

<a id="q3-8"></a>
### 3.8 Сортировки; quick sort vs merge sort

`QuickSort` обычно быстрый и сортирует **in-place**, но в плохом случае может упасть до O(n²), если плохо выбирается pivot. Он нестабильный. `MergeSort` стабилен и гарантирует **O(n log n)**, но требует дополнительную память O(n). В Java для примитивов используются быстрые варианты quicksort, а для объектов — стабильная сортировка на базе TimSort, потому что для объектов часто важна стабильность и частичная упорядоченность данных.

<a id="q3-9"></a>
### 3.9 Коммивояжёр / рюкзак — подход

Обе задачи показывают, что не всё решается простым перебором. **Рюкзак 0/1** обычно объясняют через динамическое программирование: строим таблицу по предметам и вместимости, сложность O(n·W). **Коммивояжёр** в общем виде NP-трудный: точные решения быстро становятся слишком дорогими, поэтому для больших данных используют эвристики и приближённые алгоритмы. На интервью важно не выдумывать “идеальное быстрое решение”, а показать подход: маленькие размеры — точный алгоритм, большие — эвристика или ограничение постановки.

<a id="q3-10"></a>
### 3.10 Жадный алгоритм

Жадный алгоритм на каждом шаге выбирает локально лучший вариант и не откатывается назад. Он работает только там, где есть свойство **greedy choice**: локальный выбор ведёт к глобальному оптимуму. Примеры: выбор непересекающихся интервалов по раннему окончанию, алгоритм Дейкстры с неотрицательными весами, код Хаффмана. Контрпример — рюкзак 0/1: если брать предметы просто по выгоде, можно получить не оптимальный результат.

<a id="q3-11"></a>
### 3.11 Композиция/ассоциация: has-a vs is-a

**is-a** — это наследование: объект является разновидностью другого объекта. **has-a** — композиция или агрегация: объект содержит или использует другой объект. Наследование сильнее связывает классы и часто приводит к проблеме fragile base class: изменение родителя ломает наследников. Композиция гибче: можно заменить зависимость, проще тестировать, проще расширять поведение. Поэтому в прикладном коде обычно предпочитают **composition over inheritance**, если нет настоящей модели “является”.

<a id="q3-12"></a>
### 3.12 CAP-теорема

CAP говорит о распределённых системах: при сетевом разделении (**Partition**) нельзя одновременно сохранить строгую **Consistency** и полную **Availability**. Partition — не выбор, сеть может сломаться сама, поэтому реальный выбор обычно между CP и AP. CP-система при проблемах сети может отказать части запросов, но не отдаст противоречивые данные. AP-система продолжит отвечать, но допускает eventual consistency. Одна обычная PostgreSQL-нода не “выбирает CAP”, потому что это не распределённая система в смысле теоремы.

<a id="q3-13"></a>
### 3.13 Map/Reduce и divide and conquer: метод / класс / сервис / приложение

Идея одна: **разделить работу на независимые части**, обработать их отдельно и потом собрать результат. На уровне метода это может быть `stream.map(...).reduce(...)`. На уровне класса — `ForkJoinPool` и рекурсивная задача. На уровне сервиса — параллельная обработка батчей. На уровне приложения — Kafka-партиции, где разные consumer'ы обрабатывают разные части потока. Главное ограничение: части должны быть достаточно независимыми, иначе синхронизация съест весь выигрыш.

<a id="q3-14"></a>
### 3.14 Любимый паттерн — где и почему

Хороший ответ — **Strategy**. Он заменяет длинные `if-else` по типу поведения на набор классов с общим интерфейсом. Например, есть разные обработчики событий: каждый реализует `EventHandler`, а диспетчер держит `Map<EventType, EventHandler>`. Чтобы добавить новый тип события, мы добавляем новый класс, а не переписываем центральный метод. Это даёт **Open/Closed Principle**, лучшее тестирование и меньше риска сломать старую логику.

<a id="q3-15"></a>
### 3.15 Паттерн Фабрика: преимущество перед явным созданием

Фабрика прячет логику создания объекта в одном месте. Клиентскому коду не нужно знать, какой конкретный класс создать, какие зависимости собрать и по какому условию выбрать реализацию. Это упрощает замену реализации, тестирование и расширение. Пример: вместо `new PdfReportGenerator()` в коде вызываем `reportGeneratorFactory.get(type)`. Если завтра добавится Excel или XML, меняется фабрика и новая реализация, а клиентский код остаётся спокойным, редкий праздник.

<a id="q3-16"></a>
### 3.16 🔥 Паттерн Стратегия

**Strategy** — паттерн, где семейство алгоритмов выносится за общий интерфейс, а нужная реализация выбирается в runtime. Это хороший способ убрать ветвления вроде `if type == A`, `if type == B`. В Spring это удобно делается через список бинов: внедряем `List<Handler>`, собираем `Map` по ключу и выбираем обработчик. Примеры из JDK: `Comparator` как стратегия сравнения. От `State` отличается тем, что стратегию обычно выбирает внешний код, а состояние часто само переключает поведение объекта по жизненному циклу.

<a id="q3-17"></a>
### 3.17 🟣 Декоратор vs Адаптер

**Adapter** нужен, когда интерфейсы несовместимы: он переводит один контракт в другой. Например, оборачиваем legacy-клиент в интерфейс, который ожидает наш сервис. **Decorator** сохраняет тот же интерфейс, но добавляет поведение: логирование, буферизацию, сжатие, кеширование. В Java IO это видно хорошо: потоки можно оборачивать друг в друга. Коротко: адаптер помогает “подружить несовместимое”, декоратор — “добавить слой поведения, не меняя контракт”.

<a id="q3-18"></a>
### 3.18 Паттерн Visitor

Visitor выносит операции из классов объектов в отдельный объект-посетитель. Объект принимает visitor через `accept`, а visitor имеет отдельные `visit`-методы для разных типов. Плюс: можно добавлять новые операции без изменения самих классов. Минус: если появился новый тип элемента, приходится менять всех visitor'ов. Типичное применение — обход AST, парсеры, компиляторы, файловые деревья. В современном Java часть таких задач можно решать через sealed-классы и pattern matching.

<a id="q3-19"></a>
### 3.19 Инкапсуляция, наследование, полиморфизм

**Инкапсуляция** — это не просто private-поля, а защита инвариантов объекта через понятный API. **Наследование** — отношение “является”, когда один класс расширяет поведение другого, но с ним нужно быть аккуратным. **Полиморфизм** — возможность работать через общий контракт, а реальное поведение получать от конкретной реализации в runtime. Например, сервис зависит от интерфейса `PaymentProvider`, а настоящая реализация может быть Stripe, PayPal или заглушка для теста.

<a id="q3-20"></a>
### 3.20 Принципы чистого кода по Роберту Мартину

Чистый код — это код, который можно читать и менять без ощущения, что тебе выдали карту минного поля. Главное: **понятные имена**, маленькие функции, один уровень абстракции, минимум аргументов, отсутствие дублирования, явная обработка ошибок, тестируемость. Комментарии должны объяснять “почему”, а не пересказывать очевидное “что”. И важное практическое правило: оставлять код чуть чище, чем был до тебя, иначе проект постепенно превращается в болото с CI.

<a id="section-4"></a>
## Секция 4. Spring

<a id="q4-1"></a>
### 4.1 🟣 IoC и как Spring его реализует

**IoC** — инверсия управления: не объект сам создаёт зависимости, а контейнер создаёт объекты и связывает их между собой. В Spring это делает `ApplicationContext`: он читает конфигурации и аннотации, создаёт `BeanDefinition`, строит граф зависимостей, создаёт бины и внедряет зависимости через конструктор, сеттер или поле. **DI** — конкретный механизм IoC. Польза простая: код меньше зависит от конкретных классов, реализации легче менять, а тесты проще писать с моками.

<a id="q4-2"></a>
### 4.2 Виды autowiring

Spring может внедрять зависимости через **конструктор**, **сеттер** или **поле**. Лучше всего constructor injection: зависимость обязательная, поле можно сделать `final`, объект проще тестировать, циклы видны на старте. Поиск бина обычно идёт по типу. Если кандидатов несколько, помогают `@Primary`, `@Qualifier` или имя бина. Field injection короче, но хуже для тестов и скрывает реальные зависимости класса.

<a id="q4-3"></a>
### 4.3 Бин и жизненный цикл

Бин — объект, которым управляет Spring-контейнер. Жизненный цикл примерно такой: контейнер находит описание бина, создаёт объект, внедряет зависимости, вызывает aware-callback'и, потом `BeanPostProcessor` до инициализации, затем `@PostConstruct`, потом `BeanPostProcessor` после инициализации. Именно на этой стадии часто появляются **AOP-прокси**. При остановке вызывается `@PreDestroy` или destroy-method. Если понимать lifecycle, магия Spring становится чуть меньше похожей на шаманство.

<a id="q4-4"></a>
### 4.4 Транзакционность в Spring: настройка и проблемы

`@Transactional` обычно ставят на методы сервисного слоя. Под капотом Spring создаёт **прокси**, который открывает транзакцию перед методом и коммитит или откатывает после. Главные ловушки: **self-invocation** не проходит через прокси, private-методы не транзакционны, checked exception по умолчанию не вызывает rollback, проглоченное исключение тоже не откатит транзакцию. Ещё нельзя держать транзакцию слишком долго: внешние HTTP-вызовы внутри транзакции держат соединение и блокировки дольше, чем надо.

<a id="q4-5"></a>
### 4.5 Как Boot упрощает создание приложений

Spring Boot убирает много ручной настройки. Он даёт **starters**, автоконфигурацию по classpath, встроенный Tomcat/Jetty/Undertow, внешний конфиг через `application.yml`, профили, Actuator и разумные дефолты. Например, добавил `spring-boot-starter-data-jpa` и драйвер PostgreSQL — Boot уже понимает, что нужен `DataSource`, `EntityManagerFactory`, transaction manager. При этом всё можно переопределить своим бином, если дефолт не подходит.

<a id="q4-6"></a>
### 4.6 🔵 AOP: как работает и когда использовать

**AOP** выносит сквозную инфраструктурную логику отдельно от бизнес-кода: транзакции, логирование, безопасность, метрики, кеширование. В Spring это в основном работает через **прокси**: JDK dynamic proxy для интерфейсов или CGLIB для классов. Ограничения идут от прокси: self-invocation не перехватывается, final/private методы проблемны, обычно работают public-методы бинов. Хорошее применение AOP — инфраструктура. Плохое — спрятать бизнес-логику в аспектах и потом всем отделом гадать, почему метод делает больше, чем написано.

<a id="q4-7"></a>
### 4.7 Actuator и мониторинг

Actuator добавляет production endpoints: **health**, **metrics**, **prometheus**, **info**, **loggers**, **threaddump**, **heapdump**. Через `/health` удобно делать liveness/readiness probes в Kubernetes. Через metrics и Micrometer данные уходят в Prometheus/Grafana: latency, status codes, JVM, pools. Важно не выставлять всё наружу: публичным обычно оставляют только health, остальное закрывают авторизацией или внутренней сетью.

<a id="q4-8"></a>
### 4.8 🟣 Автоконфигурация Boot

Автоконфигурация Boot — это набор конфигураций, которые подключаются **условно**. Boot смотрит, какие классы есть в classpath, какие properties заданы и какие бины уже созданы. Например, если есть Spring MVC, он настроит web-инфраструктуру; если есть DataSource и JPA — настроит JPA. Условия типа `@ConditionalOnClass`, `@ConditionalOnMissingBean`, `@ConditionalOnProperty` решают, включать конфигурацию или нет. Если ты объявляешь свой бин, он часто переопределяет дефолтный вариант.

<a id="q4-9"></a>
### 4.9 Spring и микросервисная архитектура

Spring помогает быстро собрать микросервис как самостоятельное приложение: Boot даёт jar с embedded server, Actuator — health/metrics, Spring Cloud — Gateway, Config, Feign, LoadBalancer, интеграции с Resilience4j, Kafka/RabbitMQ. В Kubernetes часть задач уже решается платформой: service discovery, config/secrets, scaling. Поэтому Spring в микросервисах ценен не “магией микросервисов”, а готовыми интеграциями, стандартизацией и быстрым запуском сервисов.

<a id="q4-10"></a>
### 4.10 @Transactional: как работает, ограничения

`@Transactional` работает через Spring-прокси: внешний вызов метода проходит через прокси, тот открывает транзакцию, вызывает метод и потом делает commit или rollback. Важные параметры: **propagation** (`REQUIRED`, `REQUIRES_NEW`, `NESTED`), **isolation**, **readOnly**, **timeout**, **rollbackFor**. Ограничения: self-invocation, private/final методы, checked exceptions без `rollbackFor`, вызов метода не из Spring-бина. Поэтому транзакции обычно держат на public-методах сервисного слоя.

<a id="q4-11"></a>
### 4.11 ApplicationContext vs BeanFactory

`BeanFactory` — базовый контейнер Spring: умеет хранить и создавать бины. `ApplicationContext` — более богатая надстройка: события, ресурсы, i18n, автоматическая регистрация `BeanPostProcessor`, интеграция с AOP, lifecycle и обычно eager initialization singleton-бинов. На практике почти всегда работают именно с **ApplicationContext**, потому что он даёт весь нормальный Spring-функционал, а не минимальный контейнер для учебника.

<a id="q4-12"></a>
### 4.12 @Bean — как и когда

`@Bean` ставят на метод в конфигурационном классе, когда объект нужно создать вручную: сторонний класс без `@Component`, сложная настройка клиента, factory-метод, условное создание. `@Component` подходит для наших классов, которые Spring может сам найти сканированием. `@Bean` — когда мы явно говорим контейнеру: “вот так создай этот объект”. В `@Configuration` Spring проксирует методы, чтобы повторный вызов `@Bean`-метода возвращал тот же singleton, а не создавал новый объект.

<a id="q4-13"></a>
### 4.13 🔵 @Component vs @Service vs @Repository

Все три аннотации технически являются стереотипами Spring и позволяют найти бин при сканировании. **`@Component`** — общий вариант. **`@Service`** показывает, что класс содержит бизнес-логику. **`@Repository`** — слой доступа к данным; дополнительно Spring может переводить низкоуровневые исключения БД в `DataAccessException`. То есть разница не столько в создании бина, сколько в **семантике слоя** и дополнительных возможностях для инфраструктуры.

<a id="q4-14"></a>
### 4.14 🟣 @Autowired vs @Resource vs @Inject

`@Autowired` — Spring-аннотация, обычно ищет бин **по типу**, а при неоднозначности помогают `@Qualifier`, `@Primary` или имя. `@Resource` из Jakarta/JSR-250 сначала ищет **по имени**, потом по типу. `@Inject` — стандарт JSR-330, похож на `@Autowired`, но менее Spring-specific. В современном Spring часто вообще не пишут `@Autowired` на единственном конструкторе: контейнер сам понимает, что зависимости надо внедрить туда.

<a id="q4-15"></a>
### 4.15 Внедрение в static-поле

Spring управляет **экземплярами** бинов, а static-поле принадлежит классу, а не объекту. Поэтому нормальный `@Autowired` в static не работает. Можно сделать костыль через нестатический setter, который запишет значение в static, но это почти всегда плохой дизайн. Если классу нужна зависимость, лучше сделать его обычным Spring-бином и внедрять зависимость через конструктор. Static и DI плохо дружат, как корпоративный чат и здравый смысл.

<a id="q4-16"></a>
### 4.16 Инжект примитивов

Примитивы и простые значения обычно внедряют через **`@Value`**: `@Value("${app.timeout:30}")`. Для группы связанных настроек лучше использовать **`@ConfigurationProperties`**: это типобезопасно, удобно валидировать и проще поддерживать. Например, вместо десяти отдельных `@Value` сделать класс `AppProperties` с полями timeout, url, retries. Это меньше похоже на конфиг, рассыпанный по проекту как конфетти после релиза.

<a id="q4-17"></a>
### 4.17 Инжект коллекции

Spring умеет внедрять сразу все бины одного типа: `List<Handler>` даст список всех обработчиков, `Map<String, Handler>` даст имя бина → бин. Порядок в списке можно задать через `@Order`. Это удобно для паттернов вроде Strategy: каждая реализация — отдельный бин, а диспетчер собирает их в map по типу операции. Так добавление новой логики не требует переписывать центральный `switch`.

<a id="q4-18"></a>
### 4.18 @Primary и @Qualifier

Когда в контейнере несколько бинов одного типа, Spring не знает, какой выбрать. **`@Primary`** назначает бин вариантом по умолчанию. **`@Qualifier`** в точке внедрения явно говорит, какой бин нужен. Если есть и `@Primary`, и `@Qualifier`, побеждает `@Qualifier`, потому что он конкретнее. Хорошая практика — не плодить неоднозначность, а давать реализациям понятные имена или свои qualifier-аннотации.

<a id="q4-19"></a>
### 4.19 🟣 @Conditional

`@Conditional` включает бин или конфигурацию только при выполнении условия. В Boot часто используются готовые условия: `@ConditionalOnProperty`, `@ConditionalOnClass`, `@ConditionalOnMissingBean`. На этом построена автоконфигурация. Пример: если property `feature.payment.enabled=true`, создаём реальный payment-клиент; если нет — не создаём. Это удобнее, чем держать в коде ручные `if` вокруг создания бинов.

<a id="q4-20"></a>
### 4.20 @Profile

`@Profile` включает бин только для определённого окружения: `dev`, `test`, `prod`. Активные профили задаются через `spring.profiles.active`. Обычно через профили разделяют реализации и конфиги по окружениям: fake-клиент в dev, реальный клиент в prod, отдельные настройки БД для тестов. Отличие от `@Conditional`: profile — грубое деление по окружениям, conditional — произвольное условие.

<a id="section-5"></a>
## Секция 5. Базы данных, SQL, Hibernate, JPA, Spring Data

<a id="q5-1"></a>
### 5.1 🟣 Виды индексов и как использовать для оптимизации

Индекс — это дополнительная структура, которая ускоряет поиск, но замедляет запись и занимает место. В PostgreSQL чаще всего используют **B-tree**: равенство, диапазоны, сортировка, `LIKE 'abc%'`. **GIN** хорош для `jsonb`, массивов, полнотекста и trigram-поиска. **GiST** — гео, диапазоны и похожие задачи. **BRIN** полезен на огромных append-only таблицах, где данные физически идут примерно по порядку. Оптимизация начинается не с “добавим индекс на всё”, а с `EXPLAIN ANALYZE`: смотрим реальный план, фильтры, join'ы, сортировку и селективность.

<a id="q5-2"></a>
### 5.2 Change Data Capture: реализация и уведомления между БД/сервисами

**CDC** — способ получать поток изменений из базы. Самый простой вариант — polling по `updated_at`, но он даёт лаг и плохо видит удаления. Триггеры с audit/outbox-таблицей надёжнее, но добавляют работу в транзакцию. Более зрелый вариант — читать WAL через logical decoding, часто через **Debezium → Kafka**. Для микросервисов хороший паттерн — **transactional outbox**: сервис в одной транзакции пишет бизнес-данные и событие в outbox, а отдельный процесс публикует событие наружу. Так не теряется атомарность между БД и брокером.

<a id="q5-3"></a>
### 5.3 Триггеры

Триггер — код в БД, который автоматически запускается на `INSERT`, `UPDATE` или `DELETE`. Его используют для аудита, технических полей, денормализованных счётчиков, проверок на уровне базы. Плюс — логика срабатывает независимо от приложения. Минус — она становится **скрытой**: разработчик может не ожидать, что простой `UPDATE` запускает ещё пачку действий. В микросервисах триггеры используют осторожно, потому что бизнес-логика, спрятанная в БД, сложнее тестируется, версионируется и отлаживается.

<a id="q5-4"></a>
### 5.4 Хранимые процедуры и функции

Хранимые функции и процедуры — это логика, которая выполняется прямо в БД. Они хороши, когда нужно массово обработать данные рядом с ними и не гонять миллионы строк через приложение. Например, сложная миграция, отчётная агрегация, batch-операция. Минусы: vendor lock-in, отдельное тестирование, сложнее code review и деплой. В backend-приложениях обычно стараются держать бизнес-логику в приложении, а в БД оставлять то, что реально выгодно выполнять на стороне данных.

<a id="q5-5"></a>
### 5.5 🟣 Materialized Views

Обычное view хранит **запрос**, а не данные: при обращении БД выполняет запрос заново. **Materialized view** хранит уже готовый результат, почти как таблицу. Поэтому читать его быстро, можно добавлять индексы, удобно делать отчёты и тяжёлые агрегаты. Цена — данные **устаревают**, пока view не обновили через `REFRESH MATERIALIZED VIEW`. Если нужен refresh без блокировки чтения, используют `CONCURRENTLY`, но для этого обычно нужен unique-индекс. Это классический trade-off: скорость чтения против свежести данных.

<a id="q5-6"></a>
### 5.6 Репликация между разными БД

Внутри PostgreSQL есть физическая streaming-репликация, где реплика почти побайтно повторяет primary, и логическая репликация, где можно передавать отдельные таблицы или изменения. Между разными СУБД чаще используют **CDC/ETL pipeline**: Debezium читает изменения, Kafka переносит поток, consumer пишет в целевую БД. Главные вопросы не “как скопировать строку”, а **лаг**, порядок событий, идемпотентность, обработка удалений и конфликтов, если запись возможна с двух сторон.

<a id="q5-7"></a>
### 5.7 Entity в JPA и маппинг

JPA entity — объект, который соответствует таблице и управляется persistence context. Обычно ставят **`@Entity`**, `@Table`, `@Id`, поля мапятся через `@Column`, связи — через `@ManyToOne`, `@OneToMany`, `@OneToOne`, `@ManyToMany`. Нужен no-args constructor, потому что Hibernate создаёт объекты через рефлексию и прокси. Практические правила: enum хранить как `STRING`, деньги — `BigDecimal`, связи по умолчанию держать `LAZY`, а `equals/hashCode` для entity писать аккуратно, чтобы не сломать коллекции и прокси.

<a id="q5-8"></a>
### 5.8 Транзакции и уровни изоляции

Транзакция нужна, чтобы группа операций выполнилась как единое целое: **ACID**. Уровни изоляции определяют, какие аномалии возможны: dirty read, non-repeatable read, phantom read. В PostgreSQL dirty read фактически не бывает, дефолт — **Read Committed**. Более строгие уровни дают более стабильную картину данных, но могут повышать риск конфликтов и ретраев. Важно понимать: изоляция не заменяет бизнес-логику. Для баланса, остатков, лимитов часто нужны атомарные `UPDATE`, `SELECT FOR UPDATE` или optimistic locking.

<a id="q5-9"></a>
### 5.9 🟣 Ленивая загрузка в Hibernate и проблемы

**Lazy loading** означает, что связанный объект или коллекция не грузятся сразу. Hibernate подставляет proxy или специальную коллекцию и делает SQL только при первом обращении. Проблемы две: **LazyInitializationException**, если обратиться к lazy-связи вне открытой сессии, и **N+1**, если в цикле для каждой entity догружается связь отдельным запросом. Решения: `JOIN FETCH`, `@EntityGraph`, DTO-проекции, `@BatchSize`. OSIV может скрывать проблему, но часто держит соединение слишком долго, поэтому его лучше не считать нормальным решением.

<a id="q5-10"></a>
### 5.10 Cascade в JPA

Cascade говорит Hibernate: если операция выполняется над родителем, применить её и к связанным объектам. Например, `PERSIST` сохранит детей вместе с родителем, `REMOVE` удалит их, `MERGE` сольёт изменения. Это удобно для **агрегатов**, где родитель реально владеет дочерними объектами, например Order и OrderItem. Опасность — поставить `CascadeType.ALL` куда попало. Особенно страшно на `ManyToMany`: можно удалить объект, которым пользуются другие сущности. `orphanRemoval=true` удаляет ребёнка, если его убрали из коллекции родителя.

<a id="q5-11"></a>
### 5.11 🔵 N+1 и предотвращение

**N+1** — когда сначала выполняется один запрос за списком, а потом ещё N запросов за связями каждого элемента. На маленьких данных это маскируется, на проде превращается в весёлую катастрофу с SQL-логами на простыню. Диагностика: SQL logs, Hibernate statistics, p6spy/datasource-proxy. Решения: **`JOIN FETCH`**, **`@EntityGraph`**, DTO-проекции, `@BatchSize`, правильные запросы под конкретный экран/API. Главное — не делать одну универсальную entity-граф-мясорубку на все случаи, иначе победишь N+1 и проиграешь память.

<a id="q5-12"></a>
### 5.12 Стратегии генерации PK

`IDENTITY` использует автоинкремент в БД, просто работает, но Hibernate хуже батчит insert'ы, потому что id нужен сразу. `SEQUENCE` обычно лучше для PostgreSQL: можно брать id пачками через `allocationSize`, и batch insert работает эффективнее. `TABLE` эмулирует sequence через отдельную таблицу и почти всегда не нужен. `UUID` удобен в распределённых системах, потому что id можно создать в приложении, но он тяжелее для индексов; лучше использовать более упорядоченные варианты вроде UUIDv7, если это доступно.

<a id="q5-13"></a>
### 5.13 Мониторинг производительности запросов

Сначала нужно видеть, какие запросы реально выполняются и сколько они стоят. На уровне приложения помогают Hibernate statistics, p6spy/datasource-proxy, метрики HikariCP, APM и трассировка. На уровне PostgreSQL — **`pg_stat_statements`**, `EXPLAIN ANALYZE`, slow query log, auto_explain. Нормальный подход: найти медленный endpoint, увидеть конкретный SQL, проверить план, индексы, количество строк и только потом оптимизировать. Без измерений “оптимизация” быстро превращается в гадание на YAML.

<a id="q5-14"></a>
### 5.14 🟣 Оптимистическая vs пессимистическая блокировка

**Optimistic locking** предполагает, что конфликты редкие. В entity добавляют `@Version`, и при обновлении Hibernate проверяет, что версия не изменилась. Если изменилась — конфликт, нужен retry или ошибка пользователю. **Pessimistic locking** сразу блокирует строку, например через `SELECT FOR UPDATE`, и другие транзакции ждут. Оптимистическая блокировка хороша для редких конфликтов и обычных форм редактирования. Пессимистическая — для горячих данных: балансы, остатки, очереди, где лучше подождать, чем потом разгребать конфликт.

<a id="q5-15"></a>
### 5.15 Пагинация в Spring Data + большие объёмы

`Pageable` позволяет передать страницу, размер и сортировку. `Page` возвращает данные плюс общее количество, но для этого делает count-запрос, который может быть дорогим. `Slice` проще: говорит, есть ли следующая страница, без полного count. Для больших таблиц `OFFSET` начинает деградировать, потому что БД всё равно должна пропустить много строк. Тогда лучше **keyset pagination**: идти от последнего `(created_at, id)` и брать следующие N записей. Обязательно нужна стабильная сортировка с tie-breaker, обычно `id`.

<a id="q5-16"></a>
### 5.16 Second Level Cache

First-level cache — это persistence context внутри одной сессии/транзакции. **Second-level cache** живёт дольше и может переиспользоваться между сессиями. Его включают через провайдер вроде Ehcache/Infinispan и настраивают для отдельных entity/collections. Хорошо подходит для почти неизменяемых справочников. Для часто меняющихся данных опасен: нужно думать про инвалидацию, консистентность, кластер и stale data. Поэтому во многих проектах проще и прозрачнее использовать service-level cache через Caffeine/Redis с явным TTL.

<a id="q5-17"></a>
### 5.17 Кастомные запросы в Spring Data, @Query

Spring Data умеет строить запросы по имени метода, например `findByStatusAndCreatedAtAfter`, но для сложной логики лучше использовать **`@Query`**. Это может быть JPQL или native SQL. Для `UPDATE`/`DELETE` нужен `@Modifying`. Для динамических фильтров подходят `Specification`, Criteria API или Querydsl. Практическое правило: простые запросы — derived methods, понятные нестандартные запросы — `@Query`, сложная динамика — Specification/Querydsl, а совсем специфичный SQL — native query или кастомный репозиторий.

<a id="q5-18"></a>
### 5.18 🟣 Liquibase/Flyway, автоматизация миграций

Liquibase и Flyway версионируют схему БД вместе с кодом. **Flyway** обычно работает через SQL-файлы `V1__init.sql`, `V2__add_table.sql`. **Liquibase** использует changeset'ы в XML/YAML/SQL с id, author и checksum. Главная идея: схема меняется контролируемо, а не руками в проде, потому что мы всё-таки цивилизованные люди, местами. В CI/CD миграции запускают до старта приложения или отдельным шагом. Важное правило: уже применённые миграции не редактируют; для rolling deploy используют подход **expand → migrate → contract**.

<a id="q5-19"></a>
### 5.19 Связи One-to-One/Many + подводные камни

В JPA владельцем связи обычно является сторона с **foreign key** и `@JoinColumn`, обратная сторона использует `mappedBy`. `ManyToOne` по умолчанию EAGER, но на практике его часто переводят в `LAZY`. Коллекции обычно LAZY. Подводные камни: N+1, рекурсия в `toString`, плохой `equals/hashCode` с ассоциациями, каскады не туда, `ManyToMany`, который потом приходится заменять явной entity-связью. `OneToOne LAZY` тоже может неприятно удивить без bytecode enhancement.

<a id="q5-20"></a>
### 5.20 Пагинация при работе с БД (общий вопрос)

Есть два основных подхода. **Offset pagination** — `LIMIT/OFFSET`: просто, удобно для UI, но на глубоких страницах дорого и нестабильно при новых вставках. **Keyset/seek pagination** — идти от последнего значения сортировки: `WHERE (created_at, id) > (...) ORDER BY created_at, id LIMIT N`. Это быстрее на больших данных и стабильнее для бесконечных лент, export'ов и delta-endpoint'ов. Главное — иметь индекс под сортировку и фильтры.

<a id="q5-21"></a>
### 5.21 Фильтрация/сортировка/Pageable, спецификации

Для простых фильтров хватает derived query или `@Query` с параметрами и `Pageable`. Когда фильтров много и они комбинируются динамически, удобны **`Specification`** и `JpaSpecificationExecutor`: каждый фильтр становится маленьким предикатом, которые можно соединять через `and/or`. Сортировку из API нельзя слепо прокидывать в БД: нужен whitelist разрешённых полей, иначе можно получить тяжёлые запросы, ошибки или утечку внутренней структуры модели.

<a id="section-6"></a>
## Секция 6. REST, микросервисы, межсетевое взаимодействие

<a id="q6-1"></a>
### 6.1 🔵 Что такое REST, принципы

**REST** — архитектурный стиль для API, где мы работаем с ресурсами через стандартные HTTP-методы. Ресурс — это существительное в URI: `/users`, `/orders/42`, а действие выражается методом: `GET`, `POST`, `PUT`, `DELETE`. Главные принципы: **stateless**, client-server, cacheable, uniform interface, layered system. Stateless особенно важен: каждый запрос должен содержать всё нужное для обработки, поэтому сервис проще масштабировать горизонтально.

<a id="q6-2"></a>
### 6.2 Идемпотентность методов HTTP

Идемпотентность значит: повтор одного и того же запроса приводит к тому же состоянию системы. **GET**, **PUT**, **DELETE** считаются идемпотентными по смыслу, хотя ответ может отличаться, например второй `DELETE` вернёт 404. **POST** обычно не идемпотентен: два POST могут создать две сущности. Для платежей и заказов POST часто делают идемпотентным через **Idempotency-Key**, чтобы ретрай после сетевого сбоя не создал дубль и не списал деньги дважды, ведь банковские системы и так достаточно любят драму.

<a id="q6-3"></a>
### 6.3 Статус-коды

Статус-код должен честно отражать результат. **2xx** — успех: `200 OK`, `201 Created`, `204 No Content`, `202 Accepted`. **4xx** — ошибка клиента: `400` невалидный запрос, `401` нет аутентификации, `403` нет прав, `404` не найдено, `409` конфликт состояния, `422` семантическая ошибка, `429` rate limit. **5xx** — проблема сервера или зависимостей: `500`, `502`, `503`, `504`. Главное правило: не превращать все ошибки в `500`, потому что клиенту нужно понимать, можно ли исправить запрос или надо повторить позже.

<a id="q6-4"></a>
### 6.4 Пагинация в REST API

В REST чаще встречаются два подхода. **Offset pagination**: `page`, `size`, `sort`; просто для UI, но плохо на глубоких страницах и может “ехать”, если данные меняются. **Cursor pagination**: клиент получает `nextCursor` и запрашивает следующую порцию; внутри часто keyset pagination. Cursor лучше для лент, больших списков и стабильной навигации. В любом случае нужно ограничивать максимальный `size`, задавать стабильную сортировку и возвращать клиенту понятный признак `hasNext` или следующий cursor.

<a id="q6-5"></a>
### 6.5 Безопасность REST API: аутентификация и авторизация

**Аутентификация** отвечает на вопрос “кто ты?”, **авторизация** — “что тебе можно?”. Для REST используют HTTPS, JWT/Bearer token, OAuth2/OIDC, API keys для простых интеграций, mTLS для service-to-service в строгих окружениях. Важно проверять права не только на endpoint, но и на конкретный ресурс, иначе будет **IDOR**: пользователь меняет id в URL и видит чужие данные. Ещё нужны validation, rate limiting, маскирование секретов в логах и короткий TTL access token'ов.

<a id="q6-6"></a>
### 6.6 🟣 Обработка ошибок: @ExceptionHandler, @ControllerAdvice

`@RestControllerAdvice` позволяет централизованно обрабатывать исключения из контроллеров. Вместо try/catch в каждом endpoint можно написать `@ExceptionHandler`, который мапит `NotFoundException` в 404, `ValidationException` в 400, конфликт состояния в 409. В Boot 3 удобно использовать **ProblemDetail** в стиле RFC 7807. В ответ не нужно отдавать stack trace и внутренние классы. В логах, наоборот, нужен traceId/correlationId, чтобы потом найти реальную причину, а не шаманить по скриншоту фронтенда.

<a id="q6-7"></a>
### 6.7 CORS

**CORS** — браузерный механизм, который решает, может ли frontend с одного origin читать ответ API с другого origin. Это не защита backend от Postman, это именно политика браузера. При “сложных” запросах браузер сначала делает preflight `OPTIONS`, а сервер отвечает заголовками `Access-Control-Allow-Origin`, `Allow-Methods`, `Allow-Headers`. Важно не ставить бездумно `*`, особенно если используются credentials/cookies. В Spring CORS настраивают глобально или через Spring Security, обычно с whitelist конкретных origin'ов.

<a id="q6-8"></a>
### 6.8 Синхронные vs асинхронные запросы в микросервисах

**Синхронный вызов** REST/gRPC подходит, когда нужен ответ прямо сейчас: проверить лимит, получить профиль, создать сущность с immediate result. Минусы — зависимость по времени: если downstream тормозит, тормозит и вызывающий сервис. **Асинхронное взаимодействие** через Kafka/RabbitMQ лучше для событий, уведомлений, аналитики, интеграций и буферизации пиков. Минусы — eventual consistency и сложнее отладка. Нормальное правило: команда с немедленным результатом — sync с timeout/circuit breaker; факт “что-то произошло” — event.

<a id="q6-9"></a>
### 6.9 Circuit Breaker

**Circuit Breaker** защищает сервис от зависимостей, которые начали падать или отвечать слишком долго. В состоянии closed запросы идут нормально. Если ошибок/таймаутов слишком много, breaker переходит в open и быстро отклоняет запросы, не добивая зависимость. Потом half-open пропускает несколько тестовых запросов и решает, восстановилась ли зависимость. Это помогает избежать каскадного отказа, когда один медленный сервис съедает все потоки у соседей. Обычно используют вместе с timeout, retry и bulkhead.

<a id="q6-10"></a>
### 6.10 Согласованность данных между микросервисами: сага и 2PC

В микросервисах нельзя просто открыть одну транзакцию на пять баз и жить счастливо. **2PC/XA** теоретически даёт распределённую транзакцию, но он тяжёлый, блокирующий и плохо подходит для автономных сервисов. **Saga** разбивает процесс на локальные транзакции и компенсирующие действия. В choreography сервисы реагируют на события друг друга, в orchestration есть координатор процесса. Для надёжности саге почти всегда нужны **outbox**, идемпотентные обработчики и понятная модель статусов.

<a id="q6-11"></a>
### 6.11 API Gateway

**API Gateway** — единая точка входа перед набором сервисов. Он маршрутизирует запросы, может проверять токены, делать rate limiting, TLS termination, CORS, агрегацию, канареечную маршрутизацию. Но gateway не должен превращаться в место, куда свалили всю бизнес-логику, потому что тогда получается распределённый монолит с красивым фасадом. Хороший gateway решает инфраструктурные задачи на периметре, а доменная логика остаётся в сервисах.

<a id="q6-12"></a>
### 6.12 gRPC vs REST, когда gRPC

**REST** обычно проще для публичных API: JSON, HTTP, легко дебажить curl'ом и читать глазами. **gRPC** использует HTTP/2 и Protobuf, даёт строгий контракт через `.proto`, кодогенерацию, быстрый бинарный формат, streaming и deadlines. gRPC хорошо подходит для внутренней service-to-service коммуникации, где важны скорость, контрактность и полиглотность. Минусы: сложнее ручная отладка, браузеру нужен grpc-web, меньше “человеческой” читаемости. Наружу часто дают REST, внутрь — gRPC или события.

<a id="q6-13"></a>
### 6.13 🔵 SOAP vs REST

**SOAP** — протокол с XML-envelope, WSDL-контрактом и стандартами WS-* для безопасности, транзакций и надёжной доставки. Он тяжёлый, но в enterprise, банках и госсистемах до сих пор встречается, потому что там любят строгие контракты и XML как семейную реликвию. **REST** легче: ресурсы, HTTP-методы, JSON, меньше церемоний. Для новых публичных API обычно выбирают REST, для старых интеграций с банками/госами часто приходится жить с SOAP.

<a id="q6-14"></a>
### 6.14 FeignClient vs RestTemplate vs WebClient

**FeignClient** — декларативный клиент: описываешь интерфейс, Spring генерирует реализацию. Удобно для межсервисных вызовов, меньше boilerplate, хорошо дружит с interceptors, load balancing и resilience. **RestTemplate** — старый синхронный клиент, всё ещё работает, но считается legacy/maintenance. **RestClient** — современная синхронная замена в новых Spring. **WebClient** — неблокирующий reactive-клиент, хорош для WebFlux, streaming и большого числа IO-вызовов. Выбор зависит не от моды, а от стека приложения.

<a id="q6-15"></a>
### 6.15 Логирование и мониторинг REST API

Для REST важно видеть метод, путь, статус, длительность, traceId и ошибку. Тела запросов логируют осторожно: там могут быть пароли, токены, персональные данные. Метрики обычно собирают через Micrometer: количество запросов, latency p95/p99, error rate, статусы. Трейсинг через OpenTelemetry/Micrometer Tracing помогает пройти путь запроса через несколько сервисов. Нормальная картина: логи отвечают “что произошло”, метрики — “насколько всё плохо”, трейсы — “где именно всё умерло”.

<a id="q6-16"></a>
### 6.16 Сжатие в REST API

Сжатие уменьшает размер ответа через `Content-Encoding: gzip/br`, если клиент прислал `Accept-Encoding`. В Spring Boot можно включить `server.compression.enabled=true`, но часто это делают на nginx/gateway. Сжимать полезно большие JSON-ответы, текст, статику. Мелкие ответы иногда невыгодно сжимать из-за CPU overhead. И важный момент: иногда лучший способ уменьшить трафик — не gzip, а нормальная пагинация и DTO без лишних полей. Какая неожиданная идея: не отправлять мусор.

<a id="q6-17"></a>
### 6.17 🟣 OpenAPI/Swagger

**OpenAPI** — спецификация REST-контракта: endpoints, параметры, схемы запросов/ответов, коды ошибок, security. Swagger UI — удобная визуализация и “потыкать руками”. В Spring часто используют springdoc-openapi, который генерирует спецификацию из контроллеров и аннотаций. Есть два подхода: **code-first**, когда spec генерируется из кода, и **contract-first**, когда сначала пишется контракт, а потом по нему генерируются клиенты/заглушки. Для командной разработки contract-first часто надёжнее.

<a id="q6-18"></a>
### 6.18 Тестирование REST API

REST API тестируют на нескольких уровнях. `@WebMvcTest` + MockMvc проверяет контроллеры, валидацию и `ControllerAdvice` без полного приложения. `@SpringBootTest` с random port поднимает приложение ближе к реальности. RestAssured даёт удобный DSL для HTTP-проверок. Внешние сервисы мокают через WireMock, БД поднимают через Testcontainers. Для контрактов между сервисами используют Pact или Spring Cloud Contract. Нагрузку проверяют k6/Gatling/JMeter, когда уже хочется узнать, где система треснет первой.

<a id="q6-19"></a>
### 6.19 RPC vs REST

**RPC** выглядит как вызов функции по сети: `createOrder`, `calculatePrice`, `getUser`. Контракт обычно ближе к операциям. **REST** строится вокруг ресурсов и стандартных HTTP-методов: `/orders`, `/users/42`. RPC удобен для внутренних жёстких контрактов, особенно gRPC. REST удобен для публичных API, кэширования, uniform interface и понятной модели ресурсов. Разница не в том, что одно “хорошее”, а другое “плохое”; просто они по-разному моделируют взаимодействие.

<a id="q6-20"></a>
### 6.20 Cookie vs Bearer Token

**Cookie** браузер отправляет автоматически, поэтому удобна для web-сессий, особенно если она `HttpOnly`, `Secure`, `SameSite`. Но автоматическая отправка создаёт риск **CSRF**, если не настроить защиту. **Bearer token** передаётся в `Authorization` header; он не отправляется браузером сам, поэтому CSRF меньше актуален, но если хранить токен в `localStorage`, XSS может его украсть. Для SPA часто хороший вариант — HttpOnly Secure SameSite cookie или BFF-подход, для mobile/service-to-service — bearer token.

<a id="q6-21"></a>
### 6.21 Access и refresh token

**Access token** короткоживущий и используется в обычных запросах. Если его украдут, ущерб ограничен временем жизни. **Refresh token** живёт дольше и нужен, чтобы получить новый access token без повторного логина. Его нужно хранить безопаснее: HttpOnly cookie, secure storage, rotation. Хорошая практика — **refresh token rotation**: при каждом обновлении выдаётся новый refresh, старый инвалидируется, а повторное использование старого считается сигналом компрометации.

<a id="q6-22"></a>
### 6.22 NGINX как reverse proxy

NGINX как **reverse proxy** принимает запросы от клиентов и проксирует их на backend-сервисы. Он может делать TLS termination, gzip, отдачу статики, кэширование, rate limiting, буферизацию медленных клиентов и скрывать внутреннюю топологию. Для приложения важны заголовки вроде `X-Forwarded-For` и `X-Forwarded-Proto`, чтобы правильно понимать реальный IP и схему запроса. Часто NGINX стоит перед несколькими инстансами приложения и распределяет трафик.

<a id="q6-23"></a>
### 6.23 Балансировка NGINX

В NGINX upstream может балансировать запросы разными способами. **Round-robin** — по очереди, вариант по умолчанию. **Least connections** отправляет на сервер с меньшим числом активных соединений. **ip_hash** держит одного клиента на одном backend, если нужна липкая сессия. Можно задавать веса, если инстансы разной мощности. В open-source NGINX health-check в основном пассивный: сервер помечается плохим после ошибок. Для активных health checks обычно нужен NGINX Plus или отдельная инфраструктура.

<a id="q6-24"></a>
### 6.24 HTTP/2 и HTTP/3 vs 1.1

HTTP/1.1 текстовый и исторически ограничен проблемой head-of-line blocking на соединении. HTTP/2 вводит бинарные фреймы, **multiplexing** нескольких потоков в одном TCP-соединении и сжатие заголовков. Это снижает накладные расходы, особенно при множестве запросов. HTTP/3 работает поверх **QUIC/UDP**, быстрее устанавливает соединение и убирает head-of-line blocking на уровне TCP. Для backend-разработчика важно понимать: протокол влияет на latency, соединения, прокси и поведение клиентов, но REST-модель ресурсов от этого не исчезает.

<a id="q6-25"></a>
### 6.25 Микросервисные паттерны + один подробно

Частые паттерны: **API Gateway**, Service Discovery, Circuit Breaker, Retry/Timeout, Bulkhead, Saga, Transactional Outbox, Idempotent Consumer, CQRS, Event Sourcing, Strangler Fig. Один хороший паттерн для подробного ответа — **Transactional Outbox**. Проблема: нельзя атомарно закоммитить БД и отправить сообщение в брокер обычным способом. Решение: в той же транзакции пишем бизнес-данные и запись в outbox-таблицу. Отдельный publisher читает outbox и отправляет событие. Получаем at-least-once, поэтому consumer должен быть идемпотентным.

<a id="section-7"></a>
## Секция 7. Брокеры сообщений: Kafka, RabbitMQ, JMS

<a id="q7-1"></a>
### 7.1 Pub/sub vs point-to-point

**Point-to-point** — сообщение из очереди обрабатывает один consumer: это модель распределения работы. **Pub/sub** — одно событие получают все подписчики, которым оно интересно. В Kafka обе модели достигаются через consumer groups: внутри одной группы сообщение из партиции читает один consumer, а разные группы читают поток независимо. В RabbitMQ pub/sub обычно строится через exchange, который раскладывает сообщения в разные очереди. В JMS это классическое различие Queue vs Topic.

<a id="q7-2"></a>
### 7.2 Kafka vs RabbitMQ: отличия и выбор

**Kafka** — распределённый commit log. Сообщения хранятся по retention, consumer сам читает по offset, можно делать replay, высокая пропускная способность, порядок гарантирован внутри партиции. **RabbitMQ** — брокер очередей с гибкой маршрутизацией через exchanges, ack/nack на сообщение, TTL, DLQ, priority и удобные work queues. Kafka лучше для событийных потоков, аналитики, интеграций многих подписчиков. RabbitMQ лучше для задач, команд, сложного роутинга и сценариев “взял задачу → обработал → подтвердил”.

<a id="q7-3"></a>
### 7.3 Партиционирование в Kafka

Топик Kafka состоит из **партиций**. Партиция — единица порядка и параллелизма. Сообщения с одинаковым key обычно попадают в одну партицию, поэтому по одному `orderId` сохраняется порядок. Consumer group распределяет партиции между consumer'ами: одна партиция в рамках группы читается только одним consumer'ом одновременно. Если consumer'ов больше, чем партиций, лишние простаивают. Подводные камни: горячий ключ может перегрузить одну партицию, а увеличение числа партиций меняет распределение ключей.

<a id="q7-4"></a>
### 7.4 Уникальность сообщений / предотвращение дублей

В распределённых системах дубли — нормальная реальность, а не редкий зверь из учебника. Kafka producer может включить **idempotence**, чтобы брокер отфильтровывал дубли при retry. Но на стороне consumer всё равно нужна **идемпотентная обработка**: eventId, business key, таблица `processed_events`, `ON CONFLICT DO NOTHING`, upsert вместо blind insert. Обычно целятся в at-least-once delivery и делают обработчик безопасным к повтору. Exactly-once вне узкого Kafka-пайплайна почти всегда требует дополнительных гарантий на уровне БД и бизнес-операций.

<a id="q7-5"></a>
### 7.5 🟣 Offset в Kafka

**Offset** — позиция сообщения внутри партиции. Consumer group хранит committed offset в специальном топике `__consumer_offsets`: это точка, с которой группа продолжит читать после рестарта или rebalance. Если коммитить offset до обработки, можно потерять сообщение. Если после обработки — возможны дубли при падении между обработкой и коммитом. Поэтому стандартный безопасный подход: обработал → зафиксировал результат → закоммитил offset, а обработчик сделал идемпотентным. Offset ещё позволяет делать replay через `seek`.

<a id="q7-6"></a>
### 7.6 Механизмы маршрутизации RabbitMQ

В RabbitMQ producer отправляет сообщение в **exchange**, а exchange маршрутизирует его в очереди по binding'ам. `direct` отправляет по точному routing key. `topic` поддерживает шаблоны вроде `order.*.paid` и `order.#`. `fanout` рассылает во все связанные очереди, игнорируя ключ. `headers` маршрутизирует по заголовкам. Сила RabbitMQ именно в такой гибкой маршрутизации: можно описать сложные схемы доставки без того, чтобы каждый consumer сам фильтровал весь поток.

<a id="q7-7"></a>
### 7.7 🟣 Durable Queue и сохранность при сбоях

Чтобы сообщение пережило рестарт RabbitMQ, мало сделать очередь durable. Нужна вся цепочка: **durable exchange**, **durable queue**, **persistent message** и желательно **publisher confirms**, чтобы producer знал, что брокер принял сообщение. Если очередь durable, но сообщение transient, очередь после рестарта останется, а сообщение исчезнет. Для высокой доступности в кластере используют quorum queues, а не старые mirrored queues. На стороне consumer ack нужно отправлять **после** успешной обработки, иначе сообщение можно потерять.

<a id="q7-8"></a>
### 7.8 Acknowledgements в RabbitMQ и Kafka, проблемы настройки

В RabbitMQ consumer отправляет **ack**, когда сообщение успешно обработано. Если поставить autoAck, брокер считает сообщение обработанным сразу после доставки, и при падении consumer оно потеряется. Если забыть ack, сообщения зависнут unacked и очередь начнёт “стоять”. В Kafka ack как отдельной операции нет: аналог — **commit offset**. Коммит до обработки даёт потерю, коммит после обработки даёт возможный повтор. Поэтому в обоих мирах главная идея одна: подтверждать после результата и делать обработку идемпотентной.

<a id="q7-9"></a>
### 7.9 Порядок доставки и семантики (at-most/at-least/exactly-once)

В Kafka порядок гарантирован **только внутри одной партиции**. Если важен порядок по заказу, ключом должен быть `orderId`, чтобы все события заказа попали в одну партицию. **At-most-once** — сообщение может потеряться, но не повторится. **At-least-once** — сообщение не потеряется, но может повториться; это самый практичный вариант. **Exactly-once** в Kafka возможен для Kafka-to-Kafka processing с транзакциями, но если есть внешняя БД или HTTP-сервис, всё равно нужна идемпотентность. Магии “ровно один раз во вселенной” обычно не бывает, как и нормального ТЗ с первого раза.

<a id="q7-10"></a>
### 7.10 Dead Letter Queue в RabbitMQ

**DLQ** — очередь для сообщений, которые не удалось обработать нормально. В RabbitMQ сообщение может попасть в DLQ после reject/nack без requeue, истечения TTL или превышения лимита доставок. Обычно настраивают dead-letter-exchange и routing key. DLQ нужна, чтобы плохое сообщение не блокировало всю очередь бесконечными retry. Нормальная схема: несколько retry с задержкой, потом DLQ, алерт, ручной анализ или отдельный recovery-процесс. Важно сохранять причину ошибки и payload, иначе DLQ превращается в кладбище без надгробий.

<a id="q7-11"></a>
### 7.11 Проблемы брокеров в распределённых системах

Брокер не отменяет сложность распределённых систем, он просто переносит её в другое место, спасибо прогрессу. Типичные проблемы: дубли, потеря порядка, lag, backpressure, poison messages, schema evolution, недоступность брокера, split brain/cluster issues, рост storage из-за retention, неидемпотентные consumer'ы. Поэтому вокруг брокера нужны retry policy, DLQ, мониторинг lag, трассировка, versioned schemas и идемпотентная обработка. Сообщение “отправлено” ещё не значит “бизнес-операция завершена”.

<a id="q7-12"></a>
### 7.12 Spring Kafka и Spring AMQP

Spring Kafka и Spring AMQP дают удобные абстракции поверх клиентов брокеров. В Kafka обычно используют `KafkaTemplate` для отправки и `@KafkaListener` для чтения. Можно настраивать consumer factory, error handler, retry, manual ack, deserialization, transactions. В RabbitMQ через Spring AMQP используют `RabbitTemplate`, `@RabbitListener`, декларацию queues/exchanges/bindings, retry и DLQ. Смысл Spring-обёрток — меньше boilerplate и единый стиль интеграции, но семантику брокера всё равно нужно понимать.

<a id="q7-13"></a>
### 7.13 🔵 Consumer groups в Kafka

**Consumer group** — способ масштабировать чтение топика. Все consumer'ы с одним `group.id` делят между собой партиции топика. Каждая партиция назначается только одному consumer'у внутри группы, поэтому порядок внутри партиции сохраняется. Если появляется новый consumer или падает старый, происходит **rebalance**. Разные группы читают один и тот же топик независимо, каждая со своими offset'ами. Поэтому один топик может одновременно читать billing, analytics и notifications, не мешая друг другу.

<a id="q7-14"></a>
### 7.14 Транзакционность в Kafka и RabbitMQ

В Kafka есть транзакции producer'а: можно атомарно записать сообщения в несколько партиций и закоммитить offsets в рамках Kafka-процессинга. Это нужно для exactly-once в Kafka Streams/consume-transform-produce сценариях. Но транзакция Kafka не делает автоматически атомарной запись в PostgreSQL. Для БД + Kafka обычно используют **outbox**. В RabbitMQ транзакции есть, но чаще используют **publisher confirms**, потому что они производительнее. Бизнес-транзакционность с RabbitMQ тоже обычно строят через outbox/idempotency, а не через магическое “всё вместе”.

<a id="q7-15"></a>
### 7.15 Мониторинг и метрики брокеров

Для Kafka важны consumer lag, throughput, error rate, rebalance count, under-replicated partitions, ISR, broker disk, request latency. Для RabbitMQ — queue depth, unacked messages, publish/ack rate, consumer count, redeliveries, DLQ size, memory/disk alarms. Метрики без алертов — просто красивые графики для посмертного вскрытия. Хороший мониторинг отвечает на вопросы: растёт ли очередь, отстают ли consumer'ы, есть ли дубли/ошибки, хватает ли диска, жив ли кластер.

<a id="q7-16"></a>
### 7.16 🔥 Retention в Kafka

**Retention** в Kafka определяет, сколько сообщений хранить: по времени (`retention.ms`) или по размеру (`retention.bytes`). Consumer после чтения не удаляет сообщение; Kafka хранит лог независимо от consumer'ов. Это позволяет делать replay, подключать новых потребителей и восстанавливать обработку. Есть ещё **log compaction**: Kafka хранит последнее значение по ключу, что удобно для changelog/state topics. Важно: если consumer отстал дольше retention, старые сообщения уже удалятся, и догнать их будет невозможно.

<a id="q7-17"></a>
### 7.17 Backpressure и message flooding

**Backpressure** — ситуация, когда producer производит быстрее, чем consumer успевает обрабатывать. В Kafka это проявляется ростом consumer lag. В RabbitMQ — ростом queue depth и unacked messages. Решения: ограничивать скорость producer'ов, масштабировать consumer'ы, увеличивать партиции/очереди, настраивать prefetch, batch processing, retry с задержкой, DLQ для poison messages. Важно не делать бесконечные retry без паузы: так система не лечится, а просто громче страдает.

<a id="q7-18"></a>
### 7.18 Балансировка нагрузки RabbitMQ

В RabbitMQ несколько consumer'ов могут читать одну очередь, и брокер распределяет сообщения между ними. Чтобы один consumer не получил слишком много работы заранее, настраивают **prefetch**: сколько unacked-сообщений можно выдать одному consumer'у. Ack после обработки позволяет брокеру выдавать новые сообщения по мере освобождения consumer'ов. Для отказоустойчивости используют quorum queues. Если нужна маршрутизация по типам задач, делают exchange + несколько очередей, а не одну гигантскую очередь “для всего”.

<a id="q7-19"></a>
### 7.19 Распределённые транзакции с брокерами: сага и 2PC

2PC с брокером и БД технически возможен в некоторых стеках, но на практике тяжёлый, хрупкий и плохо масштабируется. В микросервисах чаще используют **Saga** и **Transactional Outbox**. Сервис в локальной транзакции меняет свою БД и пишет событие в outbox. Publisher отправляет событие в брокер. Другие сервисы обрабатывают событие идемпотентно и при ошибках запускают компенсации. Это не “простая транзакция”, но зато система остаётся более живучей и автономной.

<a id="q7-20"></a>
### 7.20 Шифрование и аутентификация в Kafka/RabbitMQ

Для брокеров нужны две вещи: **шифрование транспорта** и **проверка клиента**. Kafka поддерживает TLS/SSL, SASL/PLAIN, SCRAM, Kerberos, ACL на topics и groups. RabbitMQ поддерживает TLS, username/password, vhosts, permissions на exchanges/queues. Важно не хранить пароли в репозитории, использовать secrets, ротацию, минимальные права и разделение окружений. Для межсервисной коммуникации в строгих системах могут использовать mTLS. Брокер — не внутренняя игрушка; через него часто проходит почти вся бизнес-жизнь системы.

<a id="section-8"></a>
## Секция 8. Многопоточность, GC, JVM

<a id="q8-1"></a>
### 8.1 Immutable: как сделать и зачем

**Immutable** объект нельзя изменить после создания. Делают класс `final`, поля `private final`, не дают сеттеров, все значения задают в конструкторе, а mutable-поля копируют на входе и выходе. Польза: такие объекты проще использовать в многопоточке, безопаснее кэшировать, передавать между слоями и использовать как ключи в map. Пример — `String`, record'ы и value object'ы. Главное не забыть про defensive copy для `List`, `Date`, массивов и прочих изменяемых радостей.

<a id="q8-2"></a>
### 8.2 wait() — как работает и почему аккуратно

`wait()` вызывается только внутри `synchronized` на том же объекте-мониторе. Поток освобождает монитор и засыпает, пока другой поток не вызовет `notify/notifyAll` или не истечёт timeout. Проверку условия нужно делать в **while**, а не в `if`, потому что возможны spurious wakeups и пробуждение не того потока. `wait/notify` — низкоуровневый механизм, поэтому в обычном коде чаще используют `BlockingQueue`, `CountDownLatch`, `Semaphore`, `Condition` и другие готовые конструкции.

<a id="q8-3"></a>
### 8.3 Поток vs процесс; создание и запуск

**Процесс** — отдельный экземпляр программы со своей памятью и ресурсами ОС. **Поток** живёт внутри процесса и разделяет с другими потоками heap, открытые файлы и другие ресурсы процесса, но имеет свой stack. В Java поток можно создать через `new Thread(runnable).start()`, но в реальном backend-коде почти всегда используют **ExecutorService**, потому что создавать потоки вручную дорого и неудобно контролировать. Важно: `run()` просто вызовет метод в текущем потоке, а `start()` реально запускает новый поток.

<a id="q8-4"></a>
### 8.4 volatile

`volatile` даёт **видимость изменений** между потоками и запрещает некоторые переупорядочивания инструкций. Если один поток записал значение в volatile-поле, другой поток увидит эту запись при чтении. Но volatile не делает сложные операции атомарными: `count++` всё равно состоит из read-modify-write и может проиграть race condition. Хорошее применение: флаги остановки, публикация ссылки на уже корректно созданный объект, простые состояния. Для счётчиков лучше `AtomicInteger` или lock.

<a id="q8-5"></a>
### 8.5 Happens-before и за счёт чего достигается

**Happens-before** — правило Java Memory Model: если операция A happens-before операции B, то результаты A видны B. Это достигается через `synchronized` unlock/lock, volatile write/read, `Thread.start`, `Thread.join`, завершение `Future`, операции concurrent collections и другие гарантии. Смысл вопроса: в многопоточке важна не только атомарность, но и **видимость**. Код может “работать на моей машине”, пока CPU/JIT/cache не покажут, что без happens-before ты просто верил в сказку.

<a id="q8-6"></a>
### 8.6 wait()/notify() — где целесообразно

`wait/notify` уместны, когда нужно вручную построить координацию потоков вокруг общего условия. Но это низкий уровень, где легко ошибиться: забыть lock, использовать `if` вместо `while`, вызвать `notify` вместо `notifyAll`, потерять сигнал. В реальном коде чаще лучше взять готовую конструкцию: `BlockingQueue` для producer-consumer, `CountDownLatch` для ожидания завершения, `Semaphore` для ограничения доступа, `Condition` с `ReentrantLock` для более явных условий.

<a id="q8-7"></a>
### 8.7 sleep() vs wait()

`Thread.sleep()` просто приостанавливает текущий поток на время и **не освобождает монитор**, если поток был внутри `synchronized`. `wait()` вызывается на объекте-мониторе, освобождает этот монитор и ждёт `notify/notifyAll` или timeout. `sleep` — про паузу по времени. `wait` — про ожидание условия и взаимодействие потоков. Поэтому заменить одно другим нельзя: они решают разные задачи, хотя оба, разумеется, умеют сделать debugging ещё веселее.

<a id="q8-8"></a>
### 8.8 Как избежать deadlock

**Deadlock** возникает, когда потоки ждут ресурсы друг друга и никто не может продолжить. Профилактика: захватывать locks всегда в одном порядке, не держать lock во время внешних вызовов/IO, минимизировать критические секции, использовать timeout через `tryLock`, не смешивать разные механизмы блокировок без нужды. Диагностика: thread dump, где видно потоки в `BLOCKED` и мониторы, которые они держат/ждут. Главное правило: чем меньше ручных locks, тем меньше шансов построить комнату без дверей.

<a id="q8-9"></a>
### 8.9 ReentrantLock

`ReentrantLock` — явная блокировка из `java.util.concurrent.locks`. Как и `synchronized`, она **reentrant**: поток может захватить её несколько раз и должен столько же раз освободить. Преимущества: `tryLock`, lock с timeout, interruptible lock, fairness mode, несколько `Condition` для разных ожиданий. Минус — нужно обязательно делать `unlock()` в `finally`, иначе lock останется захваченным. `synchronized` проще, `ReentrantLock` гибче, но за гибкость платим шансом выстрелить себе в ногу красивее.

<a id="q8-10"></a>
### 8.10 Проблемы Executors-фабрик

Фабрики `Executors.newFixedThreadPool`, `newCachedThreadPool`, `newSingleThreadExecutor` удобны, но прячут важные настройки. `newFixedThreadPool` использует **unbounded queue**, из-за чего при наплыве задач можно получить OOM. `newCachedThreadPool` может создать слишком много потоков. `newSingleThreadExecutor` тоже имеет неограниченную очередь. Для production лучше явно создавать `ThreadPoolExecutor` с понятными параметрами: размер пула, размер очереди, rejection policy, thread factory и мониторинг.

<a id="q8-11"></a>
### 8.11 Race condition

**Race condition** — ошибка, когда результат зависит от случайного порядка выполнения потоков. Например, два потока одновременно делают `count++`: оба прочитали 5, оба записали 6, одно увеличение потерялось. Лечат через `synchronized`, locks, atomic-классы, concurrent collections, immutable objects или изменение дизайна без общего mutable-состояния. Важно: race condition может проявляться редко, поэтому “я прогнал 10 раз и норм” — не доказательство, а маленький ритуал самообмана.

<a id="q8-12"></a>
### 8.12 Fork-Join Framework

**Fork-Join** нужен для задач, которые можно рекурсивно делить на независимые подзадачи: fork — разбить и запустить, join — собрать результат. Работает через `ForkJoinPool` и work-stealing: потоки, которые освободились, забирают задачи у других. Хорошо подходит для CPU-bound divide-and-conquer задач: сортировки, агрегации, обработка больших массивов. Не стоит использовать для блокирующего IO без понимания последствий, потому что блокировки могут съесть рабочие потоки пула.

<a id="q8-13"></a>
### 8.13 ThreadLocal: использование и проблемы

`ThreadLocal` хранит значение **отдельно для каждого потока**. Это удобно для context data: traceId, security context, transaction context. Проблема начинается с thread pools: поток переиспользуется, и старое значение может случайно попасть в новую задачу, если не сделать `remove()`. Ещё ThreadLocal может держать ссылки и вызывать memory leak, особенно в контейнерах и долгоживущих пулах. Правило: установил ThreadLocal — очисти в `finally`.

<a id="q8-14"></a>
### 8.14 Видит ли дочерний поток ThreadLocal родителя

Обычный `ThreadLocal` **не наследуется** дочерним потоком. Для наследования есть `InheritableThreadLocal`: при создании нового потока он копирует значение от родителя. Но с thread pools это часто бесполезно или опасно, потому что потоки создаются заранее и переиспользуются. Для передачи контекста в асинхронном коде лучше использовать явную передачу, wrappers/decorators задач, специальные context propagation механизмы или возможности фреймворка.

<a id="q8-15"></a>
### 8.15 🔥 Типы экзекьюторов и их проблемы

Основные варианты: fixed pool, cached pool, single-thread executor, scheduled executor, work-stealing/ForkJoinPool. Проблема не в самих типах, а в дефолтах: неограниченные очереди, неограниченное число потоков, отсутствие rejection policy и мониторинга. Для CPU-bound задач размер часто близок к числу ядер. Для IO-bound может быть больше, но нужен контроль очереди и таймауты. В production лучше явно настраивать `ThreadPoolExecutor`, а не доверять фабрике, которая “ну вроде удобно”.

<a id="q8-16"></a>
### 8.16 submit() vs execute()

`execute()` принимает `Runnable` и просто запускает задачу; если внутри будет unchecked exception, она попадёт в uncaught exception handler потока. `submit()` возвращает **`Future`** и прячет исключение внутрь него: ошибка всплывёт только при `future.get()`. Поэтому с `submit()` легко потерять исключения, если Future никто не читает. `submit()` нужен, когда нужен результат, отмена или контроль выполнения. Если задача fire-and-forget, `execute()` часто честнее.

<a id="q8-17"></a>
### 8.17 Безопасная публикация объектов

**Safe publication** означает, что другой поток увидит объект полностью сконструированным, а не в полусобранном состоянии. Способы: записать ссылку в `volatile` поле, опубликовать через `final` поле корректно созданного объекта, использовать `synchronized`, concurrent collection, static initialization. Особенно важны `final` поля: Java Memory Model даёт им специальные гарантии после завершения конструктора. Нельзя отдавать `this` из конструктора наружу, потому что другой поток может увидеть объект до окончания инициализации.

<a id="q8-18"></a>
### 8.18 RecursiveTask и RecursiveAction

`RecursiveTask<V>` и `RecursiveAction` — базовые классы для Fork-Join. **RecursiveTask** возвращает результат, **RecursiveAction** ничего не возвращает. Внутри реализуется `compute()`: если задача маленькая — считаем напрямую, если большая — делим на подзадачи, запускаем `fork`, потом собираем через `join`. Важно выбрать нормальный threshold: слишком мелкие задачи дадут overhead, слишком крупные — плохой параллелизм.

<a id="q8-19"></a>
### 8.19 Атомарность long и double

По современному Java Memory Model чтение и запись `long`/`double`, объявленных как **volatile**, атомарны и дают видимость. Для обычных non-volatile `long/double` спецификация исторически допускала неатомарное чтение/запись на 32-битных JVM, хотя на современных JVM это редко встречается. Но даже если запись атомарна, операция вроде `value++` всё равно не атомарна. Для счётчиков и конкурентных обновлений нужны `AtomicLong`, `LongAdder`, lock или другая синхронизация.

<a id="q8-20"></a>
### 8.20 Phaser vs CyclicBarrier

`CyclicBarrier` ждёт фиксированное число потоков: все дошли до барьера — все пошли дальше, и барьер можно использовать снова. `Phaser` гибче: участники могут **регистрироваться и сниматься** динамически, есть фазы, можно строить более сложную координацию. `CyclicBarrier` хорош для простых сценариев “N потоков на каждом этапе”. `Phaser` — когда число участников меняется или этапов больше и они сложнее.

<a id="q8-21"></a>
### 8.21 🟣 Блокировки и lock-free алгоритмы

Блокировки дают взаимное исключение: один поток держит lock, другие ждут. Это просто понимать, но возможны deadlock, contention, priority inversion. **Lock-free** алгоритмы используют атомарные операции вроде CAS: поток пытается обновить значение, и если кто-то изменил его раньше, повторяет попытку. Плюс — нет классического ожидания lock'а. Минус — сложность, ABA problem, spinning под нагрузкой. В Java lock-free подход виден в atomic-классах и некоторых concurrent structures.

<a id="q8-22"></a>
### 8.22 Атомарные операции, классы, AtomicInteger

`AtomicInteger`, `AtomicLong`, `AtomicReference` дают атомарные операции без явного lock'а: `incrementAndGet`, `compareAndSet`, `updateAndGet`. Внутри обычно используется **CAS**: сравнить текущее значение с ожидаемым и заменить, если оно не изменилось. Для высококонкурентных счётчиков лучше `LongAdder`, потому что он снижает contention за счёт разнесения обновлений. Atomic-классы хороши для простого состояния, но если нужно атомарно менять несколько полей сразу, лучше lock или другой дизайн.

<a id="q8-23"></a>
### 8.23 Мониторинг JVM

Мониторинг JVM включает heap/non-heap memory, GC pauses, thread count, CPU, class loading, metaspace, direct memory, connection pools и latency приложения. Инструменты: Actuator + Micrometer, Prometheus/Grafana, JMX, Java Flight Recorder, VisualVM, jcmd/jstat/jstack. Важно смотреть не один график, а связь: рост latency может быть из-за GC, пула БД, блокировок, CPU или внешнего сервиса. JVM обычно оставляет следы, просто люди героически их игнорируют до падения.

<a id="q8-24"></a>
### 8.24 Профилирование

Профилирование отвечает, где приложение тратит CPU, память или время ожидания. CPU profiling показывает горячие методы. Allocation profiling показывает, где создаётся много объектов и давится GC. Thread profiling показывает блокировки и ожидания. Инструменты: Java Flight Recorder, async-profiler, YourKit, VisualVM, IntelliJ profiler. Правильный подход: сначала воспроизвести проблему, снять профиль под похожей нагрузкой, потом оптимизировать конкретное место, а не “улучшать код” по вдохновению.

<a id="q8-25"></a>
### 8.25 GC — все сборщики

GC освобождает heap от объектов, на которые больше нет достижимых ссылок. В HotSpot есть несколько сборщиков. **Serial GC** простой, для маленьких приложений. **Parallel GC** делает throughput, но с паузами. **G1** — дефолтный универсальный сборщик для серверных приложений, делит heap на регионы и старается держать паузы под контролем. **ZGC** и **Shenandoah** — low-latency сборщики с очень короткими паузами, полезны для больших heap и чувствительных к latency сервисов. Выбор GC зависит от цели: throughput, latency, размер heap, предсказуемость пауз.

<a id="section-9"></a>
## Секция 9. Тестирование

<a id="q9-1"></a>
### 9.1 🟣 Unit-тестирование и принципы

**Unit-тест** проверяет маленькую единицу логики изолированно: метод, класс, сервис без реальной БД, сети и брокера. Хороший unit-тест быстрый, детерминированный, читаемый и проверяет поведение, а не внутреннюю реализацию без нужды. Обычно используют паттерн Arrange-Act-Assert. Моки нужны для внешних зависимостей, но если тест превращается в театр из 20 моков, это часто сигнал, что класс делает слишком много.

<a id="q9-2"></a>
### 9.2 🟣 Unit vs Integration: когда что

**Unit** проверяет логику быстро и изолированно. Он хорош для правил, расчётов, ветвлений, ошибок. **Integration** проверяет, что несколько частей реально работают вместе: Spring context, БД, Kafka, REST-клиент, сериализация. Unit-тесты дают быструю обратную связь, integration-тесты ловят проблемы конфигурации и взаимодействия. Нормальная стратегия — много unit-тестов на бизнес-логику и меньше, но качественных integration-тестов на границы системы.

<a id="q9-3"></a>
### 9.3 JUnit 5: как работает, нововведения

JUnit 5 состоит из Platform, Jupiter и Vintage. **Jupiter** — новый API для тестов: `@Test`, `@BeforeEach`, `@AfterEach`, `@Nested`, `@ParameterizedTest`, `@DisplayName`, extensions. Вместо старых rules/runner используется extension model. Параметризованные тесты позволяют прогнать один тест на разных входных данных. `@Nested` помогает группировать сценарии. JUnit 5 лучше интегрируется с современными инструментами и читается приятнее, что уже почти подвиг для тестового кода.

<a id="q9-4"></a>
### 9.4 🔥 Testcontainers

**Testcontainers** поднимает реальные сервисы в Docker для тестов: PostgreSQL, Redis, Kafka, RabbitMQ и так далее. Это лучше H2, когда важны реальные SQL-диалекты, индексы, транзакции, jsonb, Kafka semantics. В Spring Boot можно подключить container как datasource или service dependency. Минус — тесты медленнее и требуют Docker. Плюс — они ловят настоящие интеграционные проблемы, а не проверяют фантазию H2 о том, как якобы работает PostgreSQL.

<a id="q9-5"></a>
### 9.5 Мокирование: Mockito, PowerMock

Mockito позволяет заменить зависимость тестовым объектом: задать поведение через `when(...).thenReturn(...)` и проверить взаимодействие через `verify(...)`. Это удобно для unit-тестов сервисов без реальной БД/HTTP. PowerMock раньше использовали для static/private/final сложностей, но в современном коде лучше не доводить дизайн до необходимости PowerMock. Если приходится мокать static-ад, конструкторы и приватные кишки, возможно, проблема не в тесте, а в дизайне кода.

<a id="q9-6"></a>
### 9.6 Stub vs mock

**Stub** просто отдаёт заранее заданные данные, чтобы тестируемый код мог выполниться. **Mock** ещё проверяет взаимодействие: был ли вызван метод, с какими аргументами, сколько раз. Если нам важно только получить ответ от зависимости — stub. Если важно проверить, что сервис отправил уведомление или вызвал gateway — mock. Не стоит проверять каждую внутреннюю мелочь через verify, иначе тесты станут хрупкими и будут падать от любого рефакторинга без изменения поведения.

<a id="q9-7"></a>
### 9.7 @MockBean vs @Mock

`@Mock` — Mockito-мок в обычном unit-тесте, Spring context не нужен. `@MockBean` создаёт мок и кладёт его в **Spring ApplicationContext**, заменяя реальный бин. Его используют в Spring-тестах, например `@WebMvcTest`, когда нужно подменить сервис в контексте. `@MockBean` тяжелее, потому что связан с Spring context. Если можно протестировать класс без Spring — лучше `@Mock` + `@InjectMocks` или ручное создание объекта.

<a id="q9-8"></a>
### 9.8 Фикстуры (test fixtures)

Фикстуры — подготовленные данные и объекты для тестов: user, order, DTO, entity, состояние БД. Хорошие фикстуры делают тесты короче и понятнее. Плохие фикстуры превращаются в общий огромный “test data god object”, где изменение одного поля ломает половину проекта. Удобные подходы: builders, object mother, factory methods, SQL/dataset для интеграционных тестов. Главное — чтобы из теста было видно, какие данные важны для сценария.

<a id="q9-9"></a>
### 9.9 E2E vs Unit vs Integration

**Unit** проверяет маленькую логику изолированно. **Integration** проверяет взаимодействие компонентов: БД, Spring, брокер, внешние клиенты. **E2E** проверяет полный пользовательский или бизнес-сценарий через всю систему. Чем выше уровень, тем тест дороже, медленнее и нестабильнее, но тем больше уверенности в реальном сценарии. Поэтому пирамида тестирования: много unit, меньше integration, совсем немного E2E на критические пути.

<a id="q9-10"></a>
### 9.10 Boundary testing

Boundary testing проверяет **границы допустимых значений**: минимум, максимум, прямо перед границей и сразу после неё. Если поле принимает 1–100, тестируем 0, 1, 100, 101. Именно на границах часто живут баги: off-by-one, неправильные `>=` вместо `>`, пустые коллекции, null, максимальная длина строки. Это простой способ получить много пользы без героического написания тысячи случайных тестов.

<a id="q9-11"></a>
### 9.11 JUnit vs TestNG vs Spock

**JUnit** — стандарт де-факто в Java, особенно JUnit 5. **TestNG** исторически давал сильные возможности параметризации, групп и зависимостей между тестами, но сейчас используется реже. **Spock** — framework на Groovy с очень читаемым given/when/then синтаксисом и мощной параметризацией. Для обычного Java/Spring проекта JUnit 5 + Mockito + AssertJ чаще всего достаточно. Spock хорош, если команда готова к Groovy в тестах.

<a id="q9-12"></a>
### 9.12 Тестирование асинхронного кода

Асинхронный код нельзя нормально тестировать через `Thread.sleep(5000)`, потому что это медленно и нестабильно. Лучше использовать Awaitility: ждать условие до timeout. Для `CompletableFuture` можно проверять результат через `get` с timeout или подставлять controlled executor. Для Kafka/RabbitMQ в интеграционных тестах проверяют, что сообщение дошло и обработалось, часто через Testcontainers. Главное — тест должен ждать **событие/условие**, а не просто “поспим и надеемся”.

<a id="q9-13"></a>
### 9.13 Тестирование взаимодействия с внешними сервисами (REST, Kafka)

REST-интеграции удобно тестировать через **WireMock**: поднимаем fake HTTP server, задаём ожидаемые запросы и ответы. Kafka/RabbitMQ лучше проверять через Testcontainers или embedded-решения, если они достаточно близки к реальности. Для контрактов между сервисами используют Pact или Spring Cloud Contract. Важно тестировать не только happy path, но и timeout, 500, malformed response, retry, idempotency. Внешний сервис всегда когда-нибудь ответит так, будто его писал враг.

<a id="q9-14"></a>
### 9.14 🔵 TDD и BDD

**TDD** — подход “сначала тест, потом код”: red → green → refactor. Он помогает лучше продумать API и писать только нужную логику. **BDD** фокусируется на поведении системы с точки зрения бизнеса: given/when/then, сценарии, понятные не только разработчикам. TDD чаще про developer-level дизайн, BDD — про общую формулировку поведения. На практике можно не быть религиозным фанатиком, но полезно писать тесты так, чтобы они описывали поведение, а не случайную реализацию.

<a id="q9-15"></a>
### 9.15 H2 / встроенная БД в Spring Boot

H2 удобна для быстрых тестов: не нужен Docker, стартует быстро, легко сбрасывать состояние. Но H2 не равна PostgreSQL: другой SQL-диалект, другие типы, индексы, транзакционные нюансы, jsonb и locking. Поэтому H2 подходит для простых repository-тестов или учебных случаев, но для серьёзной интеграции с PostgreSQL лучше **Testcontainers**. Иначе тесты зелёные, а прод красный, классика жанра.

<a id="q9-16"></a>
### 9.16 Test data

Тестовые данные должны быть понятными, минимальными и стабильными. Если тест проверяет скидку, в данных должны выделяться именно поля, влияющие на скидку, а не гигантский объект заказа на 80 строк. Для unit-тестов удобны builders и factory methods. Для интеграционных — SQL scripts, migration-based setup, container clean state. Важно изолировать тесты друг от друга: один тест не должен зависеть от данных, оставленных предыдущим.

<a id="q9-17"></a>
### 9.17 In-memory БД vs Testcontainers

In-memory БД быстрая и простая, но часто отличается от настоящей БД. Testcontainers медленнее, зато поднимает реальный PostgreSQL/MySQL/Kafka и ловит настоящие проблемы диалекта, миграций, индексов, типов и транзакций. Если тест проверяет чистую бизнес-логику — БД вообще не нужна. Если тест проверяет SQL/JPA/миграции — лучше Testcontainers. H2 в роли PostgreSQL — это как резиновая гантеля: форму напоминает, нагрузка другая.

<a id="q9-18"></a>
### 9.18 Contract testing

**Contract testing** проверяет, что provider и consumer одинаково понимают API. Consumer описывает ожидания: endpoint, request, response, поля, статусы. Provider потом проверяет, что реально соответствует этому контракту. Это помогает ловить breaking changes до деплоя. Инструменты: Pact, Spring Cloud Contract. Особенно полезно в микросервисах, где команды независимы, а “мы просто переименовали поле” внезапно ломает чужой production.

<a id="q9-19"></a>
### 9.19 Изоляция тестов: @DirtiesContext, @Transactional

Тесты должны быть независимыми. В Spring `@Transactional` на тесте часто откатывает изменения после выполнения, что удобно для БД. `@DirtiesContext` говорит Spring, что context испачкан и его надо пересоздать, но это дорого, поэтому использовать только когда реально изменили global state/context. Для изоляции также применяют очистку таблиц, отдельные schemas, Testcontainers lifecycle, уникальные test data. Плохой признак — тесты проходят отдельно, но падают пачкой.

<a id="q9-20"></a>
### 9.20 🟣 Покрытие кода: Jacoco

Jacoco измеряет, какие строки и ветки кода были выполнены тестами. Это полезная метрика, но не доказательство качества. 90% coverage может ничего не значить, если тесты не проверяют assertions, а просто вызывают методы ради зелёного отчёта. Лучше смотреть **branch coverage**, критические участки бизнес-логики и mutation testing, если хочется глубже. Покрытие — индикатор, а не цель. Иначе команда начинает тестировать геттеры, а баги всё равно идут в прод.

<a id="section-10"></a>
## Секция 10. Прочее: CI/CD, безопасность, UNIX и т.д.

<a id="q10-1"></a>
### 10.1 CI/CD и инструменты

**CI** — автоматическая проверка изменений: build, tests, static analysis, security checks, Docker image. **CD** — автоматическая доставка дальше по окружениям: dev, staging, prod. Инструменты: GitLab CI, GitHub Actions, Jenkins, TeamCity, Argo CD, Helm, Docker registry. Хороший pipeline быстро ловит ошибки и делает релиз повторяемым. Плохой pipeline — это когда “на моей машине работает”, а production собирается вручную в полночь по древнему ритуалу.

<a id="q10-2"></a>
### 10.2 Доставка кода от разработчика до продакшена (свой проект)

Типичный путь: разработчик создаёт ветку, пишет код и тесты, открывает merge request, проходит review и CI. CI собирает проект, запускает unit/integration tests, проверяет стиль и, если всё хорошо, собирает Docker image. Потом image попадает в registry, деплой идёт на dev/stage/prod через pipeline, Helm/Kubernetes или другой инструмент. Перед prod обычно нужны миграции БД, smoke tests, monitoring и возможность rollback. Главная идея — один и тот же артефакт проходит окружения, а не пересобирается вручную.

<a id="q10-3"></a>
### 10.3 Maven/Gradle: механизм сборки

Maven и Gradle управляют зависимостями, сборкой, тестами и packaging. **Maven** более декларативный и стандартный: lifecycle phases `compile`, `test`, `package`, `verify`, `install`. **Gradle** гибче, быстрее на больших проектах за счёт incremental build/cache, но build scripts могут стать творческой помойкой. Оба тянут зависимости из repositories, решают транзитивные зависимости и запускают plugins/tasks. Для Java backend важно понимать dependency scopes/configurations и как смотреть dependency tree.

<a id="q10-4"></a>
### 10.4 Git из командной строки: частые команды

Базовые команды: `git status`, `git add`, `git commit`, `git push`, `git pull`, `git fetch`, `git checkout`/`switch`, `git branch`, `git merge`, `git rebase`, `git log`, `git diff`, `git stash`. Для поиска проблем полезны `git blame` и `git bisect`. Важно понимать разницу между working tree, staging area и commit history. Git — не просто кнопка в IDE, хотя IDE делает вид, что история коммитов сама себя спасёт.

<a id="q10-5"></a>
### 10.5 Squash и rebase

**Rebase** переносит коммиты поверх другой базы, делая историю линейнее. Его часто используют перед merge, чтобы подтянуть актуальный master/main. **Squash** объединяет несколько коммитов в один, чтобы в main не попадала история вида `fix`, `fix2`, `please work`. Rebase переписывает историю, поэтому нельзя бездумно делать его на общих ветках, которыми уже пользуются другие. В feature-ветке — нормально, в shared branch — осторожно.

<a id="q10-6"></a>
### 10.6 git branch -f master HEAD~1 и git reset --soft HEAD~1

`git branch -f master HEAD~1` принудительно передвинет указатель ветки `master` на предыдущий коммит, но не обязательно переключит тебя на неё. Это операция над ссылкой ветки. `git reset --soft HEAD~1` двигает текущую ветку на предыдущий коммит, но оставляет изменения из убранного коммита в staging area. То есть soft reset удобен, когда нужно “разобрать” последний коммит и закоммитить иначе. Обе команды переписывают историю, так что применять с пониманием, а не как заклинание из StackOverflow.

<a id="q10-7"></a>
### 10.7 LF vs CRLF

**LF** (`\n`) — перевод строки в Unix/Linux/macOS. **CRLF** (`\r\n`) — стиль Windows. Разница может ломать shell-скрипты, Docker entrypoint, форматирование и диффы. В Git это контролируют через `core.autocrlf` и `.gitattributes`. Для backend-проектов обычно фиксируют правила в репозитории: например, исходники с LF. Иначе половина diff'а внезапно состоит из “изменений”, которые никто не делал, потому что переносы строк решили поиграть в диверсантов.

<a id="q10-8"></a>
### 10.8 Безопасная остановка Spring Boot

Graceful shutdown означает, что приложение перестаёт принимать новые запросы, но даёт текущим завершиться. В Spring Boot это настраивается через `server.shutdown=graceful` и timeout lifecycle phase. В Kubernetes важно сочетать это с readiness probe, preStop hook и достаточным `terminationGracePeriodSeconds`. Также нужно корректно закрывать connection pools, consumers, executors. Иначе pod умер, запросы оборвались, Kafka offsets не зафиксировались, а потом все делают вид, что “так и задумано”.

<a id="q10-9"></a>
### 10.9 TCP vs UDP

**TCP** — соединение, порядок, доставка, retransmission, контроль потока. Подходит для HTTP/1.1, HTTP/2, БД, большинства надёжных протоколов. **UDP** — без соединения и гарантий доставки/порядка, зато меньше overhead и latency. Используется для DNS, стриминга, игр, VoIP, а HTTP/3 построен поверх QUIC, который использует UDP и добавляет свои гарантии. Коротко: TCP — надёжный поток байтов, UDP — быстрые датаграммы без встроенной заботы о твоих чувствах.

<a id="q10-10"></a>
### 10.10 🟣 Docker-контейнер vs виртуальная машина

**Виртуальная машина** виртуализирует целую ОС с собственным ядром, поэтому тяжелее и стартует дольше. **Контейнер** использует ядро хостовой ОС и изолирует процесс через namespaces/cgroups, поэтому легче и быстрее. Контейнер — не маленькая VM, а изолированный процесс с файловой системой, сетью и лимитами. VM лучше для сильной изоляции разных ОС, контейнеры — для упаковки и деплоя приложений. В проде часто контейнеры запускаются внутри VM или managed nodes.

<a id="q10-11"></a>
### 10.11 Частые Docker-команды

Частые команды: `docker build`, `docker run`, `docker ps`, `docker logs`, `docker exec -it`, `docker stop`, `docker rm`, `docker images`, `docker rmi`, `docker pull`, `docker push`, `docker compose up/down`, `docker inspect`, `docker network ls`, `docker volume ls`. Для диагностики обычно нужны `logs`, `exec`, `inspect`, `ps`. Главное понимать разницу между image и container: image — шаблон, container — запущенный экземпляр.

<a id="q10-12"></a>
### 10.12 🟣 Dockerfile и Docker Compose

**Dockerfile** описывает, как собрать image: base image, копирование файлов, установка зависимостей, build, entrypoint. Для Java часто используют multi-stage build: сначала Maven/Gradle собирает jar, потом runtime image запускает его. **Docker Compose** описывает несколько контейнеров для локального окружения: приложение, PostgreSQL, Redis, Kafka. Compose удобен для dev/test, но не заменяет Kubernetes в production. В Dockerfile важно не тащить лишнее, не хранить секреты и использовать понятный entrypoint.

<a id="q10-13"></a>
### 10.13 Kubernetes Pod

**Pod** — минимальная единица запуска в Kubernetes. Внутри pod может быть один или несколько контейнеров, которые делят network namespace и могут общаться через localhost. Обычно один pod — один основной контейнер приложения плюс иногда sidecar. Pod сам по себе недолговечен: его могут пересоздать, перенести, удалить. Поэтому состояние выносят наружу, а управление количеством pod'ов делают через Deployment/ReplicaSet/StatefulSet.

<a id="q10-14"></a>
### 10.14 🟣 ReplicaSet

**ReplicaSet** следит, чтобы было запущено нужное число одинаковых pod'ов. Если pod умер, ReplicaSet создаст новый. В реальной работе ReplicaSet обычно не создают руками: его создаёт **Deployment**, который дополнительно умеет rolling update и rollback. ReplicaSet выбирает pod'ы по labels/selectors. Важно, чтобы selector не пересекался случайно с чужими pod'ами, иначе Kubernetes начнёт управлять не тем, чем нужно. Автоматизация, как обычно, прекрасна до момента, пока автоматизирует катастрофу.

<a id="q10-15"></a>
### 10.15 Ingress

**Ingress** управляет внешним HTTP/HTTPS-доступом к сервисам в Kubernetes. Он описывает правила: какой host/path направить в какой Service. Сам по себе Ingress — только ресурс; нужен Ingress Controller, например nginx-ingress, Traefik, HAProxy. Через Ingress часто настраивают TLS, path-based routing, host-based routing, redirects, иногда rate limits. Он стоит выше Service: Service даёт стабильный доступ внутри кластера, Ingress пускает трафик снаружи.

<a id="q10-16"></a>
### 10.16 Свободное место на диске / размер директории в Unix

Свободное место смотрят через `df -h`: показывает файловые системы и сколько места занято/свободно. Размер директории — `du -sh /path`, а детализация по подпапкам — `du -h --max-depth=1 /path`. Чтобы найти крупные файлы, используют `find /path -type f -size +100M` или сортировку `du`. Важно отличать место на диске от inode'ов: иногда место есть, а новые файлы не создаются, потому что закончились inode.

<a id="q10-17"></a>
### 10.17 Список работающих процессов

Процессы смотрят через `ps aux`, `top`, `htop`, `pgrep`, `pidof`. `ps aux | grep java` — классика поиска Java-процесса, хотя `pgrep -af java` обычно чище. Для дерева процессов есть `pstree`. В контейнерах полезно помнить, что внутри может быть свой PID namespace, и процесс с PID 1 — это приложение или entrypoint. Для диагностики важны PID, CPU, memory, command, user и время работы.

<a id="q10-18"></a>
### 10.18 Thread dump работающего Java-процесса

Thread dump можно снять через `jstack <pid>`, `jcmd <pid> Thread.print`, `kill -3 <pid>` в Linux или через Actuator `/threaddump`, если он включён. Dump показывает потоки, их состояния, stack trace, locks, deadlocks. Он полезен при зависаниях, deadlock, высокой latency, подозрении на исчерпание пула потоков. Читать нужно не одну строку, а группы потоков: кто RUNNABLE, кто BLOCKED, кто WAITING и на чём именно.

<a id="q10-19"></a>
### 10.19 grep, tail, less, more

`grep` ищет строки по шаблону: `grep -i error app.log`, `grep -R "text" .`. `tail -f` показывает конец файла в реальном времени, удобно для логов. `less` открывает файл с навигацией, поиском `/pattern`, прокруткой и без загрузки всего файла в редактор. `more` проще и старее, обычно less удобнее. Частая связка: `tail -f app.log | grep ERROR`, потому что смотреть логи без фильтра — это как читать роман из stack trace.

<a id="q10-20"></a>
### 10.20 IDOR (Insecure Direct Object References)

**IDOR** — уязвимость, когда пользователь может получить доступ к чужому объекту, просто поменяв id в URL или запросе. Например, `/users/123/documents/5`, и сервер проверил только, что пользователь залогинен, но не проверил, что документ принадлежит ему. Защита: проверка прав на каждый ресурс, ownership checks, ACL/roles/scopes, не доверять id от клиента. UUID вместо numeric id может усложнить перебор, но не заменяет авторизацию.

<a id="q10-21"></a>
### 10.21 🔵 SQL-инъекция

**SQL-инъекция** возникает, когда пользовательский ввод склеивают со строкой SQL. Атакующий может изменить смысл запроса: обойти логин, прочитать чужие данные, удалить таблицы, устроить маленький праздник безумия. Защита: **prepared statements**, bind parameters, JPA parameters, whitelist для сортировки/имен колонок, минимальные права пользователя БД. ORM не спасает автоматически, если писать native query через конкатенацию строк.

<a id="q10-22"></a>
### 10.22 CSRF

**CSRF** — атака, где браузер жертвы автоматически отправляет запрос с её cookie на доверенный сайт. Например, пользователь залогинен, а вредная страница заставляет браузер отправить POST. Защита: CSRF token, SameSite cookie, проверка Origin/Referer, не использовать cookie-auth без защиты. Для Bearer token в Authorization header CSRF обычно менее актуален, потому что браузер сам не прикрепляет такой header к чужому запросу.

<a id="q10-23"></a>
### 10.23 Reflected XSS

**Reflected XSS** — когда вредный скрипт приходит в запросе и сразу отражается в HTML-ответе без экранирования. Например, параметр `?q=<script>...` попадает на страницу. Защита: output encoding по контексту, шаблонизаторы с автоэкранированием, Content Security Policy, валидация ввода, запрет inline scripts по возможности. Backend должен не отдавать опасный HTML из пользовательских данных, frontend — не делать `innerHTML` из всего подряд.

<a id="q10-24"></a>
### 10.24 Clickjacking

**Clickjacking** — сайт встраивают в iframe и накладывают обманный UI, чтобы пользователь кликнул не туда, куда думает. Защита: заголовок **`X-Frame-Options: DENY/SAMEORIGIN`** или современный CSP `frame-ancestors`. Для админок, платёжных страниц и личных кабинетов это особенно важно. Смысл простой: запретить чужим сайтам встраивать твою страницу в iframe, если это не предусмотрено явно.

<a id="q10-25"></a>
### 10.25 Угроза токена сессии в URL

Токен в URL — плохая идея, потому что URL попадает в browser history, server logs, proxy logs, analytics, referer headers и скриншоты. Такой токен легко утечь случайно. Сессионные токены нужно передавать в **HttpOnly Secure cookie** или Authorization header, а не в query parameter. Если токен всё же оказался в URL, нужен короткий TTL, одноразовость и немедленная ротация после использования.

<a id="q10-26"></a>
### 10.26 Защита сессионных токенов, распределённые системы

Токены защищают через HTTPS, короткий TTL, refresh rotation, HttpOnly/Secure/SameSite cookies, хранение секретов вне кода, ротацию ключей подписи и отзыв скомпрометированных токенов. В распределённой системе проблема — logout/revoke: JWT stateless, и сервер не обязан ходить в БД на каждый запрос. Решения: короткий access TTL, blacklist/revocation list в Redis, version/session id в токене, introspection для opaque tokens. Баланс всегда между производительностью и контролем.

<a id="q10-27"></a>
### 10.27 Защита от DDoS

DDoS полностью не “чинится” кодом приложения. Защита строится слоями: CDN/WAF, rate limiting, load balancer, autoscaling, circuit breakers, очереди, кэширование, ограничения размера request body, timeout'ы, защита дорогих endpoint'ов. На уровне приложения важно не позволять одному запросу съедать слишком много CPU/памяти/БД. Для API полезны quotas по user/api key/IP и деградация функциональности вместо полного падения.

<a id="q10-28"></a>
### 10.28 🔵 IDE и любимые сочетания клавиш

Нормальный ответ можно дать по IntelliJ IDEA: поиск везде `Shift Shift`, перейти к классу `Cmd/Ctrl+N`, к файлу `Cmd/Ctrl+Shift+N`, usages `Alt/F7`, refactor rename `Shift+F6`, extract variable/method, generate `Cmd/Ctrl+N` или `Alt+Insert`, форматирование, optimize imports, run/debug, breakpoints. Важно не просто перечислить кнопки, а сказать, что IDE используешь для навигации, рефакторинга, отладки, анализа usages и быстрого чтения чужого кода.

<a id="q10-29"></a>
### 10.29 Самая большая/запоминающаяся задача

Хороший ответ строится как короткая история: **контекст → проблема → действия → результат**. Например: “На проекте с аукционной/e-commerce платформой была проблема производительности: endpoint или обработка грузили слишком много данных/делали N+1/лишние запросы. Я посмотрел SQL и метрики, нашёл узкое место, изменил fetch strategy/запросы/индексы, добавил тесты и проверил результат. В итоге время ответа снизилось, количество запросов уменьшилось, а логика осталась понятной”. Важно говорить не “я делал таску”, а какой был impact и какие решения ты принял.
