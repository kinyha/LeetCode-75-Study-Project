# ТИ-1 IBS — банк вопросов для интервью

> Формат: вопрос можно раскрывать как карточку. Внутри каждого ответа есть строка **«Вспомнить за 5 секунд»**, чтобы быстро освежить суть перед повторением, а не читать простыню, потому что мозг не обязан страдать за чужой Excel.

## Как пользоваться

1. Сначала пройди вопросы с метками **🔥 / 🟣 / 🔵**.
2. Потом повторяй по секциям: открыл вопрос → попытался ответить вслух → раскрыл карточку → сверил структуру.
3. Для ответа на интервью держи формулу: **суть → под капотом → пример из проекта → масштаб / edge cases**.

## Легенда

- **🔥** — спрашивали обоих кандидатов, максимальная вероятность повтора.
- **🟣** — спрашивали Мишу.
- **🔵** — спрашивали Вадима.

## Формула ответа на 4 балла

**Суть:** сначала дай короткое определение, без захода через лес терминов.

**Под капотом:** объясни механизм: структура данных, прокси, транзакция, offset, GC, индекс, брокер, поток выполнения.

**Практика:** привяжи к своему опыту: AuctionPort, RabbitMQ, PostgreSQL, Liquibase, Kubernetes, k6, поиск N+1, HikariCP и так далее.

**Масштаб:** добавь, что меняется под нагрузкой: latency, contention, retries, идемпотентность, кластеры, мониторинг, деградация.

<details>
<summary><strong>2.5 🟣 Почему HashMap неэффективна при плохой хэш-функции: метод / сервис / приложение + оптимизации</strong></summary>

> **Вспомнить за 5 секунд:** Метод: плохой hash → все ключи в один бакет → связный список → поиск O(n) вместо O(1); с Java 8 бакет ≥8 элементов при capacity ≥64 превращается в красно-чёрное дерево → O(log n).

**Ключевые слова:** **Метод**, **Сервис**, **Приложение**, **Оптимизации**, **hashCode**, **Objects.hash**

**Ответ:**

- **Метод:** плохой hash → все ключи в один бакет → связный список → поиск O(n) вместо O(1); с Java 8 бакет ≥8 элементов при capacity ≥64 превращается в красно-чёрное дерево → O(log n).
- **Сервис:** деградация горячих операций (кэши, индексы в памяти), лишние аллокации и нагрузка на GC при resize.
- **Приложение:** в высоконагруженной системе — рост латентности, при HashMap-подобном шардировании — перекос нагрузки на узлы.
- **Оптимизации:** качественный `hashCode` (`Objects.hash`/все значимые поля), задать `initialCapacity`, если размер известен (меньше resize), для конкурентности — `ConcurrentHashMap`, для кэшей — Caffeine с нормальной политикой.

</details>

<a id="q-2-6"></a>
<details>
<summary><strong>2.6 Fail-fast vs fail-safe итераторы</strong></summary>

> **Вспомнить за 5 секунд:** Fail-fast (ArrayList, HashMap): проверяют modCount, при структурной модификации во время итерации бросают ConcurrentModificationException (best-effort, не гарантия). Удаление при итерации — только iterator.remove()/removeIf.

**Ключевые слова:** **Fail-fast**, **Fail-safe**, **ArrayList**, **HashMap**, **modCount**, **ConcurrentModificationException**

**Ответ:**

- **Fail-fast** (`ArrayList`, `HashMap`): проверяют `modCount`, при структурной модификации во время итерации бросают `ConcurrentModificationException` (best-effort, не гарантия). Удаление при итерации — только `iterator.remove()`/`removeIf`.
- **Fail-safe:** `CopyOnWriteArrayList` — итерация по снапшоту массива; `ConcurrentHashMap` — weakly consistent iterator (не бросает, может видеть/не видеть конкурентные изменения).

</details>

<details>
<summary><strong>2.7 Абстрактный класс vs интерфейс, когда что</strong></summary>

> **Вспомнить за 5 секунд:** Интерфейс — контракт: множественная реализация, default/static методы, состояния нет (только static final).

**Ключевые слова:** **Интерфейс**, **Абстрактный класс**, **static final**, **protected**

**Ответ:**

- **Интерфейс** — контракт: множественная реализация, default/static методы, состояния нет (только `static final`).
- **Абстрактный класс** — общая реализация + состояние: поля, конструкторы, `protected`.
- Интерфейс — «что умеет» (может реализовать кто угодно), абстрактный класс — «является разновидностью» с общим кодом. Со времён default-методов граница тоньше; состояние — главный водораздел.

</details>

<details>
<summary><strong>2.8 Как работает «перегрузка» статического метода</strong></summary>

> **Вспомнить за 5 секунд:** Перегрузка (overload) static-методов работает как обычная — выбор по типам параметров на этапе компиляции. Подвох вопроса — «переопределение»: static-методы не переопределяются, а скрываются (hiding): вызов резолвится по типу ссылки, а не объекта — Parent p = new Child(); p.staticMethod() вызовет метод Parent.

**Ключевые слова:** **не переопределяются**, **вызовет метод**

**Ответ:**

Перегрузка (overload) static-методов работает как обычная — выбор по типам параметров на этапе компиляции. Подвох вопроса — «переопределение»: static-методы **не переопределяются**, а скрываются (hiding): вызов резолвится по типу ссылки, а не объекта — `Parent p = new Child(); p.staticMethod()` вызовет метод `Parent`. Поэтому static-методы зовут через имя класса.

</details>

<details>
<summary><strong>2.9 Проблемы try-with-resources: метод / сервис / приложение</strong></summary>

> **Вспомнить за 5 секунд:** Метод: ресурс должен быть AutoCloseable; исключение из close() становится suppressed (не теряется, но и не главное — смотреть getSuppressed()); порядок закрытия обратный объявлению.

**Ключевые слова:** **Метод**, **Сервис**, **Приложение**, **AutoCloseable**, **close()**, **getSuppressed()**

**Ответ:**

- **Метод:** ресурс должен быть `AutoCloseable`; исключение из `close()` становится suppressed (не теряется, но и не главное — смотреть `getSuppressed()`); порядок закрытия обратный объявлению.
- **Сервис:** если не использовать — незакрытые соединения/файлы; с пулом БД — исчерпание пула и зависание сервиса.
- **Приложение:** утечка файловых дескрипторов → «too many open files». Сам по себе оверхеда почти не добавляет — проблемы от неиспользования.

<a id="q-2-10"></a>
<details>
<summary><strong>2.10 Механизм AutoCloseable, последствия для файлов и БД</strong></summary>

> **Вспомнить за 5 секунд:** Интерфейс с единственным close(); try-with-resources гарантирует вызов в finally-семантике даже при исключении. Для БД важно: Connection из пула при close() возвращается в пул, а не закрывается физически — забытый close = утечка соединения из пула (HikariCP leak detection ловит именно это — мой случай с MassIndexer).

**Ключевые слова:** **close()**, **Connection**, **close**, **Closeable**

**Ответ:**

Интерфейс с единственным `close()`; try-with-resources гарантирует вызов в finally-семантике даже при исключении. Для БД важно: `Connection` из пула при `close()` возвращается в пул, а не закрывается физически — забытый `close` = утечка соединения из пула (HikariCP leak detection ловит именно это — мой случай с MassIndexer). `Closeable` (IO) — подинтерфейс с требованием идемпотентности `close`.

</details>

<summary><strong>2.11 ClassLoader, ClassNotFoundException</strong></summary>

> **Вспомнить за 5 секунд:** Иерархия: Bootstrap (java.*) → Platform → Application (classpath); parent delegation — загрузчик сначала спрашивает родителя (защита от подмены java.lang.String).

**Ключевые слова:** **Иерархия**, **Этапы**, **Решение**, **java.***, **java.lang.String**, **ClassNotFoundException**

**Ответ:**

- **Иерархия:** Bootstrap (`java.*`) → Platform → Application (classpath); parent delegation — загрузчик сначала спрашивает родителя (защита от подмены `java.lang.String`).
- **Этапы:** loading → linking (verification, preparation, resolution) → initialization (static-блоки).
- `ClassNotFoundException` — класс не найден при явной загрузке (`Class.forName`, отсутствует зависимость в runtime-classpath); отличать от `NoClassDefFoundError` — класс был при компиляции, но исчез в рантайме или упала его статическая инициализация.
- **Решение:** проверить classpath/скоупы зависимостей, конфликты версий (`mvn dependency:tree`).

</details>

<summary><strong>2.12 Проблемы рефлексии: производительность, безопасность, защита</strong></summary>

> **Вспомнить за 5 секунд:** Производительность: обход JIT-оптимизаций (нет инлайна), проверки доступа на каждый вызов — на горячем пути в разы медленнее (фреймворки кэшируют Method/MethodHandle).

**Ответ:**

- **Защита:** модульная система JPMS (пакет закрыт без `opens`/`--add-opens`), enum-синглтон (защищён от рефлексивного создания), валидация инвариантов не только в конструкторе.

<summary><strong>2.16 🔵 Generics, примеры, PECS</strong></summary>

> **Вспомнить за 5 секунд:** Параметризация типов с проверкой на компиляции: List вместо List + касты. В рантайме — type erasure: параметр стирается до Object/bound, отсюда нельзя new T(), new T[], instanceof List.

**Ключевые слова:** **Уровни**, **PECS**, **List<String>**, **List**, **Object**, **new T()**

**Ответ:**

Параметризация типов с проверкой на компиляции: `List<String>` вместо `List` + касты. В рантайме — type erasure: параметр стирается до `Object`/bound, отсюда нельзя `new T()`, `new T[]`, `instanceof List<String>`.
- **Уровни:** обобщённый класс `class Box<T>`, обобщённый метод `<T> T first(List<T> l)`, bounded `<T extends Comparable<T>>`.
- **PECS:** Producer Extends — `List<? extends Number>` (откуда читаем, положить нельзя), Consumer Super — `List<? super Integer>` (куда пишем).
- Пример из JDK: `Collections.copy(List<? super T> dest, List<? extends T> src)`.

<details>
<summary><strong>2.19 Кастомная аннотация, retention</strong></summary>

> **Вспомнить за 5 секунд:** RetentionPolicy: SOURCE (только компилятор — @Override, Lombok), CLASS (в байткоде, невидима рефлексии — дефолт), RUNTIME (читается рефлексией — всё Spring).

**Ключевые слова:** **RetentionPolicy**, **Обработка**, **SOURCE**, **@Override**, **CLASS**, **RUNTIME**

**Ответ:**

```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
```

<summary><strong>8.1 Immutable: как сделать и зачем</strong></summary>

> **Вспомнить за 5 секунд:** final class, все поля private final, никаких сеттеров, защитные копии мутабельных полей на входе и выходе (или List.copyOf), this не утекает из конструктора. Зачем: потокобезопасность без синхронизации (иммутабельный объект можно свободно шарить — JMM гарантирует видимость final-полей после конструктора),

**Ключевые слова:** **final class**, **private final**, **List.copyOf**, **this**, **Map**, **String**

**Ответ:**

`final class`, все поля `private final`, никаких сеттеров, защитные копии мутабельных полей на входе и выходе (или `List.copyOf`), `this` не утекает из конструктора. Зачем: потокобезопасность без синхронизации (иммутабельный объект можно свободно шарить — JMM гарантирует видимость final-полей после конструктора), безопасные ключи `Map`, простота рассуждений. Примеры: `String`, `BigDecimal`, records.

<summary><strong>8.2 wait() — как работает и почему аккуратно</strong></summary>

> **Вспомнить за 5 секунд:** Вызывается только владея монитором (внутри synchronized на том же объекте, иначе IllegalMonitorStateException); отпускает монитор и паркует поток до notify/notifyAll/таймаута/interrupt. Аккуратность: проверять условие в цикле while (spurious wakeups + условие могло измениться),

**Ключевые слова:** **synchronized**, **IllegalMonitorStateException**, **notify**, **notifyAll**, **while**, **wait**

**Ответ:**

Вызывается только владея монитором (внутри `synchronized` на том же объекте, иначе `IllegalMonitorStateException`); отпускает монитор и паркует поток до `notify`/`notifyAll`/таймаута/interrupt. Аккуратность: проверять условие в цикле `while` (spurious wakeups + условие могло измениться), `notify` может потеряться (если `wait` ещё не вызван) — поэтому состояние + проверка, предпочитать `notifyAll`. Современный код — `BlockingQueue`/`Condition`/`CountDownLatch` вместо ручного wait/notify.

</details>

<summary><strong>8.4 volatile</strong></summary>

> **Вспомнить за 5 секунд:** Гарантия видимости: запись видна всем последующим чтениям (значение не кэшируется в регистрах/локально), запрет переупорядочивания вокруг (happens-before: запись → чтение). Не атомарность: volatile int counter++ всё равно гонка (read-modify-write). Применение: флаги остановки, публикация ссылки на иммутабельный объект,

**Ключевые слова:** **Не атомарность**, **volatile int counter++**

**Ответ:**

Гарантия видимости: запись видна всем последующим чтениям (значение не кэшируется в регистрах/локально), запрет переупорядочивания вокруг (happens-before: запись → чтение). **Не атомарность:** `volatile int counter++` всё равно гонка (read-modify-write). Применение: флаги остановки, публикация ссылки на иммутабельный объект, DCL-синглтон.

</details>

<a id="q-8-5"></a>
<details>
<summary><strong>8.5 Happens-before и за счёт чего достигается</strong></summary>

> **Вспомнить за 5 секунд:** Отношение JMM: если A happens-before B, эффекты A видимы B. Источники: program order внутри потока; unlock → lock того же монитора; volatile write → read; Thread.start() → действия потока; действия потока → join(); финализация final-полей в конструкторе.

**Ключевые слова:** **unlock**, **lock**, **Thread.start()**, **join()**

**Ответ:**

Отношение JMM: если A happens-before B, эффекты A видимы B. Источники: program order внутри потока; `unlock` → `lock` того же монитора; volatile write → read; `Thread.start()` → действия потока; действия потока → `join()`; финализация final-полей в конструкторе. Механика: компилятор/JIT/CPU не переупорядочивают через эти границы + барьеры памяти/когерентность кэшей. Без happens-before между потоками — никаких гарантий видимости вообще.

</details>

<summary><strong>8.7 sleep() vs wait()</strong></summary>

> **Вспомнить за 5 секунд:** sleep — static Thread, усыпляет текущий поток на время, монитор не отпускает, монитор не нужен. wait — метод Object, требует владения монитором и отпускает его, будится notify/timeout. sleep — пауза/троттлинг; wait — ожидание условия.

**Ключевые слова:** **не**, **отпускает**, **sleep**, **Thread**, **wait**, **Object**

**Ответ:**

`sleep` — static `Thread`, усыпляет текущий поток на время, монитор **не** отпускает, монитор не нужен. `wait` — метод `Object`, требует владения монитором и **отпускает** его, будится notify/timeout. `sleep` — пауза/троттлинг; `wait` — ожидание условия.

</details>

<summary><strong>8.8 Как избежать deadlock</strong></summary>

> **Вспомнить за 5 секунд:** Дедлок = циклическое ожидание ресурсов. Профилактика: глобальный порядок захвата локов (перевод денег: лок по min(id), потом max(id)); не держать несколько локов; tryLock с таймаутом и откатом; уменьшать область — конкурентные структуры вместо локов; lock-free. Диагностика: thread dump (jstack)

**Ключевые слова:** **min(id)**, **max(id)**, **tryLock**, **jstack**

**Ответ:**

Дедлок = циклическое ожидание ресурсов. Профилактика: глобальный порядок захвата локов (перевод денег: лок по `min(id)`, потом `max(id)`); не держать несколько локов; `tryLock` с таймаутом и откатом; уменьшать область — конкурентные структуры вместо локов; lock-free. Диагностика: thread dump (`jstack`) — JVM сама помечает найденный дедлок.

</details>

<summary><strong>8.9 ReentrantLock</strong></summary>

> **Вспомнить за 5 секунд:** Явный лок (AQS внутри): реентерабельный (поток может брать повторно, счётчик), lock/unlock — unlock строго в finally. Плюсы над synchronized: tryLock(timeout) — избегание дедлока, lockInterruptibly, честный режим (FIFO), несколько Condition на один лок. synchronized проще и с Java 6+ не медленнее — ReentrantLock,

**Ключевые слова:** **lock**, **unlock**, **synchronized**, **tryLock(timeout)**, **lockInterruptibly**, **Condition**

**Ответ:**

Явный лок (AQS внутри): реентерабельный (поток может брать повторно, счётчик), `lock`/`unlock` — `unlock` строго в finally. Плюсы над `synchronized`: `tryLock(timeout)` — избегание дедлока, `lockInterruptibly`, честный режим (FIFO), несколько `Condition` на один лок. `synchronized` проще и с Java 6+ не медленнее — `ReentrantLock`, когда нужны его фичи.

</details>

<summary><strong>8.10 Проблемы Executors-фабрик</strong></summary>

> **Вспомнить за 5 секунд:** newFixedThreadPool/newSingleThreadExecutor — неограниченная LinkedBlockingQueue: при отстающих потребителях очередь растёт до OOM. newCachedThreadPool — неограниченное число потоков: всплеск нагрузки = тысячи потоков, исчерпание ресурсов. newScheduledThreadPool

**Ключевые слова:** **неограниченная**, **отменяет**, **newFixedThreadPool**, **newSingleThreadExecutor**, **LinkedBlockingQueue**, **newCachedThreadPool**

**Ответ:**

`newFixedThreadPool`/`newSingleThreadExecutor` — **неограниченная** `LinkedBlockingQueue`: при отстающих потребителях очередь растёт до OOM. `newCachedThreadPool` — неограниченное число потоков: всплеск нагрузки = тысячи потоков, исчерпание ресурсов. `newScheduledThreadPool` — исключение в периодической задаче **отменяет** все последующие запуски молча (обязателен try/catch внутри задачи). Правильно: явный `ThreadPoolExecutor` (core/max, ограниченная очередь, `RejectedExecutionHandler`, именованная `ThreadFactory`).

</details>

<summary><strong>8.11 Race condition</strong></summary>

> **Вспомнить за 5 секунд:** Результат зависит от порядка выполнения потоков: неатомарные read-modify-write (counter++), check-then-act (containsKey→put, проверка баланса→списание). Избежать: атомики/CAS, synchronized/локи, атомарные методы конкурентных коллекций (computeIfAbsent, putIfAbsent), иммутабельность, confinement (данные не шарить).

**Ключевые слова:** **counter++**, **containsKey**, **put**, **synchronized**, **computeIfAbsent**, **putIfAbsent**

**Ответ:**

Результат зависит от порядка выполнения потоков: неатомарные read-modify-write (`counter++`), check-then-act (`containsKey`→`put`, проверка баланса→списание). Избежать: атомики/CAS, `synchronized`/локи, атомарные методы конкурентных коллекций (`computeIfAbsent`, `putIfAbsent`), иммутабельность, confinement (данные не шарить). В БД тот же класс проблем — блокировки/атомарные `UPDATE`.

</details>

Используется под капотом parallel streams и CompletableFuture (commonPool). Когда: большие делимые вычисления;

**Ключевые слова:** **не**, **RecursiveTask**, **RecursiveAction**, **fork()**, **join()**, **CompletableFuture**

**Ответ:**

Пул для рекурсивно делимых CPU-bound задач: work stealing — свободные воркеры крадут подзадачи из деков занятых. `RecursiveTask`/`RecursiveAction`: `fork()` подзадач + `join()`. <summary><strong>8.14 Видит ли дочерний поток ThreadLocal родителя</strong></summary>

> **Вспомнить за 5 секунд:** Обычный ThreadLocal — нет. InheritableThreadLocal — да: значение копируется в момент создания дочернего потока (дальнейшие изменения родителя не видны). В пулах бесполезен (потоки созданы заранее) — поэтому контексты (MDC, security) в пулы передают явно: обёртки задач, TaskDecorator, micrometer context-propagation.

**Ключевые слова:** **ThreadLocal**, **InheritableThreadLocal**, **TaskDecorator**

**Ответ:**

Обычный `ThreadLocal` — нет. `InheritableThreadLocal` — да: значение копируется в момент создания дочернего потока (дальнейшие изменения родителя не видны). В пулах бесполезен (потоки созданы заранее) — поэтому контексты (MDC, security) в пулы передают явно: обёртки задач, `TaskDecorator`, micrometer context-propagation.

</details>

<details>
<summary><strong>8.15 🔥 Типы экзекьюторов и их проблемы</strong></summary>

> **Вспомнить за 5 секунд:** SingleThreadExecutor — последовательность + порядок; проблема: неограниченная очередь (OOM), одна застрявшая задача стопорит всё.

**Ключевые слова:** **SingleThreadExecutor**, **FixedThreadPool**, **CachedThreadPool**, **ScheduledExecutor**, **fixedRate**, **ThreadPoolExecutor**

**Ответ:**

- **SingleThreadExecutor** — последовательность + порядок; проблема: неограниченная очередь (OOM), одна застрявшая задача стопорит всё.
- **FixedThreadPool** — фиксированный параллелизм; та же неограниченная очередь.
- **CachedThreadPool** — потоки под нагрузку, idle 60с умирают; проблема: неограниченное число потоков.
- **ScheduledExecutor** — отложенные/периодические; проблема: молча отменяет периодику после исключения + `fixedRate` копит запуски при долгой задаче.

Итог для senior: в проде — явный `ThreadPoolExecutor` с ограниченной очередью и политикой отказа, метрики пула.

</details>

<summary><strong>8.16 submit() vs execute()</strong></summary>

> **Вспомнить за 5 секунд:** execute(Runnable) — void: исключение улетает в UncaughtExceptionHandler потока (в логах видно). submit — возвращает Future: исключение запаковано в Future и всплывёт только при get() (ExecutionException); если get() не зовут — задача «падает молча». Плюс submit принимает Callable (результат).

**Ключевые слова:** **execute(Runnable)**, **UncaughtExceptionHandler**, **submit**, **Future**, **get()**, **ExecutionException**

**Ответ:**

`execute(Runnable)` — void: исключение улетает в `UncaughtExceptionHandler` потока (в логах видно). `submit` — возвращает `Future`: исключение запаковано в `Future` и всплывёт только при `get()` (`ExecutionException`); если `get()` не зовут — задача «падает молча». Плюс `submit` принимает `Callable` (результат).

</details>

<details>
<summary><strong>8.17 Безопасная публикация объектов</strong></summary>

> **Вспомнить за 5 секунд:** Проблема: другой поток может увидеть ссылку на объект до завершения его конструирования (переупорядочивание) или устаревшие поля. Безопасные способы: static-инициализатор/holder, volatile-поле, final-поля (гарантия JMM после конструктора), передача через конкурентные структуры (BlockingQueue, ConcurrentHashMap),

**Ключевые слова:** **BlockingQueue**, **ConcurrentHashMap**

**Ответ:**

Проблема: другой поток может увидеть ссылку на объект до завершения его конструирования (переупорядочивание) или устаревшие поля. Безопасные способы: static-инициализатор/holder, volatile-поле, final-поля (гарантия JMM после конструктора), передача через конкурентные структуры (`BlockingQueue`, `ConcurrentHashMap`), под синхронизацией. Незащищённая публикация через обычное поле — источник «невозможных» багов (DCL без volatile).

</details>

<a id="q-8-19"></a>
<details>

## Приложение. Критерии оценки уровня

Как интервьюер отличает уровни (дословно по их методичке, сжато):

- **Junior** — базовые концепции, без нюансов производительности и масштабирования, примеры из учебных проектов.
- **Regular** — знает «под капотом», называет альтернативы и trade-off'ы, примеры из реальных проектов, думает об оптимизации.
- **Senior** — видит систему целиком: уточняет контекст вопроса, называет edge cases, примеры из крупных/нагруженных систем, архитектурные последствия решений.

**Практическая формула ответа на 4 балла:** суть (1 фраза) → как работает под капотом → пример из своего проекта → что меняется на масштабе/в кластере. И минимум один уточняющий вопрос за интервью — это буквально их маркер сеньорности.
