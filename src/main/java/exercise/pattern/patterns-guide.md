# Порождающие паттерны

## Singleton

:fire: :spring:

**Что это:** Гарантирует единственный экземпляр класса с глобальной точкой доступа.

**Когда использовать:**
- Конфигурация приложения
- Connection pool / Thread pool
- Logger
- Кэш
- Доступ к shared resource

**Примеры в Java/Spring:**
- `Runtime.getRuntime()`
- `Logger` (log4j, slf4j)
- Spring beans (`@Scope("singleton")` по умолчанию)
- `Desktop.getDesktop()`

**Схема:**
```
┌────────────────────────────────┐
│         Singleton              │
├────────────────────────────────┤
│ - instance: Singleton          │
│ - Singleton()                  │
├────────────────────────────────┤
│ + getInstance(): Singleton     │
│ + businessMethod()             │
└────────────────────────────────┘
```

**Варианты реализации:**

| Вариант | Thread-safe | Lazy | Простота | Рекомендация |
|---------|-------------|------|----------|--------------|
| Eager | Да | Нет | Простая | Для лёгких объектов |
| Synchronized | Да | Да | Простая | Медленный |
| DCL + volatile | Да | Да | Средняя | Классика |
| Holder (Bill Pugh) | Да | Да | Простая | **Рекомендуется** |
| Enum | Да | Нет | Простая | **Лучший вариант** |

**Плюсы:**
- Единственный экземпляр
- Ленивая инициализация (некоторые варианты)
- Глобальный доступ

**Минусы:**
- Скрытые зависимости (нарушает DI)
- Сложность тестирования (mock сложно)
- Нарушает SRP (контроль жизненного цикла + бизнес-логика)
- Проблемы в многопоточности (если неправильно реализован)

**Вопросы на собеседовании:**
- Как сделать Singleton thread-safe?
- Почему DCL без volatile не работает?
- Как сломать Singleton через Reflection/Serialization?
- Почему Enum Singleton — лучший вариант?
- Чем отличается от Spring singleton scope?

## Factory Method

:fire:

**Что это:** Определяет интерфейс для создания объекта, но позволяет подклассам решать, какой класс создавать.

**Когда использовать:**
- Неизвестен конкретный тип объекта заранее
- Делегирование создания объектов подклассам
- Нужно расширять библиотеку без изменения её кода

**Примеры в Java/Spring:**
- `Calendar.getInstance()`
- `NumberFormat.getInstance()`
- `Collection.iterator()`
- `BeanFactory` в Spring

**Схема:**
```
┌─────────────────┐       ┌─────────────────┐
│    Creator      │       │    Product      │
├─────────────────┤       ├─────────────────┤
│ +factoryMethod()│◄──────│                 │
│ +operation()    │       └─────────────────┘
└────────┬────────┘               ▲
         │                        │
         ▼                        │
┌─────────────────┐       ┌───────┴─────────┐
│ ConcreteCreator │       │ ConcreteProduct │
├─────────────────┤       └─────────────────┘
│ +factoryMethod()│───────────────┘
└─────────────────┘
```

**Плюсы:**
- Избавляет от привязки к конкретным классам
- Код создания в одном месте (SRP)
- Легко добавлять новые типы (Open/Closed)

**Минусы:**
- Много подклассов
- Сложнее для простых случаев

**Вопросы на собеседовании:**
- В чём разница между Factory Method и Abstract Factory?
- Когда использовать Factory Method вместо конструктора?
- Что такое Simple Factory и чем отличается от Factory Method?

## Abstract Factory

:star:

**Что это:** Создаёт семейства связанных объектов без указания конкретных классов.

**Когда использовать:**
- Система должна быть независима от способа создания объектов
- Нужно создавать семейства связанных объектов
- UI для разных ОС (Windows/Mac/Linux)

**Примеры в Java/Spring:**
- `DocumentBuilderFactory`
- `TransformerFactory`
- Spring `FactoryBean`

**Схема:**
```
┌───────────────────────┐
│   AbstractFactory     │
├───────────────────────┤
│ +createProductA()     │
│ +createProductB()     │
└───────────┬───────────┘
            │
   ┌────────┴────────┐
   ▼                 ▼
┌──────────┐   ┌──────────┐
│ Factory1 │   │ Factory2 │
├──────────┤   ├──────────┤
│ createA()│   │ createA()│
│ createB()│   │ createB()│
└──────────┘   └──────────┘
      │              │
      ▼              ▼
  ProductA1      ProductA2
  ProductB1      ProductB2
```

**Плюсы:**
- Гарантирует совместимость создаваемых объектов
- Изолирует конкретные классы
- Легко менять семейства объектов

**Минусы:**
- Сложно добавлять новые виды продуктов
- Много классов

**Вопросы на собеседовании:**
- Чем отличается от Factory Method?
- Когда выбрать Abstract Factory?

## Builder

:fire:

**Что это:** Пошаговое создание сложных объектов. Отделяет конструирование от представления.

**Когда использовать:**
- Объект с множеством параметров (>3-4)
- Много опциональных параметров
- Нужен immutable объект
- Пошаговое создание

**Примеры в Java/Spring:**
- `StringBuilder`
- `Stream.Builder`
- `HttpRequest.newBuilder()` (Java 11+)
- Lombok `@Builder`

**Схема:**
```
┌─────────────────────────────────────┐
│              Product                │
├─────────────────────────────────────┤
│ - field1, field2, field3...         │
│ - Product(Builder b)                │
└─────────────────────────────────────┘
                  ▲
                  │
┌─────────────────────────────────────┐
│           Product.Builder           │
├─────────────────────────────────────┤
│ - field1, field2, field3...         │
│ + field1(val): Builder              │
│ + field2(val): Builder              │
│ + build(): Product                  │
└─────────────────────────────────────┘
```

**Плюсы:**
- Читаемый код (именованные параметры)
- Гибкость (опциональные параметры)
- Immutable объекты
- Валидация в build()

**Минусы:**
- Много кода (хотя Lombok решает)
- Дублирование полей

**Вопросы на собеседовании:**
- Когда использовать Builder vs конструктор vs Factory?
- Как реализовать Builder для иерархии классов?
- Зачем Builder возвращает this?

# Структурные паттерны

## Adapter

:fire:

**Что это:** Преобразует интерфейс класса в другой интерфейс, ожидаемый клиентом.

**Когда использовать:**
- Интеграция с legacy кодом
- Использование библиотеки с несовместимым интерфейсом
- Работа с внешними API

**Примеры в Java/Spring:**
- `Arrays.asList()` — адаптер массива к List
- `InputStreamReader` — адаптер InputStream к Reader
- `Collections.enumeration()`
- Spring `HandlerAdapter`

**Схема:**
```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   Client    │───►│   Target    │    │   Adaptee   │
└─────────────┘    ├─────────────┤    ├─────────────┤
                   │ +request()  │    │ +specific() │
                   └──────▲──────┘    └──────▲──────┘
                          │                  │
                   ┌──────┴──────┐           │
                   │   Adapter   │───────────┘
                   ├─────────────┤
                   │ +request()  │ // вызывает specific()
                   └─────────────┘
```

**Два варианта:**
- **Object Adapter** — композиция (рекомендуется)
- **Class Adapter** — наследование (множественное, в Java только через интерфейсы)

**Плюсы:**
- Разделение интерфейса и реализации
- Гибкость (можно менять адаптируемый объект)

**Минусы:**
- Дополнительная сложность

**Вопросы на собеседовании:**
- Чем Adapter отличается от Decorator?
- Object Adapter vs Class Adapter?

## Decorator

:fire: :spring:

**Что это:** Динамически добавляет объекту новые обязанности. Альтернатива наследованию.

**Когда использовать:**
- Добавление функциональности без изменения класса
- Комбинирование поведений
- Невозможно/нежелательно использовать наследование

**Примеры в Java/Spring:**
- Java I/O: `BufferedInputStream(new FileInputStream(file))`
- `Collections.synchronizedList()`
- `Collections.unmodifiableList()`
- Spring AOP (декорирование бинов)

**Схема:**
```
┌─────────────────┐
│    Component    │◄────────────────────┐
├─────────────────┤                     │
│ +operation()    │                     │
└────────┬────────┘                     │
         │                              │
    ┌────┴────┐                  ┌──────┴──────┐
    ▼         ▼                  │  Decorator  │
┌────────┐ ┌────────────┐        ├─────────────┤
│Concrete│ │  Decorator │───────►│ -component  │
│Component│ └────────────┘        │ +operation()│
└────────┘         │              └─────────────┘
                   ▼
          ┌───────────────┐
          │ConcreteDecorator│
          ├───────────────┤
          │ +operation()  │ // super.operation() + extra
          │ +addedBehavior()│
          └───────────────┘
```

**Плюсы:**
- Гибче наследования
- Комбинирование декораторов
- Добавление поведения в runtime

**Минусы:**
- Много мелких объектов
- Порядок декораторов важен
- Сложно убрать конкретный декоратор из цепочки

**Вопросы на собеседовании:**
- Чем отличается от Proxy?
- Приведите пример из Java I/O
- Как реализовать в Spring?

## Proxy

:star: :spring:

**Что это:** Предоставляет суррогат для другого объекта для контроля доступа.

**Когда использовать:**
- Ленивая инициализация (Virtual Proxy)
- Контроль доступа (Protection Proxy)
- Логирование, кэширование (Logging/Caching Proxy)
- Удалённый вызов (Remote Proxy)

**Примеры в Java/Spring:**
- `java.lang.reflect.Proxy`
- JPA ленивая загрузка (Hibernate Proxy)
- Spring AOP, `@Transactional`, `@Cacheable`
- RMI

**Схема:**
```
┌─────────────────┐
│    Subject      │
├─────────────────┤
│ +request()      │
└────────┬────────┘
         │
    ┌────┴────────────┐
    ▼                 ▼
┌────────┐       ┌─────────┐
│  Real  │◄──────│  Proxy  │
│ Subject│       ├─────────┤
├────────┤       │ -real   │
│+request()│     │+request()│ // контроль + real.request()
└────────┘       └─────────┘
```

**Виды Proxy:**

| Вид | Назначение | Пример |
|-----|------------|--------|
| Virtual | Ленивая загрузка | Hibernate Lazy Loading |
| Protection | Контроль доступа | Security Proxy |
| Remote | Удалённый вызов | RMI |
| Caching | Кэширование | Spring @Cacheable |
| Logging | Логирование | Spring AOP |

**Плюсы:**
- Контроль над объектом
- Отложенная инициализация
- Прозрачность для клиента

**Минусы:**
- Увеличение времени отклика
- Сложность кода

**Вопросы на собеседовании:**
- Чем Proxy отличается от Decorator?
- Как работает Dynamic Proxy в Java?
- Как Spring создаёт proxy для @Transactional?

## Facade

:star: :spring:

**Что это:** Предоставляет унифицированный интерфейс к набору интерфейсов подсистемы.

**Когда использовать:**
- Упрощение сложного API
- Разделение слоёв приложения
- Интеграция с legacy системой

**Примеры в Java/Spring:**
- `javax.faces.context.FacesContext`
- Spring `JdbcTemplate`
- SLF4J (фасад к разным логгерам)
- Сервисный слой в приложении

**Схема:**
```
┌─────────────────────────────────────────┐
│                 Facade                   │
├─────────────────────────────────────────┤
│ +simpleOperation()                       │
│ +anotherOperation()                      │
└──────────────────┬──────────────────────┘
                   │
       ┌───────────┼───────────┐
       ▼           ▼           ▼
┌──────────┐ ┌──────────┐ ┌──────────┐
│ SubsystemA│ │SubsystemB│ │SubsystemC│
├──────────┤ ├──────────┤ ├──────────┤
│ +methodA()│ │+methodB()│ │+methodC()│
│ +methodA2()││+methodB2()││+methodC2()│
└──────────┘ └──────────┘ └──────────┘
```

**Плюсы:**
- Упрощает использование подсистемы
- Уменьшает связанность
- Не запрещает прямой доступ к подсистеме

**Минусы:**
- Может стать God Object
- Дополнительный слой абстракции

**Вопросы на собеседовании:**
- Чем отличается от Adapter?
- Когда Facade становится антипаттерном?

## Composite

:star:

**Что это:** Компонует объекты в древовидные структуры. Позволяет работать с отдельными объектами и композициями единообразно.

**Когда использовать:**
- Древовидные структуры (файловая система, меню, организации)
- Нужно единообразно обрабатывать простые и составные объекты
- Рекурсивные структуры

**Примеры в Java/Spring:**
- `java.awt.Container` / `Component`
- `javax.swing.JComponent`
- DOM дерево
- Организационные структуры

**Схема:**
```
┌─────────────────┐
│    Component    │◄─────────────────────┐
├─────────────────┤                      │
│ +operation()    │                      │
└────────┬────────┘                      │
         │                               │
    ┌────┴────────────┐                  │
    ▼                 ▼                  │
┌────────┐       ┌──────────┐            │
│  Leaf  │       │Composite │────────────┘
├────────┤       ├──────────┤  children
│+operation()│   │+operation()│
└────────┘       │+add(Component)│
                 │+remove(Component)│
                 │+getChild(int)│
                 └──────────┘
```

**Пример: Файловая система**
```
Directory (Composite)
├── File (Leaf)
├── File (Leaf)
└── Directory (Composite)
    ├── File (Leaf)
    └── File (Leaf)
```

**Плюсы:**
- Единообразная работа с деревом
- Легко добавлять новые типы компонентов
- Упрощает клиентский код

**Минусы:**
- Сложно ограничить типы детей
- Общий интерфейс может быть слишком общим

**Вопросы на собеседовании:**
- Как реализовать getSize() для файловой системы?
- Чем отличается от Decorator?
- Когда Leaf и Composite должны иметь разные интерфейсы?

# Поведенческие паттерны

## Strategy

:fire:

**Что это:** Определяет семейство алгоритмов, инкапсулирует каждый и делает их взаимозаменяемыми.

**Когда использовать:**
- Много похожих классов, отличающихся поведением
- Нужно выбирать алгоритм в runtime
- Алгоритм использует данные, о которых клиент не должен знать

**Примеры в Java/Spring:**
- `Comparator<T>`
- `java.util.zip.Checksum`
- `javax.servlet.Filter`
- Spring `Resource` (разные стратегии загрузки ресурсов)

**Схема:**
```
┌─────────────────┐       ┌─────────────────┐
│    Context      │──────►│    Strategy     │
├─────────────────┤       ├─────────────────┤
│ -strategy       │       │ +execute()      │
│ +setStrategy()  │       └────────┬────────┘
│ +doSomething()  │                │
└─────────────────┘       ┌────────┴────────┐
                          ▼                 ▼
                   ┌──────────────┐  ┌──────────────┐
                   │StrategyA    │  │ StrategyB    │
                   ├──────────────┤  ├──────────────┤
                   │ +execute()   │  │ +execute()   │
                   └──────────────┘  └──────────────┘
```

**Плюсы:**
- Замена алгоритмов в runtime
- Изоляция кода алгоритмов
- Замена наследования композицией

**Минусы:**
- Клиент должен знать о стратегиях
- Увеличение числа объектов

**Вопросы на собеседовании:**
- Чем Strategy отличается от State?
- Как реализовать Strategy через лямбды?
- Когда использовать Strategy vs Template Method?

## Observer

:fire: :spring:

**Что это:** Определяет зависимость один-ко-многим, при изменении объекта все зависимые уведомляются.

**Когда использовать:**
- Событийная модель
- Публикация/подписка
- MVC (Model уведомляет View)

**Примеры в Java/Spring:**
- `java.util.Observer` (deprecated)
- `PropertyChangeListener`
- `ApplicationListener` в Spring
- `@EventListener` в Spring
- Reactive Streams (`Publisher`/`Subscriber`)

**Схема:**
```
┌─────────────────┐       ┌─────────────────┐
│    Subject      │◄──────│    Observer     │
├─────────────────┤       ├─────────────────┤
│ -observers      │       │ +update()       │
│ +attach()       │       └────────┬────────┘
│ +detach()       │                │
│ +notify()       │       ┌────────┴────────┐
└─────────────────┘       ▼                 ▼
         │         ┌──────────────┐  ┌──────────────┐
         ▼         │ ConcreteObs1 │  │ ConcreteObs2 │
┌─────────────────┐├──────────────┤  ├──────────────┤
│ConcreteSubject  ││ +update()    │  │ +update()    │
├─────────────────┤└──────────────┘  └──────────────┘
│ -state          │
│ +getState()     │
└─────────────────┘
```

**Push vs Pull модель:**
- **Push:** Subject передаёт данные в update()
- **Pull:** Observer запрашивает данные у Subject

**Плюсы:**
- Слабая связанность
- Поддержка broadcast
- Добавление/удаление observers в runtime

**Минусы:**
- Порядок уведомления не гарантирован
- Утечки памяти (забыли отписаться)
- Каскадные обновления

**Вопросы на собеседовании:**
- Как избежать memory leak в Observer?
- Push vs Pull модель?
- Как реализовать thread-safe Observer?

## State

:star:

**Что это:** Позволяет объекту менять поведение при изменении внутреннего состояния.

**Когда использовать:**
- Объект с конечным числом состояний
- Поведение зависит от состояния
- Много условных операторов по состоянию

**Примеры в Java/Spring:**
- `Thread.State`
- Конечные автоматы (заказ, платёж)
- Spring State Machine

**Схема:**
```
┌─────────────────┐       ┌─────────────────┐
│    Context      │──────►│     State       │
├─────────────────┤       ├─────────────────┤
│ -state          │       │ +handle()       │
│ +setState()     │       └────────┬────────┘
│ +request()      │                │
└─────────────────┘       ┌────────┴────────┐
                          ▼                 ▼
                   ┌──────────────┐  ┌──────────────┐
                   │ StateA       │  │ StateB       │
                   ├──────────────┤  ├──────────────┤
                   │ +handle()    │  │ +handle()    │
                   └──────────────┘  └──────────────┘
```

**Диаграмма состояний (пример: Заказ):**
```
    ┌─────┐  pay()   ┌──────┐  ship()  ┌─────────┐  deliver()  ┌───────────┐
    │ NEW ├─────────►│ PAID ├─────────►│ SHIPPED ├────────────►│ DELIVERED │
    └──┬──┘          └──┬───┘          └─────────┘             └───────────┘
       │                │
       │ cancel()       │ cancel()
       ▼                ▼
    ┌─────────────────────┐
    │     CANCELLED       │
    └─────────────────────┘
```

**Плюсы:**
- Локализация поведения по состояниям
- Явные переходы между состояниями
- Убирает условные операторы

**Минусы:**
- Много классов состояний
- Сложность при множестве переходов

**Вопросы на собеседовании:**
- Чем State отличается от Strategy?
- Кто управляет переходами — Context или State?
- Как тестировать State машину?

## Chain of Responsibility

:star: :spring:

**Что это:** Позволяет передавать запрос по цепочке обработчиков. Каждый решает обработать или передать дальше.

**Когда использовать:**
- Несколько объектов могут обработать запрос
- Обработчик неизвестен заранее
- Набор обработчиков задаётся динамически

**Примеры в Java/Spring:**
- `javax.servlet.Filter`
- `java.util.logging.Logger` (handlers)
- Spring Security Filter Chain
- Exception handling в try-catch

**Схема:**
```
┌──────────────────┐
│     Handler      │
├──────────────────┤
│ -next: Handler   │
│ +setNext()       │
│ +handle()        │
└────────┬─────────┘
         │
    ┌────┴────────────────────┐
    ▼                         ▼
┌──────────────┐       ┌──────────────┐
│  HandlerA    │──────►│  HandlerB    │──────► ...
├──────────────┤       ├──────────────┤
│ +handle()    │       │ +handle()    │
└──────────────┘       └──────────────┘
```

**Варианты:**
- **Цепочка с прерыванием:** первый подходящий обрабатывает
- **Цепочка с обработкой:** все обработчики выполняются

**Плюсы:**
- Ослабление связанности
- Гибкость (порядок обработчиков)
- Каждый обработчик — одна ответственность

**Минусы:**
- Нет гарантии обработки
- Сложность отладки

**Вопросы на собеседовании:**
- Как гарантировать обработку запроса?
- Как реализовать с помощью стрима?

## Template Method

:star: :spring:

**Что это:** Определяет скелет алгоритма, делегируя некоторые шаги подклассам.

**Когда использовать:**
- Общий алгоритм с вариативными шагами
- Избежать дублирования кода
- Расширение через наследование

**Примеры в Java/Spring:**
- `AbstractList.get()`, `AbstractSet`
- `InputStream.read()`
- Spring `JdbcTemplate.execute()`
- `HttpServlet.service()`

**Схема:**
```
┌─────────────────────────────────────┐
│          AbstractClass              │
├─────────────────────────────────────┤
│ +templateMethod()                   │  // final - нельзя переопределить
│ +step1()                            │  // конкретный шаг
│ +step2()                            │  // абстрактный - подклассы реализуют
│ +hook()                             │  // хук - можно переопределить
└──────────────────┬──────────────────┘
                   │
          ┌────────┴────────┐
          ▼                 ▼
┌──────────────────┐ ┌──────────────────┐
│  ConcreteClassA  │ │  ConcreteClassB  │
├──────────────────┤ ├──────────────────┤
│ +step2()         │ │ +step2()         │
│ +hook()          │ │                  │
└──────────────────┘ └──────────────────┘
```

**Структура алгоритма:**
```java
// В абстрактном классе
public final void templateMethod() {
    step1();        // конкретный
    step2();        // абстрактный
    if (hook()) {   // хук
        step3();
    }
}
```

**Плюсы:**
- Контроль над алгоритмом
- Устранение дублирования
- Расширение через хуки

**Минусы:**
- Наследование (менее гибко чем композиция)
- Сложность при большом числе шагов

**Вопросы на собеседовании:**
- Чем отличается от Strategy?
- Что такое hook method?
- Почему templateMethod() должен быть final?

## Command

:star:

**Что это:** Инкапсулирует запрос как объект. Позволяет параметризовать клиентов с различными запросами, ставить в очередь, логировать и отменять операции.

**Когда использовать:**
- Undo/Redo функциональность
- Очереди задач, отложенное выполнение
- Макросы (последовательность команд)
- Логирование операций
- Транзакции

**Примеры в Java/Spring:**
- `Runnable` — простейшая команда
- `Callable<T>`
- `javax.swing.Action`
- Spring `@Async` задачи

**Схема:**
```
┌─────────────┐       ┌─────────────┐
│   Invoker   │──────►│   Command   │
├─────────────┤       ├─────────────┤
│ -command    │       │ +execute()  │
│ +invoke()   │       │ +undo()     │
└─────────────┘       └──────┬──────┘
                             │
                    ┌────────┴────────┐
                    ▼                 ▼
             ┌─────────────┐   ┌─────────────┐
             │ConcreteCmd1 │   │ConcreteCmd2 │
             ├─────────────┤   ├─────────────┤
             │ -receiver   │   │ -receiver   │
             │ +execute()  │   │ +execute()  │
             │ +undo()     │   │ +undo()     │
             └──────┬──────┘   └─────────────┘
                    │
                    ▼
             ┌─────────────┐
             │  Receiver   │  (выполняет реальную работу)
             └─────────────┘
```

**Участники:**
- **Command** — интерфейс с execute()/undo()
- **ConcreteCommand** — связывает Receiver с действием
- **Receiver** — знает как выполнить операцию
- **Invoker** — вызывает команду
- **Client** — создаёт команду и связывает с Receiver

**Плюсы:**
- Разделение инициатора и исполнителя
- Undo/Redo
- Очереди и отложенное выполнение
- Композиция команд (макросы)

**Минусы:**
- Много мелких классов команд
- Усложнение кода для простых операций

**Вопросы на собеседовании:**
- Как реализовать Undo/Redo?
- Чем Command отличается от Strategy?
- Как сделать макрос из команд?

## Iterator

:star:

**Что это:** Предоставляет способ последовательного доступа к элементам коллекции без раскрытия внутреннего представления.

**Когда использовать:**
- Обход коллекции без раскрытия структуры
- Единый интерфейс для разных коллекций
- Несколько способов обхода одной коллекции
- Ленивая итерация (генераторы)

**Примеры в Java:**
- `java.util.Iterator<T>`
- `java.util.Enumeration<T>`
- `java.util.Scanner`
- `java.util.stream.Stream<T>`
- Enhanced for-loop (`for-each`)

**Схема:**
```
┌─────────────────┐       ┌─────────────────┐
│    Iterable     │──────►│    Iterator     │
├─────────────────┤       ├─────────────────┤
│ +iterator()     │       │ +hasNext()      │
└─────────────────┘       │ +next()         │
         ▲                │ +remove()       │
         │                └────────┬────────┘
         │                         │
┌────────┴────────┐       ┌────────┴────────┐
│ConcreteCollection│      │ ConcreteIterator│
├─────────────────┤       ├─────────────────┤
│ -elements       │       │ -collection     │
│ +iterator()     │       │ -position       │
└─────────────────┘       │ +hasNext()      │
                          │ +next()         │
                          └─────────────────┘
```

**Виды итераторов:**
| Вид | Описание | Пример |
|-----|----------|--------|
| Forward | Только вперёд | Iterator |
| Bidirectional | Вперёд и назад | ListIterator |
| Random Access | Произвольный доступ | List.get(i) |
| External | Клиент контролирует итерацию | for-each |
| Internal | Коллекция контролирует | forEach(Consumer) |

**Плюсы:**
- Единый интерфейс обхода
- Несколько итераторов одновременно
- Скрытие внутренней структуры
- Ленивая загрузка элементов

**Минусы:**
- Избыточен для простых коллекций
- Не всегда эффективен (vs индекс)

**Вопросы на собеседовании:**
- Чем Iterator отличается от Enumeration?
- Что такое fail-fast и fail-safe итераторы?
- Как реализовать свой итератор?
- Почему нельзя модифицировать коллекцию во время итерации?

# Сравнительные таблицы

## Порождающие паттерны

| Паттерн | Цель | Когда использовать |
|---------|------|-------------------|
| Singleton | Один экземпляр | Config, Cache, Logger |
| Factory Method | Делегировать создание | Неизвестен конкретный тип |
| Abstract Factory | Семейства объектов | UI для разных ОС |
| Builder | Пошаговое создание | Много параметров, immutable |

## Factory Method vs Abstract Factory

| Критерий | Factory Method | Abstract Factory |
|----------|----------------|------------------|
| Создаёт | Один продукт | Семейство продуктов |
| Через | Наследование | Композицию |
| Когда | Один тип, много реализаций | Связанные объекты |

## Constructor vs Factory vs Builder

| Критерий | Constructor | Factory | Builder |
|----------|-------------|---------|---------|
| Параметров | 1-3 | Любое | >3-4 |
| Опциональные | Через перегрузку | Да | Да |
| Валидация | В конструкторе | В фабрике | В build() |
| Читаемость | Низкая | Средняя | Высокая |
| Immutable | Да | Да | Да |

## Decorator vs Proxy vs Adapter

| Критерий | Decorator | Proxy | Adapter |
|----------|-----------|-------|---------|
| Цель | Добавить поведение | Контроль доступа | Изменить интерфейс |
| Интерфейс | Тот же | Тот же | Другой |
| Создаёт объект | Нет | Может (lazy) | Нет |
| Пример | BufferedInputStream | Hibernate Proxy | Arrays.asList() |

## Strategy vs State vs Template Method

| Критерий | Strategy | State | Template Method |
|----------|----------|-------|-----------------|
| Меняет | Алгоритм | Поведение при изменении состояния | Шаги алгоритма |
| Кто выбирает | Клиент | Контекст/State | Подкласс |
| Через | Композицию | Композицию | Наследование |
| Runtime изменение | Да | Да | Нет |

# Частые вопросы на собеседованиях

### Q: Чем Factory Method отличается от Abstract Factory?
**A:** Factory Method создаёт один продукт через наследование — подкласс переопределяет фабричный метод. Abstract Factory создаёт семейство связанных продуктов через композицию — фабрика передаётся как параметр.

### Q: Когда использовать Strategy vs State?
**A:** Strategy — когда алгоритмы взаимозаменяемы и клиент выбирает. State — когда поведение зависит от внутреннего состояния объекта и меняется автоматически.

### Q: Почему DCL (Double-Checked Locking) без volatile не работает?
**A:** Без volatile возможен reordering: ссылка на объект может быть опубликована до завершения инициализации полей. Другой поток увидит частично созданный объект.

### Q: Как сломать Singleton?
**A:**
- Reflection: `setAccessible(true)` на конструкторе
- Serialization: десериализация создаёт новый объект
- Клонирование: если реализован Cloneable
- Classloader: разные classloader — разные экземпляры

### Q: Чем Decorator отличается от Proxy?
**A:** Decorator добавляет функциональность (обогащает поведение). Proxy контролирует доступ (ленивая загрузка, security, кэширование). У Decorator клиент всегда создаёт обёртку, у Proxy — proxy создаётся автоматически.

### Q: Почему Template Method использует наследование, а Strategy — композицию?
**A:** Template Method определяет скелет алгоритма в базовом классе, подклассы только переопределяют отдельные шаги. Strategy позволяет полностью заменить алгоритм в runtime, что требует композиции.

### Q: Что такое Hollywood Principle?
**A:** "Don't call us, we'll call you". Высокоуровневые компоненты вызывают низкоуровневые, не наоборот. Применяется в Template Method, Observer, Dependency Injection.

### Q: Как Spring реализует паттерны?
**A:**
- **Singleton:** Bean scope по умолчанию
- **Factory:** BeanFactory, FactoryBean
- **Proxy:** AOP, @Transactional, @Cacheable
- **Template Method:** JdbcTemplate, RestTemplate
- **Observer:** ApplicationEventPublisher, @EventListener
- **Strategy:** Resource loading strategies
