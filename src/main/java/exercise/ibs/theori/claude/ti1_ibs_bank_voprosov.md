## Секция 2. Java Core (25)

### 2.1 Разница между && и &, || и |
- **&&, ||** — короткое замыкание: правый операнд не вычисляется, если результат уже ясен (`a != null && a.isEmpty()` — идиома защиты от NPE).
- **&, |** — на `boolean` работают без короткого замыкания (оба операнда вычисляются всегда); на числах — это побитовые операции.

### 2.2 Назовите все методы Object
`equals`, `hashCode`, `toString`, `getClass`, `clone` (protected), `finalize` (deprecated), `wait` (3 перегрузки), `notify`, `notifyAll` — итого 11 методов с учётом перегрузок. `wait`/`notify` — база работы с мониторами, поэтому они в `Object`.

### 2.3 Ошибки от неправильных equals/hashCode на уровне метода / сервиса / приложения
- **Метод:** `contains`/`remove` в `HashSet` не находит объект (равные объекты с разными `hashCode` лежат в разных бакетах); ключ `HashMap` «теряется».
- **Сервис:** дубли в дедупликации, сломанный кэш по ключу-объекту, некорректный `distinct()` в стримах.
- **Приложение:** у JPA-entity — дубли в `Set`-коллекциях после `merge`, сломанное сравнение detached/proxy.
- **Как избежать:** переопределять парой, только по стабильным полям (для entity — по id с учётом null и `instanceof` вместо `getClass` из-за прокси), тестировать контракт.

### 2.4 this и super: когда применять, правила
- **this** — ссылка на текущий объект: разрешение shadowing (`this.field = field`), вызов другого конструктора `this(...)`, передача себя.
- **super** — доступ к членам родителя (`super.method()`), вызов конструктора родителя `super(...)`.
- **Правила:** вызов `this(...)`/`super(...)` — только первой строкой конструктора и только один из них; `this` недоступен в static-контексте.

### 2.5 🟣 Почему HashMap неэффективна при плохой хэш-функции: метод / сервис / приложение + оптимизации
- **Метод:** плохой hash → все ключи в один бакет → связный список → поиск O(n) вместо O(1); с Java 8 бакет ≥8 элементов при capacity ≥64 превращается в красно-чёрное дерево → O(log n).
- **Сервис:** деградация горячих операций (кэши, индексы в памяти), лишние аллокации и нагрузка на GC при resize.
- **Приложение:** в высоконагруженной системе — рост латентности, при HashMap-подобном шардировании — перекос нагрузки на узлы.
- **Оптимизации:** качественный `hashCode` (`Objects.hash`/все значимые поля), задать `initialCapacity`, если размер известен (меньше resize), для конкурентности — `ConcurrentHashMap`, для кэшей — Caffeine с нормальной политикой.

### 2.6 Fail-fast vs fail-safe итераторы
- **Fail-fast** (`ArrayList`, `HashMap`): проверяют `modCount`, при структурной модификации во время итерации бросают `ConcurrentModificationException` (best-effort, не гарантия). Удаление при итерации — только `iterator.remove()`/`removeIf`.
- **Fail-safe:** `CopyOnWriteArrayList` — итерация по снапшоту массива; `ConcurrentHashMap` — weakly consistent iterator (не бросает, может видеть/не видеть конкурентные изменения).

### 2.7 Абстрактный класс vs интерфейс, когда что
- **Интерфейс** — контракт: множественная реализация, default/static методы, состояния нет (только `static final`).
- **Абстрактный класс** — общая реализация + состояние: поля, конструкторы, `protected`.
- Интерфейс — «что умеет» (может реализовать кто угодно), абстрактный класс — «является разновидностью» с общим кодом. Со времён default-методов граница тоньше; состояние — главный водораздел.

### 2.8 Как работает «перегрузка» статического метода
Перегрузка (overload) static-методов работает как обычная — выбор по типам параметров на этапе компиляции. Подвох вопроса — «переопределение»: static-методы **не переопределяются**, а скрываются (hiding): вызов резолвится по типу ссылки, а не объекта — `Parent p = new Child(); p.staticMethod()` вызовет метод `Parent`. Поэтому static-методы зовут через имя класса.

### 2.9 Проблемы try-with-resources: метод / сервис / приложение
- **Метод:** ресурс должен быть `AutoCloseable`; исключение из `close()` становится suppressed (не теряется, но и не главное — смотреть `getSuppressed()`); порядок закрытия обратный объявлению.
- **Сервис:** если не использовать — незакрытые соединения/файлы; с пулом БД — исчерпание пула и зависание сервиса.
- **Приложение:** утечка файловых дескрипторов → «too many open files». Сам по себе оверхеда почти не добавляет — проблемы от неиспользования.

### 2.10 Механизм AutoCloseable, последствия для файлов и БД
Интерфейс с единственным `close()`; try-with-resources гарантирует вызов в finally-семантике даже при исключении. Для БД важно: `Connection` из пула при `close()` возвращается в пул, а не закрывается физически — забытый `close` = утечка соединения из пула (HikariCP leak detection ловит именно это — мой случай с MassIndexer). `Closeable` (IO) — подинтерфейс с требованием идемпотентности `close`.

### 2.11 ClassLoader, ClassNotFoundException
- **Иерархия:** Bootstrap (`java.*`) → Platform → Application (classpath); parent delegation — загрузчик сначала спрашивает родителя (защита от подмены `java.lang.String`).
- **Этапы:** loading → linking (verification, preparation, resolution) → initialization (static-блоки).
- `ClassNotFoundException` — класс не найден при явной загрузке (`Class.forName`, отсутствует зависимость в runtime-classpath); отличать от `NoClassDefFoundError` — класс был при компиляции, но исчез в рантайме или упала его статическая инициализация.
- **Решение:** проверить classpath/скоупы зависимостей, конфликты версий (`mvn dependency:tree`).

### 2.12 Проблемы рефлексии: производительность, безопасность, защита
- **Производительность:** обход JIT-оптимизаций (нет инлайна), проверки доступа на каждый вызов — на горячем пути в разы медленнее (фреймворки кэшируют `Method`/`MethodHandle`).
- **Безопасность:** `setAccessible` ломает инкапсуляцию — доступ к private, изменение полей.
- **Защита:** модульная система JPMS (пакет закрыт без `opens`/`--add-opens`), enum-синглтон (защищён от рефлексивного создания), валидация инвариантов не только в конструкторе.
- Spring/Hibernate целиком на рефлексии — поэтому им нужны open-пакеты.

### 2.13 var — что это и когда уместен
Локальный вывод типа (Java 10): тип выводится из инициализатора на компиляции, это **не** динамическая типизация. Только локальные переменные с инициализатором (не поля, не параметры, не `null`). Уместен, когда тип очевиден справа: `var users = new ArrayList<User>()`; вреден, когда тип теряется: `var result = service.process()`.

### 2.14 Variable shadowing
Локальная переменная/параметр с тем же именем скрывает поле (`name` vs `this.name` — классика багов в сеттерах) или переменную внешней области; поле наследника скрывает поле родителя (резолвится по типу ссылки!). Избегать: `this.field` в конструкторах/сеттерах, разные имена, предупреждения компилятора/IDE.

### 2.15 final vs effectively final
`final` — явный запрет переприсваивания. Effectively final — не объявлена `final`, но фактически не переприсваивается; такие (и только такие) локальные переменные можно захватывать в лямбдах и анонимных классах. Причина ограничения: захватывается копия значения, изменение создало бы рассинхрон.

### 2.16 🔵 Generics, примеры, PECS
Параметризация типов с проверкой на компиляции: `List<String>` вместо `List` + касты. В рантайме — type erasure: параметр стирается до `Object`/bound, отсюда нельзя `new T()`, `new T[]`, `instanceof List<String>`.
- **Уровни:** обобщённый класс `class Box<T>`, обобщённый метод `<T> T first(List<T> l)`, bounded `<T extends Comparable<T>>`.
- **PECS:** Producer Extends — `List<? extends Number>` (откуда читаем, положить нельзя), Consumer Super — `List<? super Integer>` (куда пишем).
- Пример из JDK: `Collections.copy(List<? super T> dest, List<? extends T> src)`.

### 2.17 Циклические зависимости: примеры и инструменты
- **Уровень классов:** A хранит B, B хранит A — пересмотреть ответственность, выделить третий класс, интерфейс.
- **Уровень Spring-бинов:** `BeanCurrentlyInCreationException` при constructor injection (Boot 2.6+ циклы запрещены по умолчанию); костыль — `@Lazy`/сеттер-инжекция, правильно — рефакторинг или инверсия через события (`ApplicationEventPublisher`).
- **Уровень микросервисов:** перекрёстные вызовы API — оркестратор или event-driven.
- **Уровень БД:** взаимные FK — отложенные (deferrable) ограничения.

### 2.18 Suppressed exceptions
Подавленные исключения try-with-resources: если и тело, и `close()` бросили, основное — из тела, а исключение close добавляется через `addSuppressed` и доступно через `getSuppressed()`. До Java 7 (ручной finally) исключение из finally **затирало** основное — suppressed это чинит.

### 2.19 Кастомная аннотация, retention
```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Audited { String value() default ""; }
```
- **RetentionPolicy:** `SOURCE` (только компилятор — `@Override`, Lombok), `CLASS` (в байткоде, невидима рефлексии — дефолт), `RUNTIME` (читается рефлексией — всё Spring).
- **Обработка:** рефлексия (`method.isAnnotationPresent`), аспект AOP, annotation processor на компиляции.

### 2.20 @SneakyThrows и @SuppressWarnings
`@SneakyThrows` (Lombok) — бросает checked-исключение без объявления в `throws`: Lombok на уровне байткода обходит проверку компилятора (JVM-то всё равно). Использовать точечно (лямбды, где checked неудобен); злоупотребление прячет контракт ошибок. `@SuppressWarnings` — стандартная Java-аннотация (не Lombok), подавляет предупреждения компилятора ("unchecked", "deprecation"); максимально узкая область применения.

### 2.21 🟣 Optional: когда и какие конструкции
Явный контракт «значения может не быть» для **возвращаемых** значений (`repository.findById`).
- **Конструкции:** `map`/`flatMap` (цепочки), `filter`, `orElse` (аргумент вычисляется всегда) vs `orElseGet` (лениво), `orElseThrow`, `ifPresent`/`ifPresentOrElse`, `or`.
- **Антипаттерны:** поля и параметры методов, `get()` без проверки, `Optional` вокруг коллекций (возвращай пустую), `isPresent()+get()` вместо функциональных методов.

### 2.22 Утечки памяти в Java и как избежать
GC не соберёт то, на что жива ссылка: static-коллекции и кэши без евикшена/лимита, слушатели без отписки, `ThreadLocal` в пулах без `remove()`, незакрытые ресурсы, внутренние не-static классы с неявной ссылкой на outer, ClassLoader-leaks при redeploy. Диагностика: `-XX:+HeapDumpOnOutOfMemoryError` → heap dump → Eclipse MAT (dominator tree). Профилактика: Caffeine/`WeakHashMap` для кэшей, try-with-resources, `remove()` в finally.

### 2.23 Records
Компактные иммутабельные носители данных (Java 16): `record Point(int x, int y)` — final-поля, канонический конструктор, аксессоры `x()`, корректные `equals`/`hashCode`/`toString`. Компактный конструктор — для валидации. Идеальны для DTO, ключей, value-объектов. Ограничения: не наследуются, все поля final.

### 2.24 Sealed классы
Ограничение иерархии (Java 17): `sealed interface Shape permits Circle, Square`. Наследники — final, sealed или non-sealed. Даёт исчерпывающий switch с pattern matching без default — компилятор проверяет полноту. Применение: замкнутые доменные модели (типы событий, результатов).

### 2.25 Последняя версия Java, что добавлено
На момент собеса: последний LTS — Java 25 (сентябрь 2025); в марте 2026 вышла Java 26. Из 25-й: compact object headers (меньше памяти на объект), финализация scoped values и module import declarations, компактные main-методы/файлы, structured concurrency (preview). ## Секция 3. Алгоритмы, структуры данных, паттерны, чистый код и ООП (20)

### 3.1 Основные интерфейсы коллекций + EnumSet/EnumMap
`Collection` → `List` (порядок, дубли), `Set` (уникальность), `Queue`/`Deque` (FIFO/двусторонняя); `Map` — отдельная иерархия ключ-значение. `EnumSet` — битовая маска по ordinal (компактнее и быстрее `HashSet` для enum), `EnumMap` — массив по ordinal (быстрее `HashMap` для enum-ключей). Используй их всегда, когда ключ/элемент — enum (Map переходов статусов заказа!).

### 3.2 Внутренняя структура HashMap, разрешение коллизий
Массив бакетов (степень двойки), индекс = `spread(hashCode) & (n−1)`. Коллизии: связный список; при ≥8 узлах и capacity ≥64 — красно-чёрное дерево (O(log n)); при удалении до 6 — обратно в список. Load factor 0.75 → resize ×2 с перераспределением. Один null-ключ (бакет 0).

### 3.3 🟣 Красно-чёрное дерево: как работает, где применяется
Самобалансирующееся BST: каждый узел красный/чёрный, корень чёрный, у красного нет красных детей, одинаковая «чёрная высота» до листьев — гарантирует высоту O(log n), балансировка перекрасками и поворотами. Применение: `TreeMap`/`TreeSet` (сортированные, navigable-методы), бакеты `HashMap` с Java 8, планировщик Linux CFS. Vs AVL: чуть менее строгий баланс, дешевле вставка/удаление.

### 3.4 Основные структуры данных, что применяю в работе
Массив/`ArrayList` (основная рабочая), `HashMap`/`HashSet` (индексация O(1)), `ArrayDeque` (стек/очередь), `TreeMap` (сортированный доступ), `PriorityQueue` (топ-N, задачи по приоритету), `LinkedList` — почти никогда. Из практики: `Map<EventType, Handler>` для диспетчеризации, `PriorityQueue` для отложенных задач, `Deque` для скобочных/undo-задач.

### 3.5 Fail-fast и fail-safe коллекции
См. [Секцию 2, п. 2.6](#26-fail-fast-vs-fail-safe-итераторы) — тот же вопрос про итераторы.

### 3.6 Hashtable vs ConcurrentHashMap
`Hashtable` — легаси: один монитор на всю таблицу, `synchronized` на каждом методе → потоки сериализуются. `ConcurrentHashMap` (Java 8+): чтение без локов, запись — CAS в пустой бакет / `synchronized` на первом узле бакета → конкуренция только на одном бакете; атомарные `compute`/`merge`/`putIfAbsent`; weakly consistent итераторы; `null` запрещён. `Hashtable` в новом коде — никогда.

### 3.7 🟣 Big O: сложность по времени и памяти
Асимптотика роста стоимости от размера входа, константы отбрасываются. Лестница: O(1) → O(log n) (бинарный поиск) → O(n) → O(n log n) (сортировки) → O(n²) → O(2ⁿ). Память — та же нотация для дополнительной памяти (merge sort O(n), quicksort O(log n) на стек). Оцениваем худший/средний случай; амортизированная сложность (`add` в `ArrayList` — O(1) амортизированно из-за редких resize).

### 3.8 Сортировки; quick sort vs merge sort
Пузырьковая/вставки O(n²) (вставки хороша на почти отсортированных), quick/merge/heap O(n log n).
- **Quick:** in-place, средний O(n log n), худший O(n²) (плохие pivot), нестабильная.
- **Merge:** гарантированный O(n log n), стабильная, +O(n) памяти, дружит с внешней сортировкой.
- JDK: примитивы — dual-pivot quicksort, объекты — TimSort (merge+insertion, стабильный).

### 3.9 Коммивояжёр / рюкзак — подход
Обе NP-трудные, полный перебор — экспонента. Рюкзак 0/1: динамическое программирование по таблице [предмет][вместимость], O(n·W). TSP: ДП Хелда-Карпа O(n²·2ⁿ) для малых n; для реальных размеров — эвристики (ближайший сосед, 2-opt) и метаэвристики. Ответ строится так: распознать класс задачи → точное решение малых размеров (ДП/branch&bound) → приближённое для больших.

### 3.10 Жадный алгоритм
На каждом шаге локально оптимальный выбор без пересмотра. Работает, когда есть свойство жадного выбора (доказуемо): сдача канонических монет, интервальное планирование (по времени окончания), Хаффман, Дейкстра, Прим/Крускал. Контрпример честности: рюкзак 0/1 жадным по удельной цене решается неверно.

### 3.11 Композиция/ассоциация: has-a vs is-a
**is-a** — наследование (`Car is-a Vehicle`): жёсткая связь с реализацией родителя (fragile base class). **has-a** — объект содержит другой: ассоциация (знает о), агрегация (владеет, но жизненные циклы разные), композиция (часть не живёт без целого). Правило: предпочитай композицию — гибче, тестируемее, заменяемо в рантайме (Strategy — ровно это).

### 3.12 CAP-теорема
В распределённой системе при сетевом разделении (Partition) нельзя одновременно сохранить Consistency (все узлы видят одно) и Availability (каждый запрос получает ответ). P — не выбор, а реальность сети → фактический выбор CP vs AP: CP — ZooKeeper, etcd (при расколе часть узлов недоступна, но данные согласованы); AP — Cassandra, DynamoDB (отвечают всегда, eventual consistency). Одна PostgreSQL-нода — вне теоремы; с репликами выбор проявляется в синхронной/асинхронной репликации.

### 3.13 Map/Reduce и divide and conquer: метод / класс / сервис / приложение
Идея: разбить данные на независимые части (divide/map), обработать параллельно, агрегировать (conquer/reduce).
- **Метод:** Stream `map().reduce()`/`collect()`.
- **Класс:** `ForkJoinPool` + `RecursiveTask` (merge sort — классический divide&conquer).
- **Сервис:** параллельная обработка батчей через `ExecutorService` + агрегация.
- **Приложение:** Kafka-партиции как «map» по ключам + агрегирующий консюмер; Hadoop/Spark как канон.

### 3.14 Любимый паттерн — где и почему
Готовый ответ: **Strategy**. «На AuctionPort обработчики событий аукциона: интерфейс `EventHandler` с методом `type()`, реализации — Spring-бины, диспетчер собирает `Map<EventType, Handler>` из `List<EventHandler>` в конструкторе. Новый тип события — новый бин без правки диспетчера (Open/Closed), каждый обработчик тестируется изолированно». Это одновременно ответ на review-задачу про if-else.

### 3.15 Паттерн Фабрика: преимущество перед явным созданием
Инкапсуляция логики создания: клиент не знает конкретный класс, выбор реализации в одном месте (по конфигу/типу), проще подменять в тестах и добавлять варианты. Factory Method — переопределяемый метод создания; Abstract Factory — семейство связанных продуктов. Примеры: `BeanFactory`, `ConnectionFactory`, `List.of`/Collections-фабрики.

### 3.16 🔥 Паттерн Стратегия
*(у Миши спрашивали на уровне junior — не повторяй этот уровень ответа)*

Семейство взаимозаменяемых алгоритмов за общим интерфейсом; выбор в рантайме; замена if-else-цепочек по типу. Не ограничивайся определением — дай механику Spring-реализации (`List<Handler>` → `Map` по ключу), пример из JDK (`Comparator` — стратегия сравнения), своё применение (обработчики событий, политики скидок из coding task 2.12) и отличие от State (стратегию выбирает клиент; состояние переключает себя само по жизненному циклу объекта).

### 3.17 🟣 Декоратор vs Адаптер
Оба — обёртки, разница в цели.
- **Адаптер** меняет интерфейс: делает несовместимое совместимым (`InputStreamReader`: byte-поток → char-поток; обёртка над legacy-API под наш интерфейс).
- **Декоратор** сохраняет интерфейс и добавляет поведение, обёртки компонуются: `new BufferedInputStream(new GZIPInputStream(new FileInputStream(...)))`.
- Рядом **Proxy**: интерфейс тот же, но цель — контроль доступа/ленивость, а не новое поведение (Spring `@Transactional`, Hibernate lazy proxy).

### 3.18 Паттерн Visitor
Вынос операции из иерархии объектов: элементы принимают visitor (`accept`), visitor имеет `visit`-метод на каждый тип — double dispatch. Плюс: новые операции без правки классов иерархии; минус: новый тип элемента требует правки всех visitor'ов. Применение: обход AST (компиляторы, ANTLR), `java.nio.file.FileVisitor`. Сегодня частично вытесняется pattern matching switch по sealed-иерархии.

### 3.19 Инкапсуляция, наследование, полиморфизм
**Инкапсуляция** — скрытие состояния за контрактом: private-поля, инварианты в методах (`Account` не даёт установить отрицательный баланс). **Наследование** — переиспользование через is-a; осторожно, предпочитать композицию. **Полиморфизм** — один контракт, разные реализации, выбор метода по фактическому типу в рантайме: `List l = new ArrayList()`; плюс ad-hoc полиморфизм — перегрузка (compile-time). Четвёртый кит — абстракция (интерфейс отделён от реализации).

### 3.20 Принципы чистого кода по Роберту Мартину
Говорящие имена (без расшифровки в комментарии); маленькие функции, один уровень абстракции, одно дело; мало аргументов (0–2, объект-параметр вместо пяти); DRY; комментарии — только «почему», код говорит «что»; обработка ошибок исключениями, не кодами возврата; тесты как первоклассный код (FIRST); boy scout rule — оставь код чище, чем нашёл. Плюс SOLID как архитектурный слой этих принципов.

## Секция 4. Spring (20)

### 4.1 🟣 IoC и как Spring его реализует
Инверсия управления: не объект создаёт свои зависимости, а контейнер создаёт и внедряет. Реализация: `ApplicationContext` сканирует классы/конфигурации → строит `BeanDefinition`'ы → создаёт бины с разрешением графа зависимостей → внедряет (конструктор/сеттер/поле). DI — конкретный механизм IoC. Профит: слабая связность, подмена реализаций, тестируемость (в тест — моки без контейнера).

### 4.2 Виды autowiring
По типу (`@Autowired` — основной), по имени (`@Resource` / имя параметра при неоднозначности), конструктор/сеттер/поле как места внедрения. При нескольких кандидатах: `@Primary` (дефолт), `@Qualifier` (явный выбор), имя поля как fallback. Рекомендация — constructor injection: final-поля, обязательность, тестируемость, циклы падают сразу.

### 4.3 Бин и жизненный цикл
Бин — объект под управлением контейнера. Цикл: `BeanDefinition` → инстанцирование → внедрение зависимостей → Aware-колбэки (`BeanNameAware`...) → `BeanPostProcessor.postProcessBeforeInitialization` → `@PostConstruct` / `afterPropertiesSet` → `postProcessAfterInitialization` (здесь оборачивание в AOP-прокси) → использование → `@PreDestroy`. `BeanPostProcessor` — механизм, которым сделана половина магии Spring.

### 4.4 Транзакционность в Spring: настройка и проблемы
`@EnableTransactionManagement` (в Boot автоматом) + `@Transactional` на методах сервиса; под капотом — прокси, открывающий/коммитящий транзакцию вокруг вызова. Проблемы — классическая пятёрка:
- self-invocation (мимо прокси);
- private-методы (игнор);
- проглоченное исключение (нет rollback);
- checked-исключение без `rollbackFor` (коммит!);
- внешние вызовы внутри транзакции (держат соединение, рассинхрон при откате → outbox).

Плюс длинные транзакции = блокировки и bloat.

### 4.5 Как Boot упрощает создание приложений
Автоконфигурация по classpath (есть PostgreSQL-драйвер + DataSource-стартер → готовый `DataSource`), starters — согласованные наборы зависимостей, встроенный сервер (jar = приложение), externalized config (yml, env, профили), Actuator, разумные дефолты с точечным переопределением своим бином.

### 4.6 🔵 AOP: как работает и когда использовать
Сквозная функциональность (логирование, транзакции, безопасность, метрики) выносится из бизнес-кода в аспекты. Механика Spring: прокси вокруг бина (JDK dynamic по интерфейсам / CGLIB подклассом); advice (before/after/around) применяется по pointcut-выражениям. Ограничения из-за прокси: только public, только внешние вызовы (self-invocation мимо), final-методы не перехватить CGLIB. Использовать для инфраструктурных сквозных вещей; бизнес-логика в аспектах — антипаттерн. `@Transactional`/`@Cacheable`/`@Async` — готовые аспекты.

### 4.7 Actuator и мониторинг
Производственные endpoints: `/health` (liveness/readiness для K8s), `/metrics` + `/prometheus` (Micrometer), `/info`, `/env`, `/loggers` (смена уровня на лету), `/threaddump`, `/heapdump`. Опыт: метрики Actuator/Prometheus при 8-часовом soak-тесте — пулы, латентность, GC. Безопасность: наружу только `/health`, остальное за авторизацией.

### 4.8 🟣 Автоконфигурация Boot
`@SpringBootApplication` включает `@EnableAutoConfiguration`: из `META-INF/.../AutoConfiguration.imports` грузятся конфигурации-кандидаты, каждая применяется по условиям — `@ConditionalOnClass` (есть в classpath), `@ConditionalOnMissingBean` (пользователь не определил свой), `@ConditionalOnProperty`. Свой бин выигрывает у автоконфигурационного. Отладка: `--debug` → condition evaluation report.

### 4.9 Spring и микросервисная архитектура
Boot — сервис как самодостаточный jar; Spring Cloud: Gateway (API-шлюз), Config (централизованная конфигурация), OpenFeign (декларативные клиенты), LoadBalancer, Resilience4j-интеграция (CB/retry), Micrometer Tracing (распределённый трейсинг), Spring Kafka/AMQP (события). Kubernetes забирает часть (discovery, config) — Spring дополняет health-пробами и graceful shutdown.

### 4.10 @Transactional: как работает, ограничения
Механика и пять ловушек — см. [п. 4.4](#44-транзакционность-в-spring-настройка-и-проблемы) выше. Добавь параметры: `propagation` (`REQUIRED` дефолт, `REQUIRES_NEW` — отдельная транзакция для аудита, `NESTED` — savepoint), `isolation`, `readOnly` (оптимизация Hibernate), `timeout`, `rollbackFor`.

### 4.11 ApplicationContext vs BeanFactory
`BeanFactory` — базовый контейнер: реестр бинов, ленивое создание. `ApplicationContext` — надстройка: eager-инициализация singleton'ов (ошибки на старте, а не в проде), события, ресурсы, i18n, автоматическая регистрация `BeanPostProcessor`'ов. На практике — всегда `ApplicationContext`.

### 4.12 @Bean — как и когда
Фабричный метод в `@Configuration`: для сторонних классов (не пометишь `@Component`), сложной сборки, условной логики создания. В `@Configuration` (`proxyBeanMethods=true`) повторный вызов `@Bean`-метода вернёт тот же singleton — CGLIB-прокси конфигурации.

### 4.13 🔵 @Component vs @Service vs @Repository
Технически все — `@Component` (сканирование). Семантика слоёв: `@Service` — бизнес-логика; `@Repository` — доступ к данным + **бонус**: трансляция низкоуровневых исключений (`SQLException`, Hibernate) в единую иерархию `DataAccessException`; `@Controller`/`@RestController` — веб-слой + обработка через `DispatcherServlet`. Стереотипы = читаемость + точки для аспектов по слоям.

### 4.14 🟣 @Autowired vs @Resource vs @Inject
`@Autowired` (Spring): по типу, required по умолчанию, + `@Qualifier` для уточнения. `@Resource` (JSR-250, jakarta.annotation): сначала по имени, потом по типу. `@Inject` (JSR-330): как `@Autowired` по типу, но без required, стандарт вне Spring. На практике: `@Autowired`/конструктор; `@Resource` — когда нужен выбор по имени.

### 4.15 Внедрение в static-поле
Напрямую `@Autowired` на static не работает (контейнер управляет экземплярами, поле не принадлежит экземпляру). Обходы — нестатический `@Autowired`-сеттер, пишущий в static, или `@PostConstruct` — работают, но это признак кривого дизайна: static + DI несовместимы идейно; правильно — убрать static, внедрять экземпляр.

### 4.16 Инжект примитивов
`@Value("${app.timeout:30}")` — из конфигурации с дефолтом, SpEL поддерживается. Правильнее для групп настроек — `@ConfigurationProperties(prefix="app")`: типизировано, валидируемо (`@Validated`), видно в metadata.

### 4.17 Инжект коллекции
`List<PaymentHandler> handlers` — все бины типа (порядок — `@Order`); `Map<String, PaymentHandler>` — имя бина → бин. База паттерна «диспетчер стратегий»: собрать `Map` по ключу из списка реализаций в конструкторе.

### 4.18 @Primary и @Qualifier
При нескольких кандидатах одного типа: `@Primary` — кандидат по умолчанию (один на тип); `@Qualifier("name")` в точке внедрения — явный выбор, приоритетнее `@Primary`. Свои qualifier-аннотации — типобезопасный вариант.

### 4.19 🟣 @Conditional
Включение бина/конфигурации по условию: `@ConditionalOnProperty` (фича-флаг), `OnClass`, `OnMissingBean`, `OnExpression`; своё условие — `implements Condition`. Фундамент всей автоконфигурации Boot. Пример: включение интеграции по проперти окружения (заглушка в dev, реальный клиент в prod).

### 4.20 @Profile
Группировка бинов по окружениям: `@Profile("prod")` + `spring.profiles.active`. Конфигурации `application-{profile}.yml`. Отличие от `@Conditional`: профиль — крупная гранулярность «окружение», Conditional — произвольные условия.

## Секция 5. Базы данных, SQL, Hibernate, JPA, Spring Data (21)

### 5.1 🟣 Виды индексов и как использовать для оптимизации
PostgreSQL:
- **B-tree** (дефолт: `=`, диапазоны, `ORDER BY`, `LIKE 'x%'`)
- **Hash** (только `=`, ниша)
- **GIN** (инвертированный: jsonb, массивы, полнотекст, pg_trgm для `%x%`)
- **GiST** (геометрия, диапазоны, KNN)
- **SP-GiST** (префиксные структуры)
- **BRIN** (min/max по блокам — огромные append-only таблицы)

Разновидности: unique, составной (порядок колонок — leading!), partial (`WHERE status='ACTIVE'`), covering (`INCLUDE` → index-only scan), функциональный (`lower(email)`), `CONCURRENTLY` на проде. Использование: индексировать колонки `WHERE`/`JOIN`/`ORDER BY` с высокой селективностью, проверять `EXPLAIN (ANALYZE)`, помнить цену — замедление записи и место.

### 5.2 Change Data Capture: реализация и уведомления между БД/сервисами
Варианты по возрастанию зрелости: polling по `updated_at` (просто, лаг, пропуски удалений), триггеры + таблица аудита (нагрузка на транзакции), чтение WAL — logical decoding/replication slots, Debezium → Kafka (стандарт де-факто: не грузит транзакции, даёт поток изменений). Уведомления: PG `LISTEN`/`NOTIFY` (в рамках одной БД, без гарантий доставки), события через Kafka. Связка с outbox: Debezium читает outbox-таблицу — транзакционная публикация событий.

### 5.3 Триггеры
Процедуры на события таблицы: `BEFORE`/`AFTER` `INSERT`/`UPDATE`/`DELETE`, row/statement level. Применение: аудит изменений, денормализованные счётчики, инварианты уровня БД. Минусы: скрытая от приложения логика, сложно отлаживать/версионировать, нагрузка в транзакции — в микросервисах предпочитают логику в приложении + CDC.

### 5.4 Хранимые процедуры и функции
Логика в БД (PL/pgSQL): функции возвращают значение и зовутся из SQL, процедуры (PG 11+) управляют транзакциями. Оправданы: массовые операции над данными без гоняния по сети, сложные миграции. Минусы: версионирование, тестирование, vendor lock-in, размазанная логика — в бэкенд-мире по умолчанию логика в приложении.

### 5.5 🟣 Materialized Views
*(у Миши — на уровне junior, но подготовься основательно)*

Обычное view — сохранённый **запрос**, выполняется при каждом обращении. Материализованное — физически сохранённый **результат**: читается как таблица (быстро, можно индексировать), но данные устаревают — обновление вручную/по расписанию `REFRESH MATERIALIZED VIEW`, с `CONCURRENTLY` (нужен unique-индекс) — без блокировки читателей. Применение: тяжёлые агрегаты для отчётов/дашбордов, где допустим лаг. Trade-off: скорость чтения ↔ свежесть + место.

### 5.6 Репликация между разными БД
Внутри PostgreSQL: физическая streaming (WAL, реплика байт-в-байт, read-only standby) и логическая (publication/subscription, выборочные таблицы, между версиями). Между разными СУБД: CDC-пайплайн Debezium → Kafka → консюмер пишет в целевую; ETL/ELT-инструменты. Ключевые вопросы: лаг, идемпотентность применения, порядок, конфликт-резолюшн при двусторонней.

### 5.7 Entity в JPA и маппинг
`@Entity` + `@Table`, `@Id` (+генерация), поля → `@Column`, no-arg конструктор (Hibernate создаёт через рефлексию/прокси), ассоциации `@ManyToOne`/`@OneToMany(mappedBy)`. Практики: enum — `@Enumerated(STRING)`, деньги — `BigDecimal` + precision/scale, `LAZY` для ассоциаций, `equals`/`hashCode` по id аккуратно.

### 5.8 Транзакции и уровни изоляции
ACID; уровни: read uncommitted → read committed → repeatable read → serializable, отсекают dirty read → non-repeatable read → phantom. PostgreSQL: dirty read невозможен вовсе (RU=RC), дефолт RC; RR на снапшотах — фантомов нет, но при конфликте serialization failure `40001` → ретрай; Serializable = SSI. Пример для ответа: два перевода читают баланс 100 под RC и оба списывают 80 — нужна блокировка или атомарный `UPDATE`, изоляция сама не спасёт.

### 5.9 🟣 Ленивая загрузка в Hibernate и проблемы
*(уровень senior)*

`LAZY`: вместо ассоциации — прокси, SQL при первом обращении. Проблемы: `LazyInitializationException` вне сессии/транзакции; N+1 при обращении в цикле; OSIV маскирует (держит соединение весь запрос — выключать). Решения: `JOIN FETCH` / `@EntityGraph` под конкретный сценарий, DTO-проекции для чтения, `@BatchSize` (IN-пачки). Боевой кейс: инвойсы 8–11 с → 180 мс.

### 5.10 Cascade в JPA
Распространение операций на связанные сущности: `PERSIST`, `MERGE`, `REMOVE`, `ALL` — на ассоциации; `orphanRemoval=true` — удаление ребёнка при отвязке. Осторожно: `REMOVE` на `@ManyToMany` и на «общих» сущностях (удалишь чужое); каскады — для агрегатов «родитель владеет детьми» (заказ → позиции).

### 5.11 🔵 N+1 и предотвращение
1 запрос за списком + N за ассоциациями каждого элемента. Диагностика: SQL-лог, счётчик запросов, hypersistence-optimizer/datasource-proxy. Решения: `JOIN FETCH` (JPQL), `@EntityGraph`, `@BatchSize`/subselect (пачки IN), DTO-проекции. Боевой кейс: поиск инвойсов ~1900 запросов → 15, 8–11 с → ~180 мс (`JOIN FETCH` критичных + `@BatchSize` остальных).

### 5.12 Стратегии генерации PK
`IDENTITY` — авто-инкремент колонки: просто, но Hibernate не может батчить `INSERT` (id нужен сразу). `SEQUENCE` — рекомендуемая в PG: `allocationSize` кэширует диапазон, дружит с батчингом. `TABLE` — эмуляция таблицей, медленно, не используйте. `UUID` — генерация в приложении, распределённые системы; минус — размер индекса/случайность (v7 упорядочен). `AUTO` — выбор диалектом.

### 5.13 Мониторинг производительности запросов
Уровень приложения: `hibernate.show_sql` (только dev), `hibernate.generate_statistics`, datasource-proxy/p6spy (время + параметры), метрики пула HikariCP через Micrometer. Уровень БД: `pg_stat_statements` (топ по времени), `auto_explain`, slow query log. Плюс APM/трейсинг — какой endpoint породил запрос.

### 5.14 🟣 Оптимистическая vs пессимистическая блокировка
**Оптимистическая:** `@Version`, `UPDATE ... WHERE version=?`; конфликт → `OptimisticLockException` → ретрай; для редких конфликтов, не держит блокировки. **Пессимистическая:** `SELECT FOR UPDATE` (`@Lock(PESSIMISTIC_WRITE)`) — строка залочена до конца транзакции; для горячих строк (балансы, остатки), где конфликт вероятен и ретраи дороги. Выбор: частота конфликтов × цена ретрая; помнить о порядке захвата (дедлоки) и `SKIP LOCKED` для очередей.

### 5.15 Пагинация в Spring Data + большие объёмы
`Pageable`/`PageRequest.of(page, size, Sort)` → `Page` (с count-запросом — дорого) или `Slice` (без). Большие объёмы: `OFFSET` деградирует линейно → keyset/seek (`WHERE (created_at, id) > (:ts, :id) ORDER BY ... LIMIT`); стабильная сортировка с tie-breaker (id) обязательна; для выгрузок — `Stream<T>` с fetch size или батчи по ключу.

### 5.16 Second Level Cache
Кэш сущностей между сессиями (1-й уровень — persistence context внутри сессии). Подключение: провайдер (Ehcache/Infinispan/Redis), `@Cacheable`/`@Cache(usage=CacheConcurrencyStrategy...)`, стратегии `READ_ONLY` (справочники — идеал) / `READ_WRITE`. Включать точечно: инвалидация в кластере и консистентность — главная боль; чаще выгоднее кэш уровня сервиса (Caffeine/Redis) с явным TTL.

### 5.17 Кастомные запросы в Spring Data, @Query
Иерархия: derived queries (`findByStatusAndCreatedAtAfter`) → `@Query` JPQL → `@Query(nativeQuery=true)` → Specification/Criteria (динамика) → кастомный репозиторий с `EntityManager`. `@Modifying` + `@Query` для `UPDATE`/`DELETE` (+`clearAutomatically`). Именованные параметры `:status` читабельнее позиционных.

### 5.18 🟣 Liquibase/Flyway, автоматизация миграций
Версионирование схемы в коде: Flyway — SQL-файлы `V1__init.sql`, таблица `flyway_schema_history`; Liquibase — changeset'ы (XML/YAML/SQL) с id+author+checksum, `DATABASECHANGELOG`, есть rollback и preconditions. Автоматизация: миграции едут с приложением (запуск при старте) или отдельным шагом CI/CD до деплоя; правила: changeset неизменяем после выката, обратная совместимость схемы при rolling-деплое (expand → migrate → contract). Опыт: Liquibase на AuctionPort.

### 5.19 Связи One-to-One/Many + подводные камни
Владелец связи — сторона с FK (`@JoinColumn`), обратная — `mappedBy`; `@ManyToOne` — EAGER по умолчанию (переводи в LAZY!), коллекции — LAZY. Подводные: N+1; `@OneToOne LAZY` на не-владеющей стороне не работает без bytecode enhancement; many-to-many лучше явной сущностью связи (поля на связи появятся всегда); `toString`/`equals` с ассоциациями — рекурсия и лишние загрузки; каскады см. выше.

### 5.20 Пагинация при работе с БД (общий вопрос)
`LIMIT`/`OFFSET` — просто, но `OFFSET n` читает и выбрасывает n строк; keyset — по индексируемому ключу, стабильно на любой глубине; курсоры БД — для стриминга в рамках транзакции. Мой delta-endpoint `/lots/changedBids` — keyset по timestamp, −95% трафика.

### 5.21 Фильтрация/сортировка/Pageable, спецификации
Статичные фильтры — derived/`@Query` + `Pageable` параметром. Динамичные комбинации — `Specification<T>` (`JpaSpecificationExecutor`): предикаты как функции, композиция and/or; альтернатива — Querydsl (типобезопасно). `Sort` из API валидировать whitelist'ом полей (иначе сортировка по любому полю = DoS/утечка).

## Секция 6. REST, микросервисы, межсетевое взаимодействие (25)

### 6.1 🔵 Что такое REST, принципы
Архитектурный стиль поверх HTTP: ресурсы (существительные в URI), представления (JSON), стандартные методы. Принципы: client-server, stateless (каждый запрос самодостаточен → горизонтальное масштабирование), cacheable, uniform interface (единообразие: URI + методы + коды), layered system, (code on demand — опционально). REST — не протокол, а набор ограничений; уровень зрелости обычно Richardson L2 (ресурсы + глаголы + коды).

### 6.2 Идемпотентность методов HTTP
Идемпотентный = повтор даёт тот же результат состояния: `GET`/`HEAD` (ещё и safe), `PUT`, `DELETE`. `POST` — нет (повтор создаст дубль), `PATCH` — не гарантированно. Важность: ретраи и сетевые повторы безопасны только для идемпотентных; `POST` делают идемпотентным через `Idempotency-Key`. Для финтеха — критично (двойное списание).

### 6.3 Статус-коды
- **2xx:** 200, 201 Created (+Location), 202 Accepted (асинхрон), 204 No Content.
- **4xx (клиент):** 400 невалидный запрос, 401 не аутентифицирован, 403 нет прав, 404, 409 Conflict (конфликт состояния — недопустимый переход статуса, version conflict), 422 семантическая ошибка, 429 rate limit.
- **5xx:** 500, 502/503/504 — шлюз/недоступен/таймаут апстрима.
- Правило: ошибки клиента ≠ 500.

### 6.4 Пагинация в REST API
**Offset-based:** `?page=0&size=20` + метаданные (`totalElements` или `hasNext`); просто, глубокие страницы дороги и «едут» при вставках. **Cursor-based:** `?after=<opaque_cursor>&limit=20` — стабильно и быстро (keyset под капотом) — выбор для лент/больших данных. Обязательно: cap на size, детерминированная сортировка, Link-заголовки или next-курсор в теле.

### 6.5 Безопасность REST API: аутентификация и авторизация
- **Транспорт:** только HTTPS.
- **Аутентификация:** JWT bearer (stateless, подпись RS256/HS256, короткий TTL + refresh), OAuth2 (делегирование: authorization code + PKCE для фронта, client credentials между сервисами), API-ключи (простые интеграции), mTLS (сервис-сервис в банках).
- **Авторизация:** роли/скоупы, проверка прав на каждый ресурс (IDOR!), method security `@PreAuthorize`.
- Плюс: rate limiting, валидация входа, без секретов в URL/логах.

### 6.6 🟣 Обработка ошибок: @ExceptionHandler, @ControllerAdvice
`@RestControllerAdvice` — глобальный перехватчик: `@ExceptionHandler(OrderNotFoundException.class)` → 404, `InvalidTransition` → 409, `MethodArgumentNotValidException` → 400 со списком полей. Формат — единый: `ProblemDetail` (RFC 7807) в Boot 3. Правила: не светить стектрейсы наружу (ИБ), логировать с traceId внутри, маппинг домен→HTTP в одном месте, а не try/catch по контроллерам.

### 6.7 CORS
Браузерная same-origin policy: JS с origin A не может читать ответы API на origin B без разрешения. Preflight `OPTIONS` → сервер отвечает `Access-Control-Allow-Origin`/`Methods`/`Headers` (+`Allow-Credentials` для кук). Настройка: глобальная `CorsConfiguration` или `@CrossOrigin`; со Spring Security — `http.cors()` до остальных правил. Не «отключить безопасность», а явный whitelist origin'ов; `*` с credentials — запрещено спекой.

### 6.8 Синхронные vs асинхронные запросы в микросервисах
**Sync** (REST/gRPC): нужен немедленный ответ; минусы — временнáя связность (оба живы), каскадные отказы, суммирование латентности. **Async** (Kafka/Rabbit): развязка по времени, буферизация пиков, fan-out; минусы — eventual consistency, сложнее отладка. Правило: команды с ответом — sync + таймауты/CB; факты «произошло X» — события. Пример: создание заказа sync, уведомления/аналитика/экспорт — событиями.

### 6.9 Circuit Breaker
Защита от каскадных отказов: считает ошибки/таймауты вызовов зависимости; `CLOSED` → (порог превышен) `OPEN` — мгновенный отказ без вызова (даём зависимости отдышаться) → `HALF-OPEN` — пробные запросы → обратно. Resilience4j: `@CircuitBreaker(name=..., fallbackMethod=...)`, настройка окна/порогов; в связке timeout + retry + bulkhead. Без CB медленная зависимость съедает все потоки вызывающего — падение домино.

### 6.10 Согласованность данных между микросервисами: сага и 2PC
2PC/XA — блокирующий координатор, плохо переживает отказы, в микросервисах почти не используют. Сага — последовательность локальных транзакций с компенсациями на откат: хореография (через события; просто, но поток размазан) vs оркестрация (координатор со state machine; для финансовых флоу — предпочтительно: видимость и контроль). Плюс транзакционный outbox для атомарности «БД + событие» и идемпотентность шагов.

### 6.11 API Gateway
Единая входная точка: маршрутизация к сервисам, аутентификация/авторизация на периметре, rate limiting, TLS-терминация, агрегация ответов, версии/канареечная маршрутизация. Реализации: Spring Cloud Gateway, Kong, nginx. Риск — не превращать в «умный» слой с бизнес-логикой (распределённый монолит).

### 6.12 gRPC vs REST, когда gRPC
gRPC: HTTP/2 + Protobuf (бинарный, контракт-first из `.proto`, кодогенерация), 4 режима включая двунаправленный стриминг, дедлайны из коробки. Быстрее и строже REST; минусы: не для браузера напрямую (grpc-web), сложнее отлаживать, курлом не потыкать. Когда: внутренняя высокочастотная связь сервис-сервис, стриминг, полиглотные команды. Наружу — REST, внутрь — gRPC/события.

### 6.13 🔵 SOAP vs REST
SOAP — протокол: XML-конверт (envelope/header/body), строгий контракт WSDL, транспорт-агностичен, WS-* стандарты (WS-Security, надёжная доставка), встроенные fault'ы. REST — стиль поверх HTTP, легче и проще. SOAP оправдан: банковские/государственные легаси-интеграции, жёсткие контракты и enterprise-требования. Контекст: госфинансы — там SOAP/XML ещё живы; для ЦФА-downstream может встретиться.

### 6.14 FeignClient vs RestTemplate vs WebClient
Feign (OpenFeign) — декларативный клиент: интерфейс + аннотации, реализацию генерирует Spring; меньше boilerplate, встроенная интеграция с LoadBalancer/Resilience4j, перехватчики. `RestTemplate` — императивный, в maintenance mode; `RestClient` — его современная замена (Boot 3.2+, fluent). `WebClient` — реактивный (WebFlux), неблокирующий, стриминг. Выбор: межсервисные типовые вызовы — Feign/RestClient; реактивный стек или высокая конкуренция IO — WebClient.

### 6.15 Логирование и мониторинг REST API
Логирование: фильтр/интерцептор (метод, путь, статус, длительность), correlation/traceId через MDC во все записи, тела — осторожно (PII/секреты маскировать). Метрики: Micrometer из коробки (`http.server.requests`: p95/p99, коды) → Prometheus/Grafana, алерты на error rate и латентность. Трейсинг: Micrometer Tracing/OpenTelemetry — путь запроса через сервисы. Опыт: Actuator+Prometheus на soak-тесте.

### 6.16 Сжатие в REST API
`server.compression.enabled=true` + mime-types/min-response-size; согласование через `Accept-Encoding`/`Content-Encoding` (gzip, brotli — лучше жмёт статику). Часто выносится на nginx/gateway. Trade-off CPU↔трафик; мелкие ответы не жать (min-size). Больший выигрыш иногда — пагинация и DTO-диета, а не компрессия.

### 6.17 🟣 OpenAPI/Swagger
*(уровень senior)*

OpenAPI — спецификация описания REST-контракта (yaml/json): endpoints, схемы, коды, security. springdoc-openapi генерирует из кода (+аннотации `@Operation`, `@Schema`) → Swagger UI для интерактивной документации. Подходы: code-first (генерация из кода) vs contract-first (spec → кодогенерация клиентов/серверных заглушек — лучше для межкомандных контрактов). Плюс: клиентские SDK, contract-тесты, ревью API до реализации.

### 6.18 Тестирование REST API
Слои: `@WebMvcTest` + `MockMvc` (контроллер+advice+валидация без сервера), `@SpringBootTest(webEnvironment=RANDOM_PORT)` + `TestRestTemplate`/`WebTestClient` (интеграционно), RestAssured (читаемый DSL), WireMock (заглушки внешних API), Testcontainers (реальная БД). Контрактные: Pact/Spring Cloud Contract. Нагрузочное: k6/Gatling/JMeter — опыт: k6 в Kubernetes, 8-часовой soak 50 VU / 2.68M запросов.

### 6.19 RPC vs REST
RPC — «вызов процедуры»: глаголы, плотная связность контрактов (gRPC, RMI, JSON-RPC); REST — ресурсы и единообразный интерфейс, слабее связность, кэшируемость HTTP. RPC — внутренние вызовы с жёсткими контрактами и требованиями к латентности; REST — публичные/универсальные API. Опыт: RMI/RPC на АЦК (интеграция с legacy), JSON-RPC с блокчейн-нодой.

### 6.20 Cookie vs Bearer Token
**Cookie:** браузер шлёт автоматически → уязвимость CSRF (нужны токены/SameSite), зато HttpOnly защищает от кражи через XSS; серверные сессии или JWT в куке. **Bearer** (Authorization header): CSRF не страшен (не прикрепляется автоматически), но хранение в localStorage уязвимо к XSS. Практика: SPA — httpOnly+Secure+SameSite cookie (или BFF-паттерн); межсервисно и мобильным — bearer.

### 6.21 Access и refresh token
**Access** — короткоживущий (минуты), в каждом запросе, компрометация ограничена TTL. **Refresh** — долгоживущий, хранится безопаснее, обменивается на новые access (+ротация refresh: старый гасится, повторное использование = сигнал кражи → отзыв семейства). Риски: access — перехват (mitigation: TTL, HTTPS), refresh — кража = долгий доступ (mitigation: ротация, привязка к клиенту, хранение httpOnly).

### 6.22 NGINX как reverse proxy
Принимает клиентские соединения и проксирует на upstream-бэкенды: TLS-терминация, отдача статики, кэширование, сжатие, буферизация медленных клиентов (защита потоков приложения), скрытие топологии, единая точка для WAF/rate limiting. `proxy_pass` + заголовки `X-Forwarded-For`/`Proto` (приложению — `server.forward-headers-strategy`).

### 6.23 Балансировка NGINX
upstream-методы: round-robin (дефолт), `least_conn` (наименьшее число соединений — для неравномерных запросов), `ip_hash` (стики по клиенту), weights (разные мощности), hash по ключу. Пассивные health-checks (`max_fails`/`fail_timeout`) в open source, активные — в Plus. Вместе с ретраями `proxy_next_upstream` — переживание падения инстанса.

### 6.24 HTTP/2 и HTTP/3 vs 1.1
**1.1:** текстовый, одно соединение — один запрос за раз (head-of-line blocking), keep-alive. **HTTP/2:** бинарные фреймы, мультиплексирование множества стримов в одном соединении, HPACK-сжатие заголовков; HOL остаётся на уровне TCP. **HTTP/3:** QUIC поверх UDP — устраняет TCP-HOL, быстрый handshake (0-RTT), миграция соединения при смене сети. NGINX: `http2 on; listen 443 quic;`.

### 6.25 Микросервисные паттерны + один подробно
Перечень: API Gateway, Service Discovery, Circuit Breaker, Retry/Timeout/Bulkhead, Saga, Transactional Outbox, Idempotent Consumer, CQRS, Event Sourcing, Sidecar, Strangler Fig. Подробно бери **Outbox** (коронный номер + профиль вакансии): проблема атомарности «БД-коммит + публикация события/в блокчейн» → запись намерения в outbox-таблицу в той же транзакции → отдельный воркер (поллинг с `FOR UPDATE SKIP LOCKED` или Debezium CDC) публикует и помечает → at-least-once + идемпотентный консюмер на другой стороне.

## Секция 7. Брокеры сообщений: Kafka, RabbitMQ, JMS (20)

### 7.1 Pub/sub vs point-to-point
**Point-to-point:** очередь, одно сообщение получает один консюмер (распределение работы). **Pub/sub:** топик, сообщение получают все подписчики (рассылка факта). В JMS — Queue vs Topic; в Kafka обе модели через consumer groups: внутри группы — point-to-point (партиции делятся), разные группы — pub/sub (каждая читает всё); в RabbitMQ — fanout/topic exchange на несколько очередей.

### 7.2 Kafka vs RabbitMQ: отличия и выбор
**Kafka** — распределённый лог: сообщения не удаляются при чтении (retention), replay, порядок в партиции, высокая пропускная способность, консюмер тянет по offset'у. **RabbitMQ** — умный брокер очередей: гибкая маршрутизация exchange'ами, per-message ack, приоритеты, TTL, push-модель. Kafka — потоки событий, event sourcing, интеграция многих потребителей, большие объёмы; RabbitMQ — задачи/команды, сложный роутинг, RPC-паттерны. Опыт: RabbitMQ в бою (AuctionPort), Kafka — Gauntlet.

### 7.3 Партиционирование в Kafka
Топик = N партиций — единиц параллелизма и порядка: порядок гарантирован только внутри партиции; ключ сообщения → hash → партиция (одинаковый ключ = одна партиция = порядок по сущности, например `orderId`). Масштабирование: партиций ≥ консюмеров в группе (лишние консюмеры простаивают). Подводное: увеличение числа партиций ломает соответствие ключ→партиция для старых данных; перекос по горячему ключу.

### 7.4 Уникальность сообщений / предотвращение дублей
Дубли неизбежны при at-least-once (ретраи продюсера, повторная доставка после ребаланса). Со стороны продюсера: `enable.idempotence=true` (producer id + sequence — дедуп ретраев брокером), транзакции для exactly-once внутри Kafka-пайплайна. Со стороны консюмера — идемпотентность: дедуп по бизнес-ключу/eventId (таблица `processed_events` + `INSERT ... ON CONFLICT DO NOTHING` в одной транзакции с эффектом) или естественно идемпотентные операции (upsert).

### 7.5 🟣 Offset в Kafka
Позиция сообщения в партиции (монотонный номер). Консюмер-группа хранит committed offset в `__consumer_offsets` — «докуда обработали»; при рестарте/ребалансе чтение продолжается с него. Автокоммит (`enable.auto.commit`, по интервалу) — риск потери/дублей при падении между обработкой и коммитом; ручной коммит после обработки — at-least-once (стандарт). `auto.offset.reset` (earliest/latest) — где начинать без сохранённого offset'а. Возможность перечитать (`seek`) — сила лога.

### 7.6 Механизмы маршрутизации RabbitMQ
Producer шлёт в exchange с routing key; binding'и связывают exchange с очередями. Типы: **direct** (точное совпадение ключа), **topic** (маски: `orders.*.paid`, `orders.#`), **fanout** (во все привязанные очереди, ключ игнорируется), **headers** (по заголовкам). Гибкость роутинга — главный козырь Rabbit перед Kafka.

### 7.7 🟣 Durable Queue и сохранность при сбоях
*(уровень senior)*

Три независимых флага, нужны все: durable queue (метаданные очереди переживают рестарт брокера), persistent message (`deliveryMode=2` — сообщение пишется на диск), durable exchange. Плюс publisher confirms — гарантия, что брокер принял/записал. Классическая ошибка — durable очередь с transient-сообщениями: очередь выживет, сообщения нет. В кластере — quorum queues (Raft-репликация) вместо устаревших mirrored. Полная цепочка гарантий: confirms → durable+persistent+quorum → consumer ack после обработки.

### 7.8 Acknowledgements в RabbitMQ и Kafka, проблемы настройки
**RabbitMQ:** consumer ack/nack (с requeue или в DLX) после обработки; `autoAck=true` — потеря при падении обработчика; забытый ack — сообщения копятся unacked до prefetch-лимита и «зависает» очередь. **Kafka:** у продюсера `acks=0/1/all` (`all` + `min.insync.replicas` — без потерь при падении брокера); у консюмера «ack» = коммит offset'а: коммит до обработки — потеря, после — дубли (нужна идемпотентность).

### 7.9 Порядок доставки и семантики (at-most/at-least/exactly-once)
Порядок в Kafka — только внутри партиции: ключ = сущность (`orderId`); `max.in.flight` >1 без idempotence может переупорядочить ретраи. Семантики: at-most-once (коммит до обработки — можно терять), at-least-once (коммит после — дубли, стандарт + идемпотентность), exactly-once — только внутри Kafka-экосистемы (транзакции: producer `transactional.id` + `isolation.level=read_committed`, паттерн consume-process-produce); с внешней БД честного exactly-once нет — есть at-least-once + идемпотентность.

### 7.10 Dead Letter Queue в RabbitMQ
Очередь для «мёртвых» сообщений: nack без requeue, TTL истёк, переполнение очереди → через `x-dead-letter-exchange` уходит в DLX/DLQ. Зачем: poison message не блокирует обработку и не крутится вечно; DLQ разбирается отдельно (алерт, ручной replay после фикса). В Kafka аналог — Dead Letter Topic (`DeadLetterPublishingRecoverer` в Spring Kafka после исчерпания ретраев).

### 7.11 Проблемы брокеров в распределённых системах
Дубли (→ идемпотентность), нарушение порядка (→ ключи партиционирования), poison messages (→ retry с backoff + DLQ), lag консюмеров (→ мониторинг, масштабирование), ребалансы (→ пауза обработки, дубли; cooperative rebalancing), backpressure, потеря при неправильных acks, схемы данных (→ Schema Registry, совместимость), «брокер = единая точка» (→ кластер, репликация).

### 7.12 Spring Kafka и Spring AMQP
**Kafka:** `KafkaTemplate.send`, `@KafkaListener(topics, groupId)`, `ConcurrentKafkaListenerContainerFactory` (конкаренси), `DefaultErrorHandler` (backoff-ретраи) + `DeadLetterPublishingRecoverer`, `JsonSerializer`/`Deserializer` (typed listener). **AMQP:** `RabbitTemplate`, `@RabbitListener(queues)`, декларация топологии бинами (Queue/Exchange/Binding), `MessageConverter` (Jackson), retry-interceptor + DLX. Общее: контейнер слушателей поверх клиента, маппинг в POJO, обработка ошибок конфигурацией, а не try/catch в листенере.

### 7.13 🔵 Consumer groups в Kafka
Группа = логический подписчик: партиции топика распределяются между консюмерами группы (партиция — ровно одному консюмеру группы в момент времени) → горизонтальное масштабирование чтения. Разные группы независимы (каждая со своими offset'ами) — fan-out. Ребаланс при изменении состава: партиции переезжают, обработка приостанавливается (cooperative протокол смягчает). Практика: консюмеров ≤ партиций; `groupId` — идентичность сервиса-подписчика.

### 7.14 Транзакционность в Kafka и RabbitMQ
**Kafka:** транзакции продюсера — атомарная запись в несколько партиций/топиков + коммит offset'ов в той же транзакции (consume-process-produce = exactly-once в пайплайне); `transactional.id`, консюмеры с `read_committed`. **RabbitMQ:** channel tx (`txSelect`/`txCommit`) — медленно, вместо них publisher confirms (асинхронное подтверждение записи). «Транзакция брокер+БД» — не бывает двухфазно: outbox/inbox паттерны.

### 7.15 Мониторинг и метрики брокеров
Ключевая метрика консюмеров — **LAG** (committed offset vs log end): растёт → не успеваем. Инструменты: JMX-метрики Kafka → kafka-exporter/JMX-exporter → Prometheus/Grafana; Burrow для lag; UI (AKHQ, Kafka UI, RabbitMQ management). Брокер: under-replicated partitions, ISR shrink, disk; RabbitMQ: глубина очередей, unacked, connections. Алерты: lag, глубина DLQ > 0, under-replicated > 0.

### 7.16 🔥 Retention в Kafka
Сообщения удаляются не при чтении, а по политике: `retention.ms` (время, дефолт 7 дней) и/или `retention.bytes` (размер) — сегменты лога удаляются целиком (`log.segment.bytes`). `cleanup.policy=delete` (обычные потоки) vs `compact` (лог-компакция: хранится последнее значение по ключу — снапшот состояния; `__consumer_offsets` так живёт) или `delete,compact`. Выбор retention = баланс replay-окна и диска; consumer должен успеть вычитать до удаления.

### 7.17 Backpressure и message flooding
Продюсеры пишут быстрее, чем читают консюмеры. **Kafka:** буфер — сам лог (lag растёт, но брокер держит), решения — масштабировать консюмеров/партиции, батчить обработку, pause/resume контейнера, квоты продюсеров. **RabbitMQ:** очереди раздуваются в память/диск → prefetch (QoS) ограничивает поток на консюмера, flow control тормозит продюсеров, lazy/quorum queues на диск, TTL+DLX на переполнение. Общее: лимитировать на входе (rate limiting), деградировать изящно.

### 7.18 Балансировка нагрузки RabbitMQ
Кластер узлов; клиентские подключения балансируются HAProxy/LB или списком хостов в клиенте. Нюанс: очередь живёт на своём узле (лидере) — подключение к другому узлу проксируется внутри кластера; quorum queues реплицируются (Raft), выдерживают падение узла. Потребление балансируется несколькими консюмерами одной очереди + prefetch. Federation/shovel — между кластерами/ДЦ.

### 7.19 Распределённые транзакции с брокерами: сага и 2PC
2PC (XA) между брокером и БД — тяжело и хрупко, в современном стеке избегают (Kafka XA не поддерживает). Практика: транзакционный outbox (атомарность БД+событие) + сага (последовательность локальных транзакций с компенсациями, события как транспорт) + идемпотентный консюмер (inbox). Оркестрация саги для денежных флоу — явный state machine.

### 7.20 Шифрование и аутентификация в Kafka/RabbitMQ
**Kafka:** TLS на транспорт (encryption in transit), аутентификация SASL (SCRAM-SHA-512 логин/пароль, mTLS сертификатами, OAUTHBEARER, в энтерпрайзе Kerberos/GSSAPI), авторизация ACL (кто в какие топики пишет/читает; listener'ы SASL_SSL). **RabbitMQ:** TLS, пользователи/пароли или x509, vhost + permissions (configure/write/read). Данные at rest Kafka сам не шифрует — диск/приложение. Для банка упомяни: секреты в Vault, ротация, принцип наименьших привилегий на топик.

## Секция 8. Многопоточность, GC, JVM (24)

### 8.1 Immutable: как сделать и зачем
`final class`, все поля `private final`, никаких сеттеров, защитные копии мутабельных полей на входе и выходе (или `List.copyOf`), `this` не утекает из конструктора. Зачем: потокобезопасность без синхронизации (иммутабельный объект можно свободно шарить — JMM гарантирует видимость final-полей после конструктора), безопасные ключи `Map`, простота рассуждений. Примеры: `String`, `BigDecimal`, records.

### 8.2 wait() — как работает и почему аккуратно
Вызывается только владея монитором (внутри `synchronized` на том же объекте, иначе `IllegalMonitorStateException`); отпускает монитор и паркует поток до `notify`/`notifyAll`/таймаута/interrupt. Аккуратность: проверять условие в цикле `while` (spurious wakeups + условие могло измениться), `notify` может потеряться (если `wait` ещё не вызван) — поэтому состояние + проверка, предпочитать `notifyAll`. Современный код — `BlockingQueue`/`Condition`/`CountDownLatch` вместо ручного wait/notify.

### 8.3 Поток vs процесс; создание и запуск
Процесс — изолированное адресное пространство; потоки процесса делят heap (общие объекты — отсюда и гонки), у каждого свой стек и program counter. Создание: `Thread` + `Runnable` (`new Thread(task).start()` — именно `start`, не `run`!), пулы `ExecutorService` (правильный путь), с Java 21 — виртуальные потоки (`Thread.ofVirtual`, дешёвые, для IO-bound).

### 8.4 volatile
Гарантия видимости: запись видна всем последующим чтениям (значение не кэшируется в регистрах/локально), запрет переупорядочивания вокруг (happens-before: запись → чтение). **Не атомарность:** `volatile int counter++` всё равно гонка (read-modify-write). Применение: флаги остановки, публикация ссылки на иммутабельный объект, DCL-синглтон.

### 8.5 Happens-before и за счёт чего достигается
Отношение JMM: если A happens-before B, эффекты A видимы B. Источники: program order внутри потока; `unlock` → `lock` того же монитора; volatile write → read; `Thread.start()` → действия потока; действия потока → `join()`; финализация final-полей в конструкторе. Механика: компилятор/JIT/CPU не переупорядочивают через эти границы + барьеры памяти/когерентность кэшей. Без happens-before между потоками — никаких гарантий видимости вообще.

### 8.6 wait()/notify() — где целесообразно
Низкоуровневая координация «жди условия на общем объекте»: ручные producer-consumer, ожидание завершения инициализации. Сегодня целесообразно редко — `java.util.concurrent` покрывает: `BlockingQueue` (producer-consumer), `CountDownLatch` (дождаться N событий), `Condition` (несколько наборов ожидания на одном локе). На собесе умей написать ping-pong на wait/notify (задача K3 из дока задач).

### 8.7 sleep() vs wait()
`sleep` — static `Thread`, усыпляет текущий поток на время, монитор **не** отпускает, монитор не нужен. `wait` — метод `Object`, требует владения монитором и **отпускает** его, будится notify/timeout. `sleep` — пауза/троттлинг; `wait` — ожидание условия.

### 8.8 Как избежать deadlock
Дедлок = циклическое ожидание ресурсов. Профилактика: глобальный порядок захвата локов (перевод денег: лок по `min(id)`, потом `max(id)`); не держать несколько локов; `tryLock` с таймаутом и откатом; уменьшать область — конкурентные структуры вместо локов; lock-free. Диагностика: thread dump (`jstack`) — JVM сама помечает найденный дедлок.

### 8.9 ReentrantLock
Явный лок (AQS внутри): реентерабельный (поток может брать повторно, счётчик), `lock`/`unlock` — `unlock` строго в finally. Плюсы над `synchronized`: `tryLock(timeout)` — избегание дедлока, `lockInterruptibly`, честный режим (FIFO), несколько `Condition` на один лок. `synchronized` проще и с Java 6+ не медленнее — `ReentrantLock`, когда нужны его фичи.

### 8.10 Проблемы Executors-фабрик
`newFixedThreadPool`/`newSingleThreadExecutor` — **неограниченная** `LinkedBlockingQueue`: при отстающих потребителях очередь растёт до OOM. `newCachedThreadPool` — неограниченное число потоков: всплеск нагрузки = тысячи потоков, исчерпание ресурсов. `newScheduledThreadPool` — исключение в периодической задаче **отменяет** все последующие запуски молча (обязателен try/catch внутри задачи). Правильно: явный `ThreadPoolExecutor` (core/max, ограниченная очередь, `RejectedExecutionHandler`, именованная `ThreadFactory`).

### 8.11 Race condition
Результат зависит от порядка выполнения потоков: неатомарные read-modify-write (`counter++`), check-then-act (`containsKey`→`put`, проверка баланса→списание). Избежать: атомики/CAS, `synchronized`/локи, атомарные методы конкурентных коллекций (`computeIfAbsent`, `putIfAbsent`), иммутабельность, confinement (данные не шарить). В БД тот же класс проблем — блокировки/атомарные `UPDATE`.

### 8.12 Fork-Join Framework
Пул для рекурсивно делимых CPU-bound задач: work stealing — свободные воркеры крадут подзадачи из деков занятых. `RecursiveTask`/`RecursiveAction`: `fork()` подзадач + `join()`. Используется под капотом parallel streams и `CompletableFuture` (commonPool). Когда: большие делимые вычисления; **не** для блокирующего IO (задушит общий пул — parallel stream с IO это антипаттерн).

### 8.13 ThreadLocal: использование и проблемы
Переменная со значением на поток: контексты без протаскивания параметров (`SecurityContext`, транзакция, MDC). Проблемы: в **пулах** поток живёт вечно → значение остаётся между задачами (утечка памяти + утечка данных между запросами!) — `remove()` в finally; не передаётся в дочерние задачи/пулы автоматически.

### 8.14 Видит ли дочерний поток ThreadLocal родителя
Обычный `ThreadLocal` — нет. `InheritableThreadLocal` — да: значение копируется в момент создания дочернего потока (дальнейшие изменения родителя не видны). В пулах бесполезен (потоки созданы заранее) — поэтому контексты (MDC, security) в пулы передают явно: обёртки задач, `TaskDecorator`, micrometer context-propagation.

### 8.15 🔥 Типы экзекьюторов и их проблемы
- **SingleThreadExecutor** — последовательность + порядок; проблема: неограниченная очередь (OOM), одна застрявшая задача стопорит всё.
- **FixedThreadPool** — фиксированный параллелизм; та же неограниченная очередь.
- **CachedThreadPool** — потоки под нагрузку, idle 60с умирают; проблема: неограниченное число потоков.
- **ScheduledExecutor** — отложенные/периодические; проблема: молча отменяет периодику после исключения + `fixedRate` копит запуски при долгой задаче.

Итог для senior: в проде — явный `ThreadPoolExecutor` с ограниченной очередью и политикой отказа, метрики пула.

### 8.16 submit() vs execute()
`execute(Runnable)` — void: исключение улетает в `UncaughtExceptionHandler` потока (в логах видно). `submit` — возвращает `Future`: исключение запаковано в `Future` и всплывёт только при `get()` (`ExecutionException`); если `get()` не зовут — задача «падает молча». Плюс `submit` принимает `Callable` (результат).

### 8.17 Безопасная публикация объектов
Проблема: другой поток может увидеть ссылку на объект до завершения его конструирования (переупорядочивание) или устаревшие поля. Безопасные способы: static-инициализатор/holder, volatile-поле, final-поля (гарантия JMM после конструктора), передача через конкурентные структуры (`BlockingQueue`, `ConcurrentHashMap`), под синхронизацией. Незащищённая публикация через обычное поле — источник «невозможных» багов (DCL без volatile).

### 8.18 RecursiveTask и RecursiveAction
Задачи `ForkJoinPool`: `RecursiveTask<V>` возвращает результат (сумма отрезка, merge sort), `RecursiveAction` — void (параллельная обработка in-place). Паттерн `compute()`: если мало — считаем; иначе split, `left.fork()`, `right.compute()`, `left.join()`.

### 8.19 Атомарность long и double
JLS разрешает **не**-атомарную запись 64-битных значений без `volatile` (две 32-битные половины) → теоретически «порванное» чтение на 32-битных JVM. Гарантия атомарности: `volatile long`/`double`, `AtomicLong`, синхронизация. На практике 64-битные JVM пишут атомарно, но код по спецификации требует `volatile`. Плюс не путать: атомарность записи ≠ атомарность инкремента.

### 8.20 Phaser vs CyclicBarrier
`CyclicBarrier` — фиксированное число участников, все ждут друг друга на барьере, переиспользуемый (+barrierAction). `Phaser` — динамические участники (`register`/`arriveAndDeregister`), многофазность (`arriveAndAwaitAdvance` по фазам), иерархия фазеров. `Phaser` — когда число потоков меняется по ходу; `CountDownLatch` — одноразовое «дождаться N».

### 8.21 🟣 Блокировки и lock-free алгоритмы
Блокировки: мониторы (`synchronized`: biased→thin→fat эволюция; сейчас thin/inflated), явные локи на AQS (очередь ожидающих потоков + CAS на state). Lock-free — прогресс системы гарантирован без блокировок, на CAS-циклах: атомики, `ConcurrentLinkedQueue` (алгоритм Майкла-Скотта), стек Трейбера. Wait-free — прогресс каждого потока (сильнее, редко). Плюсы: нет дедлоков/инверсии приоритетов; минусы: сложность, contention-спины, ABA. Пример написать — incrementAndGet-цикл (см. ниже).

### 8.22 Атомарные операции, классы, AtomicInteger
`java.util.concurrent.atomic`: `AtomicInteger`/`Long`/`Boolean`/`Reference` (+`FieldUpdater`, `LongAdder`). Внутри — CAS (compare-and-swap, процессорная инструкция) через `VarHandle`/`Unsafe`:

```java
public final int incrementAndGet() {
    int prev, next;
    do {
        prev = get();                    // volatile read
        next = prev + 1;
    } while (!compareAndSet(prev, next)); // CAS: retry при проигрыше гонки
    return next;
}
```

ABA-проблема (A→B→A незаметно для CAS) → `AtomicStampedReference`. Высокий contention на счётчике → `LongAdder` (страйпинг ячеек, сумма при чтении).

### 8.23 Мониторинг JVM
Метрики в проде: Micrometer/JMX → Prometheus/Grafana — heap по поколениям, GC-паузы и частота, потоки (число, состояния), CPU, пулы, classloading. Точечно: `jcmd` (универсальный: `GC.heap_info`, `Thread.print`, `VM.flags`), `jstat -gc` (динамика), JFR (Java Flight Recorder — низкооверхедная запись событий JVM) + Mission Control для анализа. Алерты: рост old gen после full GC (утечка), длинные паузы, thread count.

### 8.24 Профилирование
Поиск, куда уходят CPU/память/время: sampling (периодические снимки стеков, низкий оверхед — можно в проде) vs instrumentation (точнее, дороже). Инструменты: async-profiler (CPU/alloc/lock, flame graphs), JFR+JMC, VisualVM (dev). Порядок: метрики показывают что плохо → профайлер показывает где → чинишь горячее место, а не гадаешь. Аллокационный профиль часто важнее CPU (давление на GC).

### 8.25 GC — все сборщики
*(доп. вопрос «кроме G1» уже задавали Мише)*

Полная таблица сборщиков — во внешнем документе (`interview_ti1_ibs_structured.md § 8.2`), здесь только конспект: Serial, Parallel (throughput, дефолт J8), CMS (deprecated 9, удалён 14), G1 (дефолт с 9: регионы, цель по паузе), ZGC (паузы <1 мс, colored pointers, generational с 21), Shenandoah, Epsilon + generational hypothesis, minor/major/full, safepoints и фраза «как выбрать».

## Секция 9. Тестирование (20)

### 9.1 🟣 Unit-тестирование и принципы
*(уровень senior)*

Тест одного модуля в изоляции от внешних зависимостей (они мокаются). Принципы **FIRST**: Fast, Isolated (независимы друг от друга и порядка), Repeatable (детерминированы: без реального времени/сети — `Clock` мокается), Self-validating (assert, не глазами), Timely. Структура given/when/then, один сценарий — один тест, имя описывает поведение: `updateStatus_shouldThrow_whenTransitionNotAllowed`. Тестируем поведение через публичный контракт, не приватную реализацию.

### 9.2 🟣 Unit vs Integration: когда что
**Unit** — логика класса/метода, миллисекунды, моки границ: вся бизнес-логика, ветвления, edge cases (основание пирамиды). **Integration** — взаимодействие компонентов: репозитории на реальной БД (Testcontainers), контроллер+advice (`@WebMvcTest`), связка с Kafka/Rabbit. Правило: логику покрывай unit'ами (дёшево), интеграционные — на швы, где unit врёт: SQL, маппинги, сериализация, конфигурация, транзакции.

### 9.3 JUnit 5: как работает, нововведения
Архитектура: Platform (запуск) + Jupiter (API) + Vintage (JUnit 4). Нововведения против 4: `@BeforeEach`/`@AfterEach`/`@BeforeAll`, `@DisplayName`, `@Nested` (группировка), `@ParameterizedTest` (`@ValueSource`/`@CsvSource`/`@MethodSource`/`@EnumSource`), `assertThrows`/`assertAll`, `@Tag`, Extension model вместо Runner'ов (`@ExtendWith(MockitoExtension.class)`), условное выполнение (`@EnabledOnOs`), лямбда-сообщения в assert.

### 9.4 🔥 Testcontainers
Библиотека, поднимающая реальные зависимости в Docker из теста: PostgreSQL, Kafka, Redis; контейнер живёт время теста/класса. Когда оправданы: интеграционные тесты репозиториев и SQL (H2 ≠ PostgreSQL: диалект, типы jsonb, `DISTINCT ON`, блокировки), Kafka-листенеры, миграции Liquibase/Flyway. Подключение: `@Testcontainers` + `@Container` + `@ServiceConnection` (Boot 3.1+, конфиг сам) или `@DynamicPropertySource`. Ускорение: static-контейнер на класс/сьют, reuse. Trade-off: медленнее unit — держать на уровне швов, не для каждой мелочи.

### 9.5 Мокирование: Mockito, PowerMock
`Mockito`: `mock()` (все методы — заглушки), `spy()` (обёртка над реальным объектом, перехват точечно), `when`/`thenReturn`, `verify` (взаимодействия), `ArgumentCaptor` (перехват аргументов), `@Mock`/`@InjectMocks` + `MockitoExtension`. `PowerMock` — легаси для static/private/final/конструкторов: нужда в нём — маркер плохого дизайна; современный Mockito сам умеет `mockStatic` (mockito-inline) и final. Правило: мокать свои границы (репозитории, клиенты, `Clock`), не мокать value-объекты и тестируемый класс.

### 9.6 Stub vs mock
**Stub** — заранее заданные ответы, проверяем состояние/результат (state-based): репозиторий возвращает фиксированный список. **Mock** — проверяем взаимодействие (behavior-based): `verify(notificationService).notifyUser(...)` — что вызвано, с чем, сколько раз. В Mockito один объект играет обе роли (`when` = stubbing, `verify` = mocking). Перебор `verify` на каждый чих — хрупкие тесты, привязанные к реализации.

### 9.7 @MockBean vs @Mock
`@Mock` — чистый Mockito, без Spring: объект создаётся и внедряется через `@InjectMocks`, для unit-тестов. `@MockBean` — Spring Boot: **заменяет** бин в `ApplicationContext` моком — для slice/интеграционных тестов, где контекст поднимается (`@WebMvcTest` + `@MockBean` сервиса). Нюансы: каждый уникальный набор `@MockBean` = новый контекст (медленно, кэш контекстов мимо); с Boot 3.4 `@MockBean` deprecated в пользу `@MockitoBean` (то же самое, новый пакет).

### 9.8 Фикстуры (test fixtures)
Фиксированное окружение/данные теста: `@BeforeEach`-подготовка, builder'ы тестовых объектов (`OrderTestBuilder.paid().withItems(2).build()`), Object Mother, `@Sql`-скрипты для БД. Ценность: убирают дублирование setup'а, тесты читаются как сценарий; свежая фикстура на тест = изоляция. Анти-паттерн: гигантская общая фикстура на все случаи — тесты становятся связанными.

### 9.9 E2E vs Unit vs Integration
E2E — весь путь как у пользователя: реальное окружение, API-вызовы → БД → брокеры, максимум уверенности и максимум цены (медленные, хрупкие, дорогая диагностика). Пирамида: много unit → меньше integration → единицы E2E на критические флоу (создание заказа, оплата). E2E не заменяет нижние уровни — ловит конфигурацию и связку, а не логику.

### 9.10 Boundary testing
Ошибки живут на границах: min/max, 0/1/N, пустая строка/коллекция, `null`, граница диапазона включительно/исключительно, переполнение, граничные даты. Пример из их же домена: рейтинг 0..5 — тестируем -1, 0, 5, 6; переходы статусов — каждый допустимый и хотя бы один запрещённый; пагинация — page=0, последняя страница, за пределами. Инструмент — `@ParameterizedTest` с граничными значениями.

### 9.11 JUnit vs TestNG vs Spock
JUnit 5 — стандарт де-факто Java, вся экосистема Spring на нём. TestNG — исторически богаче (группы, зависимости тестов, параллельность из коробки) — встречается в legacy/QA-автоматизации. Spock — Groovy: given/when/then прямо в синтаксисе, мощные data tables (`where:`), встроенное мокирование — очень читаемо, цена — Groovy в стеке. Честный ответ: боевой опыт — JUnit 5 + Mockito + AssertJ.

### 9.12 Тестирование асинхронного кода
Проблема: assert срабатывает раньше результата. Подходы: Awaitility — `await().atMost(5, SECONDS).untilAsserted(...)` (поллинг условия вместо sleep — sleep в тестах это флаки и медленно); `CompletableFuture` — `join()`/`get(timeout)` и проверка результата; для `@Async`/листенеров — синхронный executor в тестовом профиле (выполнить сразу) либо `verify` с `timeout()`: `verify(sender, timeout(1000)).send(any())`; виртуальное время в реактивных стеках (`StepVerifier`).

### 9.13 Тестирование взаимодействия с внешними сервисами (REST, Kafka)
REST-клиенты: WireMock — реальный HTTP-сервер с заглушками (стабы ответов, задержки, ошибки 5xx — проверка ретраев/CB), `@RestClientTest` + `MockRestServiceServer` для `RestTemplate`/`RestClient`. Kafka: Testcontainers Kafka (или EmbeddedKafka) — реальный брокер, продюсер шлёт, тест ждёт через Awaitility. Принцип: юнитом мокаешь клиент как интерфейс; интеграционно — стабишь провод (HTTP/брокер), а не код.

### 9.14 🔵 TDD и BDD
**TDD:** цикл red (падающий тест на требование) → green (минимальный код) → refactor; тесты как спецификация, дизайн через тестируемость, высокая уверенность при рефакторинге. Честный ответ про практику: чистый TDD — не всегда; test-first для багфиксов (тест, воспроизводящий баг, до фикса) — обязательная привычка. **BDD:** поведение бизнес-языком Given-When-Then (Cucumber/Gherkin как инструмент, но суть — в формулировке сценариев из требований); в коде — то же given/when/then в структуре тестов.

### 9.15 H2 / встроенная БД в Spring Boot
H2 in-memory: зависимость + `spring.datasource.url=jdbc:h2:mem:test` (или автоконфигурация в тестах), `MODE=PostgreSQL` для частичной совместимости. `@DataJpaTest` по умолчанию подменяет `DataSource` на embedded. Честная позиция: H2 годится для быстрых тестов простого CRUD; для нативных запросов, диалектных фич и миграций — врёт, поэтому продовые интеграционные — на Testcontainers с реальным PostgreSQL (`@AutoConfigureTestDatabase(replace = NONE)`).

### 9.16 Test data
Данные под сценарий, а не «дамп прода»: builder'ы/Object Mother с осмысленными дефолтами (меняешь только значимое для теста поле), `@Sql` для реляционных предусловий, Instancio/EasyRandom для массовой генерации, фабрики в `@TestConfiguration`. Правила: минимум данных на тест, говорящие значения (сумма 100.00, а не 42.37 без смысла), никаких зависимостей от данных чужих тестов; PII в тестах — синтетика.

### 9.17 In-memory БД vs Testcontainers
In-memory (H2): мгновенно, без Docker; но другой движок — другой SQL-диалект, типы, блокировки, планы; нативные запросы и миграции не проверяются честно. Testcontainers: реальный PostgreSQL — тестируешь то, что поедет в прод; цена — Docker и секунды на старт (амортизируется reuse/static). Правило выбора: логика сервисов — вообще без БД (моки); слой данных — Testcontainers; H2 — только если Docker недоступен и SQL тривиальный.

### 9.18 Contract testing
Проверка контракта между потребителем и провайдером API без совместного E2E: consumer-driven (Pact) — консюмер фиксирует ожидания (запрос→ответ) в pact-файле, провайдер проверяет себя против него в своём CI; Spring Cloud Contract — контракты у провайдера, генерируются тесты провайдера + стабы для консюмеров. Ловит ломающие изменения API до деплоя; дешевле и стабильнее интеграционных стендов.

### 9.19 Изоляция тестов: @DirtiesContext, @Transactional
Тесты не должны влиять друг на друга: общий `ApplicationContext` кэшируется — грязное состояние (изменённые бины, данные в БД) утекает между тестами. `@Transactional` на тесте — откат транзакции после каждого теста (дёшево); ловушки: `REQUIRES_NEW` и явные коммиты не откатятся, поведение отличается от прода (не видно проблем flush/commit), с `@KafkaListener`/async — бесполезен. `@DirtiesContext` — пересоздать контекст (надёжно, но очень дорого — крайняя мера). Практика: транзакционный откат или явная очистка (truncate) в `@AfterEach` + независимые данные (уникальные ключи на тест).

### 9.20 🟣 Покрытие кода: Jacoco
Jacoco инструментирует байткод, метрики line/branch coverage, отчёт HTML/XML, quality gate в CI (jacoco-maven-plugin check, интеграция с SonarQube). Здравая позиция: покрытие — индикатор дыр, а не цель; 100% не гарантирует корректность (покрыто ≠ проверено assert'ами); важнее branch coverage бизнес-логики, чем line coverage геттеров. Дополнение для глубины: mutation testing (PIT) — проверяет, что тесты ловят искусственные мутации кода.

## Секция 10. Прочее: CI/CD, безопасность, UNIX и т.д. (30)

### 10.1 CI/CD и инструменты
CI — каждое изменение автоматически собирается и проверяется (build → unit/integration тесты → статанализ → артефакт/образ); CD — автоматическая доставка до стейджа/прода (continuous delivery — кнопка, deployment — без кнопки). Инструменты: GitHub Actions, GitLab CI, Jenkins; деплой — ArgoCD (GitOps), Helm. Стек: GitHub Actions + ArgoCD.

### 10.2 Доставка кода от разработчика до продакшена (свой проект)
Готовый рассказ (AuctionPort): feature-ветка → PR с ревью + прогон CI (сборка, тесты, статанализ) → merge в main → GitHub Actions собирает Docker-образ (multi-stage), тегает, пушит в registry → обновление манифестов/Helm values в git-репо конфигурации → ArgoCD видит diff и синхронизирует Kubernetes (GitOps: желаемое состояние в git, деплой через PR, откат через revert) → rolling update с readiness-пробами, миграции Liquibase на старте → мониторинг Actuator/Prometheus.

### 10.3 Maven/Gradle: механизм сборки
**Maven:** декларативный `pom.xml`, lifecycle-фазы `validate` → `compile` → `test` → `package` → `verify` → `install` → `deploy` (вызов фазы тянет предыдущие), плагины на фазах, зависимости с транзитивностью и scope (compile/provided/runtime/test), `dependencyManagement`/BOM для версий, конфликты — nearest wins (`mvn dependency:tree`). **Gradle:** Groovy/Kotlin DSL, граф задач, инкрементальность и кэш — быстрее на больших проектах.

### 10.4 Git из командной строки: частые команды
`status`, `add -p` (по кускам), `commit`, `pull --rebase`, `push`, `checkout`/`switch`, `branch`, `merge`, `rebase -i` (squash/reword), `stash`, `log --oneline --graph`, `diff`, `cherry-pick`, `reset`, `revert`, `bisect` (поиск коммита с багом). Честно: IDE для просмотра, консоль для операций.

### 10.5 Squash и rebase
**Squash** — склейка серии коммитов в один (`rebase -i` / squash-merge PR): чистая история «фича = коммит». **Rebase** — перенос коммитов на новое основание: линейная история без merge-коммитов; интерактивный — редактирование истории. Золотое правило: не ребейзить опубликованные общие ветки (переписывание истории ломает коллег); свою фичу до PR — можно и нужно.

### 10.6 git branch -f master HEAD~1 и git reset --soft HEAD~1
`git branch -f master HEAD~1` — принудительно передвигает указатель ветки master на коммит назад, **не** трогая текущую ветку и рабочую директорию (нельзя для checked-out ветки). `git reset --soft HEAD~1` — сдвигает **текущую** ветку на коммит назад, изменения последнего коммита остаются в staged (index) — типично «переделать последний коммит». (`--mixed` — изменения в working dir unstaged, `--hard` — изменения уничтожены.)

### 10.7 LF vs CRLF
Окончания строк: LF (`\n`) — Unix/macOS, CRLF (`\r\n`) — Windows. Проблемы: «изменён весь файл» в диффах, битые шелл-скрипты (bash не понимает `\r`), хеши/подписи. Решение: `.gitattributes` (`* text=auto eol=lf`) — принудительный LF в репозитории; `core.autocrlf=input` на Windows; IDE — LF по умолчанию.

### 10.8 Безопасная остановка Spring Boot
Graceful shutdown: `server.shutdown=graceful` + `spring.lifecycle.timeout-per-shutdown-phase=30s` — по SIGTERM перестаёт принимать новые запросы, дорабатывает активные, останавливает пулы, `@PreDestroy`-хуки. В Kubernetes: SIGTERM → preStop-хук (пауза, чтобы LB убрал под из endpoints) → `terminationGracePeriodSeconds` > таймаута приложения; `kill -9` — только если завис (без доработки запросов). Actuator `/shutdown` — выключен по умолчанию, в проде не открывать.

### 10.9 TCP vs UDP
**TCP:** соединение (handshake), гарантия доставки и порядка, ретрансмиссии, flow/congestion control — HTTP, БД, брокеры. **UDP:** датаграммы без соединения и гарантий — минимальная латентность: DNS, стриминг, игры, метрики (statsd). Современный мостик: HTTP/3 = QUIC поверх UDP — надёжность реализована в user space, быстрее устанавливает соединение.

### 10.10 🟣 Docker-контейнер vs виртуальная машина
Контейнер — изоляция процессов средствами ядра хоста: namespaces (pid, net, mount — своя видимость мира) + cgroups (лимиты CPU/памяти); ядро общее, стартует за миллисекунды, образ — мегабайты. VM — гипервизор эмулирует железо, внутри полная ОС со своим ядром: сильнее изоляция, дороже (гигабайты, минуты старта). Контейнеры — плотность и скорость доставки; VM — граница безопасности/мультитенантность (и то, на чём крутятся ноды Kubernetes).

### 10.11 Частые Docker-команды
`docker build -t app:tag .`, `run` (`-d`, `-p 8080:8080`, `-e`, `-v`, `--name`), `ps` (`-a`), `logs -f`, `exec -it <c> sh`, `images`, `pull`/`push`, `stop`/`rm`/`rmi`, `inspect`, `stats`, `system prune`; compose: `docker compose up -d` / `down` / `logs -f`. Плюс диагностика: `docker exec` в контейнер посмотреть процессы/конфиг.

### 10.12 🟣 Dockerfile и Docker Compose
*(уровень senior)*

Dockerfile — рецепт образа: каждая инструкция = слой с кэшем, порядок критичен (зависимости раньше кода), multi-stage (build на JDK+Maven → runtime на JRE: меньше образ и поверхность атаки), exec-форма `ENTRYPOINT` (сигналы доходят до JVM → graceful shutdown), non-root `USER`, JVM-флаги под контейнер (`-XX:MaxRAMPercentage`). Compose — декларация многоконтейнерного окружения (app + postgres + kafka): сервисы, сети (имена как DNS), volumes, `depends_on` + healthcheck; локальная разработка и интеграционные стенды.

*Полный пример Dockerfile — во внешнем документе (`interview_ti1_ibs_structured.md § 10.1`).*

### 10.13 Kubernetes Pod
Минимальная деплой-единица: один или несколько контейнеров с общими сетью (localhost между ними, один IP) и томами; sidecar-паттерн (прокси, лог-агент рядом с приложением). Поды эфемерны — создаются/умирают, напрямую их не создают: управляют Deployment'ы, стабильный доступ — через Service.

### 10.14 🟣 ReplicaSet
Контроллер, поддерживающий заданное число реплик пода: под упал/нода умерла → создаёт новый (reconciliation loop: desired vs actual state). Напрямую не используют — Deployment управляет ReplicaSet'ами: rolling update = новый RS постепенно масштабируется вверх, старый вниз; rollback = откат на предыдущий RS.

### 10.15 Ingress
Маршрутизация внешнего HTTP(S)-трафика к Service'ам: правила host/path (`api.example.com/orders` → `order-service`), TLS-терминация. Работает через ingress-контроллер (nginx-ingress, traefik) — сам ресурс лишь декларация. Аннотации контроллера: таймауты, лимиты тела, rewrite. Альтернатива/развитие — Gateway API.

### 10.16 Свободное место на диске / размер директории в Unix
`df -h` — файловые системы и свободное место; `du -sh /path` — размер директории (`du -h --max-depth=1 | sort -h` — найти, что распухло). Классика прода: диск забит логами — `du` по `/var/log`, ротация.

### 10.17 Список работающих процессов
`ps aux` (все процессы: CPU, память, команда), `ps aux | grep java`; интерактивно — `top`/`htop` (сортировка по CPU/памяти); дерево — `ps -ef --forest` / `pstree`. По порту: `ss -tlnp | grep 8080` (или `lsof -i :8080`) → PID.

### 10.18 Thread dump работающего Java-процесса
Снять: `jstack <PID>`; `jcmd <PID> Thread.print` (современный путь); `kill -3 <PID>` (SIGQUIT — дамп в stdout процесса, приложение не убивает); через Actuator `/threaddump`. Смотреть: глазами/в IDE, fastthread.io, VisualVM. Видно: все потоки, состояния (RUNNABLE/WAITING/BLOCKED), стеки — кто что делает, на каком мониторе заблокирован, найденные дедлоки (JVM помечает сама). Практика: высокий CPU → 3 дампа с интервалом → какие потоки стабильно RUNNABLE в одном месте.

### 10.19 grep, tail, less, more
`grep` — поиск по шаблону: `-r` (рекурсивно), `-n` (номера строк), `-i` (без регистра), `-v` (инверсия), `-E` (regex), `-A`/`-B` (контекст). `tail` — конец файла: `tail -f`/`-F` — follow лога в реальном времени (`-F` переживает ротацию). `less` — постраничный просмотр с поиском (`/pattern`, Shift+F = follow); `more` — примитивный предок `less` (только вперёд). Связка: `grep ERROR app.log | less`; `tail -f app.log | grep -i exception`.

### 10.20 IDOR (Insecure Direct Object References)
Доступ к объекту по идентификатору без проверки прав: `GET /orders/123` отдаёт **чужой** заказ, перебор id — выгрузка всей базы. Защита: авторизация на **каждый** доступ к ресурсу (`findByIdAndUserId` / проверка владельца или роли — ровно пункт из code review задачи!), непредсказуемые идентификаторы (UUID) как дополнительный слой (не замена проверке прав). В банке — прямой путь к инцидентам, проговори это.

### 10.21 🔵 SQL-инъекция
Пользовательский ввод конкатенируется в SQL: `"... WHERE name = '" + input + "'"` → input `' OR '1'='1' --` меняет запрос (чтение чужого, обход аутентификации, запись). Предотвращение: параметризованные запросы всегда (`PreparedStatement`, `:param` в JPA/JPQL — параметры передаются отдельно от текста запроса и не парсятся как SQL), никакой конкатенации даже для «внутренних» значений, для динамических имён колонок/сортировки — whitelist, минимальные привилегии учётки БД, WAF как эшелон. ORM не панацея: нативные запросы с конкатенацией так же уязвимы.

### 10.22 CSRF
Сайт злоумышленника инициирует запрос на ваш API от имени залогиненного пользователя — браузер сам прикрепит cookie сессии. Работает против cookie-аутентификации и state-changing запросов. Защита: CSRF-токены (синхронизатор: токен в форме/заголовке, проверка на сервере — Spring Security даёт из коробки), `SameSite=Lax/Strict` на cookie, проверка Origin. Для stateless API на bearer-токенах CSRF неактуален (заголовок не прикрепляется автоматически) — поэтому в таких API его осознанно отключают.

### 10.23 Reflected XSS
Ввод из запроса отражается в HTML-ответе без экранирования (`search?q=<script>...`) → скрипт исполняется в браузере жертвы (кража токенов, действия от её имени); доставка — вредоносная ссылка. Защита: экранирование вывода по контексту (шаблонизаторы делают по умолчанию — не отключать), Content-Security-Policy, валидация входа, HttpOnly на cookie (украсть сессию сложнее). Для чистого JSON-API риск ниже, но заголовки (`X-Content-Type-Options: nosniff`, CSP) всё равно ставят.

### 10.24 Clickjacking
Ваш сайт грузится в невидимый iframe поверх приманки — жертва кликает «в игру», а реально жмёт кнопки вашего интерфейса (перевод, смена настроек). Защита: запрет фрейминга — заголовок `X-Frame-Options: DENY/SAMEORIGIN` и современный CSP `frame-ancestors 'none'`; Spring Security ставит DENY по умолчанию. Плюс подтверждения критических действий.

### 10.25 Угроза токена сессии в URL
URL утекают массово: логи серверов/прокси, история браузера, заголовок Referer при переходе по внешней ссылке, закладки, скриншоты, шаринг ссылки. Токен в query = токен скомпрометирован. Правильно: Authorization-заголовок или httpOnly cookie; если legacy требует URL — одноразовые короткоживущие токены и Referrer-Policy.

### 10.26 Защита сессионных токенов, распределённые системы
Cookie-флаги: HttpOnly (недоступна JS — защита от XSS-кражи), Secure (только HTTPS), SameSite (CSRF); короткий TTL + ротация, инвалидация на сервере (logout, смена пароля), привязка к контексту (аномалии IP/устройства — сигнал). Распределённо два подхода: центральный session store (Redis) — мгновенный отзыв, но общая зависимость; stateless JWT — масштабируется без хранилища, но отзыв только через короткий access + refresh с ротацией и чёрный список при инцидентах. Выбор для банковского контекста: короткий JWT access (минуты) + refresh в httpOnly cookie с ротацией — компромисс отзываемости и масштабирования.

### 10.27 Защита от DDoS
- **Периметр:** CDN/anti-DDoS провайдер (Cloudflare/облачный scrubbing, поглощает объёмные L3/L4 атаки).
- **Сеть:** SYN cookies, лимиты соединений.
- **Приложение (L7):** rate limiting по IP/токену/endpoint'у (bucket4j, лимиты на gateway/nginx), дорогие операции — за аутентификацией, кэширование, autoscaling как амортизация, деградация изящно (shed load, очереди).
- Плюс: не иметь «дешёвого запроса с дорогим ответом» (тяжёлый поиск без лимитов), мониторинг и алерты на аномалии трафика.

### 10.28 🔵 IDE и любимые сочетания клавиш

### 10.29 Самая большая/запоминающаяся задача

## Приложение. Критерии оценки уровня

Как интервьюер отличает уровни (дословно по их методичке, сжато):

- **Junior** — базовые концепции, без нюансов производительности и масштабирования, примеры из учебных проектов.
- **Regular** — знает «под капотом», называет альтернативы и trade-off'ы, примеры из реальных проектов, думает об оптимизации.
- **Senior** — видит систему целиком: уточняет контекст вопроса, называет edge cases, примеры из крупных/нагруженных систем, архитектурные последствия решений.

**Практическая формула ответа на 4 балла:** суть (1 фраза) → как работает под капотом → пример из своего проекта → что меняется на масштабе/в кластере. И минимум один уточняющий вопрос за интервью — это буквально их маркер сеньорности.
