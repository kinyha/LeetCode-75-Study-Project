## 2. Java Core

**Q: JDK / JRE / JVM?**
JVM исполняет байткод, JRE = JVM + библиотеки, JDK = JRE + инструменты разработки (javac, jconsole, jstack).

**Q: `==` vs `equals()`?**
`==` сравнивает ссылки (для примитивов — значения), `equals()` — логическое равенство, если переопределён. По умолчанию `Object.equals` == `==`.

**Q: Контракт equals/hashCode?**
equals: рефлексивность, симметричность, транзитивность, консистентность, `x.equals(null) == false`. hashCode: равные объекты обязаны иметь равный hashCode (обратное не требуется — коллизии допустимы). Переопределяются всегда вместе, иначе объект «теряется» в HashMap/HashSet: положили по одному хэшу, ищем по другому бакету.

**Q: Почему String immutable? String pool?**
Безопасность (ключи, class loading), потокобезопасность, кэширование hashCode, string pool — переиспользование литералов. `new String("a")` создаёт объект в куче мимо пула; `intern()` кладёт/берёт из пула.

**Q: String vs StringBuilder vs StringBuffer?**
String неизменяем — конкатенация в цикле порождает мусор. StringBuilder — изменяемый, не потокобезопасный (используй его). StringBuffer — synchronized, легаси.

**Q: Кэш Integer / подвохи автобоксинга?**
`Integer.valueOf` кэширует −128..127 → `Integer a = 127; a == b` даёт true, для 128 — false. Сравнивать только equals. Ещё подвохи: NPE при анбоксинге null, скрытый боксинг в циклах (перформанс).

**Q: final / finally / finalize?**
final — константа/запрет наследования/переопределения. finally — блок, выполняющийся всегда (кроме System.exit / падения JVM). finalize — deprecated (с Java 9), не использовать; альтернативы — try-with-resources, Cleaner.

**Q: Checked vs unchecked исключения?**
Throwable → Error (не ловим) и Exception. Checked (IOException, SQLException) — компилятор требует обработки, восстановимые ситуации. Unchecked = RuntimeException (NPE, IllegalArgumentException) — ошибки программирования/контрактов. В Spring-мире почти всё unchecked (DataAccessException). Важно для @Transactional: по умолчанию rollback только на unchecked.

**Q: try-with-resources?**
Автозакрытие AutoCloseable в обратном порядке объявления. Исключения при close становятся suppressed (`getSuppressed()`), не затирают основное — в отличие от ручного finally.

**Q: Generics, type erasure, PECS?**
Дженерики — compile-time проверка типов, в рантайме стираются до Object/bound (erasure) → нельзя `new T[]`, `instanceof List<String>`. PECS: producer extends (`List<? extends Number>` — читаем), consumer super (`List<? super Integer>` — пишем).

**Q: Как написать immutable-класс?**
final класс, все поля private final, без сеттеров, защитные копии изменяемых полей (в конструкторе и геттерах), не отдавать this из конструктора. Или `record` (Java 16+): компактный immutable-носитель данных с готовыми equals/hashCode/toString.

**Q: Интерфейс vs абстрактный класс?**
Интерфейс — контракт, множественная реализация, default/static методы (с Java 8), нет состояния (только static final). Абстрактный класс — состояние, конструкторы, любой доступ, одиночное наследование. Diamond с default-методами → компилятор требует явного override и `Interface.super.method()`.

**Q: Java 8 фичи? (клиент требует Java 8 — знай отлично)**
Лямбды + функциональные интерфейсы (Function, Supplier, Consumer, Predicate), Stream API, Optional, default/static методы в интерфейсах, java.time, CompletableFuture, Metaspace вместо PermGen.

**Q: Stream API: intermediate vs terminal? map vs flatMap?**
Intermediate (map, filter, sorted) — ленивые, строят конвейер; terminal (collect, forEach, reduce, count) — запускают. map: 1→1, flatMap: 1→N, разворачивает вложенные стримы (`List<List<T>>` → поток T). collect(groupingBy/toMap/joining) — знать. Стрим одноразовый.

**Q: Parallel stream — когда нельзя?**
Общий ForkJoinPool на всё приложение → блокирующие IO-операции в parallel stream душат всех. Годится только для CPU-bound на больших коллекциях без общего изменяемого состояния. В веб-приложениях почти никогда.

**Q: Optional — зачем и антипаттерны?**
Явный контракт «может отсутствовать» для возвращаемых значений. orElse вычисляется всегда, orElseGet — лениво. Антипаттерны: Optional-поля, Optional-параметры, `opt.get()` без проверки, оборачивание коллекций (возвращай пустую коллекцию).

**Q: Что появилось в 11/17/21?**
11: var, HTTP Client, String.isBlank/lines. 17: records, sealed classes, switch expressions, text blocks, pattern matching instanceof. 21: virtual threads, record patterns, pattern matching switch. У тебя живой опыт Java 8→17 миграций — упомяни.

**Q: Память JVM?**
Heap (young: eden + survivors; old), Stack (фреймы вызовов, на поток), Metaspace (метаданные классов). StackOverflowError — глубокая рекурсия; OutOfMemoryError — куча/метаспейс. GC: поколенческая модель, minor/major GC, stop-the-world; G1 по умолчанию с Java 9.

**Q: Java передаёт по значению или по ссылке?**
Всегда по значению. Для объектов копируется значение ссылки: поменять поля объекта внутри метода можно, переприсвоить ссылку снаружи — нет.

## 3. Collections

**Q: Иерархия коллекций?**
Collection → List (ArrayList, LinkedList), Set (HashSet, LinkedHashSet, TreeSet), Queue/Deque (ArrayDeque, PriorityQueue). Map — отдельно: HashMap, LinkedHashMap, TreeMap, ConcurrentHashMap.

**Q: ArrayList vs LinkedList?**
ArrayList — массив: get O(1), add в конец амортизированно O(1), вставка в середину O(n), рост ×1.5. LinkedList — двусвязный список: вставка/удаление у известного узла O(1), но доступ O(n) и плохая кэш-локальность. На практике почти всегда ArrayList; LinkedList — почти никогда (даже как Deque лучше ArrayDeque).

**Q: Как устроен HashMap? (обязательный вопрос)**
Массив бакетов (capacity — степень двойки, дефолт 16). hashCode ключа → дополнительный spread (xor старших бит) → индекс `hash & (n−1)`. Коллизии — связный список; при ≥8 элементах в бакете и capacity ≥64 — красно-чёрное дерево (O(log n) вместо O(n)). Load factor 0.75 → resize ×2 с перехэшированием. Один null-ключ допустим. get/put в среднем O(1).

**Q: Требования к ключам HashMap?**
Корректные equals/hashCode + иммутабельность. Если поле ключа, участвующее в hashCode, изменить после put — элемент «потеряется» (лежит в старом бакете, ищется в новом).

**Q: HashMap vs Hashtable vs ConcurrentHashMap?**
Hashtable — легаси, synchronized на каждый метод. HashMap — не потокобезопасный (конкурентная запись → потерянные апдейты, до Java 8 — возможен бесконечный цикл при resize). CHM — потокобезопасный без глобального лока. CHM не допускает null-ключи/значения: null неотличим от «отсутствует» при конкурентном доступе.

**Q: Как CHM устроен внутри (Java 8+)?**
Не сегменты (это Java 7). Пустой бакет — CAS; непустой — synchronized на первом узле бакета. Чтение без локов (volatile-семантика). size — распределённые счётчики (идея LongAdder). compute/merge — атомарные операции на ключе.

**Q: HashSet?**
Обёртка над HashMap (элементы — ключи, значение — заглушка). LinkedHashSet — сохраняет порядок вставки, TreeSet — сортированный (NavigableSet).

**Q: TreeMap?**
Красно-чёрное дерево, O(log n), ключи по Comparable или Comparator (equals не используется — сравнение через compareTo!). Navigable-методы: firstKey, floorKey, headMap, subMap.

**Q: LinkedHashMap и LRU-кэш?**
HashMap + двусвязный список порядка. С `accessOrder=true` + переопределённым `removeEldestEntry` — готовый LRU-кэш. Классический вопрос «как сделать LRU».

**Q: Fail-fast vs fail-safe итераторы?**
Fail-fast (ArrayList, HashMap): modCount, при структурной модификации во время итерации — ConcurrentModificationException (best-effort). Удалять при итерации — только через iterator.remove() или removeIf. Fail-safe (CopyOnWriteArrayList — итерация по снапшоту; CHM — weakly consistent, исключений нет).

**Q: Comparable vs Comparator?**
Comparable — «естественный порядок», внутри класса (compareTo). Comparator — внешняя стратегия, можно несколько. `Comparator.comparing(...).thenComparing(...).reversed()`, nullsFirst/nullsLast.

**Q: List.of vs Collections.unmodifiableList?**
List.of — истинно неизменяемая копия, null запрещён. unmodifiableList — read-only view: изменения исходного списка видны через view.

**Q: CopyOnWriteArrayList — когда?**
Каждая запись копирует массив. Годится при «читаем часто, пишем редко» (список слушателей). При частой записи — недопустимо дорого.

## 4. Concurrency (ключевой блок: event-обработка + шедулеры из JD)

**Q: Процесс vs поток? Способы создания потока?**
Процесс — изолированная память; потоки делят кучу процесса, у каждого свой стек. Создание: extends Thread, Runnable, Callable + Future (возвращает значение, бросает checked), на практике — только через ExecutorService.

**Q: Жизненный цикл потока?**
NEW → RUNNABLE → BLOCKED (ждёт монитор) / WAITING (wait, join, park) / TIMED_WAITING (sleep, wait(t)) → TERMINATED.

**Q: synchronized — как работает?**
Захват монитора объекта: synchronized-метод — монитор this, static — монитор Class-объекта, блок — указанного объекта. Reentrant (поток может повторно взять свой монитор). Даёт и взаимоисключение, и видимость (happens-before между release и acquire).

**Q: volatile?**
Гарантирует видимость (чтение всегда из «основной памяти») и запрет переупорядочивания (happens-before: запись volatile → последующее чтение). НЕ даёт атомарности: `count++` под volatile всё равно гонка (read-modify-write). Кейс: флаг остановки воркера. Для счётчиков — AtomicInteger/LongAdder.

**Q: Java Memory Model, happens-before?**
JMM определяет, когда запись одного потока видна другому. Источники happens-before: порядок в одном потоке, unlock → lock того же монитора, volatile write → read, Thread.start() → действия потока, действия потока → join(), финальные поля после конструктора.

**Q: Atomics и CAS?**
Compare-And-Swap — атомарная аппаратная инструкция: «замени, если значение всё ещё X». AtomicInteger/Long/Reference: инкременты, compareAndSet, без блокировок (lock-free). Под высокой конкуренцией на счётчик — LongAdder (распределённые ячейки). ABA-проблема — AtomicStampedReference (знать название).

**Q: Race condition — пример?**
Check-then-act: `if (map.get(k) == null) map.put(k, v)` — между проверкой и записью влез другой поток. Решения: атомарные методы (putIfAbsent, computeIfAbsent), synchronized, CAS.

**Q: Deadlock — условия, как избегать, как диагностировать?**
Условия: взаимное исключение, удержание+ожидание, отсутствие вытеснения, циклическое ожидание. Избегать: единый порядок захвата локов, tryLock с таймаутом, минимизация вложенных локов. Диагностика: thread dump (jstack) — JVM сама помечает found deadlock.

**Q: wait/notify?**
Только внутри synchronized на том же мониторе (иначе IllegalMonitorStateException). wait — отпускает монитор и ждёт; notify — будит одного, notifyAll — всех. wait всегда в цикле while(условие) — spurious wakeups + перепроверка условия. Современная альтернатива — Lock + Condition или готовые BlockingQueue.

**Q: ExecutorService — типы, shutdown?**
newFixedThreadPool, newCachedThreadPool (растёт неограниченно — опасен), newSingleThreadExecutor, newScheduledThreadPool. shutdown — доработать очередь, не принимать новые; shutdownNow — interrupt всем + вернуть невыполненное; awaitTermination — дождаться.

**Q: Параметры ThreadPoolExecutor? (частый подвох)**
corePoolSize, maximumPoolSize, keepAliveTime, workQueue, RejectedExecutionHandler. Порядок: задача → потоки до core → очередь → потоки до max → rejection. Подвох: с неограниченной LinkedBlockingQueue maximumPoolSize НИКОГДА не сработает — очередь бесконечно копится. Rejection policies: AbortPolicy (исключение), CallerRunsPolicy (естественный backpressure), Discard*.

**Q: Как выбрать размер пула?**
CPU-bound ≈ число ядер (+1). IO-bound ≈ ядра × (1 + время ожидания / время вычислений). Плюс мониторинг очереди и метрики (у тебя: Actuator/Prometheus на soak-тесте).

**Q: Future vs CompletableFuture?**
Future — только блокирующий get. CF — композиция: thenApply (map), thenCompose (flatMap, цепочка асинхронных), thenCombine (объединить два), allOf/anyOf, exceptionally/handle (ошибки). По умолчанию async-методы идут в common ForkJoinPool — для IO передавай свой executor.

**Q: BlockingQueue?**
put блокируется на полной, take — на пустой. ArrayBlockingQueue (ограниченная), LinkedBlockingQueue, SynchronousQueue (передача из рук в руки), DelayQueue. База producer-consumer.

**Q: CountDownLatch vs CyclicBarrier vs Semaphore?**
Latch — одноразовый счётчик «дождаться N событий» (часто в тестах). Barrier — многоразовая точка сбора N потоков. Semaphore — N разрешений, лимит конкурентного доступа (например, ограничить параллельные вызовы внешнего API).

**Q: ReentrantLock vs synchronized?**
Lock: tryLock (с таймаутом — против дедлоков), lockInterruptibly, fair-режим, несколько Condition. Минус — надо не забыть unlock в finally. synchronized проще и достаточно в большинстве случаев. ReadWriteLock — много читателей / один писатель.

**Q: ThreadLocal?**
Значение на поток: SecurityContext, транзакционные ресурсы Spring, MDC для логирования. Опасность: в пулах поток переиспользуется → утечка/протечка данных между задачами. Всегда remove() в finally.

**Q: Virtual threads (Java 21)?**
Лёгкие потоки на carrier-потоках JVM: блокирующий IO-код масштабируется как асинхронный, thread-per-request снова ок. Проблема pinning: synchronized-блок с блокирующим вызовом внутри пиннит carrier (в 21; знать как ограничение). Для CPU-bound выгоды нет.

**Q: @Scheduled / @Async в Spring? (JD: «настройка шедулеров»)**
@EnableScheduling + @Scheduled(fixedDelay/fixedRate/cron). Подвохи, о которых стоит сказать самому:
1. По умолчанию ОДИН поток на все @Scheduled — долгая задача блокирует остальные. Лечится настройкой TaskScheduler / pool size.
2. fixedRate при долгих задачах накапливает запуски; fixedDelay отсчитывает от завершения.
3. В кластере из N подов джоба выполнится N раз → нужен распределённый лок (ShedLock поверх PostgreSQL) или вынос в очередь.
4. Исключение внутри — задача не должна умирать молча: try/catch + метрики/алерты.
5. Идемпотентность: джоба может отработать повторно — статусные апдейты пишутся так, чтобы повтор был безопасен.
Твой опыт: фоновые задачи обновления статусов лотов, ставок и инвойсов на AuctionPort — расскажи через эти пункты.

## 5. PostgreSQL / SQL

**Q: ACID?**
Atomicity (всё или ничего), Consistency (инварианты), Isolation (параллельные транзакции не мешают), Durability (закоммичено = сохранено, WAL).

**Q: Уровни изоляции и аномалии?**
Read uncommitted (dirty read) → read committed (non-repeatable read) → repeatable read (phantom read) → serializable. Специфика PostgreSQL: dirty read невозможен вообще (read uncommitted = read committed); дефолт — read committed; repeatable read на снапшотах — фантомов нет, но возможны serialization failure (ошибка 40001 → ретрай транзакции); serializable = SSI, тоже через ретраи.

**Q: MVCC?**
Каждая запись — версии строк (xmin/xmax), читатели не блокируют писателей. Мёртвые версии копятся → VACUUM/autovacuum чистит, иначе bloat. Отсюда же: длинные транзакции вредны — держат старые снапшоты.

**Q: Индексы — какие и когда не работают?**
B-tree (дефолт: =, <, >, ORDER BY), GIN (полнотекст, jsonb, массивы), GiST/BRIN — кратко. Составной индекс работает по префиксу колонок (leading column). Не используется: функция/выражение над колонкой (нужен функциональный индекс), низкая селективность, неявный каст типов, LIKE '%x'. Partial-индексы (WHERE status = 'ACTIVE') и covering (INCLUDE) — index-only scan.

**Q: EXPLAIN ANALYZE?**
Seq Scan vs Index Scan vs Bitmap Scan; фактические строки vs оценка планировщика; BUFFERS — чтения. Твоя практика: анализ SQL при устранении N+1 и оптимизации поиска инвойсов.

**Q: JOIN'ы, EXISTS vs IN?**
INNER/LEFT/RIGHT/FULL/CROSS. EXISTS обычно эффективнее IN на больших подзапросах (semi-join, короткое замыкание); NOT IN с NULL — ловушка (вернёт пусто), NOT EXISTS безопасен.

**Q: Оптимистичная vs пессимистичная блокировка?**
Оптимистичная: @Version, при конфликте OptimisticLockException → ретрай; для редких конфликтов. Пессимистичная: SELECT ... FOR UPDATE — строка залочена до конца транзакции; для горячих строк (баланс, счётчик). FOR UPDATE SKIP LOCKED — паттерн «БД как очередь задач»: несколько воркеров разбирают джобы без конфликтов. Красиво упомянуть в связке с шедулерами.

**Q: Пагинация: OFFSET vs keyset?**
OFFSET n сканирует и выбрасывает n строк — деградирует на глубине. Keyset: WHERE (created_at, id) > (:last_ts, :last_id) ORDER BY ... LIMIT — стабильное время. Твой delta-endpoint /lots/changedBids с timestamp-фильтрацией — по сути keyset-подход: клиент забирает только изменения после своей метки, −95% трафика.

**Q: Connection pool / HikariCP?**
Пул соединений: maximumPoolSize (обычно 10–20, больше ≠ лучше: считай от ядер БД), connectionTimeout, leakDetectionThreshold. Твоя история: разбирался с warning'ами leak detection — оказались false positive от долгих задач MassIndexer (Hibernate Search), а не реальными утечками. Хорошая история про «не чинить то, что не сломано, пока не разобрался».

**Q: Нормализация?**
1НФ — атомарность, 2НФ — нет частичных зависимостей от составного ключа, 3НФ — нет транзитивных. Денормализация — осознанно, ради чтения (отчёты, кэш-таблицы).

## 6. Hibernate / JPA

**Q: Состояния entity?**
Transient (new) → Managed (в persistence context, dirty checking) → Detached (контекст закрыт) → Removed. persist — только transient; merge — копирует состояние detached в managed-копию (возвращает её!).

**Q: LazyInitializationException?**
Обращение к lazy-ассоциации вне транзакции/сессии. Решения: JOIN FETCH в запросе, @EntityGraph, DTO-проекция сразу в запросе. Open Session in View — плохое решение (соединение держится на весь HTTP-запрос, ленивые запросы из view-слоя); в Boot включён по умолчанию — стоит выключать.

**Q: N+1 — твой боевой кейс (расскажи как историю)?**
Поиск инвойсов: ~8–11 с, в логах ~1900 SQL-запросов — на каждый инвойс отдельно подтягивались связанные сущности. Диагностика: SQL-лог + счётчик запросов. Решение: JOIN FETCH для критичных ассоциаций + @BatchSize для остальных (IN-подгрузка пачками). Результат: ~180 мс и 15 запросов. Альтернативы, которые стоит назвать: @EntityGraph, @Fetch(SUBSELECT), DTO-проекции.

**Q: Кэши Hibernate?**
1-й уровень — persistence context (в рамках сессии, всегда включён, на нём же dirty checking: снапшот при загрузке → сравнение при flush → UPDATE только изменённого). 2-й уровень — между сессиями (Ehcache/Infinispan/Redis), только для редко меняющихся справочников, вопросы инвалидации в кластере.

**Q: equals/hashCode для entity?**
Ловушки: id появляется после persist; прокси ломает getClass(). Практика: сравнение по id с проверкой на null, `instanceof` вместо getClass, hashCode — константный или по классу. Не включать коллекции.

**Q: Cascade и orphanRemoval?**
cascade = операции распространяются на детей (PERSIST, MERGE, REMOVE, ALL). orphanRemoval = удаление ребёнка при отвязке от родителя. CascadeType.REMOVE на @ManyToMany — не делать.

**Q: Миграция Hibernate 5→6 / Boot 3 (твой опыт)?**
javax.* → jakarta.* по всему коду, ужесточение JPQL (типизация, неявные касты), изменения в генерации SQL и типах. Плюс Spring Security конфиг на новый стиль. Рассказывай как управляемый процесс: ветка, компиляция → тесты → регресс.

**Q: Liquibase?**
Версионируемые changeset'ы (id + author + checksum), таблица DATABASECHANGELOG, миграции в CI/CD, откаты. Правило: changeset неизменяем после выката — только новый.

## 7. Spring / Spring Boot

**Q: IoC / DI? Почему constructor injection?**
Контейнер управляет созданием и связыванием бинов. Constructor injection: поля final (иммутабельность), обязательные зависимости видны сразу, тестируемость без контейнера, циклические зависимости падают явно на старте.

**Q: Scopes бинов?**
singleton (дефолт), prototype, request, session, application. Singleton не должен хранить mutable-состояние запроса. Prototype внутри singleton инжектится один раз — нужен ObjectProvider/@Lookup.

**Q: Жизненный цикл бина?**
Определения бинов → инстанцирование → внедрение зависимостей → Aware-интерфейсы → BeanPostProcessor.before → @PostConstruct/InitializingBean → BeanPostProcessor.after (здесь создаются AOP-прокси!) → работа → @PreDestroy.

**Q: @Component vs @Bean?**
@Component — авто-сканирование своего класса. @Bean — фабричный метод в @Configuration, для чужих классов и сложной сборки. @Configuration проксируется CGLIB: повторный вызов @Bean-метода внутри конфига возвращает тот же singleton.

**Q: Как работают прокси / AOP?**
JDK dynamic proxy (по интерфейсам) или CGLIB (наследование, дефолт в Boot). Следствие: аннотации-аспекты (@Transactional, @Cacheable, @Async) работают только при вызове снаружи через прокси. Self-invocation (this.method()) — аспект НЕ сработает. Классический вопрос-ловушка.

**Q: @Transactional — глубоко (спросят почти наверняка)?**
- Propagation: REQUIRED (дефолт, присоединиться/создать), REQUIRES_NEW (приостановить текущую, отдельный коммит — для аудита/логов), NESTED (savepoint), SUPPORTS/MANDATORY/NEVER.
- Rollback: по умолчанию только unchecked; checked — rollbackFor = Exception.class.
- Не работает: private-методы, self-invocation, исключение поймано внутри метода, метод не через Spring-прокси.
- readOnly = true — оптимизации Hibernate (без dirty checking snapshot'ов).
- Антипаттерн, критичный для вашего домена: внешние вызовы (HTTP, Kafka send, блокчейн-транзакция) ВНУТРИ БД-транзакции. Держат соединение, а при откате внешний эффект уже случился. Решение — outbox (раздел 8/11).

**Q: Автоконфигурация Spring Boot?**
@SpringBootApplication = @Configuration + @ComponentScan + @EnableAutoConfiguration. Конфигурации из AutoConfiguration.imports применяются по условиям (@ConditionalOnClass, OnMissingBean, OnProperty). Свой бин перекрывает дефолтный. Starters — наборы зависимостей.

**Q: Конфигурация и профили?**
application.yml, приоритет: аргументы > env vars > yml профиля > дефолтный yml. @ConfigurationProperties (типизировано, валидируемо) vs @Value. Профили: dev/stage/prod, @Profile на бинах.

**Q: Обработка ошибок REST?**
@RestControllerAdvice + @ExceptionHandler, маппинг доменных исключений на статусы, ProblemDetail (RFC 7807, Boot 3). Единый формат ошибок, без стектрейсов наружу (ИБ!).

**Q: Валидация (JD: «подготовка схем валидации»)?**
Bean Validation: @NotNull/@NotBlank/@Size/@Pattern/@Positive на DTO + @Valid в контроллере → 400 с деталями через advice. Кастомные: ConstraintValidator (у тебя — валидация входных данных ставок: шаг ставки, статус лота, права). @Validated — группы и валидация параметров методов. Плюс валидация на границе с блокчейном: схема payload'а до отправки в контракт.

**Q: Spring Security — база?**
Цепочка фильтров (SecurityFilterChain-бин в Boot 3, DSL вместо WebSecurityConfigurerAdapter). Authentication (кто) vs Authorization (что можно). SecurityContextHolder (ThreadLocal). JWT-флоу: логин → подписанный токен → фильтр валидирует подпись/exp → заполняет контекст; stateless, csrf off для чистого API. Method security: @PreAuthorize("hasRole(...)"). Пароли — BCrypt.

**Q: Actuator / метрики?**
health, metrics, prometheus endpoint; Micrometer — счётчики/таймеры. Твоя практика: анализ метрик через Actuator/Prometheus на 8-часовом soak-тесте (50 VU, 2.68M запросов), поиск узких мест — деградация времени ответа, пулы, GC.

## 8. Kafka / Messaging (JD: «уверенный опыт rest, kafka, rpc»)

**Q: Архитектура Kafka?**
Broker'ы → topic'и → partition'ы (единица параллелизма и порядка). Сообщение = key + value + offset. Consumer group: партиция читается одним консюмером группы; групп может быть много (fan-out). Репликация: leader + follower'ы, ISR.

**Q: Гарантии порядка?**
Порядок — только внутри партиции. Ключ определяет партицию (hash). Для финансовых операций ключ = ID актива/сделки/клиента — все события одной сущности идут последовательно. Без ключа — round-robin, порядок теряется.

**Q: Семантики доставки?**
At-most-once (коммит до обработки), at-least-once (обработка до коммита — дефолтный практичный выбор, но возможны дубли), exactly-once (идемпотентный продюсер enable.idempotence + Kafka-транзакции; в реальном мире чаще: at-least-once + идемпотентный консюмер). Для ЦФА-операций проговори: дубль финансового события недопустим → дедупликация по event id / бизнес-ключу.

**Q: Идемпотентный консюмер — как?**
Таблица processed_events (event_id PK) — вставка в той же БД-транзакции, что и бизнес-эффект; дубль → уникальный конфликт → скип. Либо естественная идемпотентность (UPSERT по бизнес-ключу, статусная машина «переход разрешён только из X в Y»).

**Q: Offset'ы и коммиты?**
enable.auto.commit — риск потерь/дублей; надёжнее ручной коммит после успешной обработки батча. При ребалансе необработанные-но-закоммиченные теряются, обработанные-но-незакоммиченные придут повторно (снова к идемпотентности).

**Q: Rebalance?**
Перераспределение партиций при изменении состава группы. max.poll.interval.ms превышен (долгая обработка) → консюмер выкинут → ребаланс-шторм. Cooperative rebalancing (incremental) — без stop-the-world всей группы.

**Q: Retry / DLQ?**
Ошибка обработки: ретраи с backoff (retry-топики: retry-5s, retry-1m...), после исчерпания — dead letter topic + алерт. Poison message не должен блокировать партицию.

**Q: Outbox pattern? (ключевой — свяжи с блокчейном)**
Проблема: БД-коммит и отправка в Kafka — не атомарны. Решение: в одной БД-транзакции пишем бизнес-данные + запись в outbox-таблицу; отдельный воркер (@Scheduled или Debezium CDC) читает outbox и публикует, помечая отправленное. Тот же паттерн — для публикации в блокчейн: сначала намерение в БД, потом асинхронная отправка с фиксацией txHash.

**Q: Настройки надёжности продюсера?**
acks=all + min.insync.replicas=2, retries, enable.idempotence=true (нет дублей при ретраях продюсера), linger.ms/batch.size — throughput.

**Q: Kafka vs RabbitMQ (у тебя оба — сильная позиция)?**
Kafka — распределённый лог: сообщения хранятся по retention, replay возможен, throughput выше, порядок по партициям; консюмер сам тянет. RabbitMQ — брокер очередей: гибкий роутинг (exchange: direct/topic/fanout), сообщение удаляется после ack, ниже латентность на малых объёмах, удобные приоритеты/TTL/DLX. На AuctionPort — RabbitMQ для bid-событий; на SCS — Kafka для event-driven доставки и downstream-интеграций.

**Q: RPC-опыт?**
RMI/RPC на АЦК — интеграция ICE-платформы с legacy Java-сервисами; JSON-RPC — общение с блокчейн-нодой на AuctionPort. Понимание trade-off sync RPC vs async messaging: связность, таймауты, ретраи, идемпотентность.

## 9. Микросервисы и паттерны

**Q: Sync vs async взаимодействие?**
REST/RPC — когда нужен немедленный ответ; события — для развязки, устойчивости к недоступности, fan-out. Правило: команды — можно sync, факты («произошло X») — события.

**Q: Распределённые транзакции / Saga?**
2PC в микросервисах не используют. Saga: последовательность локальных транзакций + компенсации. Хореография (через события, просто, но сложно отлаживать) vs оркестрация (координатор, явное состояние — для финансовых флоу предпочтительнее). Выпуск ЦФА — типичный кандидат: резерв средств → запись в реестр/блокчейн → подтверждение → уведомления; каждая ступень с компенсацией.

**Q: Идемпотентность API?**
Idempotency-Key от клиента + таблица обработанных ключей; PUT идемпотентен по семантике, POST — делаем таковым. Для денег/активов — обязательна.

**Q: Устойчивость: circuit breaker и др.?**
Timeout (всегда и на всё), retry с exponential backoff + jitter (только на идемпотентные операции!), circuit breaker (resilience4j: closed → open → half-open) — быстрый отказ вместо каскадного зависания, bulkhead — изоляция пулов. Для блокчейн-ноды: она может тормозить — таймауты + CB + очередь на отправку.

**Q: Наблюдаемость?**
Структурные логи + traceId через MDC, распределённый трейсинг (OpenTelemetry/Micrometer Tracing), метрики (Prometheus/Grafana), health checks (readiness/liveness в K8s). Твой опыт: Actuator/Prometheus, деплой в AKS через ArgoCD, CI/CD GitHub Actions.

**Q: Паттерны GoF на практике (просят примеры)?**
Singleton — Spring beans; Factory Method — BeanFactory, ConnectionFactory; Builder — StringBuilder, Lombok @Builder, HttpRequest; Strategy — валидаторы/обработчики событий по типу (Map<EventType, Handler>); Template Method — JdbcTemplate, AbstractController; Observer — Spring ApplicationEvent, listeners; Proxy — @Transactional/@Cacheable; Chain of Responsibility — Spring Security filter chain; Adapter — обёртки над внешними API. Приготовь 2–3 примера из СВОЕГО кода: например, Strategy для обработчиков auction events по типу.

**Q: SOLID с примерами?**
S — контроллер/сервис/репозиторий, отдельный blockchain-publisher; O — новый тип события = новый handler без правки диспетчера; L — подтипы не ломают контракт (исключения, преусловия); I — узкие интерфейсы (Reader/Writer вместо God-interface); D — зависимость от абстракций: сервис зависит от интерфейса BlockchainGateway, не от Web3j напрямую (+ это же даёт тестируемость).

## 10. Тестирование

**Q: Пирамида тестов?**
Много unit (быстрые, изолированные) → меньше integration (реальные БД/брокер через Testcontainers) → мало e2e. У тебя: JUnit 5 + Mockito, покрытие proxy-bidding unit-тестами — доменная логика ставок (перебивание, шаг, лимиты, граничные случаи).

**Q: Mockito: mock vs spy? verify?**
Mock — полностью фейковый объект (when/thenReturn); spy — обёртка над реальным, часть методов подменена. verify — проверка вызовов (times, never), ArgumentCaptor — перехват аргументов (например, проверить payload, ушедший в blockchain gateway). Не мокать то, что тестируешь; не переусердствовать с verify (хрупкие тесты).

**Q: @SpringBootTest vs slice-тесты?**
@SpringBootTest — весь контекст, медленно, для интеграционных. Slices: @WebMvcTest (контроллеры + MockMvc), @DataJpaTest (репозитории + встроенная/контейнерная БД). @MockBean — подмена бина в контексте (в новых версиях Boot переименован в @MockitoBean — можно блеснуть).

**Q: Testcontainers?**
Реальные PostgreSQL/Kafka в Docker на время теста: тестируем настоящий SQL/сериализацию, а не H2-суррогат. @DynamicPropertySource для проброса URL.

**Q: Как тестировать асинхронщину/шедулеры?**
Выносить логику джобы в отдельный метод/сервис и тестировать синхронно; Awaitility для ожидания асинхронных эффектов; для Kafka — Testcontainers + тестовый консюмер.

## 11. «Расскажи про опыт с блокчейном» — скрипт ответа

### Основной ответ (~90 секунд, разговорный)

> На AuctionPort блокчейн использовался как audit trail для результатов торгов: заказчику нужна была неизменяемая фиксация итогов аукционов, чтобы результаты нельзя было оспорить или подменить задним числом. Сеть — приватная, EVM-совместимая, на ней задеплоен смарт-контракт, куда записывались итоговые цены, winning bid и ключевые события аукциона.
>
> Моя зона — интеграционный слой между Java-бэкендом и блокчейн-контуром. Взаимодействие шло через Web3j и JSON-RPC. Пайплайн событийный: завершение аукциона порождает доменное событие → оно попадает в обработчик → я готовил payload (нормализация данных лота, цены, победителя под формат метода контракта), дальше формирование и подписание транзакции, отправка в ноду, фиксация tx hash и статуса у себя в БД.
>
> Из важного, что вынес: блокчейн-вызов — это медленная и ненадёжная внешняя система, поэтому его нельзя делать синхронно внутри бизнес-транзакции. У нас публикация была развязана от основного флоу аукциона через событийную обработку — если нода недоступна, аукцион завершается штатно, а публикация догоняет с ретраями. Смарт-контракты сам не писал — их делала отдельная команда, я работал с их ABI. Но именно ту часть, которая у вас в описании — «реализация транзакций в смарт-контракты и публикация цен в блокчейн» — я делал руками.

### Ожидаемые follow-up и ответы

**Q: Как именно работает Web3j?**
Из ABI контракта генерируется Java-обёртка (типизированные методы). Для записи: RawTransaction (nonce, gas, адрес контракта, закодированный вызов) → подпись приватным ключом (Credentials) → eth_sendRawTransaction через JSON-RPC → получаем tx hash → ждём майнинга (receipt). Для чтения view-методов — eth_call: бесплатно, синхронно, без транзакции.

**Q: Что такое nonce и какие с ним проблемы?**
Порядковый номер транзакции от аккаунта — защита от replay и упорядочивание. Проблемы: параллельная отправка с одного аккаунта → конфликты/дыры в nonce (транзакции зависают, пока дыра не заполнена). Решение: сериализация отправки — одна очередь/один воркер на «отправляющий» аккаунт, nonce-менеджер, при сбое — пересинхронизация с eth_getTransactionCount(pending).

**Q: Gas?**
Плата за вычисления: gasLimit (потолок) × цена (в EIP-1559 — baseFee + priorityFee). В приватных/permissioned сетях экономика газа часто отключена (gasPrice = 0), но лимит указывать всё равно нужно. Out of gas → транзакция откатывается, но включается в блок как failed.

**Q: Как бы ты организовал вычитывание операций из блокчейна? (пункт №1 в JD — подготовь развёрнуто)**
Смарт-контракт при выполнении эмитит events → они попадают в logs блоков. Чтение:
1. eth_getLogs с фильтром по адресу контракта и topic'ам (сигнатуры событий) по диапазону блоков; либо подписка (WebSocket), но polling надёжнее.
2. Чекпоинт: храню в БД номер последнего обработанного блока; воркер (@Scheduled) читает [checkpoint+1 … latest − N confirmations], обрабатывает, атомарно двигает чекпоинт.
3. Идемпотентность: уникальный ключ события = txHash + logIndex; повторная обработка (рестарт, перечитка) безопасна.
4. Финальность: в публичных PoW/PoS ждут N подтверждений из-за reorg; в приватных BFT-сетях (IBFT/QBFT) финальность мгновенная — reorg'ов нет, можно читать сразу.
5. Дальше стандартно: событие → внутренний формат → Kafka/обработчики → downstream.
По сути это тот же consumer-паттерн, что с Kafka: offset ↔ номер блока, идемпотентность, ретраи.

**Q: Как обеспечить надёжность записи в блокчейн?**
Outbox: в БД-транзакции бизнес-эффекта пишу «намерение публикации» (status = PENDING) → отдельный воркер отправляет, сохраняет txHash (SENT) → следующий шаг опрашивает receipt (CONFIRMED/FAILED) → на FAILED ретрай с новой транзакцией. Плюс таймауты на ноду, circuit breaker, алерты на застрявшие PENDING/SENT. Никогда не блокировать HTTP-запрос и БД-транзакцию ожиданием майнинга.

**Q: Что такое смарт-контракт / ABI?**
Контракт — код с состоянием по адресу в сети, исполняется детерминированно EVM. ABI — JSON-описание методов и событий (сигнатуры, типы) — по нему кодируются вызовы и декодируются логи; для Java по ABI генерируется обёртка Web3j.

**Q: EOA vs контрактный аккаунт?**
EOA — управляется приватным ключом, инициирует транзакции. Контрактный — код + storage, активируется входящей транзакцией/вызовом.

**Q: Где хранить приватные ключи?**
Не в коде/конфиге. Enterprise: Vault/KMS/HSM, подпись как сервис, ротация, разграничение доступа — тема ИБ, которую в банке точно поднимут. Достаточно показать, что понимаешь проблему.

## 12. Блокчейн-минимум под ЦФА (специфика вакансии)

### Что такое ЦФА
- 259-ФЗ «О цифровых финансовых активах» (31.07.2020). ЦФА — цифровые права: денежные требования, права по эмиссионным ценным бумагам, участие в капитале непубличного АО; учёт и обращение — в информационной системе на основе распределённого реестра.
- Роли: эмитент → выпуск на платформе ОИС (оператор информационной системы, реестр ведёт ЦБ) → инвесторы; вторичное обращение — операторы обмена ЦФА (Мосбиржа, СПБ Биржа).
- ЦФА ≠ криптовалюта: не средство платежа, приватные (permissioned) сети, KYC на уровне платформы, полный надзор ЦБ.
- 90–95% выпусков — по сути цифровые аналоги коротких облигаций (денежное требование с фиксированной доходностью); есть индексные (привязка к золоту, акциям, топливу) и гибридные (право получить деньги или актив).

### Рынок (актуально на середину 2026, цифры называй осторожно)
- Взрывной рост: ~60 млрд ₽ выпусков за 2023 → ~600 млрд ₽ за 2024; накопленный объём привлечений к 2025 — сотни млрд ₽ (ЦБ называл ~800 млрд к 1К2025).
- В реестре ЦБ ~15 ОИС. Основные: А-Токен (Альфа, лидер по объёму рыночных размещений), Мастерчейн, Сбер, Атомайз (первый ОИС, 2022, группа Т-Технологий), Токеон (ПСБ), Лайтхаус, НРД, СПБ Биржа, Блокчейн Хаб (МТС), Еврофинанс Моснарбанк и др.
- Боли рынка, о которых говорят: слабое вторичное обращение, нет интероперабельности между платформами, лимиты для неквалов — если спросят «что знаешь о ЦФА», упоминание этого покажет погружение.

### Технологии платформ
- Мастерчейн — российская сеть на базе Ethereum-стека с ГОСТ-криптографией (АФТ/ЦБ) → EVM-опыт и Web3j-модель мышления прямо релевантны.
- Атомайз — Hyperledger Fabric (chaincode вместо EVM-контрактов) — вероятно, но уточни на интервью, какая платформа у заказчика.
- Типовой enterprise-выбор для EVM: Hyperledger Besu / Quorum — приватные сети, консенсус PoA/IBFT/QBFT, мгновенная финальность, газ обычно нулевой.
- Твой главный тезис: «модель одна и та же — смарт-контракт, транзакции, события, JSON-RPC; я это делал на EVM-совместимой сети, перенос на вашу платформу — вопрос конкретики API».

### Вопрос себе перед 3-м этапом

## 13. FIX — краткая заготовка (в JD упомянут)

FIX (Financial Information eXchange) — стандартный протокол обмена торговыми сообщениями между брокерами, биржами и буферами ликвидности: ордера, исполнения (execution reports), котировки. Текстовый формат tag=value (8=FIX.4.4|35=D|...), сессионный уровень: логон, heartbeat, sequence numbers, resend request при пропусках. В Java — QuickFIX/J.

Честный ответ, если спросят: «Hands-on с FIX не было — на моих проектах транспортом были REST, RabbitMQ/Kafka и RPC. Модель понимаю: сессии, порядковые номера, гарантии доставки через resend — идейно близко к тому, что я делал с офсетами Kafka и идемпотентной обработкой. Разберусь быстро, если на проекте есть FIX-интеграция».
