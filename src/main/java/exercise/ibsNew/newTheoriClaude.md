## <a id="s2"></a>2. Java Core [↑](#toc)

### <a id="q2-1"></a>2.1. Разница `&&`/`&` и `||`/`|` [↑](#toc)
**Суть:** `&&`/`||` — короткое замыкание (второй операнд не вычисляется, если результат ясен); `&`/`|` — всегда вычисляют оба операнда; для чисел `&`/`|` — побитовые операции.

- `if (obj != null && obj.isValid())` — безопасно; с `&` — NPE, т.к. правая часть вычислится всегда.
- `&`/`|` для boolean нужны редко — когда правая часть имеет side effect, который обязан выполниться.

### <a id="q2-2"></a>2.2. Методы класса Object [↑](#toc)
**Суть:** `equals`, `hashCode`, `toString`, `getClass`, `clone`, `finalize` (deprecated), `wait` (3 перегрузки), `notify`, `notifyAll`.

- `wait/notify/notifyAll` — `final`, работают с монитором объекта.
- `getClass` — `final`, возвращает runtime-класс.
- `clone` — `protected`, требует `Cloneable`, иначе `CloneNotSupportedException`.
- `finalize` — deprecated for removal; замена — `Cleaner` / try-with-resources.

### <a id="q2-3"></a>2.3. Неправильные `hashCode()`/`equals()`: последствия на уровне метода / сервиса / приложения [↑](#toc)
**Суть:** контракт: равные по `equals` объекты обязаны иметь одинаковый `hashCode`. Нарушение ломает все hash-структуры.

- **Метод:** объект положили в `HashSet`/`HashMap` — `contains`/`get`/`remove` его не находит (ищет не в том бакете); дубликаты в `Set`.
- **Сервис:** кэш по ключу-объекту даёт промахи → лишние запросы к БД; дедупликация не работает; «потерянные» элементы = утечка памяти.
- **Приложение:** рост памяти, деградация O(1)→O(n) (все объекты с одинаковым hashCode в один бакет), некорректные бизнес-результаты (двойные списания, дубли заказов).
- **Как избежать:** генерировать оба метода вместе (IDE / Lombok `@EqualsAndHashCode` / records), только по неизменяемым полям, не мутировать ключи после помещения в коллекцию; тесты (EqualsVerifier).

### <a id="q2-4"></a>2.4. `this` и `super` [↑](#toc)
**Суть:** `this` — ссылка на текущий объект; `super` — доступ к членам родителя. `this(...)`/`super(...)` — вызов конструктора, должен идти первой инструкцией (классически).

- `this.field` — разрешение затенения параметром; `this(...)` — делегирование другому конструктору.
- `super.method()` — вызов переопределённой версии родителя; `super(...)` — конструктор родителя (вставляется неявно, если нет своего).
- Правила: в одном конструкторе либо `this(...)`, либо `super(...)`; недоступны в static-контексте.
- Нюанс новых версий: Java 25 (JEP 513, flexible constructor bodies) разрешает код до `super(...)`, кроме обращения к `this`.

### <a id="q2-5"></a>2.5. HashMap при плохой хэш-функции: влияние и оптимизации [↑](#toc)
**Суть:** плохой `hashCode` → все ключи в несколько бакетов → цепочки → поиск O(n) вместо O(1). С Java 8 бакет превращается в красно-чёрное дерево (≥8 элементов и capacity ≥64) → O(log n), но только если ключи `Comparable`.

- **Метод:** `get`/`put` медленные, CPU на `equals`-переборе цепочки.
- **Сервис:** рост latency под нагрузкой, узкое место в горячем кэше; уязвимость **HashDoS** (специально подобранные коллизии из внешнего ввода).
- **Приложение:** каскадная деградация зависимых сервисов, таймауты.
- **Оптимизации:** метод — качественный `hashCode` (все значимые поля, IDE/records), иммутабельные ключи, initial capacity/loadFactor под объём; сервис — профилирование, кэш с нормальными ключами (String/UUID); архитектура — лимиты на внешние данные-ключи, шардирование, мониторинг latency.

### <a id="q2-6"></a>2.6. Fail-fast vs fail-safe итераторы [↑](#toc)
**Суть:** fail-fast бросает `ConcurrentModificationException` при изменении коллекции во время обхода; fail-safe работает по копии/снапшоту и исключение не бросает.

- Fail-fast: `ArrayList`, `HashMap` — через `modCount` (best-effort, не гарантия). Легальное удаление — `iterator.remove()`.
- Fail-safe: `CopyOnWriteArrayList` (итератор по снапшоту), `ConcurrentHashMap` (weakly consistent — может видеть или не видеть параллельные изменения).
- Цена fail-safe: копирование/устаревшие данные.

### <a id="q2-7"></a>2.7. Абстрактный класс vs интерфейс [↑](#toc)
**Суть:** абстрактный класс — общее состояние + частичная реализация + конструкторы, одиночное наследование; интерфейс — контракт, множественная реализация, поля только `public static final`.

- Интерфейс: `default`/`static`/`private` методы (с Java 8/9), нет состояния экземпляра.
- Абстрактный класс: любые модификаторы доступа, поля, конструкторы (для подклассов).
- Выбор: «is-a» с общим кодом и состоянием → абстрактный класс (Template Method); «умеет X», API, множественность → интерфейс. Дефолт — интерфейс.

### <a id="q2-8"></a>2.8. Перегрузка/сокрытие статических методов [↑](#toc)
**Суть:** перегрузка (overload) static-методов работает как обычно — разные сигнатуры. Переопределить (override) static нельзя — происходит **сокрытие (hiding)**: вызов резолвится по типу ссылки на этапе компиляции.

- `Parent p = new Child(); p.staticMethod()` → вызовется версия `Parent` (нет полиморфизма).
- Зачем перегрузка: удобные варианты API (`valueOf(int)`, `valueOf(String)`).
- Вызывать static через имя класса, не через переменную — иначе маскируется сокрытие.

### <a id="q2-9"></a>2.9. Проблемы try-with-resources и производительность [↑](#toc)
**Суть:** сам механизм почти бесплатен; проблемы — семантические: подавленные исключения, порядок закрытия, ресурс вне try.

- Исключение из `close()` при уже летящем исключении становится **suppressed** — можно потерять причину, если не смотреть `getSuppressed()`.
- Ресурсы закрываются в обратном порядке объявления.
- Если конструктор ресурса бросил до входа в try — закрывать нечего, но уже созданные ранее ресурсы в том же выражении закроются.
- **Производительность:** метод — накладные ничтожны; сервис/приложение — критично обратное: *отсутствие* закрытия = утечка соединений/дескрипторов → исчерпание connection pool → отказ. `close()` Connection в пуле = возврат в пул (быстро), файлов — flush буферов (может быть заметен).

### <a id="q2-10"></a>2.10. Механизм AutoCloseable [↑](#toc)
**Суть:** интерфейс с `close() throws Exception`; компилятор разворачивает try-with-resources в try-finally с вызовом `close()` и `addSuppressed()`.

- `Closeable` (IO) — подтип с `IOException`, рекомендована идемпотентность `close()`.
- Файлы: без `close()` буферы не сбрасываются — потеря данных, незакрытые дескрипторы.
- БД: `close()` у пулированного Connection возвращает его в пул; незакрытый Statement/ResultSet — утечки на стороне драйвера/БД (курсоры).

### <a id="q2-11"></a>2.11. ClassLoader и ClassNotFoundException [↑](#toc)
**Суть:** иерархия Bootstrap → Platform → Application (system); модель **parent delegation** — сначала спросить родителя, потом искать самому. Классы грузятся лениво, идентичность класса = (имя + загрузчик).

- `ClassNotFoundException` — checked, при явной загрузке (`Class.forName`, `loadClass`) класс не найден в classpath.
- Отличать от `NoClassDefFoundError` — класс был при компиляции, но недоступен/не инициализировался в runtime (часто — упавший static-инициализатор).
- Решение: проверить classpath/зависимости (mvn dependency:tree), конфликты версий, наличие в fat-jar/образе; в контейнерах приложений — свои иерархии загрузчиков.

### <a id="q2-12"></a>2.12. Рефлексия: проблемы, производительность, безопасность, защита [↑](#toc)
**Суть:** обход инкапсуляции и проверок компилятора: медленнее (слабее JIT-оптимизации, проверки доступа), ломко при рефакторинге, риски безопасности.

- Проблемы: доступ к private, `setAccessible(true)`, изменение полей (мутация `final` — с Java 26 warning, далее будет блокироваться), конфликты с JPMS (модуль должен `opens` пакет).
- Производительность: вызовы через `Method.invoke` дороже прямых; смягчение — кэширование `Method`/`MethodHandle`.
- Защита: JPMS (не открывать пакеты), для singleton — `enum` (не создаётся рефлексией), бросать исключение в конструкторе при повторном создании, не полагаться на приватность как на безопасность.

### <a id="q2-13"></a>2.13. `var` [↑](#toc)
**Суть:** локальный вывод типа (Java 10). Тип статический, выводится компилятором из инициализатора. Не динамическая типизация.

- Только: локальные переменные с инициализатором, `for`, try-with-resources; нельзя: поля, параметры, возвращаемый тип, `var x = null`.
- Оптимально: тип очевиден из правой части (`var user = new User()`), длинные generic-типы, короткие области видимости.
- Избегать, когда тип неочевиден (`var result = service.process()`), — читаемость важнее.

### <a id="q2-14"></a>2.14. Variable shadowing [↑](#toc)
**Суть:** локальная переменная/параметр с тем же именем скрывает поле класса (или переменную внешней области) — обращение по имени идёт к ближайшей.

- Классика: `this.name = name` в конструкторе/сеттере — доступ к полю через `this`.
- Лямбды/внутренние блоки не могут переобъявлять локальные переменные метода (ошибка компиляции) — shadowing касается полей.
- Избегать: осмысленные имена, короткие методы, предупреждения IDE.

### <a id="q2-15"></a>2.15. `final` vs effectively final [↑](#toc)
**Суть:** `final` — явный запрет переприсваивания; effectively final — переменная фактически не меняется после инициализации, компилятор трактует её как final.

- Нужно для лямбд и анонимных классов: захватывать можно только final/effectively final локальные переменные.
- Причина: захват по значению (копия); изменяемость дала бы рассинхронизацию и гонки.
- Обход при необходимости — контейнер (`AtomicInteger`, массив на 1 элемент) — но чаще это сигнал переписать код.

### <a id="q2-16"></a>2.16. Generics и PECS [↑](#toc)
**Суть:** параметризация типов для compile-time безопасности; в runtime — стирание типов (erasure). **PECS: Producer Extends, Consumer Super.**

- `List<? extends Number>` — производитель: читаем как Number, добавлять нельзя (кроме null).
- `List<? super Integer>` — потребитель: пишем Integer, читаем как Object.
- Пример из JDK: `Collections.copy(List<? super T> dest, List<? extends T> src)`.
- Следствия erasure: нет `new T()`, `instanceof List<String>`, массивов `T[]`; `List<String>` и `List<Integer>` — один класс в runtime.

### <a id="q2-17"></a>2.17. Циклические зависимости и инструменты решения [↑](#toc)
**Суть:** A зависит от B, B от A — на уровне классов, пакетов, Spring-бинов или сборочных модулей. Решение — разрыв цикла через абстракцию или пересмотр границ.

- **Spring:** цикл при constructor injection → `BeanCurrentlyInCreationException` (с Boot 2.6 циклы запрещены по умолчанию). Обходы: рефакторинг (вынести общее в третий бин — правильный путь), `@Lazy` на одной из зависимостей, setter/field injection, events (`ApplicationEventPublisher`).
- **Классы/пакеты:** выделить интерфейс (DIP), инвертировать зависимость, события/колбэки.
- **Maven/Gradle:** цикл модулей не собирается — только пересмотр структуры.
- **Инструменты обнаружения:** IDEA Dependency Analysis, ArchUnit-тесты, JDepend, Sonar.

### <a id="q2-18"></a>2.18. Suppressed Exceptions [↑](#toc)
**Суть:** исключения, подавленные при закрытии ресурсов в try-with-resources, когда основное исключение уже летит; прикрепляются к основному через `addSuppressed()`, читаются `getSuppressed()`.

- Решает проблему старого try-finally: исключение из finally затирало исходное.
- Основное исключение — из тела try; из `close()` — в suppressed.

### <a id="q2-19"></a>2.19. Кастомная аннотация и RetentionPolicy [↑](#toc)
**Суть:** `@interface` + элементы с `default` + мета-аннотации. `@Retention` определяет, до какого этапа аннотация живёт.

```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface Audited { String value() default ""; }
```
- **RetentionPolicy:** `SOURCE` — только исходники (Lombok, обработка annotation processor); `CLASS` — в байткоде, невидима рефлексии (default); `RUNTIME` — доступна рефлексии (Spring, валидация).
- Прочие мета-аннотации: `@Target`, `@Documented`, `@Inherited`, `@Repeatable`.
- Обработка: рефлексия (`method.isAnnotationPresent`), AOP-аспект, annotation processor на компиляции.

### <a id="q2-20"></a>2.20. `@SneakyThrows` и `@SuppressWarnings` [↑](#toc)
**Суть:** `@SneakyThrows` (Lombok) — бросить checked-исключение без `throws` в сигнатуре; `@SuppressWarnings` — стандартная Java-аннотация (не Lombok), подавляет предупреждения компилятора.

- `@SneakyThrows` работает трюком на уровне байткода/generics: JVM не различает checked/unchecked — обманывается только компилятор. Риск: вызывающий код не ожидает checked-исключение и не может его поймать по типу напрямую.
- `@SuppressWarnings("unchecked")`, `"deprecation"` — применять максимально локально (на переменную/метод, не класс).

### <a id="q2-21"></a>2.21. Memory leaks в Java [↑](#toc)
**Суть:** утечка при GC = объекты достижимы из GC roots, но больше не нужны — GC их не соберёт.

- Типовые источники: **static-коллекции** и кэши без eviction; **слушатели/колбэки**, которые не отписали; **ThreadLocal в пулах потоков** (поток живёт вечно → значение тоже); незакрытые ресурсы; inner-классы, держащие ссылку на outer; ClassLoader leaks при редеплое.
- Избежать: кэши с ограничением и TTL (Caffeine), `WeakHashMap`/`WeakReference` для вспомогательных мапов, `threadLocal.remove()` в finally, try-with-resources.
- Диагностика: рост old gen на графиках, heap dump (`jcmd GC.heap_dump`) → Eclipse MAT (dominator tree), JFR.

### <a id="q2-22"></a>2.22. Optional [↑](#toc)
**Суть:** контейнер «значение может отсутствовать» — явная альтернатива null для **возвращаемых значений**.

- Создание: `of` (NPE на null), `ofNullable`, `empty`.
- Конструкции: `map`/`flatMap`/`filter` — цепочки; `orElse` (вычисляется всегда) vs `orElseGet` (лениво) vs `orElseThrow`; `ifPresent`/`ifPresentOrElse`; `or`; `stream()`.
- Антипаттерны: `isPresent()+get()` вместо функциональных методов; Optional в полях, параметрах, коллекциях; `Optional.of(x).orElse(...)` как замена if.

### <a id="q2-23"></a>2.23. Records [↑](#toc)
**Суть:** (Java 16) компактные иммутабельные носители данных: канонический конструктор, аксессоры `name()`, `equals`/`hashCode`/`toString` — генерируются.

- Компактный конструктор — для валидации/нормализации: `record Range(int lo, int hi) { Range { if (lo > hi) throw ...; } }`.
- Могут реализовывать интерфейсы, иметь static-члены; наследоваться нельзя (неявно final, extends Record).
- Применение: DTO, ключи Map, value objects, record patterns в switch (Java 21).
- Иммутабельность неглубокая — поля-коллекции защищать копиями (`List.copyOf`).

### <a id="q2-24"></a>2.24. Sealed классы [↑](#toc)
**Суть:** (Java 17) закрытая иерархия: `sealed interface Shape permits Circle, Square`. Наследники обязаны быть `final`, `sealed` или `non-sealed`.

- Зачем: моделирование алгебраических типов (результат: Success/Failure), контроль расширения API.
- Синергия с pattern matching: switch по sealed-типу **исчерпывающий без default** — компилятор проверит покрытие всех вариантов.

### <a id="q2-25"></a>2.25. Последняя версия Java и что в ней добавлено [↑](#toc)
**Суть:** последняя — **Java 26** (март 2026, non-LTS); последний **LTS — Java 25** (сентябрь 2025).

- **Java 26:** HTTP/3 в `HttpClient`; AOT-кэш для любых GC, включая ZGC (Project Leyden); улучшения G1 (+5–15% throughput); warning при рефлексивном изменении `final`-полей; удалён Applet API; preview: Lazy Constants, Structured Concurrency, PEM API, примитивы в pattern matching.
- **Java 25 (LTS):** compact object headers, Scoped Values (final), module import declarations, flexible constructor bodies.
- На интервью достаточно: «работаю на LTS (17/21/25), слежу за фичами: виртуальные потоки (21), records/sealed/pattern matching (17–21)».

## <a id="s3"></a>3. Алгоритмы, структуры данных, паттерны, ООП [↑](#toc)

### <a id="q3-1"></a>3.1. Интерфейсы коллекций; EnumSet/EnumMap [↑](#toc)
**Суть:** `Iterable → Collection → List / Set / Queue(Deque)`; `Map` — отдельная иерархия.

- **List** — порядок, дубликаты, доступ по индексу (ArrayList, LinkedList).
- **Set** — уникальность (HashSet, LinkedHashSet, TreeSet — sorted).
- **Queue/Deque** — FIFO/двусторонняя (ArrayDeque, PriorityQueue).
- **Map** — ключ→значение (HashMap, LinkedHashMap, TreeMap).
- **EnumSet/EnumMap** — только enum-ключи: внутри битовая маска / массив по `ordinal()` → компактнее и быстрее hash-аналогов, порядок объявления констант.

### <a id="q3-2"></a>3.2. Внутреннее устройство HashMap [↑](#toc)
**Суть:** массив бакетов `Node[]`; индекс = `(n-1) & hash`, где hash = `h ^ (h >>> 16)` (spread старших бит). Коллизии — связный список; при ≥8 элементов в бакете и capacity ≥64 — красно-чёрное дерево (O(log n)), обратно в список при ≤6.

- Resize: при `size > capacity * loadFactor` (0.75) — удвоение массива и перераспределение.
- null-ключ — один, в бакете 0.
- put: вычислить hash → бакет → сравнение по hash, затем `equals` → замена значения или добавление узла.

### <a id="q3-3"></a>3.3. Красно-чёрное дерево [↑](#toc)
**Суть:** самобалансирующееся бинарное дерево поиска: узлы красные/чёрные, инварианты (красный не имеет красного ребёнка; одинаковое число чёрных на любом пути вниз) → высота ≤ 2·log(n), операции гарантированно O(log n).

- Балансировка через повороты и перекраску при вставке/удалении.
- Где: `TreeMap`/`TreeSet`, бакеты `HashMap` (Java 8+), планировщик Linux (CFS).

### <a id="q3-4"></a>3.4. Основные структуры данных; что применяю [↑](#toc)
**Суть:** массив, динамический массив, связный список, стек, очередь/дек, хэш-таблица, дерево (BST/сбалансированные), куча, граф.

- В работе: `ArrayList` (по умолчанию), `HashMap`/`HashSet` (индексация, дедупликация), `ArrayDeque` (стек/очередь), `LinkedHashMap` (LRU-кэш через `removeEldestEntry`), `PriorityQueue` (top-K, планирование), `ConcurrentHashMap` (конкурентные кэши).

### <a id="q3-5"></a>3.5. Fail-fast и fail-safe коллекции [↑](#toc)
**Суть:** то же различие, что у итераторов ([2.6](#q2-6)): обычные коллекции (`ArrayList`, `HashMap`) — fail-fast (CME при concurrent-модификации); `CopyOnWriteArrayList` (снапшот) и `ConcurrentHashMap` (weakly consistent) — fail-safe.

- Использование: COW — редкие записи, частые чтения (слушатели); CHM — конкурентный доступ общего вида.

### <a id="q3-6"></a>3.6. Hashtable vs ConcurrentHashMap [↑](#toc)
**Суть:** оба потокобезопасны, но Hashtable — один монитор на всю таблицу (synchronized-методы), legacy; CHM — CAS + synchronized на уровне бакета, чтения без блокировок.

- Hashtable: null-ключи/значения запрещены; низкая масштабируемость; не использовать.
- CHM: weakly consistent итераторы (без CME), атомарные `putIfAbsent`, `compute`, `merge`; `size()` приблизительный по своей природе конкурентности.

### <a id="q3-7"></a>3.7. Big O [↑](#toc)
**Суть:** асимптотическая оценка роста времени/памяти от размера входа, отбрасывая константы: O(1), O(log n), O(n), O(n log n), O(n²), O(2ⁿ).

- Различать худший/средний/амортизированный случай: `ArrayList.add` — амортизированно O(1); `HashMap.get` — средне O(1), худший O(log n)/O(n).
- Сложность по памяти — дополнительная память алгоритма (merge sort O(n), quick sort O(log n) на стек).

### <a id="q3-8"></a>3.8. Сортировки: quick sort vs merge sort [↑](#toc)
**Суть:** quick — in-place, средний O(n log n), худший O(n²) (плохой pivot), нестабильная; merge — стабильная, гарантированный O(n log n), но O(n) доп. памяти.

- Merge дружелюбна к связным спискам и внешней сортировке (данные не помещаются в память).
- В JDK: примитивы — dual-pivot quicksort; объекты — **TimSort** (merge + insertion, стабильный, использует упорядоченные подпоследовательности).

### <a id="q3-9"></a>3.9. Коммивояжёр / рюкзак: подход [↑](#toc)
**Суть:** обе NP-трудные → на малых n точные методы, на больших — приближённые.

- **Рюкзак 0/1:** DP-таблица `dp[i][w]` — максимум ценности для первых i предметов и вместимости w; O(n·W) — псевдополиномиально. На интервью — это основной ожидаемый ответ.
- **Коммивояжёр:** точные — полный перебор O(n!), DP Хелда-Карпа O(2ⁿ·n²), branch and bound; приближённые — ближайший сосед, 2-opt, жадные, метаэвристики (генетические, simulated annealing).
- Проговорить trade-off: точность vs время, оценка размера входа.

### <a id="q3-10"></a>3.10. Жадный алгоритм [↑](#toc)
**Суть:** на каждом шаге локально оптимальный выбор без пересмотра. Корректен, если у задачи есть свойство жадного выбора + оптимальная подструктура.

- Примеры: выбор заявок (activity selection — брать с минимальным концом), Хаффман, Дейкстра, Прим/Крускал (MST), сдача монет для канонических номиналов.
- Контрпример: сдача монет {1,3,4} для 6 — жадный даёт 4+1+1, оптимум 3+3 → нужна DP.

### <a id="q3-11"></a>3.11. Композиция/ассоциация: has-a vs is-a [↑](#toc)
**Суть:** is-a — наследование/реализация (`Dog extends Animal`); has-a — объект содержит другой (Car has Engine).

- Ассоциация — общее «знает о»; **агрегация** — слабое владение (части живут отдельно: кафедра–преподаватели); **композиция** — сильное (часть умирает с целым: дом–комнаты).
- Принцип: предпочитать композицию наследованию — гибче, слабее связность, поведение можно менять в runtime (Strategy).

### <a id="q3-12"></a>3.12. CAP-теорема [↑](#toc)
**Суть:** в распределённой системе при сетевом разделении (Partition) нельзя одновременно сохранить Consistency (линеаризуемость) и Availability — выбираем одно.

- P — не опция, сеть ненадёжна → реальный выбор CP vs AP **во время partition**.
- CP: отказать части запросов ради согласованности (ZooKeeper, etcd, HBase). AP: отвечать всегда, согласованность eventual (Cassandra, DynamoDB).
- Расширение PACELC: без partition — trade-off latency vs consistency.

### <a id="q3-13"></a>3.13. MapReduce и divide and conquer [↑](#toc)
**Суть:** D&C — разделить задачу, решить подзадачи рекурсивно, объединить результаты. MapReduce — map (независимое преобразование элементов) + reduce (свёртка) — параллелится тривиально.

- **Метод:** `list.stream().map(...).reduce(...)`; binary search, merge sort — D&C.
- **Класс/сервис:** `parallelStream`, Fork/Join (work-stealing) для CPU-bound.
- **Приложение:** Hadoop/Spark, Kafka Streams (агрегации по ключам), map-фаза шардированной обработки + reduce-агрегация.

### <a id="q3-14"></a>3.14. Любимый паттерн (заготовка ответа) [↑](#toc)
**Суть:** готовый ответ — **Strategy + Spring DI**.

- «Чаще всего использую Strategy: инжектирую `Map<String, Handler>` или `List<Handler>` — Spring сам собирает все реализации. Выбор обработчика по типу события/платежа вместо switch.
- Реальный пример: обработчики разных типов сообщений из RabbitMQ — интерфейс `MessageHandler` с методом `supports(type)`, диспетчер выбирает нужный. Добавление нового типа = новый бин, без изменения существующего кода (OCP)».

### <a id="q3-15"></a>3.15. Паттерн Фабрика [↑](#toc)
**Суть:** инкапсуляция логики создания объектов: клиент запрашивает по параметрам, не зная конкретного класса.

- Варианты: **Factory Method** (подклассы решают, что создавать), **Abstract Factory** (семейства связанных объектов), простая статическая фабрика.
- Преимущества перед `new`: единая точка создания, слабая связанность (возврат по интерфейсу), подмена в тестах, скрытие сложной инициализации, кэширование/пулы.
- Примеры JDK/Spring: `List.of`, `Executors.newFixedThreadPool`, `BeanFactory`.

### <a id="q3-16"></a>3.16. Паттерн Стратегия [↑](#toc)
**Суть:** семейство взаимозаменяемых алгоритмов за одним интерфейсом; выбор в runtime; убирает разрастающиеся if/switch.

- Оправдано: несколько способов сделать одно (расчёт комиссии, тарификация, форматы экспорта), поведение зависит от конфигурации/типа входа.
- JDK: `Comparator` — стратегия сравнения. Spring: `Map<String, Strategy>` через DI.

### <a id="q3-17"></a>3.17. Декоратор vs Адаптер [↑](#toc)
**Суть:** оба — обёртки, разница в цели: Декоратор **добавляет поведение**, сохраняя тот же интерфейс; Адаптер **преобразует интерфейс** к ожидаемому клиентом.

- Декоратор: `new BufferedInputStream(new FileInputStream(f))` — тот же InputStream + буферизация; `Collections.unmodifiableList`.
- Адаптер: `InputStreamReader` (байтовый поток → символьный), `Arrays.asList`, Spring `HandlerAdapter`.
- Декораторы компонуются в цепочки; адаптер обычно один.

### <a id="q3-18"></a>3.18. Паттерн Visitor [↑](#toc)
**Суть:** вынести операции над иерархией объектов в отдельный класс без изменения самих классов; работает через **double dispatch**: `element.accept(visitor)` → `visitor.visit(this)`.

- Плюс: новые операции — новый visitor, элементы не трогаем. Минус: новый тип элемента — менять все visitors.
- Применение: обход AST (компиляторы), экспорт структуры в разные форматы, расчёты по иерархии документов.
- Современная альтернатива: sealed-иерархия + pattern matching switch — компилятор проверяет полноту.

### <a id="q3-19"></a>3.19. Инкапсуляция, наследование, полиморфизм [↑](#toc)
**Суть:** инкапсуляция — скрытие состояния за интерфейсом (private поля + методы, инварианты); наследование — переиспользование/расширение (extends); полиморфизм — один интерфейс, разные реализации.

- Полиморфизм: **подтипов** (переопределение, dynamic dispatch: `Animal a = new Dog(); a.sound()`), **параметрический** (generics), **ad-hoc** (перегрузка).
- Пример инкапсуляции: `BankAccount.withdraw()` проверяет баланс — извне поле не изменить.
- + четвёртый принцип — абстракция (выделение существенного, интерфейсы).

### <a id="q3-20"></a>3.20. Clean Code (Роберт Мартин) [↑](#toc)
**Суть:** код читается чаще, чем пишется — оптимизировать на читаемость.

- Осмысленные имена; функции маленькие и делают одно; мало аргументов (≤3), без флагов; без side effects.
- DRY; комментарии — последнее средство (код должен объяснять себя); единое форматирование.
- Ошибки — исключениями, не кодами возврата; не возвращать/не передавать null.
- Тесты: FIRST, чистота тестов равна чистоте кода. Boy scout rule: оставь код чище, чем нашёл.
- Рядом — SOLID: SRP, OCP, LSP, ISP, DIP.

## <a id="s4"></a>4. Spring [↑](#toc)

### <a id="q4-1"></a>4.1. IoC и его реализация в Spring [↑](#toc)
**Суть:** Inversion of Control — не объект создаёт и ищет свои зависимости, а контейнер создаёт объекты и внедряет зависимости (DI — конкретная форма IoC).

- Spring: `ApplicationContext` читает конфигурацию (аннотации/`@Configuration`/сканирование) → строит `BeanDefinition` → создаёт бины, внедряет зависимости (constructor/setter/field), управляет жизненным циклом.
- Выгода: слабая связанность, тестируемость (подмена зависимостей), декларативные сервисы (транзакции, AOP).

### <a id="q4-2"></a>4.2. Виды autowiring [↑](#toc)
**Суть:** по типу (byType), по имени (byName), через конструктор; на практике — аннотации.

- `@Autowired` — byType; при нескольких кандидатах: `@Primary`, затем сопоставление по имени поля/параметра, иначе — `NoUniqueBeanDefinitionException`; уточнение — `@Qualifier`.
- `@Resource` (JSR-250) — сначала byName, потом byType. `@Inject` (JSR-330) — как `@Autowired`, без `required`.
- Рекомендация: **constructor injection** — иммутабельность, явные обязательные зависимости, тестируемость, ловит циклы.

### <a id="q4-3"></a>4.3. Бин и его жизненный цикл [↑](#toc)
**Суть:** бин — объект под управлением контейнера. Цикл: определение → создание → внедрение → инициализация → использование → уничтожение.

1. Сканирование/чтение конфигурации → `BeanDefinition` (+ `BeanFactoryPostProcessor`, напр. properties).
2. Инстанцирование (конструктор) → внедрение зависимостей.
3. Aware-интерфейсы (`BeanNameAware`, `ApplicationContextAware`).
4. `BeanPostProcessor.postProcessBeforeInitialization`.
5. Инициализация: `@PostConstruct` → `InitializingBean.afterPropertiesSet` → `initMethod`.
6. `postProcessAfterInitialization` — здесь оборачивание в **AOP-прокси**.
7. Разрушение (только singleton): `@PreDestroy` → `DisposableBean.destroy` → `destroyMethod`.

### <a id="q4-4"></a>4.4. Настройка транзакционности и проблемы [↑](#toc)
**Суть:** `@Transactional` поверх `PlatformTransactionManager` (в Boot настраивается автоматически). Параметры: propagation, isolation, readOnly, rollbackFor, timeout.

- Типовые проблемы: **self-invocation** — вызов `this.method()` минует прокси, транзакция не открывается; метод не public; **checked-исключения не откатывают** по умолчанию (нужен `rollbackFor = Exception.class`); длинные транзакции с внешними HTTP-вызовами внутри (держат соединение); `REQUIRES_NEW` + внешняя транзакция → взаимные блокировки; `LazyInitializationException` после выхода из транзакции.
- Ключевая ошибка из скорокарты коллеги: **уведомления/внешние вызовы до коммита** — отправлять после (`TransactionalEventListener(phase = AFTER_COMMIT)` / outbox).

### <a id="q4-5"></a>4.5. Что даёт Spring Boot [↑](#toc)
**Суть:** автоконфигурация + starters + embedded server + производственные фичи → минимум конфигурации до работающего сервиса.

- Starters — согласованные версии зависимостей (BOM); embedded Tomcat/Netty — jar вместо war; внешняя конфигурация (yaml, profiles, env); Actuator; DevTools.
- Соглашения вместо конфигурации, всё переопределяемо.

### <a id="q4-6"></a>4.6. AOP в Spring [↑](#toc)
**Суть:** вынесение сквозной функциональности (логирование, транзакции, security, метрики, ретраи) из бизнес-кода в аспекты.

- Термины: aspect, advice (`@Before`, `@AfterReturning`, `@Around`), pointcut (выражение выбора точек), join point.
- Spring AOP — **прокси** (JDK dynamic для интерфейсов / CGLIB для классов): работает только на public-методах бинов, self-invocation не перехватывается; AspectJ (weaving) — без этих ограничений, но сложнее.
- На AOP построены `@Transactional`, `@Cacheable`, `@Async`, security-аннотации.

### <a id="q4-7"></a>4.7. Spring Actuator [↑](#toc)
**Суть:** production-ready эндпоинты для мониторинга и управления.

- `/health` (+liveness/readiness для k8s-проб), `/metrics` (Micrometer), `/prometheus`, `/info`, `/env`, `/loggers` (смена уровня логов на лету), `/threaddump`, `/heapdump`.
- Осторожно с экспозицией: `management.endpoints.web.exposure.include`, закрывать security, отдельный порт.

### <a id="q4-8"></a>4.8. Автоконфигурация Spring Boot [↑](#toc)
**Суть:** `@EnableAutoConfiguration` (внутри `@SpringBootApplication`) загружает список конфигураций из `META-INF/spring/...AutoConfiguration.imports` (ранее spring.factories); каждая срабатывает по условиям `@Conditional*`.

- Условия: `@ConditionalOnClass` (есть в classpath), `@ConditionalOnMissingBean` (пользовательский бин перекрывает дефолт), `@ConditionalOnProperty`.
- Пример: H2 в classpath + нет DataSource → сконфигурируется embedded DataSource.
- Отладка: флаг `--debug` → condition evaluation report; исключение: `@SpringBootApplication(exclude = ...)`.

### <a id="q4-9"></a>4.9. Spring и микросервисная архитектура [↑](#toc)
**Суть:** Boot — автономные сервисы; Spring Cloud — инфраструктурные паттерны.

- Spring Cloud Gateway (API gateway), Config (централизованная конфигурация), OpenFeign (декларативные клиенты), LoadBalancer, Resilience4j (circuit breaker/retry), discovery (Eureka/Consul/k8s), Micrometer Tracing (распределённый трейсинг).
- Асинхронность: Spring Kafka / Spring AMQP. Наблюдаемость: Actuator + Prometheus. Контейнеризация + k8s-пробы.

### <a id="q4-10"></a>4.10. `@Transactional`: как работает и ограничения [↑](#toc)
**Суть:** прокси открывает/коммитит/откатывает транзакцию вокруг метода; соединение и статус — в ThreadLocal.

- Ограничения: self-invocation мимо прокси; только public (в proxy-режиме); final-методы/классы не проксируются CGLIB; rollback по умолчанию только на RuntimeException/Error; не переживает смену потока (`@Async`, CompletableFuture — транзакция не распространяется); readOnly — подсказка (flush mode, оптимизации драйвера), не запрет.
- Propagation: REQUIRED (default), REQUIRES_NEW (приостановить текущую), NESTED (savepoint), SUPPORTS/MANDATORY/NEVER/NOT_SUPPORTED.

### <a id="q4-11"></a>4.11. ApplicationContext vs BeanFactory [↑](#toc)
**Суть:** BeanFactory — базовый контейнер (getBean, ленивое создание); ApplicationContext — его расширение с enterprise-функциями.

- ApplicationContext добавляет: eager-инициализацию singleton'ов, события (`ApplicationEvent`), `MessageSource` (i18n), доступ к ресурсам, автоматическую регистрацию `BeanPostProcessor`/`BeanFactoryPostProcessor`, Environment/profiles.
- На практике всегда ApplicationContext; BeanFactory — только очень ограниченные среды.

### <a id="q4-12"></a>4.12. `@Bean` [↑](#toc)
**Суть:** метод в `@Configuration`-классе, результат которого регистрируется как бин; для сторонних классов и тонкой настройки создания.

- Имя бина = имя метода (или `@Bean("name")`); `initMethod`/`destroyMethod`.
- В `@Configuration` (proxyBeanMethods=true) вызовы `@Bean`-методов друг из друга возвращают тот же singleton (CGLIB-прокси конфига).
- Когда: нельзя поставить `@Component` (чужая библиотека), нужно несколько вариантов одного типа, сложная инициализация.

### <a id="q4-13"></a>4.13. `@Component` vs `@Service` vs `@Repository` [↑](#toc)
**Суть:** все — стереотипы `@Component` (обнаруживаются сканированием); разница — семантика слоя + доп. поведение у `@Repository`.

- `@Repository` — трансляция исключений персистентности в иерархию `DataAccessException`.
- `@Service` — маркер бизнес-логики (без доп. поведения, но точка для аспектов/читаемость).
- `@Controller`/`@RestController` — web-слой (`@RestController` = `@Controller` + `@ResponseBody`).

### <a id="q4-14"></a>4.14. `@Autowired` vs `@Resource` vs `@Inject` [↑](#toc)
**Суть:** `@Autowired` (Spring) и `@Inject` (JSR-330) — byType; `@Resource` (JSR-250) — byName сначала.

- `@Autowired(required=false)` — необязательная зависимость; у `@Inject` нет required, но можно `Optional<T>`/`ObjectProvider`.
- `@Resource(name="...")` — явное имя; на параметры конструктора не ставится.
- Уточнение выбора: `@Qualifier`, `@Primary`.

### <a id="q4-15"></a>4.15. Внедрение бина в static-поле [↑](#toc)
**Суть:** напрямую `@Autowired static` не работает — Spring внедряет в экземпляры. Корректного способа по сути нет; это антипаттерн — избегать.

- Обходы (если очень нужно, напр. legacy/утилитный класс): нестатический сеттер, пишущий в static-поле; `@PostConstruct`, копирующий инжектированное поле в static; holder с `ApplicationContextAware`.
- Правильное решение — рефакторинг: сделать класс бином и внедрять зависимость нормально.

### <a id="q4-16"></a>4.16. `@Primary` и `@Qualifier` [↑](#toc)
**Суть:** при нескольких кандидатах одного типа: `@Primary` — выбор по умолчанию; `@Qualifier("name")` — явное указание, приоритетнее `@Primary`.

- Можно создавать собственные qualifier-аннотации (`@interface PaymentGateway` + `@Qualifier` мета).
- Альтернатива — инжект `Map<String, T>` и выбор в runtime.

### <a id="q4-17"></a>4.17. Инжект примитивов [↑](#toc)
**Суть:** `@Value("${app.timeout:30}")` — плейсхолдеры из properties/env с дефолтами, SpEL (`#{...}`); конвертация типов автоматическая.

- Для групп настроек — `@ConfigurationProperties(prefix="app")` — типизированный класс, валидация (`@Validated`), лучше для множества свойств.
- Программно — `Environment.getProperty`.

### <a id="q4-18"></a>4.18. Инжект коллекции бинов [↑](#toc)
**Суть:** `List<MyInterface>` — все бины типа; `Map<String, MyInterface>` — имя бина → бин.

- Порядок в List — через `@Order`/`Ordered`.
- База для Strategy/plugin-архитектуры: диспетчер по ключу из Map.
- Отфильтровать — кастомный `@Qualifier` на группе бинов.

### <a id="q4-19"></a>4.19. `@Conditional` [↑](#toc)
**Суть:** регистрация бина/конфигурации только при выполнении условия — интерфейс `Condition.matches()`.

- Готовые из Boot: `@ConditionalOnProperty`, `@ConditionalOnClass`, `@ConditionalOnMissingBean`, `@ConditionalOnBean`, `@ConditionalOnExpression`, `@ConditionalOnWebApplication`.
- Применение: фиче-флаги, окружения, взаимозаменяемые реализации; фундамент автоконфигурации Boot.

### <a id="q4-20"></a>4.20. `@Profile` [↑](#toc)
**Суть:** активация бинов/конфигураций только для заданных профилей: `@Profile("dev")`, `@Profile("!prod")`.

- Активация: `spring.profiles.active=dev`, env-переменная, аргумент JVM; файлы `application-dev.yml`.
- Типовое: заглушки внешних систем в dev/test, реальные интеграции в prod. По сути специализация `@Conditional`.

## <a id="s5"></a>5. Базы данных, SQL, Hibernate, JPA, Spring Data [↑](#toc)

### <a id="q5-1"></a>5.1. Виды индексов и их правильное использование [↑](#toc)
**Суть:** B-tree — стандарт (равенство, диапазоны, ORDER BY); плюс hash, составные, частичные, функциональные, покрывающие, GIN/GiST/BRIN (PostgreSQL).

- **Составной** — работает по leftmost prefix: индекс (a, b) покрывает WHERE a и WHERE a,b, но не WHERE b.
- **Частичный** — `WHERE status = 'ACTIVE'` — меньше и быстрее для горячего подмножества.
- **Функциональный** — по выражению (`lower(email)`); **покрывающий** (INCLUDE) — index-only scan без обращения к таблице.
- **GIN** — jsonb, полнотекстовый поиск, массивы; **BRIN** — огромные таблицы с естественным порядком (время).
- Правильно: индексировать под реальные запросы (WHERE/JOIN/ORDER BY), учитывать селективность, не плодить (каждый индекс замедляет запись), проверять `EXPLAIN (ANALYZE)`, следить за неиспользуемыми.

### <a id="q5-2"></a>5.2. Change Data Capture и уведомления об изменениях [↑](#toc)
**Суть:** промышленный способ — чтение журнала транзакций: **Debezium** (WAL/binlog → Kafka), логическая репликация PostgreSQL.

- Альтернативы: триггеры + таблица изменений; polling по `updated_at` (просто, но нагрузка и риск пропусков); **Transactional Outbox** — событие пишется в ту же транзакцию, relay публикует.
- Уведомления между сервисами: Kafka (Debezium/Kafka Connect), PostgreSQL `LISTEN/NOTIFY` для лёгких случаев.

### <a id="q5-3"></a>5.3. Триггеры [↑](#toc)
**Суть:** процедуры, автоматически выполняемые на INSERT/UPDATE/DELETE (BEFORE/AFTER, row-level/statement-level).

- Для отслеживания: AFTER-триггер пишет в audit-таблицу (кто, когда, старое/новое значение — `OLD`/`NEW`).
- Минусы: скрытая логика вне кода приложения, сложнее отладка и миграции, накладные на каждую запись → использовать точечно (аудит, инварианты уровня БД).

### <a id="q5-4"></a>5.4. Хранимые процедуры и функции [↑](#toc)
**Суть:** логика в БД (PL/pgSQL и т.п.); функция возвращает значение и вызывается в SELECT; процедура (PG 11+) может управлять транзакциями.

- Оправдано: массовые операции рядом с данными (исключить round-trip'ы и перекачку данных), сложные пакетные пересчёты, миграции.
- Минусы: версионирование/деплой, тестируемость, привязка к вендору, размазывание бизнес-логики.

### <a id="q5-5"></a>5.5. Материализованные представления [↑](#toc)
**Суть:** обычное представление — сохранённый запрос, выполняется при каждом обращении; материализованное — **физически сохранённый результат**, требует обновления.

- `REFRESH MATERIALIZED VIEW [CONCURRENTLY]` (CONCURRENTLY — без блокировки чтения, нужен unique-индекс).
- Применение: тяжёлая аналитика/агрегаты, дашборды. Цена: устаревание данных между refresh, место.

### <a id="q5-6"></a>5.6. Репликация между разными БД [↑](#toc)
**Суть:** одинаковый движок — встроенная (streaming/логическая репликация PG); гетерогенные — CDC (Debezium + Kafka Connect source/sink), ETL/ELT-пайплайны, foreign data wrappers.

- Учитывать: отставание (lag), порядок событий, идемпотентность приёмника, маппинг схем, initial snapshot + поток изменений.

### <a id="q5-7"></a>5.7. Entity и маппинг на таблицу [↑](#toc)
**Суть:** `@Entity` + `@Id`; `@Table`/`@Column` — соответствие таблице/колонкам.

- Требования: no-arg конструктор (protected ок), не final (прокси Hibernate), идентификатор.
- Практика: `@Enumerated(EnumType.STRING)` (не ORDINAL), `@Embedded` для value-объектов, ассоциации LAZY.
- `equals/hashCode` — осторожно: по id (null до persist) — типовое решение: `equals` по id c проверкой на null, `hashCode` константный/по классу; либо по бизнес-ключу.

### <a id="q5-8"></a>5.8. Транзакции и уровни изоляции [↑](#toc)
**Суть:** ACID; уровни изоляции — компромисс между аномалиями и конкурентностью.

| Уровень | Dirty read | Non-repeatable read | Phantom |
|---|---|---|---|
| READ UNCOMMITTED | возможно | возможно | возможно |
| READ COMMITTED (PG default) | нет | возможно | возможно |
| REPEATABLE READ | нет | нет | возможно* |
| SERIALIZABLE | нет | нет | нет |

- PostgreSQL — MVCC: READ UNCOMMITTED = READ COMMITTED; *RR в PG — snapshot isolation, фантомов фактически нет, но возможны аномалии сериализации (write skew) → SERIALIZABLE (SSI) с ретраями.
- Пример: два одновременных чтения баланса и списание — на RC оба прочитают старый баланс; лечится SELECT FOR UPDATE / оптимистичной блокировкой / атомарным UPDATE.

### <a id="q5-9"></a>5.9. Lazy loading и его проблемы [↑](#toc)
**Суть:** `FetchType.LAZY` — прокси/ленивые коллекции, загрузка при первом обращении внутри открытой сессии.

- Проблемы: **LazyInitializationException** вне сессии/транзакции; **N+1** при итерации; Open-Session-In-View маскирует проблему и держит соединение на весь запрос (в Boot включён по умолчанию — осознанно выключать).
- Решения: `JOIN FETCH` в запросе, `@EntityGraph`, DTO-проекции (не тащить сущности в web-слой), batch fetching.

### <a id="q5-10"></a>5.10. Каскадные операции (Cascade) [↑](#toc)
**Суть:** распространение операции с родителя на связанные сущности: PERSIST, MERGE, REMOVE, REFRESH, DETACH, ALL.

- `orphanRemoval=true` — удаление «осиротевшего» ребёнка при удалении из коллекции (сильнее, чем CascadeType.REMOVE — тот только при удалении родителя).
- Практика: на `@OneToMany` (владение композицией) — PERSIST+MERGE (+orphanRemoval при полном владении); **REMOVE на @ManyToMany не использовать** — удалит общие сущности.

### <a id="q5-11"></a>5.11. N+1 проблема [↑](#toc)
**Суть:** 1 запрос за родителями + N запросов за детьми при обращении к lazy-ассоциации в цикле.

- Обнаружение: SQL-логи (p6spy/datasource-proxy, счётчик запросов), Hypersistence Utils asserts в тестах, APM.
- Решения: `JOIN FETCH` (осторожно с пагинацией коллекций — in-memory pagination HHH000104), `@EntityGraph`, **batch fetching** (`default_batch_fetch_size`/`@BatchSize` — IN-запросы пачками), `@Fetch(SUBSELECT)`, DTO-проекции.
- Личный кейс для интервью: поиск инвойсов — устранение N+1 + переработка запросов: 8–11 c → ~180 мс.

### <a id="q5-12"></a>5.12. Стратегии генерации первичных ключей [↑](#toc)
**Суть:** `IDENTITY`, `SEQUENCE`, `TABLE`, `AUTO`; отдельно UUID.

- **SEQUENCE** — рекомендуемая для PG: `allocationSize` кэширует диапазон, дружит с JDBC-батчингом.
- **IDENTITY** — insert нужен для получения id → Hibernate **отключает батчинг вставок**.
- **TABLE** — эмуляция через таблицу, медленно, legacy. **AUTO** — выбор по диалекту.
- **UUID** — генерация на клиенте, распределённые системы; v4 фрагментирует B-tree индекс, лучше time-ordered (v7).

### <a id="q5-13"></a>5.13. Мониторинг производительности запросов [↑](#toc)
**Суть:** три слоя — приложение, пул, БД.

- Приложение: SQL-логирование через **p6spy/datasource-proxy** (с таймингами; show_sql в проде — нет), `hibernate.generate_statistics` + метрики, `slow query` порог.
- Пул: метрики HikariCP (active/idle/pending) через Micrometer → Prometheus/Grafana.
- БД: `pg_stat_statements` (топ по времени), `EXPLAIN (ANALYZE, BUFFERS)`, auto_explain, slow query log.

### <a id="q5-14"></a>5.14. Оптимистическая vs пессимистическая блокировка [↑](#toc)
**Суть:** оптимистическая — конфликт обнаруживаем при записи (`@Version`: UPDATE ... WHERE version = ?; иначе `OptimisticLockException` → retry); пессимистическая — блокируем строку при чтении (`SELECT ... FOR UPDATE`, `PESSIMISTIC_WRITE`).

- Выбор: конфликты редки / допустим retry / долгие «пользовательские» транзакции → оптимистическая; горячие строки, строгие инварианты (баланс, остатки), короткие транзакции → пессимистическая.
- Риски пессимистической: ожидания, deadlock (лочить в едином порядке), масштабируемость.

### <a id="q5-15"></a>5.15. Пагинация в Spring Data и большие объёмы [↑](#toc)
**Суть:** `Pageable`/`PageRequest.of(page, size, sort)` → `Page<T>` (с count-запросом) или `Slice<T>` (без count — дешевле).

- Большие объёмы: **offset-пагинация деградирует** (OFFSET читает и отбрасывает строки) → **keyset/seek**: `WHERE (created, id) < (:lastCreated, :lastId) ORDER BY created DESC, id DESC LIMIT :n`; в Spring Data 3.1+ — Scroll API (`WindowIterator`, keyset).
- count дорогой — избегать `Page`, если total не нужен; выгрузки — `Stream<T>` в транзакции с fetch size.
- `JOIN FETCH` коллекции + Pageable = пагинация в памяти (HHH000104) — разделять запросы.

### <a id="q5-16"></a>5.16. Second Level Cache [↑](#toc)
**Суть:** L1 — кэш сессии (всегда включён); L2 — кэш сущностей на уровне SessionFactory между сессиями.

- Настройка: провайдер через JCache (Ehcache, Infinispan, Caffeine), `hibernate.cache.use_second_level_cache=true`, на сущности `@Cacheable` + `@org.hibernate.annotations.Cache(usage = READ_WRITE/READ_ONLY/NONSTRICT_READ_WRITE)`; query cache отдельно (`use_query_cache`).
- Подходит: справочники, читаемые намного чаще, чем пишутся. Риски: инвалидация при изменениях мимо Hibernate, консистентность в кластере (нужен распределённый кэш).

### <a id="q5-17"></a>5.17. Кастомные запросы: `@Query` [↑](#toc)
**Суть:** derived queries по имени метода — для простого; `@Query` — JPQL или native (`nativeQuery = true`).

- Параметры: `:name` + `@Param`; `@Modifying` для UPDATE/DELETE (+ `clearAutomatically = true` — сброс persistence context).
- Проекции: интерфейсные (`interface UserView { String getName(); }`) или DTO через конструктор в JPQL.
- `Pageable`/`Sort` совместимы с `@Query`; SpEL: `#{#entityName}`.

### <a id="q5-18"></a>5.18. Миграции схемы: Liquibase / Flyway [↑](#toc)
**Суть:** версионированные миграции в репозитории, применяются автоматически (при старте приложения или шагом CI/CD); таблица истории следит за применённым.

- **Flyway** — SQL-first: `V1__init.sql`, checksum, простота. **Liquibase** — changelog (XML/YAML/SQL), rollback-сценарии, preconditions, contexts.
- Практика: `ddl-auto=validate` (никогда update в проде), миграции обратно-совместимые (expand → migrate → contract) для zero-downtime, прогон на копии прода в pipeline.

### <a id="q5-19"></a>5.19. Связи One-to-One / One-to-Many / Many-to-Many и подводные камни [↑](#toc)
**Суть:** владелец связи — сторона с FK (`@ManyToOne`/`@JoinColumn`); у обратной — `mappedBy`.

- **@OneToMany двунаправленная:** синхронизировать обе стороны helper-методами (`addChild`); забытая синхронизация — рассинхрон состояния.
- **@OneToOne:** LAZY на не-owning стороне не работает без bytecode enhancement (Hibernate должен знать, null или нет) → shared PK + `@MapsId`.
- **@ManyToMany:** лучше явная сущность связи (join-таблица с полями всё равно понадобится); CascadeType.REMOVE — опасно.
- Общие грабли: N+1, `EAGER` по умолчанию у `@ManyToOne` (ставить LAZY), Lombok `@ToString/@EqualsAndHashCode` по ассоциациям → рекурсия/инициализация lazy.

### <a id="q5-20"></a>5.20. Фильтрация, сортировка, Pageable, Specifications [↑](#toc)
**Суть:** `PageRequest.of(page, size, Sort.by("field").descending())`; динамические фильтры — `Specification<T>` + `JpaSpecificationExecutor`.

```java
Specification<Order> spec = Specification
    .where(hasStatus(status))
    .and(createdAfter(from));
orderRepository.findAll(spec, pageable);
```
- Specification — обёртка над Criteria API, компонуется `and/or`, null-безопасные фабрики условий.
- Альтернативы: Querydsl (типобезопасность), Query by Example для простых равенств.

## <a id="s6"></a>6. REST, микросервисы, межсетевое взаимодействие [↑](#toc)

### <a id="q6-1"></a>6.1. REST и его принципы [↑](#toc)
**Суть:** архитектурный стиль (Филдинг) для распределённых систем поверх HTTP.

- Принципы: **client-server**, **stateless** (сервер не хранит состояние клиента между запросами), **cacheable**, **uniform interface** (ресурсы и URI, представления, self-descriptive messages, HATEOAS), **layered system**, code on demand (опционально).
- Практика: ресурсы — существительные (`/orders/{id}`), действия — HTTP-методы, состояние — в представлении.

### <a id="q6-2"></a>6.2. Идемпотентность HTTP-методов [↑](#toc)
**Суть:** идемпотентный метод — повтор запроса даёт тот же эффект на сервере, что и один вызов.

- Идемпотентны: **GET, HEAD, OPTIONS** (ещё и safe), **PUT, DELETE**. Не идемпотентны: **POST**; PATCH — не гарантированно.
- Почему важно: сеть ненадёжна — клиенты и прокси ретраят; идемпотентность делает ретраи безопасными. Для POST — **Idempotency-Key** заголовок + дедупликация на сервере.

### <a id="q6-3"></a>6.3. HTTP статус-коды [↑](#toc)
**Суть:** 2xx успех, 3xx перенаправление, 4xx ошибка клиента, 5xx ошибка сервера.

- 200 OK; 201 Created (+Location); 202 Accepted (асинхронно принято); 204 No Content.
- 301/302 редиректы; 304 Not Modified (кэш, ETag).
- 400 Bad Request; 401 Unauthorized (не аутентифицирован); 403 Forbidden (нет прав); 404; 405 Method Not Allowed; 409 Conflict (версия/дубликат); 422 ошибки валидации; 429 Too Many Requests.
- 500; 502 Bad Gateway; 503 Service Unavailable; 504 Gateway Timeout.

### <a id="q6-4"></a>6.4. Пагинация в REST API [↑](#toc)
**Суть:** offset-based (`?page=0&size=20`) — просто, но деградирует на глубине и нестабильно при вставках; cursor/keyset (`?cursor=opaque`) — стабильно и быстро.

- Ответ: данные + метаданные (next cursor / totalElements при необходимости) или `Link`-заголовки (rel="next").
- Сортировка — обязательная и детерминированная (tiebreaker по id) для стабильных страниц.

### <a id="q6-5"></a>6.5. Безопасность REST API: аутентификация и авторизация [↑](#toc)
**Суть:** HTTPS всегда; аутентификация — кто ты; авторизация — что тебе можно.

- Механизмы: Basic (только поверх TLS, для простого/внутреннего), API keys (м2м), **JWT** (stateless: подпись, claims, exp), **OAuth2** (делегирование доступа; flows: authorization code + PKCE для пользователей, client credentials для сервис-сервис), OIDC — слой идентификации поверх OAuth2.
- Авторизация: роли/scopes, Spring Security (`@PreAuthorize("hasRole(...)")`), проверка владения ресурсом (против IDOR).
- Плюс: rate limiting, валидация входа, секреты в vault.

### <a id="q6-6"></a>6.6. Обработка ошибок: `@ExceptionHandler`, `@ControllerAdvice` [↑](#toc)
**Суть:** централизованно — `@RestControllerAdvice` с методами `@ExceptionHandler` → единый формат ошибки и маппинг на статус-коды.

```java
@RestControllerAdvice
class ApiExceptionHandler {
  @ExceptionHandler(OrderNotFoundException.class)
  ResponseEntity<ProblemDetail> notFound(OrderNotFoundException e) {
    return ResponseEntity.status(404).body(ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, e.getMessage()));
  }
}
```
- Spring 6: **ProblemDetail** (RFC 7807/9457) — стандартный формат.
- Валидация: `MethodArgumentNotValidException` → 400 со списком полей. Не отдавать stack trace наружу; логировать с correlation id.

### <a id="q6-7"></a>6.7. CORS [↑](#toc)
**Суть:** браузерная защита same-origin; сервер явно разрешает кросс-доменные запросы заголовками `Access-Control-Allow-Origin/-Methods/-Headers`.

- Не-простые запросы — **preflight OPTIONS** перед основным.
- Spring: `@CrossOrigin`, глобально `CorsConfigurationSource`/`WebMvcConfigurer.addCorsMappings`; часто настраивают на gateway.
- `allowCredentials=true` несовместим с `*` в origins — перечислять явно. CORS защищает браузерных пользователей, не сервер.

### <a id="q6-8"></a>6.8. Синхронное vs асинхронное взаимодействие микросервисов [↑](#toc)
**Суть:** sync (REST/gRPC) — простая модель запрос-ответ, но временнáя связность: latency суммируется, сбои каскадируются; async (Kafka/RabbitMQ) — развязка, буферизация, устойчивость к недоступности потребителя.

- Цена async: eventual consistency, дубликаты (нужна идемпотентность), сложнее отладка/трейсинг.
- Примеры async: события «заказ создан» → нотификации/аналитика/склад независимо; long-running задачи; сглаживание пиков.

### <a id="q6-9"></a>6.9. Circuit Breaker [↑](#toc)
**Суть:** защита от каскадных сбоев: при превышении доли ошибок перестаём дёргать умирающий сервис и падаем быстро.

- Состояния: **CLOSED** (нормально, считаем ошибки) → **OPEN** (fail fast/fallback) → **HALF_OPEN** (пробные запросы) → CLOSED/OPEN.
- Реализация: Resilience4j — `@CircuitBreaker(name="x", fallbackMethod="fb")`, конфиг: скользящее окно, порог ошибок, wait duration; в связке с timeout, retry (с jitter), bulkhead.

### <a id="q6-10"></a>6.10. Согласованность данных между микросервисами: Saga и 2PC [↑](#toc)
**Суть:** распределённые ACID-транзакции (2PC/XA) в микросервисах почти не используют (блокирующий протокол, координатор — точка отказа) → **Saga**: последовательность локальных транзакций + компенсирующие действия при сбое.

- **Оркестрация** — центральный координатор ведёт шаги (Camunda/Temporal/свой сервис): явный процесс, проще отладка. **Хореография** — сервисы реагируют на события друг друга: слабее связность, сложнее видеть целое.
- Дополнить: **Transactional Outbox** (атомарность «запись в БД + публикация события»), идемпотентные обработчики, отслеживание статуса саги.
- Пример: заказ → резерв товара → оплата; сбой оплаты → компенсация: снять резерв, отменить заказ.

### <a id="q6-11"></a>6.11. API Gateway [↑](#toc)
**Суть:** единая точка входа для клиентов: маршрутизация к сервисам + сквозные функции.

- Функции: аутентификация/валидация JWT, rate limiting, TLS termination, агрегация/композиция ответов, версионирование, кэширование, логирование/метрики, сокрытие внутренней топологии.
- Реализации: Spring Cloud Gateway, Kong, NGINX, облачные. Риски: единая точка отказа (масштабировать), дополнительный hop, риск превратить в «умный монолит».

### <a id="q6-12"></a>6.12. gRPC vs REST [↑](#toc)
**Суть:** gRPC — RPC-фреймворк поверх HTTP/2 + Protobuf: бинарная сериализация, строгая схема, кодогенерация клиентов/серверов, стриминг (server/client/bidirectional), deadlines.

- Быстрее и компактнее REST+JSON; сильные контракты.
- Когда: внутренние межсервисные вызовы, low latency, стриминг, полиглотные команды.
- Минусы: браузеры напрямую не умеют (grpc-web/шлюз), бинарный формат хуже для отладки, инструментов меньше, чем у REST.

### <a id="q6-13"></a>6.13. SOAP vs REST [↑](#toc)
**Суть:** SOAP — протокол: XML-конверты, жёсткий контракт **WSDL**, спецификации WS-* (Security, ReliableMessaging, Transactions), транспорт-агностичен. REST — архитектурный стиль, легковеснее.

- SOAP имеет смысл: enterprise-интеграции с формальными контрактами, банки/госсектор/legacy, требования WS-Security, строгая типизация и генерация из WSDL.

### <a id="q6-14"></a>6.14. FeignClient vs RestTemplate vs WebClient [↑](#toc)
**Суть:** OpenFeign — декларативный клиент: интерфейс + аннотации, реализацию генерирует Spring Cloud.

```java
@FeignClient(name = "billing", url = "${billing.url}")
interface BillingClient {
  @GetMapping("/invoices/{id}") InvoiceDto get(@PathVariable Long id);
}
```
- Преимущества: минимум boilerplate, единый стиль, из коробки интеграция с LoadBalancer, Resilience4j, перехватчики (auth-заголовки), логирование.
- RestTemplate — синхронный, maintenance mode; современная синхронная замена — RestClient (Spring 6.1). WebClient — реактивный/неблокирующий, стриминг, WebFlux.

### <a id="q6-15"></a>6.15. Логирование и мониторинг REST API [↑](#toc)
**Суть:** структурированные логи + метрики RED + распределённый трейсинг.

- Логи: JSON-формат, **correlation/trace id** через MDC (Micrometer Tracing/OpenTelemetry), фильтр логирования запрос/ответ (без тел с PII/паролями).
- Метрики: `http.server.requests` (rate, errors, duration percentiles) → Prometheus → Grafana; алерты на p99/error rate.
- Трейсинг: OpenTelemetry → Jaeger/Zipkin/Tempo — путь запроса через сервисы.

### <a id="q6-16"></a>6.16. Сжатие данных в REST API [↑](#toc)
**Суть:** согласование через `Accept-Encoding` (клиент) / `Content-Encoding` (сервер): gzip, deflate, **brotli (br)**, zstd.

- Spring Boot: `server.compression.enabled=true`, `min-response-size` (мелкие не жать), mime-types.
- Обычно выносится на NGINX/gateway/CDN. JSON сжимается отлично (в разы); уже сжатое (jpeg, zip) — не трогать. Brotli плотнее gzip для текста.

### <a id="q6-17"></a>6.17. OpenAPI / Swagger [↑](#toc)
**Суть:** OpenAPI — спецификация описания REST API (yaml/json); Swagger — экосистема инструментов (UI, codegen).

- Code-first: **springdoc-openapi** генерирует спецификацию из кода/аннотаций + Swagger UI для интерактивной документации.
- Contract-first: сначала спецификация → генерация серверных интерфейсов и клиентов (openapi-generator) → контракт как источник истины.
- Применение: документация, генерация клиентов, контрактные проверки в CI, mock-серверы.

### <a id="q6-18"></a>6.18. Тестирование REST API [↑](#toc)
**Суть:** слои: unit контроллера → интеграционные → контрактные → нагрузочные.

- `@WebMvcTest` + MockMvc — слайс web-слоя (валидация, статусы, сериализация) с моками сервисов.
- `@SpringBootTest(webEnvironment = RANDOM_PORT)` + TestRestTemplate/WebTestClient + **Testcontainers** — полные интеграционные.
- REST Assured — выразительные API-тесты; контрактные — Pact/Spring Cloud Contract.
- Нагрузочные: **k6** (мой опыт: 8-часовой soak, 50 VUs, 2.68M запросов в Kubernetes), Gatling, JMeter — метрики p95/p99, деградация, утечки.

### <a id="q6-19"></a>6.19. RPC vs REST [↑](#toc)
**Суть:** RPC — «вызов процедуры» (глаголы, действия: gRPC, JSON-RPC), тесный контракт; REST — ресурсы + единообразный интерфейс HTTP-семантики, кэшируемость, слабее связность.

- RPC: внутренние высокочастотные вызовы, эффективность, стриминг.
- REST: публичные API, эволюционируемость, HTTP-инфраструктура (кэши, прокси) из коробки.

### <a id="q6-20"></a>6.20. Cookie vs Bearer Token [↑](#toc)
**Суть:** cookie браузер отправляет автоматически (удобно для классического веба, но CSRF-риск); Bearer — заголовок `Authorization`, клиент управляет сам (SPA, мобильные, м2м), CSRF не грозит, но встаёт вопрос хранения.

- Cookie: `HttpOnly` (недоступна JS → защита от кражи через XSS), `Secure`, `SameSite`; серверные сессии — stateful.
- Bearer/JWT: stateless, масштабируется; хранение в JS-доступном месте уязвимо к XSS; отзыв сложнее.
- Выбор: браузерное приложение с сессиями — HttpOnly cookie; API/микросервисы/мобильные — Bearer.

### <a id="q6-21"></a>6.21. Access token и refresh token [↑](#toc)
**Суть:** access — короткоживущий (минуты), передаётся в каждом запросе; refresh — долгоживущий, используется только для получения новой пары, хранится максимально безопасно (HttpOnly cookie / secure storage).

- Риски: кража access — ущерб ограничен TTL; кража refresh — долгий доступ → **ротация refresh** (одноразовость) + детект повторного использования (отзыв всей семьи токенов), привязка к клиенту.
- Отзыв stateless JWT сложен: короткий TTL + blacklist/versioning в Redis при необходимости.

### <a id="q6-22"></a>6.22. NGINX как reverse proxy [↑](#toc)
**Суть:** принимает запросы клиентов и проксирует на бэкенды (`proxy_pass`), скрывая топологию.

- Даёт: TLS termination, кэширование ответов, сжатие, балансировку, буферизацию медленных клиентов (защита воркеров приложения), отдачу статики, rate limiting (`limit_req`), единую точку логирования.

### <a id="q6-23"></a>6.23. Балансировка нагрузки в NGINX [↑](#toc)
**Суть:** блок `upstream` + методы: **round robin** (по умолчанию), **weighted** (веса серверов), **least_conn** (наименьшее число активных соединений), **ip_hash** (стикость по клиенту), **hash** (consistent hashing по ключу), random.

- Пассивные health-чеки: `max_fails`/`fail_timeout` (активные — в NGINX Plus); `backup`-серверы.
- Помогает: равномерное распределение, вывод узлов из ротации, сессионная привязка при необходимости.

### <a id="q6-24"></a>6.24. NGINX и HTTP/2, HTTP/3 [↑](#toc)
**Суть:** HTTP/2 — бинарный, мультиплексирование потоков в одном TCP-соединении, сжатие заголовков HPACK → убирает HTTP-очереди, но остаётся TCP head-of-line blocking. HTTP/3 — поверх **QUIC (UDP)**: убирает TCP HOL, быстрее рукопожатие (0-RTT), миграция соединения при смене сети.

- NGINX: `listen 443 ssl http2;` / HTTP/3: `listen 443 quic;` + заголовок `Alt-Svc` (поддержка с 1.25).
- Выгода vs HTTP/1.1: одно соединение вместо шести, нет domain sharding, ниже задержки.

### <a id="q6-25"></a>6.25. Паттерны микросервисов + один подробно (Outbox) [↑](#toc)
**Суть:** перечислить: API Gateway, Service Discovery, Circuit Breaker, Retry/Timeout/Bulkhead, Saga, **Transactional Outbox**, CQRS, Event Sourcing, Database per Service, Strangler Fig, Sidecar, BFF.

- **Подробно Outbox:** проблема dual write — «записать в БД и отправить в брокер» не атомарно. Решение: в той же локальной транзакции пишем событие в таблицу `outbox`; отдельный relay (poller или Debezium CDC) читает и публикует в Kafka, помечая отправленным. Гарантия at-least-once → консьюмеры идемпотентны.
- Могу связать с опытом: идемпотентные консьюмеры и запланированные обработчики строил на AuctionPort.

## <a id="s7"></a>7. Брокеры сообщений (Kafka, RabbitMQ, JMS) [↑](#toc)

### <a id="q7-1"></a>7.1. Publish/subscribe vs point-to-point [↑](#toc)
**Суть:** p2p — очередь: сообщение получает **один** из консьюмеров (конкурирующие потребители); pub/sub — топик: сообщение получают **все** подписчики.

- JMS: Queue vs Topic. RabbitMQ: одна очередь с несколькими консьюмерами = p2p; fanout exchange на несколько очередей = pub/sub.
- Kafka объединяет: внутри consumer group — p2p (партиции делятся), разные группы — pub/sub (каждая читает всё).

### <a id="q7-2"></a>7.2. Kafka vs RabbitMQ [↑](#toc)
**Суть:** Kafka — распределённый реплицируемый **лог**: сообщения хранятся (retention), можно перечитывать, партиции, огромная пропускная способность, pull-модель. RabbitMQ — классический **брокер очередей**: умная маршрутизация (exchanges), per-message ack, TTL/приоритеты, push, сообщение удаляется после подтверждения.

- Kafka: событийные потоки, event sourcing, replay, стриминг, интеграция многих потребителей, строгий порядок в партиции.
- RabbitMQ: распределение задач (work queues), RPC-паттерны, сложная маршрутизация, низкая задержка на умеренных объёмах.

### <a id="q7-3"></a>7.3. Партиционирование в Kafka [↑](#toc)
**Суть:** топик делится на партиции — единицы параллелизма и порядка: порядок гарантирован **только внутри партиции**.

- Роутинг: `hash(key) % partitions` (murmur2); без ключа — sticky-распределение по батчам.
- Масштабирование: консьюмеров в группе можно поднять до числа партиций (лишние простаивают); больше партиций — больше параллелизм (но накладные на брокер и recovery).
- Увеличение числа партиций ломает маппинг ключ→партиция (порядок для ключа) — планировать заранее.

### <a id="q7-4"></a>7.4. Уникальность сообщений / защита от дублей [↑](#toc)
**Суть:** дубликаты неизбежны при at-least-once → бороться на двух концах.

- **Producer:** `enable.idempotence=true` (PID + sequence number — брокер отбрасывает дубли ретраев; в новых версиях включён по умолчанию с acks=all); транзакции для exactly-once в цепочках read-process-write.
- **Consumer:** идемпотентная обработка — ключ дедупликации (eventId), unique constraint в БД / таблица processed_ids (inbox), upsert-семантика.

### <a id="q7-5"></a>7.5. Offset в Kafka [↑](#toc)
**Суть:** порядковый номер сообщения в партиции; позиция консьюмер-группы хранится в топике `__consumer_offsets`.

- Commit: авто (`enable.auto.commit`) или ручной (`commitSync`/`commitAsync`) — контроль семантики (до/после обработки = at-most/at-least-once).
- `auto.offset.reset` (earliest/latest) — где начинать без сохранённого offset; `seek()` — перечитать заново.
- После ребаланса группа продолжает с последнего committed offset.

### <a id="q7-6"></a>7.6. Маршрутизация в RabbitMQ: exchanges, bindings, routing keys [↑](#toc)
**Суть:** producer публикует в **exchange**; exchange по **bindings** (правилам с routing key) раскладывает в очереди.

- Типы: **direct** — точное совпадение ключа; **topic** — шаблоны (`order.*`, `#`); **fanout** — всем связанным очередям; **headers** — по заголовкам.
- DLX (dead letter exchange) назначается аргументами очереди; alternate exchange — для немаршрутизированных.

### <a id="q7-7"></a>7.7. Durable Queue в RabbitMQ [↑](#toc)
**Суть:** durable-очередь переживает рестарт брокера (объявление хранится на диске). Для сохранности данных этого мало — нужна связка:

- durable queue + **persistent messages** (`deliveryMode=2`) + **publisher confirms** (подтверждение записи брокером) + manual consumer ack.
- HA/отказоустойчивость: **quorum queues** (Raft-репликация) — современная замена classic mirrored queues.

### <a id="q7-8"></a>7.8. Acknowledgements в RabbitMQ и Kafka [↑](#toc)
**Суть:** подтверждения определяют семантику доставки; неправильная настройка = потери или дубликаты.

- **RabbitMQ (consumer):** auto-ack — потеря при падении обработчика; manual ack/nack(requeue) — надёжно, но забытый ack копит unacked и стопорит prefetch; poison message с requeue=true — бесконечный цикл (→ DLQ).
- **Kafka (producer acks):** 0 — fire-and-forget; 1 — лидер записал; **all + min.insync.replicas=2** — надёжно.
- **Kafka (consumer):** commit offset до обработки → потеря; после → возможен дубль (норма, лечится идемпотентностью).

### <a id="q7-9"></a>7.9. Порядок доставки и гарантии (at-most/at-least/exactly-once) [↑](#toc)
**Суть:** порядок — только внутри партиции: один агрегат/ключ → одна партиция.

- Ретраи могут переупорядочить: с идемпотентным продюсером `max.in.flight.requests.per.connection ≤ 5` порядок сохраняется (без него — 1).
- **At-most-once:** commit до обработки — возможны потери. **At-least-once (default):** обработка → commit — возможны дубли. **Exactly-once:** идемпотентный продюсер + транзакции (`transactional.id`, `sendOffsetsToTransaction`) + консьюмеры `isolation.level=read_committed` — в рамках Kafka-пайплайнов (Kafka Streams EOS); с внешними системами — идемпотентность на стороне приёмника.

### <a id="q7-10"></a>7.10. Dead Letter Queue [↑](#toc)
**Суть:** очередь для сообщений, которые не удалось обработать — вместо потери или бесконечных ретраев.

- **RabbitMQ:** DLX — сообщение уходит при reject/nack с requeue=false, истечении TTL, переполнении; ретраи с задержкой строятся на TTL + DLX-циклах или delayed exchange plugin.
- **Kafka:** встроенного нет; Spring Kafka — `DefaultErrorHandler` + `DeadLetterPublishingRecoverer` → топик `.DLT` после исчерпания ретраев; `@RetryableTopic` — ретраи через промежуточные топики.
- Обязательно: мониторинг DLQ, процедура разбора и replay.

### <a id="q7-11"></a>7.11. Проблемы брокеров в распределённых системах [↑](#toc)
**Суть:** дубликаты, потери, нарушение порядка, отставание, poison messages, dual write, эволюция схем.

- Дубликаты → идемпотентность; потери → acks=all/репликация/publisher confirms; порядок → ключ партиционирования.
- **Consumer lag** → мониторинг, масштабирование группы, оптимизация обработки, батчинг.
- Poison message → DLQ; ребалансы (stop-the-world обработки) → cooperative sticky assignor, static membership.
- Dual write → Transactional Outbox; несовместимые изменения схем → Schema Registry + правила совместимости.

### <a id="q7-12"></a>7.12. Spring Kafka и Spring AMQP [↑](#toc)
**Суть:** декларативные слушатели + шаблоны отправки + обработка ошибок.

- **Spring Kafka:** `@KafkaListener(topics, groupId)`, `KafkaTemplate.send`, `ConcurrentKafkaListenerContainerFactory` (concurrency), `DefaultErrorHandler` с backoff, `@RetryableTopic`, JsonSerializer/Deserializer.
- **Spring AMQP:** `@RabbitListener(queues)`, `RabbitTemplate.convertAndSend(exchange, routingKey, msg)`, `Jackson2JsonMessageConverter`, декларация топологии бинами (Queue/Exchange/Binding), retry interceptors + DLQ.

### <a id="q7-13"></a>7.13. Consumer groups в Kafka [↑](#toc)
**Суть:** группа — логический потребитель: партиции топика распределяются между её членами, каждая партиция — ровно одному консьюмеру группы.

- Масштабирование: добавили инстанс сервиса → ребаланс → партиции перераспределились; предел параллелизма = число партиций.
- Разные группы независимы (свои offsets) — несколько подсистем читают один топик целиком.

### <a id="q7-14"></a>7.14. Транзакционность в Kafka и RabbitMQ [↑](#toc)
**Суть:** Kafka — полноценные транзакции продюсера: атомарная запись в несколько партиций + commit offsets (`transactional.id`, begin/commit, `sendOffsetsToTransaction`), консьюмеры видят только закоммиченное при `read_committed`. Нужна для exactly-once read-process-write (Kafka Streams EOS).

- RabbitMQ: tx-каналы есть, но медленные — на практике **publisher confirms**.
- С транзакцией БД брокерные транзакции не объединяются (нет общего XA на практике) → Outbox-паттерн.

### <a id="q7-15"></a>7.15. Мониторинг брокеров [↑](#toc)
**Суть:** метрики брокера + клиентов → Prometheus → Grafana + алерты.

- **Kafka:** consumer lag (kafka-exporter/Burrow), under-replicated partitions, ISR shrink, request/queue latency, disk usage; JMX-метрики клиентов.
- **RabbitMQ:** management/prometheus plugin — глубина очередей, unacked, publish/deliver rates, memory/disk alarms, connection churn.
- Алерты: рост lag/глубины очереди, URP > 0, отсутствие консьюмеров.

### <a id="q7-16"></a>7.16. Retention в Kafka [↑](#toc)
**Суть:** сообщения не удаляются по факту чтения — по политике: `retention.ms` (по умолчанию 7 дней) и/или `retention.bytes`; удаление — целыми сегментами (`segment.ms/bytes`).

- `cleanup.policy=delete` (по времени/размеру) | `compact` (**log compaction** — хранится последнее значение по ключу; удаление — tombstone с null-значением) | `compact,delete`.
- Compaction — для состояний (changelog, снапшоты сущностей).

### <a id="q7-17"></a>7.17. Backpressure и message flooding [↑](#toc)
**Суть:** продюсер быстрее консьюмера.

- **Kafka:** буфер — сам лог (растёт lag, диск); решения: масштабировать консьюмеров/партиции, ускорить обработку (батчи, асинхронность), pause/resume контейнера, алерты на lag.
- **RabbitMQ:** рост очереди → memory/disk alarms **блокируют продюсеров**; решения: prefetch tuning, больше консьюмеров, lazy/квorum queues (на диск), TTL + DLQ, rate limiting и батчинг у продюсера, шардирование очередей.
- Системно: flow control на входе (429/token bucket), деградация функциональности.

### <a id="q7-18"></a>7.18. Балансировка нагрузки в RabbitMQ [↑](#toc)
**Суть:** два уровня — подключения и работа.

- Подключения: LB (HAProxy/NGINX) перед узлами кластера; учитывать, что classic-очередь живёт на одном узле (обращения проксируются), quorum — реплицированы.
- Работа: несколько консьюмеров на очередь + `prefetch` (fair dispatch); распределение по очередям — consistent-hash exchange / sharding plugin.

### <a id="q7-19"></a>7.19. Распределённые транзакции с брокерами: Saga и 2PC [↑](#toc)
**Суть:** XA/2PC с брокерами на практике не применяют (Kafka XA не поддерживает; блокирующий, хрупкий) → паттерны:

- **Outbox** — атомарность локальной записи и публикации ([6.25](#q6-25)).
- **Saga** — цепочка локальных транзакций через события + компенсации ([6.10](#q6-10)).
- Идемпотентность + ретраи заменяют «атомарность» на уровне системы.

### <a id="q7-20"></a>7.20. Шифрование и аутентификация в Kafka и RabbitMQ [↑](#toc)
**Суть:** TLS для транспорта + аутентификация клиентов + авторизация на ресурсы.

- **Kafka:** TLS/mTLS (шифрование + аутентификация сертификатами), **SASL** (SCRAM-SHA-512, OAUTHBEARER, GSSAPI/Kerberos, PLAIN только с TLS), ACL на топики/группы (`--allow-principal`).
- **RabbitMQ:** TLS, пользователи + vhosts + permissions (configure/write/read), OAuth2/LDAP-плагины.
- При повышенных требованиях — шифрование payload на уровне приложения.

## <a id="s8"></a>8. Многопоточность и JVM [↑](#toc)

### <a id="q8-1"></a>8.1. Immutable-объекты и их роль в многопоточности [↑](#toc)
**Суть:** объект, состояние которого нельзя изменить после создания → можно свободно шарить между потоками **без синхронизации**.

- Рецепт: все поля `final`, класс `final` (или статические фабрики), нет сеттеров, защитные копии изменяемых полей (даты, коллекции) на входе и выходе, не отдавать `this` из конструктора.
- JMM-бонус: `final`-поля гарантированно видны корректно после безопасной публикации конструктора (final field freeze).
- Примеры: String, Integer, LocalDate, **records**. Минус — новые объекты на каждое «изменение» (обычно дёшево).

### <a id="q8-2"></a>8.2. Поток vs процесс; виртуальные потоки [↑](#toc)
**Суть:** процесс — изолированное адресное пространство; потоки живут внутри процесса, разделяют heap, у каждого свой стек и program counter. Переключение потоков дешевле процессов, общение — через общую память (нужна синхронизация).

- **Виртуальные потоки** (Java 21+): легковесные потоки JVM поверх небольшого пула carrier-потоков; при блокирующем I/O **отмонтируются** от носителя → миллионы конкурентных задач в стиле thread-per-request без асинхронщины.
- Нюансы: не для CPU-bound; pinning при блокировке внутри `synchronized` (до Java 24) и в нативных вызовах; ThreadLocal осторожно (много потоков → память); `Executors.newVirtualThreadPerTaskExecutor()`.

### <a id="q8-3"></a>8.3. volatile [↑](#toc)
**Суть:** гарантирует **видимость** (чтение видит последнюю запись; значение не кэшируется в регистрах/локально) и **упорядоченность** (запрет переупорядочивания, happens-before между записью и последующим чтением).

- НЕ даёт атомарности составных операций: `count++` (read-modify-write) остаётся гонкой → AtomicInteger/lock.
- Применения: флаги остановки (`volatile boolean running`), безопасная публикация ссылки, double-checked locking (`private static volatile Singleton`).
- Бонус: чтение/запись volatile `long`/`double` атомарны (см. [8.18](#q8-18)).


### <a id="q8-4"></a>8.4. Happens-before и Java Memory Model [↑](#toc)
**Суть:** JMM определяет, когда запись одного потока **гарантированно видна** чтению другого. Happens-before (hb) — отношение порядка: если A hb B, то B видит эффекты A.

- Ключевые правила: program order (внутри потока); **monitor unlock hb последующий lock** того же монитора; **volatile write hb последующий read**; `Thread.start()` hb действия потока; действия потока hb возврат из `join()`; передача задачи в executor hb её выполнение.
- Без hb-связи между потоками компилятор/CPU вправе переупорядочивать и кэшировать → «невозможные» на вид баги. Синхронизация — это не только взаимоисключение, но и **видимость**.

### <a id="q8-5"></a>8.5. wait/notify/notifyAll [↑](#toc)
**Суть:** координация потоков на мониторе объекта: `wait()` отпускает монитор и ждёт; `notify()` будит один произвольный ждущий поток, `notifyAll()` — все.

- Вызывать **только владея монитором** (внутри `synchronized` на том же объекте), иначе `IllegalMonitorStateException`.
- `wait()` всегда в цикле `while (!condition) wait();` — из-за **spurious wakeups** и конкуренции за условие после пробуждения.
- `notify` экономнее, но рискован (может разбудить «не того»); по умолчанию — `notifyAll`. Современная замена: `Lock`+`Condition` (await/signal), BlockingQueue, CompletableFuture.

```java
synchronized (lock) {
  while (queue.isEmpty()) lock.wait();
  process(queue.poll());
}
```

### <a id="q8-6"></a>8.6. sleep vs wait [↑](#toc)
**Суть:** `Thread.sleep(ms)` — статический, усыпляет текущий поток, **монитор не отпускает**, монитора не требует, просыпается по таймеру/interrupt.

- `obj.wait()` — метод объекта, требует владения монитором, **отпускает монитор**, просыпается по notify/notifyAll/таймауту/interrupt.
- sleep — пауза по времени; wait — ожидание условия/события.

### <a id="q8-7"></a>8.7. Deadlock: причины и предотвращение [↑](#toc)
**Суть:** взаимная блокировка — потоки держат ресурсы и циклически ждут друг друга. Четыре условия Коффмана: mutual exclusion, hold-and-wait, no preemption, **circular wait**.

- Классика: T1 держит A ждёт B; T2 держит B ждёт A.
- Предотвращение: **глобальный порядок захвата локов** (ломает circular wait); `tryLock(timeout)` с откатом; сужать область — не звать чужой код под локом; один лок вместо двух; неблокирующие структуры.
- Диагностика: thread dump (`jstack` пишет "Found one Java-level deadlock"), JConsole/JMC.

### <a id="q8-8"></a>8.8. ReentrantLock vs synchronized [↑](#toc)
**Суть:** обе — реентерабельные взаимоисключающие блокировки; ReentrantLock — явная, с расширенными возможностями.

- Плюсы ReentrantLock: **tryLock()/tryLock(timeout)**, `lockInterruptibly()`, **честный режим** (fairness), несколько `Condition` на один лок, опрос состояния.
- Плюсы synchronized: короче, нельзя забыть unlock, достаточно в 90% случаев.
- Шаблон: `lock.lock(); try { ... } finally { lock.unlock(); }`. Ещё: `ReentrantReadWriteLock`, `StampedLock` (оптимистичное чтение).

### <a id="q8-9"></a>8.9. Проблемы Executors-фабрик [↑](#toc)
**Суть:** удобные фабрики скрывают опасные дефолты.

- `newFixedThreadPool`/`newSingleThreadExecutor` — **неограниченная очередь** LinkedBlockingQueue → при отставании копятся задачи → OOM.
- `newCachedThreadPool` — потоки без лимита (до Integer.MAX_VALUE) → при всплеске тысячи потоков, исчерпание ресурсов.
- Продакшен: явный `new ThreadPoolExecutor(core, max, keepAlive, bounded queue, ThreadFactory, RejectedExecutionHandler)` — осознанные лимиты + политика отказа (AbortPolicy/CallerRunsPolicy) + именованные потоки для дампов.

### <a id="q8-10"></a>8.10. Race condition [↑](#toc)
**Суть:** результат зависит от порядка выполнения потоков на общем изменяемом состоянии; классика — check-then-act и read-modify-write.

- `count++` — три операции; `if (map.get(k)==null) map.put(k,v)` — гонка между проверкой и действием.
- Лечение: атомики (`incrementAndGet`), составные атомарные операции (`computeIfAbsent`, `putIfAbsent`), synchronized/lock, immutable + confinement (не шарить).

### <a id="q8-11"></a>8.11. Fork/Join Framework [↑](#toc)
**Суть:** пул для рекурсивного «разделяй и властвуй» с **work-stealing**: у каждого воркера своя deque, простаивающий крадёт задачи с хвоста чужой → автоматическая балансировка.

- API: `ForkJoinPool`, `RecursiveTask<V>`/`RecursiveAction`; паттерн: `left.fork(); right.compute(); left.join();` — порог, ниже которого считаем последовательно.
- Используется под капотом: parallel streams, `CompletableFuture` (common pool), виртуальные потоки (carrier-пул — ForkJoinPool).
- Для CPU-bound задач; блокирующий I/O в common pool — плохо (душит parallel streams всего приложения).

### <a id="q8-12"></a>8.12. ThreadLocal: применение и риски [↑](#toc)
**Суть:** у каждого потока — своё значение переменной; контекст без передачи через параметры.

- Применения: **MDC** (correlation id в логах), SecurityContext, транзакционные ресурсы Spring, несинхронизированные форматтеры (legacy SimpleDateFormat).
- Риски: в **пулах потоков** значение переживает задачу → утечки памяти и «протекание» контекста между запросами → **`remove()` в finally**; с виртуальными потоками — миллионы копий.
- Современно: **ScopedValue** (Java 21+ preview, финализация в 25) — иммутабельный контекст со структурной областью видимости.

### <a id="q8-13"></a>8.13. InheritableThreadLocal [↑](#toc)
**Суть:** дочерний поток при создании получает **копию** значений родителя (снапшот на момент `new Thread()`).

- Ограничение: в пулах бесполезен — потоки создаются один раз, не при отправке задачи → контекст не передаётся. Решения: обёртки задач (копирование контекста вручную, `TaskDecorator` в Spring), micrometer context-propagation.

### <a id="q8-14"></a>8.14. Типы экзекьюторов [↑](#toc)
**Суть:** `Executor` → `ExecutorService` → `ScheduledExecutorService`; реализации — ThreadPoolExecutor, ScheduledThreadPoolExecutor, ForkJoinPool.

- Фабрики: fixed, cached, single, `newWorkStealingPool` (ForkJoin), **`newScheduledThreadPool`** (задержки/периодика: `scheduleAtFixedRate` vs `scheduleWithFixedDelay`), **`newVirtualThreadPerTaskExecutor`** (Java 21+).
- Выбор: CPU-bound → fixed ≈ числу ядер; I/O-bound → больше потоков или виртуальные; периодика → scheduled.

### <a id="q8-15"></a>8.15. submit vs execute [↑](#toc)
**Суть:** `execute(Runnable)` — из интерфейса Executor, ничего не возвращает; необработанное исключение уходит в UncaughtExceptionHandler потока (в логах видно).

- `submit(Runnable|Callable)` — из ExecutorService, возвращает **Future**; исключение **захватывается** в Future и всплывает только при `future.get()` как ExecutionException → «молчаливое проглатывание», если get не звали.
- Правило: submit — когда нужен результат/контроль; execute — fire-and-forget с логированием ошибок.

### <a id="q8-16"></a>8.16. Безопасная публикация объекта [↑](#toc)
**Суть:** сделать объект видимым другим потокам так, чтобы они увидели его **полностью сконструированным**.

- Способы: инициализация в `static`/статическом блоке (class loading гарантирует); **volatile**-поле; `final`-поле (freeze-семантика); под общим локом; через concurrent-коллекции/BlockingQueue; holder idiom для ленивых синглтонов.
- Антипаттерн: запись в обычное поле без синхронизации → другой поток может увидеть частично инициализированный объект; утечка `this` из конструктора (листенеры, запуск потока в конструкторе).

### <a id="q8-17"></a>8.17. RecursiveTask vs RecursiveAction [↑](#toc)
**Суть:** обе — задачи Fork/Join; **RecursiveTask<V>** возвращает результат (`compute(): V`), **RecursiveAction** — нет (`compute(): void`).

- Task — суммирование, поиск, свёртки (join собирает результаты); Action — обработка на месте (параллельная сортировка сегментов, заполнение массива).

### <a id="q8-18"></a>8.18. Атомарность чтения/записи long и double [↑](#toc)
**Суть:** JLS исторически допускает **word tearing**: не-volatile `long`/`double` (64 бита) могли читаться/писаться как две 32-битные половины → поток видит «химеру» из двух значений.

- `volatile long/double` — атомарность гарантирована. На практике 64-битные JVM пишут атомарно, но по спецификации полагаться нельзя (валидный вопрос-ловушка).
- Нужна атомарность + инкременты → AtomicLong; ссылки любых типов пишутся атомарно всегда (но без видимости).

### <a id="q8-19"></a>8.19. Phaser vs CyclicBarrier [↑](#toc)
**Суть:** оба — точки синхронизации «дождаться всех». CyclicBarrier — **фиксированное** число участников, reusable, barrier action по прибытии всех.

- **Phaser** гибче: **динамическая регистрация** участников (`register`/`arriveAndDeregister`), многофазность (номер фазы), иерархия фазеров, `arriveAndAwaitAdvance`.
- CountDownLatch — одноразовый счётчик (ждать событий, не участников). Выбор: фиксированные итерации → CyclicBarrier; меняющийся состав/фазы → Phaser.

### <a id="q8-20"></a>8.20. Виды локов; lock-free; CAS и ABA [↑](#toc)
**Суть:** блокирующие: synchronized (intrinsic), ReentrantLock, ReadWriteLock (много читателей/один писатель), StampedLock (оптимистичное чтение с валидацией).

- **Lock-free** — прогресс системы гарантирован без блокировок, на **CAS** (compareAndSet: атомарно «замени, если равно ожидаемому», инструкция CPU): атомики, ConcurrentLinkedQueue, ConcurrentHashMap частично.
- Цикл CAS: read → compute → CAS, retry при неудаче. Под высокой конкуренцией — livelock-давление ретраев (→ LongAdder).
- **ABA:** значение A→B→A; CAS «не замечает» изменения. Актуально для структур с переиспользованием узлов → `AtomicStampedReference` (значение + версия).

### <a id="q8-21"></a>8.21. Атомики: AtomicInteger, LongAdder и др. [↑](#toc)
**Суть:** классы `java.util.concurrent.atomic` — атомарные операции без локов на CAS/VarHandle.

- AtomicInteger/Long: `incrementAndGet`, `addAndGet`, `compareAndSet`, `updateAndGet(lambda)`; AtomicReference — CAS над ссылкой; AtomicIntegerArray; FieldUpdater'ы.
- **LongAdder/LongAccumulator** — под высокой конкуренцией масштабируются лучше (шардированные ячейки Cell[], сумма при чтении); идеальны для счётчиков/метрик, когда точное значение нужно редко.
- Атомик решает read-modify-write одной переменной; инвариант на несколько переменных — уже лок или immutable snapshot в AtomicReference.

### <a id="q8-22"></a>8.22. Мониторинг многопоточного приложения (JVM) [↑](#toc)
**Суть:** метрики + дампы + постоянный низкооверхедный профиль.

- Метрики (Micrometer → Prometheus/Grafana): состояние пулов (`executor.active`, queue size, rejected), `jvm.threads.states` (рост BLOCKED/WAITING — симптом), GC, CPU.
- **Thread dump** ([10.18](#q10-18)) — мгновенный срез: дедлоки, где стоят потоки; несколько дампов с интервалом — динамика.
- **JFR** (Java Flight Recorder) — низкий оверхед, пишет contention (monitor blocked), allocations, safepoints; анализ в JMC. Async-profiler — lock/cpu/alloc профили, flame graphs.

### <a id="q8-23"></a>8.23. Профилирование производительности [↑](#toc)
**Суть:** сначала метрики (что деградирует: p99, CPU, GC?), затем профилировщик — где именно.

- **async-profiler** — sampling без safepoint bias: CPU (`-e cpu`), аллокации (`-e alloc`), локи (`-e lock`), wall-clock; выход — **flame graph** (широкие рамки = горячие места).
- **JFR/JMC** — встроен, можно держать включённым в проде (~1% оверхед), события: GC, компиляция, I/O, exceptions, contention.
- Микробенчмарки — только **JMH** (warmup, форки, Blackhole против DCE); наивные `System.nanoTime`-замеры лгут из-за JIT.
- Порядок охоты: metrics → flame graph → гипотеза → фикс → повторный замер.

## <a id="s9"></a>9. Тестирование [↑](#toc)

### <a id="q9-1"></a>9.1. Принципы unit-тестирования [↑](#toc)
**Суть:** тест проверяет одну единицу поведения в изоляции; **FIRST**: Fast, Independent (не зависят друг от друга и от порядка), Repeatable (детерминированы в любом окружении), Self-validating (сам говорит pass/fail), Timely.

- Структура **AAA**: Arrange — Act — Assert (given/when/then); один логический assert-блок на тест.
- Практика: тестировать поведение через публичный API, а не реализацию; говорящие имена (`shouldRejectBid_whenAuctionClosed`); без общего изменяемого состояния; крайние случаи и ошибки, не только happy path.

### <a id="q9-2"></a>9.2. Unit vs integration тесты; пирамида [↑](#toc)
**Суть:** unit — изолированная логика с моками зависимостей, миллисекунды; integration — взаимодействие компонентов с реальными частями (БД, брокер, HTTP), секунды.

- **Пирамида**: много unit → меньше integration → мало E2E (медленные, хрупкие, дорогие в поддержке).
- Spring-градация: чистый unit (без контекста) → слайсы (`@WebMvcTest`, `@DataJpaTest`) → `@SpringBootTest` + Testcontainers → E2E.

### <a id="q9-3"></a>9.3. JUnit 5 [↑](#toc)
**Суть:** архитектура: **Platform** (запуск) + **Jupiter** (API) + Vintage (JUnit 4); главное новшество против 4 — **Extension model** вместо runners (несколько расширений на класс).

- Аннотации: `@Test`, `@BeforeEach/@AfterEach`, `@BeforeAll/@AfterAll`, `@DisplayName`, `@Disabled`, `@Tag`.
- **`@ParameterizedTest`** (+`@ValueSource`, `@CsvSource`, `@MethodSource`, `@EnumSource`), `@RepeatedTest`, `@Nested` (группировка), `@TestFactory` (динамические), `@Timeout`.
- `assertThrows`, `assertAll` (мягкие проверки); в паре — AssertJ (`assertThat(...).isEqualTo(...)`) за читаемость.

### <a id="q9-4"></a>9.4. Testcontainers [↑](#toc)
**Суть:** библиотека поднимает реальные зависимости (PostgreSQL, Kafka, Redis...) в Docker-контейнерах на время тестов — прощай H2-несовместимость.

```java
@Testcontainers
class OrderRepoIT {
  @Container @ServiceConnection
  static PostgreSQLContainer<?> pg = new PostgreSQLContainer<>("postgres:16");
}
```
- **`@ServiceConnection`** (Boot 3.1+) сам настраивает datasource/bootstrap-servers — без `@DynamicPropertySource`.
- Практика: static-контейнер переиспользуется тестами класса; reusable containers локально; работает в CI (Docker-in-Docker/socket). Тестируем реальный SQL-диалект, миграции Liquibase, `@KafkaListener` против настоящего брокера.

### <a id="q9-5"></a>9.5. Мокирование static-методов; PowerMock [↑](#toc)
**Суть:** современный Mockito умеет сам: **`mockStatic`** (mockito-inline / core 5+).

```java
try (MockedStatic<Clock> mocked = Mockito.mockStatic(Clock.class)) {
  mocked.when(Clock::systemUTC).thenReturn(fixedClock);
}
```
- Обязательно закрывать (try-with-resources) — мок действует в потоке до закрытия.
- **PowerMock** — legacy для JUnit 4: static/final/private/конструкторы через свой classloader; с JUnit 5 и новыми JDK практически мёртв.
- Сигнал дизайна: потребность мокать static → лучше инжектить зависимость (Clock как бин — образцовый пример).

### <a id="q9-6"></a>9.6. Stub vs Mock [↑](#toc)
**Суть:** stub — заглушка с заранее заданными ответами, проверяем **состояние** (state-based); mock — объект с ожиданиями взаимодействий, проверяем **поведение** (behavior-based, `verify`).

- Классификация Мезароша: dummy (заполнитель), stub (ответы), spy (записывает вызовы; в Mockito — частичный мок реального объекта), mock (ожидания), fake (рабочая упрощённая реализация — in-memory репозиторий).
- Mockito-объект — и stub (`when...thenReturn`), и mock (`verify`) одновременно. Практика: стабить запросы, верифицировать команды с побочными эффектами; не перезакручивать verify на каждый чих (хрупкость).

### <a id="q9-7"></a>9.7. @Mock vs @MockBean [↑](#toc)
**Суть:** `@Mock` — чистый Mockito без Spring (`@ExtendWith(MockitoExtension.class)`, `@InjectMocks` собирает объект вручную) — быстрые unit-тесты.

- `@MockBean` — **подменяет бин в Spring-контексте** (для `@SpringBootTest`/слайсов); внимание: каждый уникальный набор @MockBean = новый контекст = кэш контекстов ломается, тесты медленнее.
- Boot 3.4+: `@MockBean` deprecated → **`@MockitoBean`** (spring-test). Правило: без контекста — @Mock; контекст нужен — слайс + @MockitoBean.

### <a id="q9-8"></a>9.8. Fixtures [↑](#toc)
**Суть:** фиксированное исходное состояние для повторяемых тестов: тестовые данные, объекты, состояние БД.

- Реализация: `@BeforeEach/@BeforeAll` (JUnit), `@Sql("/seed.sql")` (Spring), builders/Object Mother для доменных объектов, файлы-ресурсы (json).
- Принципы: минимально необходимые данные, изоляция между тестами (rollback/cleanup), видимость важных параметров прямо в тесте (не прятать критичное в общем сетапе).

### <a id="q9-9"></a>9.9. End-to-End тестирование [↑](#toc)
**Суть:** проверка полного пользовательского сценария через реальную систему: UI/API → сервисы → БД/брокеры.

- Инструменты: Playwright/Selenium (UI), REST Assured/Karate (API-уровень); окружение — staging или docker-compose/Testcontainers-сборка.
- Свойства: максимальная уверенность, но медленные, хрупкие (flaky), дорогие в поддержке → их мало, только критические пути (smoke: логин, заказ, оплата).

### <a id="q9-10"></a>9.10. Тестирование граничных значений [↑](#toc)
**Суть:** ошибки живут на границах: min/max, 0, ±1 от границы, пустые/единичные коллекции, null, пределы типов, лимиты пагинации.

- Техника: классы эквивалентности + границы каждого класса; для `1..100`: 0, 1, 2, 99, 100, 101.
- Удобно через `@ParameterizedTest` + `@CsvSource`; property-based (jqwik) — генерация случаев автоматически.

### <a id="q9-11"></a>9.11. JUnit vs TestNG vs Spock [↑](#toc)
**Суть:** JUnit 5 — стандарт де-факто в Java/Spring: экосистема, расширения, поддержка всеми инструментами.

- TestNG — исторически богаче конфигурацией: XML-сьюты, группы, зависимости тестов, параллелизм из коробки, data providers; в новых проектах редок.
- **Spock** — Groovy, BDD-блоки `given/when/then/expect`, мощный `where:` (data tables), встроенные моки; выразительнее, но плюс Groovy в стеке.

### <a id="q9-12"></a>9.12. Тестирование асинхронного кода [↑](#toc)
**Суть:** нельзя `Thread.sleep` — флаки и медленно; ждать **условие**, а не время.

- **Awaitility**: `await().atMost(5, SECONDS).untilAsserted(() -> assertThat(repo.findById(id)).isPresent());` — опрос до успеха.
- CompletableFuture: `future.get(5, SECONDS)` / `assertThat(future).succeedsWithin(...)` (AssertJ).
- Детерминизация: инжект синхронного executor'а в unit-тестах (`Runnable::run`), фиксированный Clock; `@KafkaListener` — Awaitility поверх результата в БД/consumer'е.

### <a id="q9-13"></a>9.13. Тестирование интеграций с внешними сервисами [↑](#toc)
**Суть:** реальные внешние API в тестах — флаки, медленно, деньги → подменять контролируемыми имитациями.

- **WireMock** — HTTP-стаб: сценарии ответов, задержки, ошибки (5xx, timeouts) — тестируем ретраи и circuit breaker.
- Брокеры/БД — Testcontainers (реальный Kafka/Postgres локально).
- Контрактные тесты ([9.17](#q9-17)) — гарантия совместимости с провайдером без его поднятия. Плюс отдельные smoke-тесты против sandbox-окружений вне CI-гейта.

### <a id="q9-14"></a>9.14. TDD и BDD [↑](#toc)
**Суть:** **TDD** — цикл **Red → Green → Refactor**: сначала падающий тест, затем минимальный код до зелёного, затем рефакторинг под защитой тестов. Эффекты: дизайн под тестируемость, полное покрытие нового кода, документация поведением.

- **BDD** — фокус на поведении с точки зрения бизнеса: **Given-When-Then**, общий язык с аналитиками; инструменты — Cucumber (Gherkin-сценарии), Spock.
- Практично: TDD — на уровне unit; BDD-формулировки — для приёмочных сценариев. Не догма: для исследовательского кода тесты можно писать после, но до мержа.

### <a id="q9-15"></a>9.15. H2 в тестах [↑](#toc)
**Суть:** in-memory БД: мгновенный старт, `@DataJpaTest` по умолчанию подменяет DataSource на встроенную.

- Режимы совместимости: `jdbc:h2:mem:db;MODE=PostgreSQL;DATABASE_TO_LOWER=TRUE` — эмуляция диалекта, но **неполная**: нативные функции PG (jsonb, оконные тонкости, `ON CONFLICT`, типы), поведение блокировок отличаются → тест зелёный, прод падает.
- Вердикт: H2 годится для простой JPA-логики; всё с нативным SQL/миграциями/специфичными фичами — **Testcontainers** ([9.4](#q9-4)). Тестировать на том, на чём работаешь в проде.

### <a id="q9-16"></a>9.16. Генерация тестовых данных [↑](#toc)
**Суть:** слои подхода: **Test Data Builder** (fluent, дефолты + переопределение значимого: `anOrder().withStatus(PAID).build()`), Object Mother (готовые типовые персонажи).

- Библиотеки: **Instancio** / EasyRandom — автозаполнение объектов случайными данными; DataFaker — реалистичные значения (имена, адреса); фиксировать seed для воспроизводимости.
- БД: `@Sql`-скрипты, миграции тестовых данных; массовые данные для перфоманс-тестов — генераторы + COPY.
- Принцип: в тесте видны только значимые для сценария поля, остальное — осмысленные дефолты билдера.

### <a id="q9-17"></a>9.17. Contract testing (Pact, Spring Cloud Contract) [↑](#toc)
**Суть:** проверка совместимости API между потребителем и провайдером **без совместного поднятия**; ловит ломающие изменения на CI, где E2E слишком дороги.

- **Pact** (consumer-driven): консьюмер пишет ожидания → pact-файл → Pact Broker → провайдер верифицирует контракты против реального приложения; can-i-deploy — гейт релиза.
- **Spring Cloud Contract** (provider-side): контракты Groovy/YAML у провайдера → автогенерация тестов провайдера + **стабы для консьюмеров** (WireMock-стабы из Maven-репо).
- Отличие от схемы OpenAPI: контракт проверяет реальные примеры взаимодействий, а не только форму.

### <a id="q9-18"></a>9.18. Изоляция тестов: @DirtiesContext и «грязный» контекст [↑](#toc)
**Суть:** Spring кэширует ApplicationContext между тестами (скорость); тест, мутирующий контекст/состояние (properties, кэши, состояние синглтонов), «пачкает» его для последующих.

- `@DirtiesContext` — пересоздать контекст (после класса/метода): надёжно, но **дорого** (секунды на каждый пересбор) → использовать точечно.
- Дешевле: изоляция данных — `@Transactional` на тестах (rollback после каждого; осторожно: не увидишь поведение реальных коммитов/`REQUIRES_NEW`), `@Sql`-cleanup, truncate в `@BeforeEach`; сброс моков — `@MockitoBean` сбрасывается сам.
- Одинаковая конфигурация контекстов между классами → один контекст из кэша → быстрый сьют.

### <a id="q9-19"></a>9.19. Code coverage: JaCoCo и качество тестов [↑](#toc)
**Суть:** JaCoCo — инструментирует байткод, отчёты line/branch coverage; интеграция в Maven/Gradle + гейты (`jacoco:check` — минимум % на модуль/класс), SonarQube — coverage на новый код.

- Покрытие — **необходимая, не достаточная** метрика: 100% строк можно пройти без единого осмысленного assert.
- Качество тестов проверяет **mutation testing — PIT/pitest**: мутирует код (инвертирует условия, меняет константы) — выжившие мутанты = тесты, которые ничего не проверяют.
- Практика: branch coverage важнее line; гейт на новый код (~80%) полезнее тотального процента по legacy.

### <a id="q9-20"></a>9.20. Тестирование Spring-приложения: слайсы [↑](#toc)
**Суть:** слайс-аннотации поднимают **часть** контекста — быстрее полного `@SpringBootTest`.

- `@WebMvcTest(Controller.class)` — web-слой: MockMvc, фильтры, валидация, сериализация; сервисы — `@MockitoBean`.
- `@DataJpaTest` — JPA: репозитории, entityManager, транзакция с rollback; по умолчанию встроенная БД → `@AutoConfigureTestDatabase(replace = NONE)` + Testcontainers для реального PG.
- Другие: `@JsonTest`, `@RestClientTest`, `@DataRedisTest`. Полный контекст — только там, где реально нужна вся связка.

## <a id="s10"></a>10. Прочее: CI/CD, Git, Docker/K8s, Unix, безопасность [↑](#toc)

### <a id="q10-1"></a>10.1. CI/CD [↑](#toc)
**Суть:** **CI** — частая интеграция изменений в общую ветку с автоматической сборкой и тестами (быстрая обратная связь, ловим конфликты рано). **CD**: Continuous Delivery — артефакт всегда готов к релизу (деплой по кнопке); Continuous Deployment — автодеплой в прод после зелёного пайплайна.

- Типовой пайплайн: build → unit → static analysis (Sonar) → integration (Testcontainers) → docker build/push → deploy stage → smoke → deploy prod.
- Инструменты: GitLab CI, GitHub Actions, Jenkins; артефакты — Nexus/Artifactory/registry.

### <a id="q10-2"></a>10.2. Процесс доставки фичи до прода (мой опыт) [↑](#toc)
**Суть:** описываю по AuctionPort: задача в Jira → feature branch от develop → разработка + unit/integration тесты → MR: code review (2 аппрува) + зелёный GitLab CI (build, тесты, Sonar) → merge → пайплайн собирает Docker-образ, пушит в registry → автодеплой на stage (Kubernetes) → QA/smoke на stage → релиз в прод (тэг/manual job) → мониторинг (Grafana, логи) после выката.

- Ключевые слова, которые ждут: код-ревью, автотесты как гейт, миграции БД (Liquibase — вперёд-совместимые), rollback-план, feature flags при необходимости.

### <a id="q10-3"></a>10.3. Maven vs Gradle [↑](#toc)
**Суть:** Maven — XML (pom), декларативный, жёсткий lifecycle (validate→compile→test→package→verify→install→deploy), convention over configuration, предсказуем.

- Gradle — Groovy/Kotlin DSL, гибкий task graph, **инкрементальная сборка + build cache + daemon** → быстрее на больших проектах; сложнее дебажить нестандартное.
- Общее: управление зависимостями (central), scopes (compile/runtime/test; в Gradle — implementation/api: implementation скрывает зависимость от потребителей → быстрее перекомпиляция).
- Выбор: enterprise-стабильность/простота — Maven; большие мультимодульные/Android — Gradle.

### <a id="q10-4"></a>10.4. Git из командной строки [↑](#toc)
**Суть:** да, повседневно: clone, status, add -p, commit, push/pull, fetch, checkout/switch, branch, merge, **rebase (-i)**, cherry-pick, stash, log --oneline --graph, diff, reset, reflog, bisect.

- Отвечать уверенно, с примерами из практики: интерактивный rebase для чистки истории перед MR, reflog как «страховка», bisect для поиска коммита с багом.

### <a id="q10-5"></a>10.5. Squash коммитов [↑](#toc)
**Суть:** объединить несколько коммитов в один: `git rebase -i HEAD~3` → в редакторе оставить первый `pick`, остальным `squash` (s) / `fixup` (f — без сообщения).

- Альтернативы: `git merge --squash feature` (один коммит в целевой ветке), squash-merge кнопкой в GitLab/GitHub MR.
- После rebase уже запушенной ветки — `git push --force-with-lease` (безопаснее --force). Публичную общую историю не переписывать.

### <a id="q10-6"></a>10.6. `git branch -f master HEAD~1` vs `git reset --soft HEAD~1` [↑](#toc)
**Суть:** `branch -f master HEAD~1` — принудительно переставляет указатель ветки **master** на предыдущий коммит, **не трогая** текущую ветку/рабочую директорию (работает, только если master не checked out).

- `reset --soft HEAD~1` — двигает **текущую** ветку на коммит назад; изменения последнего коммита остаются **в staged** (index) — классика «раскоммитить и перекоммитить иначе».
- Для сравнения: `--mixed` (default) — изменения в working directory, не staged; `--hard` — изменения удаляются (восстановление — reflog).

### <a id="q10-7"></a>10.7. LF vs CRLF [↑](#toc)
**Суть:** окончания строк: Unix — LF (`\n`), Windows — CRLF (`\r\n`) → «изменён весь файл» в diff, ломаются shell-скрипты с CRLF.

- Решение в репозитории: **`.gitattributes`**: `* text=auto`, `*.sh text eol=lf`, `*.bat text eol=crlf` — единые правила для всех.
- Локально: `core.autocrlf`: `true` (Windows: checkout CRLF, commit LF), `input` (Linux/Mac: commit LF, checkout как есть). В хранилище — всегда LF.

### <a id="q10-8"></a>10.8. Graceful shutdown [↑](#toc)
**Суть:** корректное завершение: перестать принимать новое, дообработать текущее, освободить ресурсы.

- Spring Boot: `server.shutdown=graceful` + `spring.lifecycle.timeout-per-shutdown-phase=30s` — на SIGTERM веб-сервер перестаёт принимать соединения и ждёт активные запросы.
- Механика JVM: shutdown hooks (`Runtime.addShutdownHook`) — Spring вешает свой, закрывает контекст (`@PreDestroy`, пулы, listener-контейнеры Kafka/Rabbit дорабатывают текущие сообщения).
- **Kubernetes:** SIGTERM → grace period (`terminationGracePeriodSeconds`, default 30s) → SIGKILL; тонкость — под сначала убирается из endpoints не мгновенно → `preStop: sleep 5` чтобы не ловить запросы после SIGTERM. SIGKILL (kill -9) обработать нельзя.

### <a id="q10-9"></a>10.9. TCP vs UDP [↑](#toc)
**Суть:** TCP — соединение (3-way handshake), **гарантия доставки и порядка**, ретрансмиссии, flow/congestion control, поток байтов; цена — latency и оверхед.

- UDP — датаграммы без соединения и гарантий: минимальная задержка; потери/порядок — забота приложения.
- TCP: HTTP, БД, брокеры. UDP: DNS, стриминг/VoIP, игры, метрики (statsd), **QUIC/HTTP-3** (надёжность реализована поверх UDP).

### <a id="q10-10"></a>10.10. Docker vs виртуальная машина [↑](#toc)
**Суть:** контейнер — изолированный **процесс** на общем ядре хоста: изоляция через **namespaces** (pid, net, mnt, user...), лимиты ресурсов через **cgroups**, слоёная ФС (overlayfs). VM — полная виртуализация железа с гостевой ОС (hypervisor).

- Следствия: контейнеры стартуют миллисекунды, лёгкие (МБ), плотность выше; VM — сильнее изоляция (своё ядро), другая ОС возможна.
- Docker Desktop на Windows/Mac гоняет Linux-контейнеры внутри лёгкой VM (WSL2).

### <a id="q10-11"></a>10.11. Основные команды Docker [↑](#toc)
**Суть:** `docker run -d -p 8080:8080 --name app -e SPRING_PROFILES_ACTIVE=prod img:tag`, `ps (-a)`, `logs -f`, `exec -it app bash`, `stop/start/rm`, `images`, `pull/push`, `build -t`, `inspect`, `stats`.

- Уборка: `system prune`, `image prune`; volumes: `-v data:/var/lib/postgresql/data`; сети: `network create/connect`.
- Compose: `docker compose up -d`, `down`, `logs -f service`, `ps`.

### <a id="q10-12"></a>10.12. Dockerfile, образы, multi-stage, Compose [↑](#toc)
**Суть:** Dockerfile — рецепт образа; каждая инструкция — слой (кэшируются сверху вниз → зависимости копировать до исходников).

```dockerfile
FROM maven:3.9-eclipse-temurin-21 AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

FROM eclipse-temurin:21-jre
COPY --from=build /app/target/app.jar app.jar
USER 1000
ENTRYPOINT ["java","-jar","/app.jar"]
```
- **Multi-stage**: сборка в тяжёлом образе, в финальный — только jar + JRE → маленький образ без исходников и Maven.
- Хорошие практики: конкретные тэги базовых образов, .dockerignore, non-root user, layered jars Spring Boot (`bootBuildImage`/layertools) для кэшируемости слоёв.
- **Compose** — декларативный запуск нескольких контейнеров (app + postgres + kafka) одной командой: сервисы, сети, volumes, depends_on, healthchecks — локальная разработка и интеграционные стенды.

### <a id="q10-13"></a>10.13. Pod в Kubernetes [↑](#toc)
**Суть:** минимальная единица деплоя: один или несколько контейнеров с **общими** сетью (один IP, localhost между собой) и volumes; планируется на ноду целиком.

- Обычно один основной контейнер + вспомогательные: **init-контейнеры** (миграции, ожидание зависимостей), **sidecar** (прокси, лог-агент).
- Pod эфемерен (пересоздаётся с новым IP) → напрямую не создают, управляют через Deployment; стабильный доступ — через Service.
- Живучесть: liveness/readiness/startup probes; ресурсы: requests/limits.

### <a id="q10-14"></a>10.14. ReplicaSet и Deployment [↑](#toc)
**Суть:** **ReplicaSet** поддерживает заданное число реплик пода (label selector: упал — пересоздал). **Deployment** — уровень выше: управляет ReplicaSet'ами и даёт декларативные обновления.

- Rolling update: новый RS постепенно масштабируется вверх, старый вниз (`maxSurge`/`maxUnavailable`) → без даунтайма; `kubectl rollout undo` — откат на предыдущий RS; история ревизий.
- Напрямую RS не создают — только через Deployment. Родственники: StatefulSet (стабильные имена/диски), DaemonSet (по поду на ноду).

### <a id="q10-15"></a>10.15. Ingress [↑](#toc)
**Суть:** объект-правила маршрутизации внешнего HTTP(S)-трафика на Services: хосты (`api.example.com`), пути (`/orders → order-service`), **TLS termination** (секрет с сертификатом).

- Работает только с **Ingress Controller** (NGINX Ingress, Traefik) — он реально принимает трафик; аннотации контроллера — таймауты, лимиты, rewrite.
- Один входной LoadBalancer на много сервисов (vs Service type=LoadBalancer на каждый). Развитие — Gateway API.

### <a id="q10-16"></a>10.16. Свободное место и размер директории в Linux [↑](#toc)
**Суть:** **`df -h`** — свободное/занятое по файловым системам (human-readable).

- **`du -sh /var/log`** — суммарный размер директории; топ пожирателей: `du -h --max-depth=1 /var | sort -rh | head`.
- df и du могут расходиться: удалённый файл, который держит процесс, виден df, не виден du (`lsof | grep deleted`).

### <a id="q10-17"></a>10.17. Процессы Linux и поиск Java-процессов [↑](#toc)
**Суть:** `ps aux | grep java`, `top`/`htop` (интерактивно, P/M — сортировка по CPU/памяти).

- Java-специфично: **`jps -lv`** — JVM-процессы с main-классом и флагами.
- Точечно: `pgrep -f myapp.jar`, `pidof java`; убить: `kill PID` (SIGTERM, graceful) / `kill -9` (SIGKILL, крайняя мера).

### <a id="q10-18"></a>10.18. Thread dump [↑](#toc)
**Суть:** срез стеков всех потоков JVM: **`jstack PID > td.txt`** или `jcmd PID Thread.print`, или `kill -3 PID` (quit-сигнал — дамп в stdout приложения, процесс живёт).

- Читаем: состояния (RUNNABLE/BLOCKED/WAITING), «waiting to lock <0x...>» — на чём стоят; jstack сам находит дедлоки («Found one Java-level deadlock»).
- Практика: 3–5 дампов с интервалом пару секунд — отличить «завис» от «медленно работает»; в K8s: `kubectl exec pod -- jcmd 1 Thread.print`.

### <a id="q10-19"></a>10.19. Поиск текста в файлах: grep и просмотр [↑](#toc)
**Суть:** **`grep -rn "OrderNotFound" src/`** — рекурсивно с номерами строк; `-i` без регистра, `-l` только имена файлов, `-C 3` контекст, `-E` extended regex, `-v` инверсия.

- Логи: `tail -f app.log | grep ERROR` (живой хвост), `less +F app.log` (скролл + follow, `/` поиск), `zgrep` по архивам.
- Современно: `rg` (ripgrep) — быстрее и удобнее, если установлен.

### <a id="q10-20"></a>10.20. IDOR [↑](#toc)
**Суть:** Insecure Direct Object Reference — доступ к чужому ресурсу подменой идентификатора (`GET /invoices/123` → `/invoices/124`), когда сервер проверяет только аутентификацию, но **не владение ресурсом**.

- Защита: авторизация на **каждый** доступ к объекту: фильтр по владельцу в запросе (`findByIdAndUserId`), `@PreAuthorize` с проверкой principal, ACL для сложных прав.
- Дополнительно: непредсказуемые id (UUID) — снижает подбор, но не заменяет авторизацию.

### <a id="q10-21"></a>10.21. SQL Injection [↑](#toc)
**Суть:** пользовательский ввод попадает в SQL как код: `"... WHERE name='" + input + "'"` → `' OR '1'='1' --` → обход условий, чтение/изменение данных.

- Защита №1: **параметризованные запросы** — PreparedStatement (`?`), JPA/JPQL именованные параметры, `@Query` с `:param`; ORM сам параметризует.
- Опасные места даже с ORM: конкатенация в нативных запросах, **динамический ORDER BY / имена колонок** (параметром не передать → whitelist).
- Слои обороны: валидация входа, минимальные права БД-пользователя, не показывать SQL-ошибки клиенту.

### <a id="q10-22"></a>10.22. CSRF [↑](#toc)
**Суть:** злой сайт заставляет браузер жертвы отправить запрос на ваш сайт — cookie сессии уедут автоматически → действие от имени пользователя.

- Защита: **CSRF-токен** (synchronizer token: непредсказуемый токен в форме/заголовке, cookie одной недостаточно), **SameSite=Lax/Strict** на cookie (браузер не шлёт cookie с чужих сайтов), проверка Origin/Referer.
- Spring Security: CSRF-защита включена по умолчанию для изменяющих методов. **Stateless API с Bearer-токеном в заголовке CSRF не подвержен** (заголовок сам не отправится) — поэтому в REST API её часто отключают осознанно.

### <a id="q10-23"></a>10.23. Reflected XSS [↑](#toc)
**Суть:** ввод из запроса возвращается в HTML-ответ без экранирования (`/search?q=<script>...`) → скрипт исполняется в браузере жертвы (доставка — вредоносная ссылка). Отличие от stored XSS: payload не сохраняется на сервере.

- Защита: **контекстное экранирование вывода** (HTML-escape; шаблонизаторы Thymeleaf/JSX экранируют по умолчанию), валидация/санитизация ввода (OWASP Java HTML Sanitizer для rich text).
- **CSP** (`Content-Security-Policy: script-src 'self'`) — режет инлайн-скрипты; cookie `HttpOnly` — токен не украсть через `document.cookie`.

### <a id="q10-24"></a>10.24. Clickjacking [↑](#toc)
**Суть:** ваш сайт грузят в невидимый iframe поверх приманки — пользователь кликает «не туда» и выполняет действие на вашем сайте.

- Защита: запрет фрейминга — **`X-Frame-Options: DENY/SAMEORIGIN`** и современный **CSP `frame-ancestors 'none'/'self'`**.
- Spring Security ставит X-Frame-Options: DENY по умолчанию.

### <a id="q10-25"></a>10.25. Токен аутентификации в URL — что не так [↑](#toc)
**Суть:** URL «протекает» повсюду: **история браузера**, **логи** серверов/прокси/gateway, **Referer** при переходе по внешним ссылкам, закладки, копипаст ссылок, аналитика.

- Плюс: URL кэшируются, видны в мониторинге, шарятся в мессенджерах.
- Правильно: `Authorization: Bearer` заголовок или HttpOnly cookie. Исключения типа подписанных ссылок (S3 presigned) — короткий TTL и ограниченный scope.

### <a id="q10-26"></a>10.26. Защита сессионных токенов [↑](#toc)
**Суть:** cookie-флаги: **Secure** (только HTTPS) + **HttpOnly** (не читается JS → XSS не украдёт) + **SameSite=Lax/Strict** (CSRF).

- Серверно: непредсказуемые id (SecureRandom), **ротация id после логина** (session fixation), TTL + idle timeout, инвалидация при logout/смене пароля.
- Распределёнка: сессии в **Redis** (Spring Session) — единое хранилище за балансировщиком, централизованный отзыв.
- JWT-вариант: короткий access + refresh с ротацией и детектом повторного использования ([6.21](#q6-21)).

### <a id="q10-27"></a>10.27. Защита от DDoS [↑](#toc)
**Суть:** эшелонированно, по уровням.

- **L3/L4** (объёмные): CDN/anycast-провайдеры (Cloudflare, AWS Shield) — поглощение трафика до вашей инфраструктуры; SYN cookies, фильтрация на edge.
- **L7** (прикладные): **rate limiting** (NGINX `limit_req`, token bucket на gateway, Bucket4j+Redis по IP/пользователю/ключу), WAF, CAPTCHA/challenge на подозрительное, кэширование и облегчение дорогих эндпоинтов, таймауты и лимиты соединений.
- Архитектурно: автоскейлинг как буфер, circuit breakers внутри (защита от каскада), мониторинг аномалий трафика. Приложению одному объёмный DDoS не пережить — нужен периметр.

### <a id="q10-29"></a>10.29. Запомнившаяся задача (STAR-каркас) [↑](#toc)
**Суть:** держать готовый рассказ на 2–3 минуты. Вариант — N+1 на инвойсах (AuctionPort):

- **S:** поиск по инвойсам на проде деградировал до 8–11 секунд, жалобы клиента.
- **T:** найти причину и ускорить без изменения контракта API.
- **A:** включил логирование SQL/p6spy → N+1 каскад по связям; переписал на fetch join/EntityGraph + пагинация keyset, добавил недостающие индексы (EXPLAIN ANALYZE), проверил под нагрузкой k6.
- **R:** ~180 мс (в ~50 раз быстрее), убран лишний трафик к БД; добавил метрики, чтобы регрессию ловить сразу.
- Запасной: 8-часовой soak-тест k6 в Kubernetes (50 VUs, 2.68M запросов) — нашёл медленную утечку соединений, настроил HikariCP + graceful shutdown.
